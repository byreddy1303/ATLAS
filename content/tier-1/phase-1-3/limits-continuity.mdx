export const continuityCheckCode = `import numpy as np

# Three functions to test at x = 1:
# 1) f(x) = x + 1        -- continuous everywhere
# 2) f(x) = 1 / (x - 1)  -- blows up at x = 1, discontinuous
# 3) f(x) = { x + 2 if x != 1;  4 if x == 1 }  -- removable discontinuity

def is_continuous_at(f, x0, tol=1e-8):
    # 3-part definition of continuity:
    #   (a) f(x0) is defined
    #   (b) limit as x -> x0 exists (left = right)
    #   (c) they are equal
    try:
        val = f(x0)
    except Exception:
        return f"({x0}) undefined -> NOT continuous"

    left  = f(x0 - 1e-6)
    right = f(x0 + 1e-6)

    if abs(left - right) > tol:
        return f"limit doesn't exist (left={left:.4f}, right={right:.4f}) -> NOT continuous"
    limit = 0.5 * (left + right)
    if abs(val - limit) > tol:
        return f"f({x0})={val:.4f} but limit={limit:.4f} -> NOT continuous"
    return f"f({x0})={val:.4f}, limit matches -> continuous"

# 1) f(x) = x + 1
print("f(x) = x + 1:  ", is_continuous_at(lambda x: x + 1, 1.0))

# 2) f(x) = 1 / (x - 1)
print("f(x) = 1/(x-1):", is_continuous_at(lambda x: 1 / (x - 1), 1.0))

# 3) piecewise with removable discontinuity
def f3(x): return 4.0 if abs(x - 1.0) < 1e-14 else x + 2.0
print("f(x) = piecewise:", is_continuous_at(f3, 1.0))
`;

export const softmaxContinuityCode = `import numpy as np

def softmax(logits):
    # Numerically stable softmax: subtract the max first
    shifted = logits - np.max(logits)
    exps = np.exp(shifted)
    return exps / exps.sum()

# Perturb the input by TINY amounts and watch the output smoothly change.
# Softmax is continuous (in fact smooth = differentiable everywhere)
base = np.array([2.0, 1.0, 0.5])
print("Base softmax:      ", np.round(softmax(base), 6))

for eps in [1e-2, 1e-4, 1e-6, 1e-10]:
    perturbed = base + np.array([eps, 0.0, 0.0])       # bump only the first logit
    diff = softmax(perturbed) - softmax(base)
    print(f"After eps = {eps:.0e}:  max output change = {np.max(np.abs(diff)):.2e}")

print()
print("Tiny input change -> tiny output change. Softmax is CONTINUOUS.")
print("That's why gradient-based training works with softmax outputs.")
print()

# Compare to a DISCONTINUOUS operation: argmax
print("argmax(base) =", np.argmax(base))
tie = np.array([1.0, 1.0, 0.999])
print("argmax([1.0, 1.0, 0.999]) =", np.argmax(tie))
tie_perturbed = np.array([1.0, 1.0 + 1e-9, 0.999])     # tiny bump breaks the tie
print("argmax([1.0, 1.0 + 1e-9, 0.999]) =", np.argmax(tie_perturbed))
print()
print("argmax is DISCONTINUOUS at ties -> cannot backprop through it,")
print("which is why we use softmax during training and argmax only at inference.")
`;

<Motivation title="Why 'continuous' is the requirement gradient descent secretly assumes">

Every gradient-based ML method — SGD, Adam, backprop, everything — requires the loss function to be **differentiable**, which in turn requires it to be **continuous**. When a loss (or an activation, or an operation in the network) is discontinuous, gradient descent breaks: tiny weight changes produce tiny loss changes almost everywhere, then suddenly a huge jump. The "descent direction" isn't well defined at the jump; the algorithm oscillates or gets stuck.

Two examples you deal with constantly:

- **Softmax vs argmax at the output.** Argmax is discontinuous — a tiny change in logits can flip which class wins — so you cannot backprop through it. Softmax is a continuous, differentiable *smoothing* of argmax; it's why we train with cross-entropy on softmax outputs and only use argmax at inference. Every classification network's output layer is a workaround for this exact fact.
- **ReLU at zero.** ReLU is continuous, but its derivative has a jump — the "corner" at z = 0 means the derivative is 0 on the left and 1 on the right, with no defined value at 0 itself. Libraries paper over this with a "subgradient" convention.

For a GenAI engineer, "limit" and "continuity" are not academic notions. They're the reason gradient-based training is possible, the reason certain hard optimization problems get "smoothed" into tractable ones, and the reason numerical issues creep in when a computation gets close to a discontinuity or a singularity. This chapter builds the vocabulary and intuition, then shows exactly how it maps to gradient-flow guarantees in ML.

</Motivation>

<Intuition title="A limit is 'the value f approaches, even if it never gets there'">

Look at f(x) = sin(x)/x. Plug in x = 0 and you get 0/0 — undefined. But if you compute f(0.1), f(0.01), f(0.001), … you notice they all approach the value 1. The **limit** of f as x approaches 0 is 1, even though f(0) itself is undefined.

More generally: `limₓ→a f(x) = L` means "as x gets arbitrarily close to a (from either side), f(x) gets arbitrarily close to L." The limit is about *approach*, not about the value at a itself. It's a statement about how the function behaves in the neighborhood of a, ignoring what happens exactly at a.

**Continuity** is when the limit actually equals the value: `limₓ→a f(x) = f(a)`. In pictures: continuous means "draw the graph without lifting the pen" — no jumps, no gaps. If you can approach a point along the graph from either side and land on the actual value f(a), the function is continuous at a.

**Every polynomial is continuous everywhere.** Same for sin, cos, e^x. So are their sums, products, and (where the denominator is nonzero) quotients. Compositions of continuous functions are continuous. In practice, almost every function you write down in ML is continuous *except* for a few named exceptions (argmax, indicator functions, hard-thresholded quantities) that we then have to work around.

</Intuition>

<Recall term="Function domain">

The domain of a function f is the set of inputs where f is defined. When we talk about a limit "as x approaches a" we don't require a to be in the domain — the limit only concerns nearby points. This is why we can meaningfully ask about the limit of sin(x)/x at x = 0 even though the function isn't defined there.

</Recall>

## The intuitive limit — approach, not arrival

`limₓ→a f(x) = L` means: **whatever tolerance you demand around L, you can find a small enough neighborhood around a (excluding a itself) where f is within that tolerance of L.**

The picture: pin a target strip of width 2ε around L on the y-axis. Then you have to find a strip of width 2δ around a on the x-axis (not including a itself) such that every point in the x-strip lands inside the y-strip when you apply f. If you can do this for *any* ε (however small), the limit exists and equals L.

The formal (ε–δ) definition:

$$
\lim_{x \to a} f(x) = L \iff \forall\, \varepsilon > 0,\, \exists\, \delta > 0 \text{ such that } 0 < |x - a| < \delta \implies |f(x) - L| < \varepsilon.
$$

You almost never *use* this definition in practice. But it's what makes limits precise. For an ML engineer, what matters is the intuition: "approaches L as x approaches a, without needing f(a) itself to be defined or equal to L."

## One-sided limits and when the two-sided limit fails

You can compute the limit as x approaches a *from below only* (`limₓ→a⁻`) or *from above only* (`limₓ→a⁺`). These are the **one-sided limits**.

The **two-sided limit** at a exists ⇔ both one-sided limits exist AND they agree. If they disagree, or either fails to exist, the two-sided limit is undefined.

Example: the Heaviside step H(x) = 0 for x &lt; 0, 1 for x ≥ 0. Left-limit at 0 is 0; right-limit is 1. They disagree ⇒ limₓ→0 H(x) doesn't exist. H is *discontinuous* at 0.

## Infinite limits and limits at infinity

Two extensions of the basic limit idea:

- **Infinite limit**: `limₓ→a f(x) = ∞` means f(x) grows without bound as x approaches a. The limit doesn't exist as a real number, but we use the shorthand ∞ to indicate the divergence pattern. Example: 1/x² blows up to +∞ as x → 0.
- **Limit at infinity**: `limₓ→∞ f(x) = L` means f(x) approaches L as x gets arbitrarily large. Example: `limₓ→∞ (1 + 1/x) = 1`. Similarly for x → −∞.

Combining both is common in ML: the sigmoid function has `limₓ→+∞ σ(x) = 1` and `limₓ→−∞ σ(x) = 0`. The two horizontal asymptotes.

## Continuity — the three-part definition

A function f is **continuous at x = a** if all three:

1. **f(a) is defined** (a is in the domain of f).
2. **limₓ→a f(x) exists** (the two-sided limit is a finite real number).
3. **limₓ→a f(x) = f(a)** (the limit equals the function value).

If any one of these fails, f is discontinuous at a. The three types of failure:

- **f(a) undefined but the limit exists** — a *removable* discontinuity. You could redefine f(a) to equal the limit and make it continuous. Example: sin(x)/x with a hole at x = 0 (fillable with the value 1).
- **Left and right limits disagree** — a *jump* discontinuity. No single value could patch this. Example: Heaviside step at 0.
- **f blows up or oscillates near a** — an *infinite* or *essential* discontinuity. Example: 1/x at 0 (blows up), sin(1/x) at 0 (oscillates infinitely fast).

## The functions that are always continuous (and the ones that aren't)

**Always continuous (everywhere on their domain):**

- Polynomials (constants, x, x², all sums with real coefficients).
- Sin, cos, e^x, ln(x) (on x &gt; 0), √x (on x ≥ 0).
- Sums, products, compositions, and quotients (where the denominator is nonzero) of continuous functions.
- Softmax, sigmoid, tanh, GELU — every standard activation used in modern deep learning.

**Not continuous somewhere:**

- 1/x at x = 0 (blows up).
- Absolute value's derivative at 0 (the function is continuous, but its derivative jumps).
- ReLU's derivative at 0 (again continuous function, discontinuous derivative — "kink").
- Heaviside step, sign function, indicator functions (jump discontinuities).
- Argmax, top-k selection, hard-thresholded quantities (jumps whenever the winner changes).

The last group is exactly the set of operations you *can't backprop through* directly in modern ML. The workarounds are various forms of **smoothing** — replacing a discontinuous operation with a continuous relaxation that gradient descent can traverse.

## Why continuity matters for gradient descent

Gradient descent adjusts parameters to reduce a loss. If the loss is continuous, small parameter changes produce small loss changes — a locally-well-behaved optimization landscape. If it has jumps, the gradient of the loss with respect to a parameter is undefined or infinite at the jump, and updates near the jump either do nothing (if the parameter change doesn't cross the jump) or explode (if it does).

**Corollary**: any discrete decision inside a differentiable model breaks gradient flow. This is why the standard tricks — softmax instead of argmax, sigmoid instead of hard-threshold, straight-through estimators for quantization, Gumbel-softmax for discrete sampling — all replace a discrete choice with a smooth relaxation, do gradient descent on the relaxation, and then optionally snap back to a discrete choice at inference.

## Numerical continuity issues you'll see

Even for genuinely continuous functions, floating-point arithmetic can make a computation *appear* discontinuous in code:

- **Subtractive cancellation**: `f(x + h) − f(x)` for tiny h loses digits when the two values are nearly equal. This is why numerical derivatives get *worse* below h ≈ 1e-8, not better.
- **Overflow/underflow**: exp(x) for large x overflows to inf; exp(x) for very negative x underflows to 0. Cross either boundary and the function's output jumps (from a normal float to inf, or from a tiny positive number to exact 0), which then propagates through the rest of the computation.
- **Softmax without max-subtraction**: `exp(x) / sum(exp(x))` overflows for large logits. The fix is to subtract the max first: `exp(x - max_x) / sum(exp(x - max_x))` — same math, no overflow. This "log-sum-exp trick" is a canonical example of preserving continuity in floating-point.

Understanding limits and continuity is what lets you diagnose these issues: they're not really about continuity of the underlying math, but about the floating-point representation *breaking* continuity in the neighborhood of certain values.

<RealWorldML title="Why softmax is the argmax you can differentiate">

Consider the final layer of any classification network. You produce a vector of logits (one per class), and you want to output a probability distribution over classes. What you'd *like* to output — "the class with the highest logit is 100% probable, all others are 0%" — is exactly `argmax` returning a one-hot vector. But argmax is a **discontinuous** function of logits: at a tie between two classes, an infinitesimal change in logits flips which class wins, and the one-hot output changes discontinuously.

If the output layer were argmax, backprop would fail: at the tied point, the gradient of the output with respect to logits is either 0 (nothing changed on my side of the tie) or infinite (I crossed the tie boundary). Either way, gradient descent can't compute a meaningful update.

**Softmax fixes this by being a smooth version of argmax:**

$$
\mathrm{softmax}(z)_i \;=\; \frac{e^{z_i}}{\sum_{j} e^{z_j}}
$$

For each class i, output the exponentiated logit divided by the sum of all exponentiated logits. Properties:

- **Values in [0, 1] that sum to 1** — a valid probability distribution.
- **Continuous and differentiable everywhere** — no ties, no jumps. Small logit changes produce proportionally small probability changes.
- **Recovers argmax in the "temperature → 0" limit**: if you divide logits by a temperature τ before softmax, then as τ → 0 the highest logit dominates and the output becomes essentially one-hot. So softmax with τ = 1 is a "soft" approximation to argmax that becomes exact as τ → 0.

This is exactly what "argmax you can differentiate" means. Every classification network uses softmax during training (so backprop works), and either softmax or argmax during inference (softmax if you need probabilities; argmax if you just need the prediction).

The same "smoothly relax a discontinuous operation" pattern shows up all over ML:

- **Gumbel-softmax** — smoothly relaxes discrete categorical sampling for gradient-based training of models that sample categorical variables (VAEs with discrete latents, differentiable neural architecture search).
- **Straight-through estimator** — used to backprop through quantization (rounding to the nearest int) by pretending the quantizer is the identity in the backward pass, keeping gradients defined.
- **Soft attention** — replaces "pick the top-k tokens" (discontinuous) with a weighted average using softmax weights (continuous, differentiable).
- **Relaxed transformer routing** in mixture-of-experts models — smooths the "assign token to expert" choice into a soft probability distribution during training.

The rule of thumb: **whenever a model needs a discrete decision, replace it with a continuous relaxation during training, and (if needed) snap back to a discrete choice at inference**. This one pattern, powered by the continuity requirement of gradient descent, is the reason so many ML architectures look the way they do.

</RealWorldML>

<Summary>

- **Limit**: `limₓ→a f(x) = L` means f(x) approaches L as x approaches a. The value f(a) itself doesn't matter to the limit — only what f does *near* a.
- **One-sided limits** (from below and above). The two-sided limit exists ⇔ both one-sided limits exist and agree.
- **Continuity at a** requires three things: f(a) is defined, the limit at a exists, and they are equal. Failure gives removable, jump, or essential discontinuities.
- **Continuous by default**: polynomials, sin/cos/exp/log (on their domain), softmax/sigmoid/tanh/GELU. Sums, products, compositions, and (nonzero-denominator) quotients preserve continuity.
- **Discontinuous** (incompatible with vanilla backprop): argmax, indicator, hard-threshold, quantize, sign — all handled in ML by smooth relaxation (softmax, Gumbel, straight-through).
- **Why ML cares**: gradient descent requires the loss to be continuous (in fact differentiable) so that tiny weight changes produce tiny loss changes. Every "smoothed" operation in a network exists to make gradient flow possible.
- **Numerical continuity**: even genuinely continuous math can look discontinuous in floating-point — subtractive cancellation, over/underflow, and softmax without max-subtraction are all cases where you have to preserve continuity in code with a numerically-stable formulation.
- The **next chapter** builds derivatives on this foundation — differentiability requires continuity plus a well-defined slope everywhere.

</Summary>
