export const partialCode = `import numpy as np

# f(x, y) = x^2 + 3xy + y^2
# analytic partials:
#   ∂f/∂x = 2x + 3y   (treat y as constant)
#   ∂f/∂y = 3x + 2y   (treat x as constant)

def f(x, y):
    return x ** 2 + 3 * x * y + y ** 2

def grad_f(x, y):
    return np.array([2 * x + 3 * y, 3 * x + 2 * y])

def finite_partial(f, i, x, h=1e-6):
    "Numerical partial derivative w.r.t. the i-th coordinate of x."
    x_plus  = x.copy(); x_plus[i]  += h
    x_minus = x.copy(); x_minus[i] -= h
    return (f(*x_plus) - f(*x_minus)) / (2 * h)

for point in [(1.0, 2.0), (-1.0, 0.5)]:
    x = np.array(point)
    g = grad_f(*x)
    g_num = np.array([finite_partial(f, 0, x), finite_partial(f, 1, x)])
    print(f"point {tuple(x)}: analytic grad = {g}  numerical = {np.round(g_num, 5)}")`;

# f(x, y) = x^2 + 3xy + y^2, gradient computed by hand above

<Motivation title="Every parameter of an LLM is one axis in a giant landscape">

A modern LLM has hundreds of billions of parameters. The loss L is a single number that depends on all of them. Give the model any specific set of weights, it computes one loss value. The optimization job is: **for each weight, figure out how much the loss would change if I nudged just this one weight by a tiny amount — while holding all the others fixed.**

That "how much does L change if I nudge only w_i" is called a **partial derivative**, written ∂L/∂w_i. The symbol ∂ (rounded, not italic d) is the standard math notation for "we're taking a derivative but pretending everything else is frozen." For a billion-parameter model, you have a billion of these partial derivatives.

Bundle all of them into a single vector — one entry per weight — and you get the **gradient**, written ∇L (nabla-L). This is one giant vector, living in the same space as your parameters. Every optimizer (SGD, Adam, etc.) takes a step *against* this vector to lower the loss.

The chain rule (last chapter) taught us how derivatives *compose*. This chapter teaches how they *bundle*. Specifically:

- Derivatives with respect to many inputs live in a single vector object (the gradient).
- The slope in *any* direction (not just along an axis) can be pulled out with a dot product.
- Gradient descent, second-order methods, natural gradients, and the whole optimizer toolkit are really operations on this vector.

</Motivation>

<Intuition title="One slope per axis, all at once">

Picture a hilly landscape: elevation z is a function of position (x, y). Stand at a specific point and ask three questions:

- **"How steep is it if I walk east?"** That's the slope in the +x direction — the **partial derivative** ∂z/∂x.
- **"How steep is it if I walk north?"** That's the slope in the +y direction — the **partial derivative** ∂z/∂y.
- **"How steep is it if I walk northeast?"** That's the slope in the (1, 1)/√2 direction — a **directional derivative**.

The **gradient** at your point is the vector (∂z/∂x, ∂z/∂y): the eastward slope and northward slope bundled together into one vector. That vector has two magical properties (proved below): (1) it points in the direction of **steepest ascent**, and (2) its magnitude is that steepest slope. So you can read the whole hill's local behavior off a single vector.

For any *other* direction u, the slope in u is just the dot product of the gradient with u (normalized). One vector, all directional slopes for the price of a dot product.

</Intuition>

<Recall term="Partial derivative notation">

∂f/∂xᵢ (read "partial f with respect to xᵢ") is the derivative of f treating all other variables as constants — the rate at which f changes if you nudge only xᵢ. In code: it's the finite-difference (f(x + hεᵢ) − f(x − hεᵢ))/(2h) as h → 0, where εᵢ is the standard basis vector in the i-th direction. Sometimes written ∂ᵢf or fₓᵢ. All identical.

</Recall>

## Partial derivatives — hold everything else still

For a function f(x_1, …, x_n) of many variables, the partial derivative with respect to x_i is:

$$
\frac{\partial f}{\partial x_i}(a) \;=\; \lim_{h \to 0} \frac{f(a_1, \ldots, a_i + h, \ldots, a_n) - f(a_1, \ldots, a_n)}{h}
$$

Read: **wiggle only x_i by h, freeze every other coordinate, see how f responds**. This is a scalar quantity, one per input coordinate. Computing it is mechanical: treat every other variable as a constant, differentiate as if only x_i is a variable, apply the usual single-variable rules.

Concrete example: f(x, y) = x² + 3xy + y².

- ∂f/∂x: treat y as constant. d(x²)/dx = 2x, d(3xy)/dx = 3y, d(y²)/dx = 0. So **∂f/∂x = 2x + 3y**.
- ∂f/∂y: treat x as constant. d(x²)/dy = 0, d(3xy)/dy = 3x, d(y²)/dy = 2y. So **∂f/∂y = 3x + 2y**.

Nothing new mathematically: partials are single-variable derivatives with the other variables held fixed. But *bundling* them into a vector opens up all the geometry below.

## The gradient — all partials at once

The **gradient** of f at a point, written ∇f (nabla-f) or grad f, is the column vector of all partial derivatives:

$$
\nabla f \;=\; \begin{pmatrix} \partial f / \partial x_1 \\ \partial f / \partial x_2 \\ \vdots \\ \partial f / \partial x_n \end{pmatrix}
$$

Same dimension as the input space — one entry per input variable. In our example, ∇f(x, y) = (2x + 3y, 3x + 2y). At the point (1, 2): ∇f = (2 + 6, 3 + 4) = (8, 7).

The gradient is not a scalar (that's the individual partials) and not a matrix (that's the Hessian, next chapter). It's a **vector living in the input space** — the same space where you're free to move around. This means you can *add* the gradient (scaled) to your current position to move, which is exactly what gradient descent does: `x_new = x_old − learning_rate · ∇f(x_old)`.

## The gradient points in the direction of steepest ascent — and its length IS that slope

Here's the payoff geometric fact. Among all unit-length directions u you could walk from a point, the direction that increases f the fastest is **u = ∇f / ‖∇f‖**, and the corresponding rate of increase (the directional derivative in that direction) is **‖∇f‖** — the gradient's length.

This is why gradient descent works: to *decrease* f as fast as possible, walk in the direction of *negative* gradient, and the magnitude of the loss decrease per unit distance is proportional to ‖∇f‖. When the gradient's norm is small, you're near a critical point (flat region, minimum, maximum, or saddle). When it's large, you're on a steep slope and should be moving fast — or the learning rate is too big.

<Derivation title="Why ∇f points in the steepest-ascent direction">

We ask: among all unit vectors u, which direction u maximizes the directional derivative D_u f = "rate at which f changes when we walk in the direction u"?

<Step n={1} math={<span>Directional derivative formula: <code>D_u f = ∇f · u</code> (dot product)</span>}>
    This is proved in the next section as a consequence of the multivariable chain rule. For now, take it as: the slope in direction u is the projection of the gradient onto u. Small nudge in direction u produces change (∇f · u) · Δs in f, to first order.
</Step>

<Step n={2} math={<span>Bound via dot-product identity: <code>∇f · u = ‖∇f‖ · ‖u‖ · cos θ</code></span>}>
    From the dot-product chapter (a · b = ‖a‖ ‖b‖ cos θ). Here u is a unit vector, so ‖u‖ = 1, and: <code>D_u f = ‖∇f‖ cos θ</code>, where θ is the angle between u and ∇f.
</Step>

<Step n={3} math={<span>Maximize over θ: cos θ is largest (=1) when θ = 0, i.e. <b>u aligned with ∇f</b>.</span>}>
    So the maximum directional slope is achieved when u points in the same direction as ∇f. Its value at that direction is ‖∇f‖ · cos 0 = <b>‖∇f‖</b>.
</Step>

<Step n={4} math={<span>Consequences: steepest descent is u = −∇f / ‖∇f‖; level-set direction is u ⊥ ∇f</span>}>
    The minimum of cos θ is at θ = 180°, giving D_u f = −‖∇f‖ — walk opposite to ∇f to descend fastest. And when u is perpendicular to ∇f (θ = 90°, cos = 0), the directional derivative is <b>zero</b> — you're walking along a level set (contour line) of f. This is the geometric fact that <b>the gradient is perpendicular to every level set at every point</b>.
</Step>

</Derivation>

In the plot, the level sets of f(x, y) = x² + y² are circles centered at the origin. The gradient at each point is (2x, 2y) — the position vector — pointing radially outward, exactly perpendicular to the circle passing through that point. This is a universal fact: **at every point of every smooth function, the gradient is perpendicular to the level set**.

## Directional derivatives — the gradient handles every direction at once

Given a unit vector u, the **directional derivative** of f in the direction u at a point is:

$$
D_u f \;=\; \nabla f \cdot u
$$

— just a dot product. This is enormously useful: with the *one* gradient vector, you get the slope in every possible direction, for the price of one dot product per direction of interest. No need to redo any calculus.

Special cases:

- u = ε_i (i-th standard basis) → D_u f = ∂f/∂x_i (the i-th partial derivative — the gradient's i-th component).
- u = ∇f/‖∇f‖ → D_u f = ‖∇f‖ (the maximum, from the derivation above).
- u = −∇f/‖∇f‖ → D_u f = −‖∇f‖ (the minimum — steepest descent).
- u ⊥ ∇f → D_u f = 0 (the level-set direction — walking without changing f).

<RunnableCode title="Directional derivatives from one gradient" code={directionalCode} packages={["numpy"]} />

The identity D_u f = ∇f · u is not magic — it's a consequence of the **multivariable chain rule**, which says that if you parametrize your walk as γ(t) = x + t · u (a straight line through x in direction u), then df(γ(t))/dt = ∇f(γ(t)) · γ'(t) = ∇f(γ(t)) · u. At t = 0, this equals ∇f(x) · u. Same construction, one line of chain rule.

## The traps that matter

**Partial derivatives are not "derivatives with a coordinate mixed in"** — they're single-variable derivatives with everything else held fixed. ∂f/∂x means: change x only, everything else is treated as a *number*, apply the ordinary d/dx rules. Beginners sometimes try to differentiate y as a "function of x" when it isn't — the whole point of a *partial* is that other variables are independent.

**The gradient is not a scalar, and its "magnitude" is meaningful.** ‖∇f‖ tells you the steepest slope at your location; near critical points (minima, maxima, saddles) it goes to zero. Optimization libraries stop when ‖∇f‖ falls below a tolerance — the "gradient norm" is the natural quantity to monitor. If your loss is stuck but ‖∇f‖ is huge, you have a learning-rate problem, not a critical point problem.

**Gradient perpendicular to level sets extends to every dimension.** In 2-D you get perpendicular-to-a-curve; in 3-D perpendicular-to-a-surface; in n-D perpendicular-to-an-(n-1)-dimensional-hypersurface. The rule "cross a level set going against its steepest slope" is the geometric heart of gradient descent in any dimension. This is why constrained optimization uses **Lagrange multipliers**: at a constrained optimum, ∇f must be parallel to the gradient of the constraint (both perpendicular to the same constraint surface).

**The gradient depends on the choice of coordinates.** ∇f as a vector has entries that depend on which basis you use. Under a change of coordinates, the *components* transform (via the Jacobian of the coordinate change), but the *geometric vector* it represents stays the same. In deep learning this rarely bites — we work in one basis (the parameter basis) — but in physics and differential geometry the distinction matters. **Natural gradient methods** (used in some RL algorithms and in second-order optimization) explicitly correct for this by applying an inverse Fisher-information matrix, effectively making the gradient basis-invariant.

**Gradient of vector-valued function → Jacobian.** If f produces a vector instead of a scalar (say f : ℝⁿ → ℝᵐ), the "gradient" generalizes to the **Jacobian matrix** (m rows, n columns), where entry (i, j) is ∂f_i/∂x_j. The gradient is the special case m = 1. Chain rule for vector-in vector-out functions is Jacobian multiplication, which is exactly matrix multiplication. That's the next chapter.

<RealWorldML title="Gradient descent on a real loss surface">

For a linear regression model y_pred = w_1·x_1 + w_2·x_2 with mean squared error loss L(w) = (1/N) Σᵢ (y_pred − y_true)², the loss depends on 2 parameters w_1, w_2 (plus a bias if we include one). The gradient ∇L is a 2-vector telling you the direction of steepest ascent in loss.

Concretely, for a single training example (x, y_true), the loss is L = (w_1 x_1 + w_2 x_2 − y_true)². By the chain rule:

- ∂L/∂w_1 = 2 (y_pred − y_true) · x_1
- ∂L/∂w_2 = 2 (y_pred − y_true) · x_2

The gradient is ∇L = 2 (y_pred − y_true) (x_1, x_2) = 2 (residual) x. In code, `grad = 2 * (X @ w - y) @ X / N` for a batch. One line: computes the gradient of the loss with respect to every weight simultaneously, using matrix operations that push the work onto BLAS. That's the entire content of "gradient computation" for linear regression, and for every layer of a neural network it's the same skeleton — just chained through many layers via backprop.

The gradient descent update: `w ← w − learning_rate · ∇L`. The negative sign takes us in the direction of steepest *descent*; the learning rate controls step size. For big learning rate, we may overshoot the minimum; for tiny, we crawl. Adam, RMSprop, and other adaptive optimizers modify the step direction and per-coordinate scale, but they all consume this same gradient vector as input. The next chapter (Jacobian & Hessian) develops the machinery for the "adaptive" and "second-order" pieces; this chapter's job is to make sure the raw ∇L object is second nature.

</RealWorldML>

<Summary>

- The **partial derivative** ∂f/∂x_i is the rate at which f changes with respect to x_i alone, holding all other variables fixed. Compute it by treating the others as constants and applying single-variable rules.
- The **gradient** ∇f is the vector of all partials, one entry per input dimension. Same dimension as the parameter space — you can add it (scaled) to your position and move.
- **Directional derivative** in unit direction u = ∇f · u — one dot product answers "how steep is f in this direction?" for every u you care about.
- **∇f points in the direction of steepest ascent**; ‖∇f‖ is that steepest slope. **−∇f is steepest descent** (basis of gradient descent). **∇f is perpendicular to level sets** at every point.
- Optimization uses these facts constantly: `x ← x − η · ∇f(x)` is the vanilla update; adaptive methods (Adam, RMSprop) rescale the step per-coordinate but still consume the gradient as input.
- **Vector-valued generalization**: if f produces a vector, the gradient becomes the **Jacobian matrix** — one row of partials per output component. Chain rule for vector functions becomes Jacobian multiplication (next chapter).

</Summary>

In practice we use w_new = w − η · ∇L, which is: take a step *proportional* to the gradient magnitude. Where the gradient is big (steep slope), we take a bigger step; where it's small (near a critical point), we take a small step. This is a helpful adaptive behavior — the gradient's magnitude carries useful information about how far to move. If we normalized, we'd take equally-sized steps everywhere, potentially oscillating around minima. So "unnormalized" trades a tiny bit of "true" steepest-descent geometry for the practical benefit of gradient-magnitude-aware step sizes. Adam and second-order methods build even more elaborate scaling schemes on top of this idea.</>
