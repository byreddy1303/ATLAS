export const medicalTestCode = `# The classic base-rate example: a 99% accurate test for a rare disease.
# 1% prior; 99% sensitivity (P(pos | disease)); 99% specificity (P(neg | healthy)).
# Q: patient tests positive — what's P(disease | positive)?
prior_disease = 0.01                       # 1% of population has the disease
p_pos_given_D  = 0.99                      # sensitivity: test catches 99% of true positives
p_pos_given_H  = 1 - 0.99                  # 1 - specificity: false-positive rate = 1%

# Bayes: P(D | pos) = P(pos | D) P(D) / P(pos)
p_pos = p_pos_given_D * prior_disease + p_pos_given_H * (1 - prior_disease)
posterior = (p_pos_given_D * prior_disease) / p_pos
print(f"P(disease | positive) = {posterior:.4f}   ({posterior*100:.2f}%)")
print(f"Even with a 99% accurate test, prob of disease given positive test is ~{posterior*100:.0f}%.")
print("Because the disease is rare, most positives are FALSE positives.  This is the base-rate fallacy.")
print()

# Sensitivity to prior — a public-health screen has different implications from a targeted test
for prior in [0.001, 0.01, 0.1, 0.5]:
    p_pos_total = p_pos_given_D * prior + p_pos_given_H * (1 - prior)
    post = p_pos_given_D * prior / p_pos_total
    print(f"prior {prior*100:>5.2f}%  ->  P(D | +) = {post*100:>6.2f}%")`;

export const spamCode = `# Naive Bayes text classifier — Bayes' theorem applied to word features.
# Simple bag-of-words: given a message, is it spam?
import numpy as np
from collections import Counter, defaultdict

# Toy training data: 5 spam, 5 ham messages
training = [
    ("free money now click", "spam"),
    ("meeting tomorrow at ten", "ham"),
    ("free vacation win prize", "spam"),
    ("lunch tomorrow bring the report", "ham"),
    ("cash prize free entry", "spam"),
    ("send the meeting notes", "ham"),
    ("free trial upgrade now", "spam"),
    ("dinner tomorrow at eight", "ham"),
    ("win million dollars easy", "spam"),
    ("report meeting update at ten", "ham"),
]

# Learn class priors and per-class word likelihoods (with Laplace smoothing)
def train(data):
    class_docs = defaultdict(list)
    for msg, label in data:
        class_docs[label].append(msg.split())
    prior = {c: len(docs) / len(data) for c, docs in class_docs.items()}
    vocab = set(w for msg, _ in data for w in msg.split())
    likelihoods = {}
    for c, docs in class_docs.items():
        counts = Counter(w for msg in docs for w in msg)
        total  = sum(counts.values()) + len(vocab)              # Laplace +1 smoothing
        likelihoods[c] = {w: (counts[w] + 1) / total for w in vocab}
    return prior, likelihoods, vocab

def classify(msg, prior, likelihoods, vocab):
    # log posterior for each class: log P(c) + sum log P(w | c)
    words = [w for w in msg.split() if w in vocab]
    scores = {}
    for c in prior:
        s = np.log(prior[c])
        for w in words:
            s += np.log(likelihoods[c].get(w, 1e-6))
        scores[c] = s
    # normalize to actual posterior (softmax)
    total = np.logaddexp.reduce(list(scores.values()))
    return {c: np.exp(s - total) for c, s in scores.items()}

prior, likelihoods, vocab = train(training)
for msg in ["free prize now", "meeting tomorrow at ten", "click free win"]:
    p = classify(msg, prior, likelihoods, vocab)
    print(f"{msg:>30}  ->  spam prob = {p['spam']:.3f}, ham prob = {p['ham']:.3f}")`;

export const bayesUpdateCode = `# Sequential Bayes: update prior as evidence arrives.
# Coin problem: is this coin fair (p=0.5) or biased toward heads (p=0.75)?
# Start with 50-50 prior, then update after each flip.
import numpy as np

prior_fair = 0.5
prior_bias = 0.5

def update(observation, posterior_fair, posterior_bias):
    # posterior_fair := P(fair | observations so far)
    # observation: "H" or "T"
    p_obs_fair = 0.5 if observation == "H" else 0.5
    p_obs_bias = 0.75 if observation == "H" else 0.25
    joint_fair = p_obs_fair * posterior_fair
    joint_bias = p_obs_bias * posterior_bias
    total = joint_fair + joint_bias
    return joint_fair / total, joint_bias / total

flips = "HHHHH"    # 5 heads in a row
p_fair, p_bias = prior_fair, prior_bias
print(f"start: P(fair) = {p_fair:.3f},  P(biased) = {p_bias:.3f}")
for i, f in enumerate(flips, 1):
    p_fair, p_bias = update(f, p_fair, p_bias)
    print(f"after {i} flip{'s' if i > 1 else ''}: {flips[:i]:>{len(flips)}}  ->  P(fair) = {p_fair:.3f}, P(biased) = {p_bias:.3f}")

print()
print("After 5 heads, our belief has shifted from 50/50 to strongly biased —")
print(f"each 'H' multiplies the ratio P(bias)/P(fair) by 0.75/0.5 = 1.5")`;

<Motivation title="Bayes' theorem — the recipe for updating beliefs">

**Bayes' theorem** is a one-liner:

$$\text{posterior} = \frac{\text{prior} \times \text{likelihood}}{\text{evidence}}$$

But it's much bigger than a formula. It's the recipe for how any rational reasoning system — a person, a doctor, a spam filter, an LLM, a self-driving car — should update its beliefs when new information arrives. Miss it and you'll misread confident-looking outputs your whole career. Get it and you'll see it everywhere.

Three pieces of vocabulary before we go further:

- **Prior**: what you believed before seeing new evidence.
- **Likelihood**: how well each possible answer *explains* the evidence you saw.
- **Posterior**: your updated belief after combining the prior and the evidence.

Bayes appears in several places you'll actually work with as a GenAI engineer:

- **Reading model outputs correctly.** When an LLM says "The answer is X with 99% confidence," Bayes helps you check whether that 99% is *calibrated* — that is, does the model actually get things right 99% of the time when it says 99%? Usually not — most models are overconfident. Bayesian thinking helps you tell.
- **RAG (retrieval-augmented generation).** RAG scores documents against a query and ranks them. Interpret the scores as unnormalized probabilities and the whole "which document is most relevant?" question becomes Bayesian inference.
- **Reasoning under uncertainty.** "The model says X — how much should I trust it?" is a Bayesian question. Your prior: how reliable is the model in general? Your evidence: this specific answer. Your posterior: your updated belief in X.
- **Bayesian deep learning.** Not mainstream in vanilla LLM training, but critical for uncertainty estimation, active learning, and hyperparameter tuning (via Bayesian optimization).

The chapter is short and payoff-heavy: derive the theorem, walk through the "medical test paradox" that trips up almost everyone the first time, build a Naive Bayes spam filter from scratch, and sketch how beliefs update as data streams in.

</Motivation>

<Intuition title="Reweight your beliefs by how well each explanation fits the evidence">

Imagine you have several possible explanations for something — call them hypotheses H₁, H₂, H₃… Before seeing any evidence, you have some starting beliefs about each. That's your **prior**. If you have no reason to prefer one, you'd say "equally likely."

Now evidence E shows up. For each hypothesis, ask: **"If this hypothesis were true, how likely would I have seen this evidence?"** That answer is the **likelihood** P(E | Hᵢ) — read as "probability of the evidence given hypothesis i."

Bayes' recipe: multiply each hypothesis's prior by its likelihood, then rescale so all your beliefs add up to 1. Hypotheses that predicted the evidence well get boosted. Hypotheses that didn't get demoted. What comes out is your **posterior** — your updated beliefs.

$$
\text{posterior} \;=\; \frac{\text{prior} \times \text{likelihood}}{\text{evidence}}
$$

That "evidence" in the denominator is just the total probability of the evidence occurring, summed over all hypotheses. It's a normalizing constant — its only job is to make the posteriors add to 1. In practice you often compute up to this constant, work in log-space (to avoid underflow), and normalize at the very end.

</Intuition>

<Recall term="Conditional probability">

P(A | B) = P(A and B) / P(B) — the probability that A is true, given we know B is true. Restricts attention to the "B is true" subset of the sample space. This is the definition that makes Bayes' theorem trivial to derive.

</Recall>

## Bayes' theorem — one line derived

Start from the definition of conditional probability:

$$
P(A \mid B) \;=\; \frac{P(A \cap B)}{P(B)} \qquad \text{and} \qquad P(B \mid A) \;=\; \frac{P(A \cap B)}{P(A)}
$$

Rearrange the second: P(A ∩ B) = P(B | A) · P(A). Substitute into the first:

$$
\boxed{\;P(A \mid B) \;=\; \frac{P(B \mid A) \cdot P(A)}{P(B)}\;}
$$

That's Bayes' theorem. In model-fitting language: A is a hypothesis (e.g., "disease present," "spam," "coin biased"), B is evidence (e.g., "test positive," "message contains word 'free'," "flip is heads"). The pieces:

- **P(A)** = **prior** — belief in A before seeing B.
- **P(B | A)** = **likelihood** — how well A explains the evidence.
- **P(A | B)** = **posterior** — belief in A after seeing B.
- **P(B)** = **evidence / marginal likelihood** — total probability of seeing B, summed over all hypotheses. For two-hypothesis problems, P(B) = P(B | A) P(A) + P(B | ¬A) P(¬A).

## The base-rate trap — the "99% accurate test" that isn't

Here's the classic Bayes example that trips up almost everyone the first time. You'll be tempted to guess wrong. That's fine — this one is *supposed* to be counter-intuitive.

**Setup.** A rare disease affects 1% of the population. There's a test with two nice properties:

- **Sensitivity** 99% — meaning if you have the disease, the test says "positive" 99% of the time. In our notation: P(positive | disease) = 0.99.
- **Specificity** 99% — meaning if you're healthy, the test says "negative" 99% of the time. In our notation: P(negative | healthy) = 0.99.

You take the test. It comes back positive. **What's the probability you actually have the disease?**

Intuition screams "about 99%." Bayes says otherwise. Let's compute:

- P(disease) = 0.01 (the prior — 1% base rate)
- P(positive | disease) = 0.99 (likelihood if you have it)
- P(positive | healthy) = 0.01 (the false-positive rate — 1 minus specificity)
- P(positive) = 0.99·0.01 + 0.01·0.99 = 0.0198 (total probability of testing positive, summed over both cases)

Bayes' theorem: P(disease | positive) = (0.99 · 0.01) / 0.0198 = **0.5**.

A 99% accurate test on a 1% disease still leaves you with only 50% probability of actually being sick. **Half of positive tests are false alarms.** Why? Because 1% of healthy people (a much larger group) test positive by mistake, and their numbers balance out the actual sick people who test positive.

This is called the **base-rate fallacy** — the human tendency to focus on the "99% accurate" (the likelihood) and ignore the "1% base rate" (the prior). Doctors get this wrong on their exams. Journalists get it wrong in health reporting. Machine learning models — if trained on class-imbalanced data without careful calibration — get it wrong too.

<RunnableCode title="Medical test paradox: 99% accurate test, still only 50% posterior" code={medicalTestCode} packages={[]} />

**Lesson for GenAI**: if a model's output has 99% confidence for a rare category (say "generate a factually correct sentence about an obscure topic"), you should reason like the base rate. Even if the model is well-calibrated on average, the prior probability of correctness on obscure topics is low, so the posterior may still be much lower than the model's stated confidence. Hallucination detection is fundamentally a base-rate problem.

## Naive Bayes — Bayes applied to features

The **naive Bayes classifier** is Bayes' theorem applied to features that are (naively) assumed independent given the class. For classifying a document with word features w_1, …, w_n:

$$
P(\text{class} \mid w_1, \ldots, w_n) \;\propto\; P(\text{class}) \prod_{i} P(w_i \mid \text{class})
$$

The "independence" is naive because words aren't actually independent (seeing "free" makes "prize" more likely), but the simplification makes the math easy AND the classifier surprisingly effective. This was the state of the art in spam filtering for years, and Bayesian classifiers still appear in pipelines today.

<RunnableCode title="Naive Bayes spam classifier from scratch" code={spamCode} packages={["numpy"]} />

Key implementation notes:

- **Log-space arithmetic.** Products of many small probabilities underflow; sum log-probabilities instead.
- **Laplace smoothing.** Add 1 to every count so no unseen word gets zero probability (which would zero out the whole product).
- **Softmax normalization.** Compute log-posteriors, subtract the max, exponentiate — the standard log-sum-exp trick for numerical stability.

For a modern GenAI engineer, Naive Bayes is less commonly used but the machinery is instructive: it's Bayes with strong independence assumptions, and its failure modes (bad calibration when features are correlated) illustrate why more sophisticated models exist.

## Sequential updating — the Bayesian iteration

Bayes composes. If evidence arrives one piece at a time, each new piece updates the posterior of the previous step, and the result is the same as if all evidence had arrived at once. This is the mathematical basis of **online learning**, **Kalman filters**, and **Bayesian belief networks**.

For sequential updates with independent-given-hypothesis observations E_1, E_2, …, E_n:

$$
P(H \mid E_1, \ldots, E_n) \;\propto\; P(H) \cdot P(E_1 \mid H) \cdot P(E_2 \mid H) \cdots P(E_n \mid H)
$$

— multiply the prior by every likelihood, normalize. Equivalent to iterating "posterior = prior · likelihood / evidence" and using the previous posterior as the next prior.

<RunnableCode title="Sequential Bayes: coin-fairness belief update after each flip" code={bayesUpdateCode} packages={["numpy"]} />

The iteration is easy to code and has a satisfying property: it converges to the *right* answer as evidence accumulates, regardless of the prior (as long as the prior didn't assign zero probability to the true hypothesis). This is the mathematical foundation of "belief updates are self-correcting" in Bayesian methods.

## The traps that matter

**Don't confuse P(A|B) with P(B|A).** They differ by the ratio of priors: P(A|B) / P(B|A) = P(A) / P(B). Journalists commit this error constantly ("99% of terrorists use encryption" ≠ "99% of encryption users are terrorists"). The base-rate fallacy is a special case of this confusion.

**Priors matter — subjective is not the same as wrong.** Different priors give different posteriors. In practice, use domain knowledge to choose a prior; if you have none, use a "flat" or "uninformative" prior (uniform over hypotheses). Objective Bayes tries to codify "uninformative" priors formally; subjective Bayes accepts that priors are user-provided and tests robustness to prior choice.

**Log-space or numerical explosion.** Multiplying many small probabilities underflows to zero. Always work in log-space: log P(posterior) = log P(prior) + log P(likelihood) − log P(evidence). Log-sum-exp is the trick for the normalizer.

**Naive Bayes is not calibrated.** Even when Naive Bayes classifies correctly, its output probabilities are often way off from true posteriors — usually too confident. For a document classified as 99% spam by naive Bayes, actual spam probability might be 60–70%. Use isotonic regression or Platt scaling to calibrate if you need meaningful probability outputs.

**Bayesian ≠ Frequentist.** These are two paradigms of interpreting probability. Bayesian: probability represents degree of belief; can talk about "P(hypothesis)". Frequentist: probability is long-run frequency in repeated trials; can't talk about "P(hypothesis)" (a hypothesis is either true or false, no randomness). Practical ML uses both — Bayes rule is a mathematical identity that works in either framework, but the *interpretation* of the outputs differs.

<RealWorldML title="Bayesian reasoning about model hallucinations">

Consider an LLM that outputs a specific fact with 90% confidence. Is that fact actually correct 90% of the time? Almost certainly not — modern LLMs are typically *overconfident*, especially on rare topics. Bayes gives us a framework for reasoning about this:

- **Prior** P(fact is true | topic): how common is truthfulness for facts of this type/domain in the training data? For common facts (capitals of countries), high. For obscure facts (specific dates in niche fields), lower.
- **Likelihood** P(model outputs this specific claim with this confidence | fact is true) vs P(same output | fact is false): how does the model behave when it "knows" vs when it's confabulating? Recent research: the *entropy* of the model's next-token distribution during generation is a useful signal — low-entropy generations for factual claims are more reliable.
- **Posterior** P(fact is true | model says X with 90% confidence): the actual belief you should hold about the fact after seeing the model output.

Practical hallucination detection systems (e.g., SelfCheckGPT, various uncertainty-quantification tools) formalize this: sample multiple generations from the model, look at agreement across samples, use the Bayesian posterior of "the model actually knows this fact" as the confidence score.

**Calibration matters for RAG systems too.** When retrieving evidence for a query, similarity scores are proxies for P(relevance | query). But these scores are not calibrated probabilities. A cosine similarity of 0.85 doesn't mean "85% probability of being the right document." Bayes lets you formalize this: if you have labeled query-doc pairs with correctness annotations, you can learn a calibrated posterior P(is-answer | similarity, query-topic) and use *that* to gate whether to trust the retrieved document. This is a genuine engineering win — better calibration means fewer wrong answers surfaced from RAG.

Every "how much should I trust this" question in GenAI has a Bayesian shape to it. You may not always write down the formula, but recognizing it in the wild is the beginning of not being fooled by overconfident model outputs.

</RealWorldML>

<Summary>

- **Bayes' theorem**: P(A|B) = P(B|A) · P(A) / P(B). Read as "posterior = prior × likelihood / evidence." Derives in one line from the definition of conditional probability.
- **Prior**: belief before evidence. **Likelihood**: how well each hypothesis predicts the evidence. **Posterior**: updated belief. **Evidence**: normalizer, sum over all hypotheses.
- **Base-rate fallacy**: even highly accurate tests give surprising posteriors when the prior is small. Formalize with Bayes; don't intuit.
- **Naive Bayes classifier**: assume features independent given class, multiply likelihoods, normalize. Simple and effective for text; poorly calibrated but often good at ranking.
- **Sequential updating**: multiply prior by each likelihood as evidence arrives; equivalent to batch Bayes. Use log-space to avoid underflow.
- **Applications in GenAI**: reasoning about model hallucinations, calibrating RAG confidences, updating beliefs about model reliability given new evaluations.
- **Not the same as frequentist inference**: Bayesian talks about P(hypothesis); frequentist talks about long-run frequencies. Bayes' theorem is a mathematical identity that works in both, but the *interpretation* of "probability of a hypothesis" is Bayesian.

</Summary>
