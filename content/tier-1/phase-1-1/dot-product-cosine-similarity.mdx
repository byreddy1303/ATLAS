rng = np.random.default_rng(42)
# Sample 5000 pairs of random unit vectors in D dimensions,
# and record their cosine similarities.
for D in [2, 3, 10, 100, 1000]:
    v = rng.standard_normal((5000, D))

# 4 "token" vectors in 3-D (imagine token embeddings)
tokens = np.array([
    [1.0, 0.0, 0.0],   # 0 : "the"
    [0.9, 0.1, 0.2],   # 1 : "big"
    [0.2, 0.9, 0.1],   # 2 : "dog"
    [0.1, 0.8, 0.2],   # 3 : "barks"
])

# query = token 2 ("dog"). We want to know how much each other token
# should "attend to" this query.
q = tokens[2]

# raw scores = dot products (this is exactly transformer attention,
# minus the /sqrt(d_k) scaling and softmax)
scores = tokens @ q                    # (4,)
print("raw scores :", scores.round(3))

# softmax to turn scores into weights
expv    = np.exp(scores - scores.max())
weights = expv / expv.sum()

# the attended output is a weighted sum of the tokens
out = weights @ tokens
print("output     :", out.round(3))`;

export const solutionCode = `import numpy as np

def most_similar_cities(home, cities, k=5):
    # cities: (n, d) matrix, home: (d,)
    dots = cities @ home
    home_norm = (home @ home) ** 0.5
    city_norms = (cities * cities).sum(axis=1) ** 0.5
    sims = dots / (city_norms * home_norm)
    return np.argsort(-sims)[:k]`;

<Motivation title="Why the dot product had to be invented">

Imagine you run a movie-streaming service. A user has just finished watching *Arrival* and stayed up until 2am rewatching a slow philosophical scene. Your job — a job you must do about ten-thousand times per second across all the users on your platform — is to answer a single question: **out of the ten thousand movies in the catalog, which are most similar to what this person just enjoyed?**

Here is the shape of the problem. Every movie is described by a handful of numbers: how much sci-fi it contains, how romantic it is, how much action, how philosophical, how slow-paced. Five numbers per movie. Ten thousand movies. That's a five-dimensional cloud of 10,000 points. The user's taste — inferred from *Arrival* — is *also* a five-dimensional point. And "how similar to your taste is movie X" is really the question **how close, in some sense, is X to the taste point?**

But *how close* means what, exactly? Suppose the user's taste is described as `(9, 3, 2, 9, 3)` — heavy on sci-fi and philosophy, light on romance and action, moderate pace. Movie A is `(9, 3, 2, 9, 3)`, the same as the taste. Movie B is `(90, 30, 20, 90, 30)` — the same *proportions*, but every axis multiplied by 10 (perhaps because the movie is much longer, so total minutes-of-each-genre are higher). Are these two movies equally good matches? Almost certainly yes — the *shape* of what B offers is identical to what the user wants. Movie C is `(3, 9, 9, 3, 9)` — an action romance with fast pace. Even though the *raw magnitudes* of C and B are pretty different (C sums to 33; B sums to 260), C is clearly a *worse* match than B, because it points in the wrong direction.

This is the deep observation: **what matters is direction, not magnitude.** The vectors `(9, 3, 2, 9, 3)` and `(90, 30, 20, 90, 30)` point exactly the same way in five-dimensional space — one is just a longer arrow. And two vectors that point the same way share the same *ratio* of one axis to another, which is precisely what "same taste, different scale" means.

The **dot product** is the arithmetic operation that computes a single number capturing "how aligned are these two arrows?" Divide it by the two vectors' lengths and every trace of magnitude vanishes; what remains is a pure measure of angular similarity we call **cosine similarity**. This one identity — a number, an angle, and a shared direction — powers every semantic search engine, every retrieval-augmented generation (RAG) pipeline, every recommender system, every attention head in every transformer. Get this chapter and you will *recognize* half of modern AI when you read its source code.

</Motivation>

<Intuition title="The idea, before any symbols">

Picture two arrows shot from the same point. If they point in nearly the same direction, they *agree*. If they point in perpendicular directions, they *disagree*. If they point in opposite directions, they *contradict*. The dot product is a single number that captures where on this "agree → perpendicular → contradict" scale two vectors sit. Positive means "kind of agree." Large positive means "strongly agree." Zero means "unrelated, right-angled." Negative means "actively opposite."

But the raw dot product doesn't know how big the arrows are. A short arrow and a long arrow pointing in exactly the same direction give a *big* dot product, not because they agree more strongly, but because one is long. To strip away that "size effect," you divide by the two lengths. What you're left with is a number between −1 and +1 — the **cosine of the angle** between the arrows. It's a pure measure of *how aligned they are*, ignoring how big either one is.

That's the whole chapter. Everything else — the algebraic formula, the projection interpretation, the code, the movie recommender, the transformer attention scores — is an implication of this single mental picture. Return to it whenever you get lost.

</Intuition>

<Recall term="Vector length (norm)">

For a 2-D vector `v = (v₁, v₂)`, the length is `‖v‖ = √(v₁² + v₂²)` — Pythagoras' theorem, unchanged since ancient Greece. For any dimension, `‖v‖ = √(v₁² + v₂² + ... + vₙ²) = √(v · v)`. Length is always non-negative. It's zero only for the zero vector `(0, 0, ..., 0)`. You will use `‖v‖` constantly for the rest of your ML life — today we already need it.

</Recall>

<Recall term="Angle between vectors">

Two arrows shot from the origin form an angle between them — call it θ (theta). By convention, θ is measured as the *smallest* positive rotation from one to the other, so it always lives in the interval [0°, 180°]. Two identical arrows: θ = 0°. Perpendicular: θ = 90°. Opposite: θ = 180°. The cosine function `cos θ` is +1 at 0°, 0 at 90°, and −1 at 180° — which is exactly the "agree → perpendicular → contradict" scale we described.

</Recall>

<Angles intro={<>The dot product is the same object described in three fundamentally different ways. Each way tells you something the others hide. Read all four passes; internalize all four pictures.</>}>

  <Angle type="geometric" title="Projection × length — the geometric heart">

  Take vector `a` and vector `b`, both drawn from the origin. Now imagine a lamp shining directly perpendicular to `b` from above — the **shadow** of `a` on the line containing `b` is called the **scalar projection** of `a` onto `b`. It's a single real number: how far along `b` does `a` reach when squashed onto `b`'s direction?

  If `a` points in exactly the same direction as `b`, the shadow is the full length of `a`. If `a` is perpendicular to `b`, the shadow is zero (the lamp shines *along* `a`, so its shadow is a single point). If `a` points opposite to `b`, the shadow is negative — it falls on the "wrong side" of the origin along `b`.

  The **geometric definition** of the dot product is: **the scalar projection of `a` onto `b`, multiplied by the length of `b`.** In symbols:

  $$
  \mathbf{a} \cdot \mathbf{b} = \|\mathbf{a}\| \, \|\mathbf{b}\| \cos\theta
  $$

  Read that formula out loud. It says: "the dot product equals the length of `a`, times the length of `b`, times the cosine of the angle between them." Every one of these three factors matters. Doubling the length of `a` doubles the dot product. Doubling the length of `b` doubles the dot product. Making the angle 90° zeros it out. Making the angle 180° flips its sign.

  Why multiply by `‖b‖` and not just report the shadow directly? Because the dot product wants to be *symmetric* — `a · b = b · a`. The shadow of `a` on `b` is not the same as the shadow of `b` on `a` (they can be different lengths). But `‖a‖ ‖b‖ cos θ` is obviously the same whether you write it as `‖a‖ (‖b‖ cos θ)` — "length of `a` times projection of `b` onto `a`" — or `‖b‖ (‖a‖ cos θ)` — "length of `b` times projection of `a` onto `b`". Symmetry preserved.

  **Mental picture to build:** two arrows, an angle, a light source. Every time you see `a · b`, imagine that scene.

  </Angle>

  <Angle type="algebraic" title="Sum of pairwise products — the arithmetic identity">

  When you learn linear algebra in school, this is often the *first* definition you see:

  $$
  \mathbf{a} \cdot \mathbf{b} = a_1 b_1 + a_2 b_2 + \dots + a_n b_n = \sum_{i=1}^n a_i b_i
  $$

  It says: *multiply matching components; sum them up.* For `a = (2, 1)` and `b = (1, 2)`: `2·1 + 1·2 = 4`. Done. It looks nothing like the geometric formula — no lengths, no angles, no projections — yet it produces the same number. That's the small miracle of the dot product: two very different-looking computations agree.

  Why should we believe them? Consider a right triangle formed by three vectors — `a`, `b`, and `a − b`. The *law of cosines* from trigonometry says:

  $$
  \|\mathbf{a} - \mathbf{b}\|^2 = \|\mathbf{a}\|^2 + \|\mathbf{b}\|^2 - 2\|\mathbf{a}\|\|\mathbf{b}\|\cos\theta
  $$

  If we expand the left side in coordinates — `‖a − b‖² = Σ(aᵢ − bᵢ)² = Σaᵢ² − 2Σaᵢbᵢ + Σbᵢ²` — and equate the two, the `Σaᵢ²` cancels with `‖a‖²` and the `Σbᵢ²` cancels with `‖b‖²`, leaving:

  $$
  -2\sum_i a_i b_i = -2 \|\mathbf{a}\|\|\mathbf{b}\|\cos\theta
  $$

  Divide both sides by −2 and you have proved:

  $$
  \sum_i a_i b_i = \|\mathbf{a}\|\|\mathbf{b}\|\cos\theta
  $$

  The algebraic and geometric definitions *must* be the same. The law of cosines glues them together. If this feels like a lot to swallow the first time, come back — the payoff is that from now on you can pick whichever description fits the moment.

  </Angle>

  <Angle type="computational" title="A NumPy one-liner — the code you'll actually write">

  In practice, no working engineer writes the sum by hand. Every scientific-computing library has a native, blazing-fast dot product. In NumPy there are three interchangeable forms:

  <CodeBlock language="python" title="dot product — three equivalent moves" code={threeWaysCode} />

  All three produce `4.0`. The `sum-of-products` version is for teaching. The `@` operator is the modern Pythonic form (Python 3.5+ added it precisely for linear-algebra libraries; nothing outside numpy/torch/jax uses it much). The `np.dot` function is the historical spelling and is still perfectly fine.

  Why does it matter that these are C-fast? Because in real work you'll compute *millions* of dot products per query — every embedding against every stored document, every attention token against every other token. A Python-loop dot product between two 1,000-dimensional vectors takes microseconds; a NumPy dot product takes nanoseconds. Multiply either by "10,000 documents × 100 queries per second" and only one of those numbers is livable.

  When you scale further — say to 100 million document embeddings — even NumPy is too slow. Vector databases (FAISS, HNSW, pgvector) precompute clever indices so that "find the 10 nearest cosine-similar vectors" runs in log time. But the primitive operation, in the innermost loop, is still the dot product.

  </Angle>

  <Angle type="physical" title="Work and force — where physics uses the same object">

  In physics, if a constant force `F` (a vector) acts on an object that moves along a displacement `d` (also a vector), the **work done** by the force is `W = F · d`. Read that formula through the geometric lens: the work equals the component of force *along the motion*, times the magnitude of motion. If the force is perpendicular to the motion (say, gravity acting on a horizontally sliding block on a frictionless surface), the dot product is zero — no work is done, energy is not transferred. If the force pushes the object exactly the way it's going, all the force is "spent" moving it — maximum work.

  This isn't a coincidence. The dot product was invented precisely because so many physical and geometric quantities have exactly this structure: *how much of one vector points along another?* Every time you see it in an ML paper, remember: physicists had it first.

  </Angle>

  <Angle type="code" title="Projection — extracting the shadow itself">

  Suppose you don't just want the *scalar* shadow of `a` onto `b`, but the actual **vector projection** — the arrow that lies along `b` and represents "the part of `a` that goes in `b`'s direction." Here's the formula, derived directly from the geometric picture:

  $$
  \text{proj}_\mathbf{b}(\mathbf{a}) = \left( \frac{\mathbf{a} \cdot \mathbf{b}}{\mathbf{b} \cdot \mathbf{b}} \right) \mathbf{b}
  $$

  Parse it slowly. The numerator `a · b` is the scalar dot product. The denominator `b · b` is `‖b‖²`. So `(a · b) / (b · b)` is a scalar; multiplying it by `b` gives a vector pointing along `b`. That scalar is exactly the right size so that the resulting vector lands where the shadow of `a` on `b`'s line falls.

  <RunnableCode title="scalar and vector projections" code={projectionCode} />

  Projection is the backbone of an enormous amount of ML: least-squares regression is literally "project the target vector onto the column space of the feature matrix." Gram-Schmidt orthogonalization is repeated projection. PCA finds the directions along which the projection captures the most variance. **If you understand the projection formula, you have a shared vocabulary with half of statistics.**

  </Angle>

<Convergence>

Five separate accounts have converged on the same object. The **geometric** view (`‖a‖‖b‖cos θ`) tells you *what the answer means* — how aligned two arrows are, weighted by their lengths. The **algebraic** view (`Σ aᵢbᵢ`) tells you *how to compute it* with a few multiplications and additions. The **computational** view (`a @ b`) is what your fingers actually type in real code. The **physical** view says the same operation shows up whenever "how much of one direction lives inside another" is the question. The **projection** view unlocks least-squares, Gram-Schmidt, and PCA.

Divide the dot product by the two lengths and every whisper of magnitude vanishes; what's left is the **cosine similarity** — a single number in [−1, +1] that answers *how aligned are these two arrows, regardless of length?* That is the ML workhorse. Read every embedding, every retrieval, every attention head through this lens; each is a variation on the same identity.

</Convergence>

<Derivation title="Deriving cos θ = (a · b) / (‖a‖ ‖b‖)">

We already have two definitions of the dot product. Setting them equal solves for cos θ. Every line of the derivation gets a plain-English partner so you can read the math without any leaps.

<Step n={1} math={<span>Start with the geometric definition: <br/> <code>a · b = ‖a‖ · ‖b‖ · cos θ</code></span>}>
    This is the picture: dot product equals (length of a) × (length of b) × (cosine of the angle between them). Convince yourself with a sanity case. If θ = 90° (the vectors are perpendicular), then cos θ = 0, so a · b = 0. Perpendicular vectors have zero dot product. Now check the other extreme: if θ = 0° (the vectors point exactly the same way), cos θ = 1, so a · b = ‖a‖ · ‖b‖ — the product of the two lengths, its largest possible value. Both cases match the intuition; the formula is trustworthy.
</Step>

<Step n={2} math={<span>Divide both sides by <code>‖a‖ · ‖b‖</code>:<br/> <code>cos θ = (a · b) / (‖a‖ · ‖b‖)</code></span>}>
    Dividing by the two lengths strips away every trace of magnitude from both sides. What remains on the right — cosine — depends only on the *angle* between the vectors. This is the single move that turns "raw similarity" into "direction-only similarity". It is the mathematical reason cosine similarity works even when your movie vectors, document embeddings, or user-taste vectors all have wildly different scales. This is why a 20-page document and a 200-page document can still be compared honestly.
</Step>

<Step n={3} math={<span>Rewrite the numerator using the algebraic form:<br/> <code>cos θ = (a₁b₁ + a₂b₂ + ... + aₙbₙ) / (‖a‖ · ‖b‖)</code></span>}>
    Now every part of the formula is computable with plain arithmetic. No angles need to be measured, no protractors used, no trigonometry looked up. The numerator is a sum of products. The denominator is a product of two square-roots-of-sums-of-squares. If you can add and multiply, you can compute a cosine similarity in any number of dimensions. This is what makes cosine practical at Google-scale search: a plain arithmetic loop, nothing exotic.
</Step>

<Step n={4} math={<span>Establish the range: <code>−1 ≤ cos θ ≤ 1</code></span>}>
    Since θ lives in [0°, 180°] by convention, cosine goes through its full range. cos θ = 1 means the vectors point *identically* (angle 0°). cos θ = 0 means they're orthogonal (angle 90°). cos θ = −1 means they point in exactly opposite directions (angle 180°). Any embedding-search algorithm you'll ever write consumes this three-point scale. When you look at retrieval results, ranked by cosine similarity, you're always reading them along this dimension.
</Step>

<Step n={5} math={<span>Sanity: what if either vector is zero?</span>}>
    The denominator becomes zero and the expression is undefined. Geometrically this makes sense — you can't measure the angle between an arrow and "no arrow at all." Real systems that use cosine similarity either (a) filter zero-vectors before comparison, (b) clamp norms above some tiny ε, or (c) use raw dot product when they know the vectors are pre-normalized. The failure mode is annoying but always debuggable if you remember step 5.
</Step>

</Derivation>

<Example title="Working it by hand — a full trace">

Let `a = (2, 1)` and `b = (1, 2)`. We'll compute every quantity and see the picture form.

**Algebraic dot product.** `a · b = a₁b₁ + a₂b₂ = 2·1 + 1·2 = 2 + 2 = 4`. That's the scalar shadow-times-length quantity from the geometric definition.

**Lengths.** `‖a‖ = √(2² + 1²) = √5 ≈ 2.236`. Same for `b`: `‖b‖ = √(1² + 2²) = √5 ≈ 2.236`. The two vectors have identical lengths.

**Cosine similarity.** `cos θ = (a · b) / (‖a‖ · ‖b‖) = 4 / (√5 · √5) = 4 / 5 = 0.8`. Read this: "the two arrows are 80% of the way toward being identical" — closer to alignment than to perpendicularity.

**Angle.** `θ = arccos(0.8) ≈ 36.87°`. Confirm with a picture: the vector `(2, 1)` points 26.57° above the x-axis; the vector `(1, 2)` points 63.43° above the x-axis. The difference is 36.87°. ✓

**Now scale `b` by 100.** Let `b' = (100, 200)`. The dot product blows up: `a · b' = 2·100 + 1·200 = 400`. But the cosine similarity is unchanged: `cos θ = 400 / (√5 · √50000) = 400 / (√5 · 100√5) = 400 / 500 = 0.8`. The cosine formula is *invariant* to any positive scaling of either vector. That is the property that makes it useful for real recommenders where document lengths vary wildly.

Move the tips around in the playground below and watch how cos θ changes. Confirm: when the vectors are exactly aligned, cos θ = 1. When they're perpendicular, cos θ = 0. When one is anti-parallel to the other, cos θ = −1. When they're close in direction, cos θ ≈ 0.9-something. Build the reflex now so it lives in your fingers by chapter 5.

<Example title="A 3-D example — the arithmetic still works">

Let `a = (1, 2, 2)` and `b = (2, 0, 1)`.

- Dot product: `a · b = 1·2 + 2·0 + 2·1 = 4`.
- `‖a‖ = √(1 + 4 + 4) = √9 = 3`.
- `‖b‖ = √(4 + 0 + 1) = √5 ≈ 2.236`.
- Cosine similarity: `4 / (3 · √5) = 4 / (3 · 2.236) ≈ 0.596`.
- Angle: `θ = arccos(0.596) ≈ 53.4°`.

Nothing changed except the number of terms. In 100 dimensions there'd be 100 terms in the sum and a 100-term sum-of-squares in each norm. That's still a millisecond of NumPy. This is why the same operation scales from a physics textbook to a trillion-parameter language model.

</Example>

<Nuance>

**Cosine ignores magnitude — by design.** A movie represented as `(4, 4)` and one as `(400, 400)` have identical cosine similarity to *any* user vector. That's usually a feature: it means "long" and "short" documents can be compared on equal footing. But sometimes magnitude is real signal — for TF-IDF, longer documents genuinely *should* get more weight for a matching term, which is why BM25 (a length-aware refinement) was invented. Choose the metric to match the semantics.

**Cosine is not a distance.** Distance functions must obey the triangle inequality: `d(a, c) ≤ d(a, b) + d(b, c)`. Cosine similarity, and even `1 − cos θ`, doesn't satisfy this in general. Real vector databases work with *cosine distance on unit-normalized vectors*, where the space behaves better; treat that as an engineering convention, not a mathematical guarantee.

**High-dimensional weirdness.** In hundreds or thousands of dimensions, two random vectors are almost always nearly *perpendicular*. Their cosine similarity clusters tightly around 0. Rankings still work — the useful signal comes from the *rank order* of scores, not their raw magnitude — but you cannot judge "is 0.35 a good cosine similarity" without knowing the distribution in your particular embedding space.

<RunnableCode title="Cosine collapse in high dimensions" code={highDimCode} />

Run the cell above. You'll see the standard deviation of cosine similarities shrink like `1/√D`. In D=1000, the spread is under 0.03. This is why "raw" cosine values from LLM embedding models can look tiny (0.5–0.7) even for very related documents — the interesting signal lives in the *tail*, not near ±1.

**Zero-length vectors** break the formula (division by zero). Real pipelines either filter them out or add a small ε to the denominator. Watch for this the first time you build a search system and a query embedding somehow comes out all-zeros.

**Sign matters (a lot) in some domains.** Cosine can be negative — meaning "actively opposite." For most modern embedding models (which pack semantic content into positive directions), negative cosines are rare and often meaningful. But in classical bag-of-words spaces where feature vectors are non-negative, cosine is *always* in [0, 1] and you never see negatives.

</Nuance>

<RealWorldML title="Transformer attention — the dot product in disguise">

Here is the same operation, dressed as the mechanism inside every modern language model. Transformer attention starts with a set of "token vectors" — one per word or sub-word — and asks: *for each token, how much should it look at each of the other tokens?* The mechanism is a scaled dot product.

<RunnableCode title="a two-line 'attention' calculation" code={attentionMiniCode} />

Read the cell. Each score is a dot product between the query token (`"dog"`) and every other token. The `softmax` turns those raw scores into a probability distribution — how much attention to pay to each token. The output is the weighted sum of token vectors, according to those weights. That's it. That's what a transformer's attention head does, minus (a) the `/ √d_k` scaling for numerical stability and (b) separate learned matrices to produce query/key/value vectors. When you build up to Tier IV you'll find that this same picture — a dot product, a softmax, a weighted sum — appears in every single layer of every single LLM.

The reason we spent so long on the dot product in this chapter is that it is the *only* operation you need to understand deeply to see how a transformer thinks. Everything else is bookkeeping.

<LifeLesson title="Direction beats magnitude — a habit worth carrying">

Cosine similarity teaches something the ML textbooks skip: **direction matters more than magnitude**. A vector with the *right* direction and a small length is often more useful than a vector with impressive length and the wrong direction. The math has been telling us this quietly for two hundred years.

The same shape appears in life. Two people can pour identical hours into a career — one with a clear direction (deliberately shaping toward a specific craft), one with volume but no compass — and end up in radically different places a decade later. The dot product is telling you: **align first, scale second.** Small-but-aligned effort compounds. Big-but-misaligned effort dissipates. It's the same reason the *Arrival*-loving user should never be recommended *Mad Max* just because *Mad Max* has more "everything." Volume without direction is just noise.

When you're overwhelmed by "how much" you're doing, ask instead: *what does the taste vector I'm trying to match actually look like — and is my day pointing at it?* If not, no amount of scaling saves you. If yes, even a short vector goes far. This is why senior engineers accomplish more in twenty focused hours than juniors in eighty scattered ones — same magnitude, better cosine.

</LifeLesson>

<MLConnection>

The dot product is not one operation among many; it is *the* operation that quietly runs modern AI. Once you internalize this chapter, you will start noticing it in every paper.

- **Semantic search / RAG:** every query embedding is compared to every document embedding by cosine similarity. The top-k documents get stuffed into the LLM's prompt. Change nothing in this chapter and you already understand how a vector database ranks results.
- **Transformer attention:** the "score" between a query token and a key token, inside self-attention, is a scaled dot product. Softmax over those scores tells the model which token to attend to. Every attention head in every LLM is a dot product.
- **Neural networks:** every fully-connected layer is `y = Wx + b`. That `Wx` is a batch of dot products — one per output neuron. A ReLU network is just a stack of dot products with kinks between them.
- **Word embeddings:** GloVe explicitly factorized a matrix so that `word_i · word_j ≈ log P(i, j)`. The dot product was the entire loss function.
- **Classical ML:** logistic regression, SVM decision function, PCA all read directly out of dot products.
- **Physics and graphics:** projections, illumination models, computer-graphics shading — anywhere "how much of one direction points along another" is the question.

The chapter you just finished is, quietly, the most reused unit of arithmetic in engineering.

</MLConnection>
