# THREE vectors in 2-D — three arrows on a page. Can ALL three be independent?
v1 = np.array([1.0, 0.0])
v2 = np.array([0.0, 1.0])

<Motivation title="How many coordinates does your world need?">

An LLM embedding says a token lives in ℝ¹²²⁸⁸, but studies keep finding models actually use far fewer axes than that — the interesting motion happens on a lower-dimensional slab inside the space. The word for that slab is a **subspace**; the word for its true axis count is **dimension**; the tool for finding both is **linear independence**.

The same concept catches bugs one step from your notebook. Two features in a table that are perfectly correlated (say, "height in cm" and "height in inches") aren't two features — they're one, wearing two hats. A linear model handed both will refuse to invert its normal equations; a neural net will learn a random split between them. The formal name for this pathology — features that can be built from each other — is **linear dependence**, and it's why libraries expose `matrix_rank`.

Later chapters lean on this hard. PCA finds a low-dimensional basis that captures the variance. Attention outputs live in the span of the value vectors. Neural-network initialization worries about the *rank* of weight matrices. Learning independence, span, and basis now unlocks all of that vocabulary — and closes the "why is my regression singular?" bug forever.

</Motivation>

<Intuition title="Independent = adds a new direction, redundant = doesn't">

Imagine you're placing arrows in the plane, one at a time, and asking after each: *did the last arrow open up a genuinely new direction, or could I have gotten there already by scaling and adding what I had?*

- **Arrow 1** (say (1, 0)): trivially new — from nothing, you can now reach a whole **line**.
- **Arrow 2** (say (0, 1), not parallel to arrow 1): also new — you can now reach the entire **plane**.
- **Arrow 3** (any vector in the plane): **not new**. Every point in ℝ² is already reachable from the first two; the third is a recipe of them wearing a hat.

That's the whole idea. A set of vectors is **linearly independent** when every one of them adds a direction you couldn't previously build; **span** is the set of everything you can reach with them; a **basis** is an *efficient* generating set — enough vectors to reach everything, no wasted redundancies. The zero vector is the vanishing point: any set containing it is automatically dependent (you can scale it by anything without moving).

</Intuition>

<Recall term="Linear combination">

From two chapters ago: c₁v₁ + c₂v₂ + … + c_kv_k — scale each vector by its own coefficient and add them. Every question in this chapter is really about linear combinations: *which* points can be reached (span), and are the recipes *unique* (independence).

</Recall>

## Span — everywhere you can reach

The **span** of a set of vectors is the set of all their linear combinations — every point you can get to by scaling and adding them:

$$
\mathrm{span}\{ \mathbf{v}_1, \dots, \mathbf{v}_k \} = \big\{\, c_1 \mathbf{v}_1 + \cdots + c_k \mathbf{v}_k \;:\; c_i \in \mathbb{R} \,\big\}
$$

The shapes are memorable:

- **One vector** (nonzero): its span is a **line** through the origin — all its scalings.
- **Two vectors** (not parallel): their span is a **plane** through the origin.
- **Two parallel vectors**: their span *collapses* back to a line — the second added no new direction.
- **The zero vector alone**: its span is just \{**0**\} — a single point.

Notice all spans pass through the origin — c = 0 for every vector produces **0**, always in the span. That's why span-based objects are called **subspaces**: closed under addition and scaling, always anchored at the origin.

## Linear independence — no free riders

A set \{v₁, …, v_k\} is **linearly independent** when the only recipe that produces the zero vector is the boring one:

$$
c_1 \mathbf{v}_1 + \cdots + c_k \mathbf{v}_k = \mathbf{0} \quad\Longrightarrow\quad c_1 = c_2 = \cdots = c_k = 0
$$

If any *non-trivial* recipe (not all zeros) sums to **0**, the set is **linearly dependent** — you can isolate one vector as a combination of the others. Two equivalent readings, both worth carrying:

- **Every reachable point has a unique recipe** (independence) — versus multiple recipes reaching the same point (dependence).
- **No vector is redundant** — no v_j can be written using the others.

Two quick reflexes: if a set contains the **zero vector**, it's dependent (multiply **0** by 999 and add zero times everything else — a nontrivial recipe for **0**). In ℝⁿ, more than **n vectors** are always dependent — you can only fit n directions before repetition sets in. This is the ceiling behind the punchline of the next section.

## Basis — a minimal, efficient frame

A **basis** for a subspace is a set of vectors that is both

1. **spanning** — every point in the subspace is some linear combination of them, and
2. **linearly independent** — no vector is redundant.

The magic follows: with a basis \{b₁, …, b_d\}, every vector x in the subspace has **exactly one** recipe

$$
\mathbf{x} = c_1 \mathbf{b}_1 + c_2 \mathbf{b}_2 + \cdots + c_d \mathbf{b}_d
$$

and the coefficients (c₁, …, c_d) are called the **coordinates** of x in that basis. This is the whole reason coordinates exist — a chosen basis is what turns "an abstract point" into "a specific list of numbers you can type into a computer."

The count d — the number of vectors in a basis — is called the **dimension** of the subspace. A remarkable theorem: *every* basis of the same subspace has the same size. This makes dimension an honest invariant of the space, not an artifact of which basis you chose.

**Standard basis of ℝⁿ.** The everyday basis of ℝⁿ is e₁ = (1, 0, …, 0), e₂ = (0, 1, 0, …, 0), …, e_n = (0, 0, …, 1) — one 1 in each slot, zeros elsewhere. In this basis, the *coordinates* of x = (x₁, …, xₙ) are just x₁, …, xₙ themselves; every array indexing operation you write is a coordinate lookup in the standard basis. Dimension of ℝⁿ = n.

<Derivation title="Uniqueness — why a basis gives coordinates, not just any recipe">

Independence isn't just "no redundancy" — it's *the* property that makes coordinates well-defined. Here's the two-line argument.

<Step n={1} math={<span>Suppose x had two recipes: <code>x = c₁b₁ + ⋯ + c_d b_d</code> and <code>x = c₁' b₁ + ⋯ + c_d' b_d</code></span>}>
    We want to show the two recipes are the same — i.e., cᵢ = cᵢ' for every i. Assume, for contradiction, that they're different; we'll derive a violation of independence.
</Step>

<Step n={2} math={<span>Subtract them: <code>0 = (c₁ − c₁') b₁ + ⋯ + (c_d − c_d') b_d</code></span>}>
    Same x minus same x is <b>0</b>; on the right, subtract the two recipes term by term. Every coefficient of bᵢ becomes cᵢ − cᵢ'. So this is a recipe for the zero vector — and we know exactly one kind of recipe reaches <b>0</b> when the bᵢ are independent.
</Step>

<Step n={3} math={<span>Independence forces every difference to be zero: <code>cᵢ − cᵢ' = 0</code>, hence <code>cᵢ = cᵢ'</code></span>}>
    By the definition of independence, the only way c₁b₁ + ⋯ + c_d b_d = <b>0</b> is <em>all coefficients zero</em>. So each cᵢ − cᵢ' = 0, so cᵢ = cᵢ' for every i, and the two recipes were the same all along. This is the guarantee that turns "there exists a recipe" (spanning) into "there is <em>exactly one</em> recipe" (coordinates), and it's why bases — not just spanning sets — are the objects worth having.
</Step>

</Derivation>

<Example title="Working it by hand — one full trace">

Question: is the set v₁ = (1, 2), v₂ = (2, 4), v₃ = (0, 1) independent, and what's its span?

**Step 1 — spot the dependence.** v₂ = 2·v₁: same direction, doubled length. So among *just* v₁, v₂, the second is a free rider — the recipe 2·v₁ + (−1)·v₂ = **0** proves it (non-trivial, sums to zero).

**Step 2 — check what remains.** v₁, v₃ = (1, 2), (0, 1): are these independent? The recipe c₁v₁ + c₂v₃ = **0** reads (c₁, 2c₁ + c₂) = (0, 0). Slot 1 forces c₁ = 0; slot 2 then forces c₂ = 0. Only the trivial recipe — independent.

**Step 3 — span.** Two independent vectors in ℝ² span *all* of ℝ² (dimension 2, two directions, done). Adding v₂ back adds no new directions — the full set still spans ℝ², just with one redundancy.

**Step 4 — dimension and rank.** The dimension of the span is **2** (rank of the matrix with these columns is 2, even though there are three columns). A basis for it is the pair v₁, v₃ (or v₁, v₂/2, or v₂, v₃ — any independent pair from the set). Bases aren't unique — but their *size* is.

</Example>

## Change of basis — the same vector, two coordinate lists

If a basis gives you coordinates, then **choosing a different basis relabels every vector's coordinates without touching the vector itself**. Think of it as swapping cameras: point stays put, its numeric address depends on which frame you photograph it in.

Concretely, let the columns of a matrix B be a basis \{b₁, …, b_d\}. A coordinate vector c in that basis represents the actual vector `x = B c` (a linear combination — the coordinates are the coefficients). To go the other way — from an actual vector x to its coordinates in the new basis — solve for c:

$$
\mathbf{x} = B\,\mathbf{c} \quad\Longleftrightarrow\quad \mathbf{c} = B^{-1} \mathbf{x}
$$

The next chapter derives a clean special case: when the basis is **orthonormal** (perpendicular unit vectors), inverting isn't needed at all — you just take dot products.

**Why this matters in ML.** PCA finds a new basis of your feature space whose first axis captures the most variance, the second captures the next-most, etc. — you then keep only the first k coordinates and throw away the rest, going from d features to k without losing much. That's dimensionality reduction, and it's *literally* a change of basis followed by a truncation. When Tier 3 introduces PCA, this section is the language it speaks.

## The traps that matter

**"Independent" is a property of the whole set, not one vector.** Asking "is v₃ independent?" is meaningless in isolation — the question is always "is the set v₁, …, v_k independent?", and the same v₃ can be redundant in one set and essential in another. Read every claim about independence as a claim about the set.

**Numerical vs exact.** Perfect dependence (v₂ = 2v₁) is rare in real data; **near-dependence** is everywhere (two features 0.99 correlated). `matrix_rank` uses a tolerance to decide when a singular value is "effectively zero," and choosing that tolerance matters — set it too tight and you'll declare noisy near-duplicates independent, too loose and you'll drop genuine directions. When it matters, look at the singular values themselves rather than reading the rank as a boolean.

**Multicollinearity in regression.** Feature dependence quietly wrecks linear models. If two columns are exact copies, the normal equations `(XᵀX)w = Xᵀy` have no unique solution — any `(w₀, w₃)` with `w₀ + w₃` fixed fits equally well. `lstsq` still returns *a* solution (the minimum-norm one), but the individual coefficients are meaningless; only sums like `w₀ + w₃` are interpretable. The clean fix is to *drop* redundant columns before fitting — or, for near-dependence, use regularization (ridge/lasso) to break the ambiguity.

**Adding features doesn't add dimension unless they're independent.** Dumping "height in cm" and "height in inches" and "height in feet" into a model gives you three columns but *one* direction of information. The rank of your data matrix — not its column count — is the honest count of features you have.

<RealWorldML title="Rank and the dimensions a model actually uses">

Modern LLMs have embedding widths in the thousands (12,288 for GPT-3), but a phenomenon called the **low-rank hypothesis** — sharpened by the LoRA fine-tuning method — observes that the *changes* a model learns during fine-tuning live in a much smaller-dimensional subspace than the full weight matrix. LoRA exploits this ruthlessly: instead of learning a full weight update ΔW ∈ ℝᵈˣᵈ (d² parameters), it learns ΔW = A·B where A ∈ ℝᵈˣʳ and B ∈ ℝʳˣᵈ for a small **rank** r (typically 4–64). By construction, that ΔW has rank at most r — because the span of A·B's columns is contained in the span of A's r columns.

Read that with this chapter's vocabulary:

1. **Independence sets the ceiling.** A d × d matrix can have at most d independent columns. Forcing it to factor through r « d dimensions caps the number of independent directions its updates can point in at r — a hard architectural bound.
2. **Parameters plummet.** ΔW = A·B has 2dr parameters instead of d². For d = 12,288 and r = 8: about 197k parameters versus 151 million per layer — a ~770× reduction, with quality that often matches full fine-tuning on downstream tasks.
3. **The empirical claim behind the trick.** The *directions* a task requires the model to move are few. Fine-tuning ChatGPT for legal documents doesn't need 12,288 independent updates per layer — it needs a handful of task-directions, and the low-rank factorization discovers them.

Same idea, other places: PCA is dimensionality reduction by keeping the k top-variance basis directions; embedding-table compression writes E ≈ U·V for U ∈ ℝⱽᵒᶜᵃᵇˣʳ, V ∈ ℝʳˣᵈ. When Tier 5 covers LoRA in full, this chapter is why the trick even exists.

</RealWorldML>

<Summary>

- **Span** \{v₁, …, v_k\} = every linear combination of them — always a subspace anchored at the origin.
- **Linearly independent** = the only recipe that reaches **0** is all-zeros. Equivalent: no vector is a combination of the others; every reachable point has a unique recipe.
- **Basis** = spanning + independent. It gives every vector *exactly one* coordinate list — this is why the concept of "coordinates" exists at all.
- **Dimension** = size of any basis (all bases of the same subspace have the same size). ℝⁿ has dimension n; its standard basis is e₁, …, eₙ.
- **Change of basis**: choosing a different basis relabels coordinates without moving the vector — `c = B⁻¹x`. PCA, LoRA, and every "new frame" trick lives on this.
- Traps: independence is a property of the *set* (not one vector); redundant features silently break linear regression's normal equations; rank of a data matrix, not its column count, is the honest feature-direction count.

</Summary>
