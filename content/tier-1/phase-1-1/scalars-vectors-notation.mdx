export const embeddingCode = `import numpy as np

# A tiny embedding table: 6 tokens, 4 dimensions each.
# Real scale: GPT-3's table is 50,257 tokens x 12,288 dims.
vocab = ["the", "dog", "cat", "barks", "meows", "runs"]
E = np.array([
    [ 0.1,  0.0,  0.2,  0.0],   # the
    [ 0.9,  0.8,  0.1,  0.3],   # dog
    [ 0.8,  0.9,  0.1,  0.2],   # cat
    [ 0.7,  0.2,  0.9,  0.1],   # barks
    [ 0.6,  0.3,  0.9,  0.1],   # meows
    [ 0.4,  0.1,  0.5,  0.9],   # runs
])
token_to_id = {tok: i for i, tok in enumerate(vocab)}

def embed(token):
    return E[token_to_id[token]]       # a "lookup" IS row indexing

sentence = ["the", "dog", "barks"]
vectors = np.stack([embed(t) for t in sentence])
print("sentence as a stack of row-vectors, shape", vectors.shape)
print(vectors)
print()
print("'dog' IS the vector", embed("dog"), "— one row of E")`;

<Motivation title="Every model sees the world as vectors">

When an LLM reads the word "dog", the first thing it does is replace it with a list of numbers — 768 of them in BERT, 12,288 in GPT-3. That list is a **vector**, and everything the model does afterward — attention, layers, the final prediction — is arithmetic on vectors. Images become vectors of pixel values, users become vectors of behavior, and a RAG system retrieves documents by comparing their vectors. Vectors are not one topic in ML; they are the *substance* models compute on.

The catch: this material lives in two languages. Papers write bold **x**, subscripts x₂, and "x ∈ ℝⁿ"; code writes `np.array`, `x[1]`, and `.shape`. Mistranslation causes real bugs — off-by-one indexing errors, and shape mismatches that silently produce a matrix where you wanted a vector.

This short chapter is the Rosetta stone: what the symbols mean, how they map to NumPy, and the two mental pictures (arrow, point) you'll reuse in every later chapter. Nothing here is hard — but everything in Tier 1 through Tier 4 is written in this language.

</Motivation>

<Intuition title="One object, three faces">

A vector is one thing with three faces. **Face 1: a list of numbers** — `(3, 4)` — this is what the computer stores. **Face 2: an arrow** — start at the origin, go 3 right and 4 up; direction and length live in those same two numbers. **Face 3: a point** — the tip of that arrow, a location in space; this is how a *dataset* looks, each sample one point.

Same numbers, three pictures. Geometry questions (which way? how far? how aligned?) use the arrow. Data questions (which samples are near each other? what does the cloud look like?) use the point. Storage and computation use the list. Fluency in this chapter means switching faces without noticing you did it — and all the notation that follows is just a compact way of saying *which vector* and *which piece of it*.

</Intuition>

<Recall term="NumPy array shape">

Every NumPy array has a `.shape` — a tuple with one entry per axis. `np.array([1, 2, 3])` has shape `(3,)`: one axis, three entries. `np.zeros((3, 1))` has shape `(3, 1)`: two axes. The *number of axes* is `ndim`; the *number of entries along an axis* is that axis's size. Keep these words separate — this chapter's biggest trap lives exactly here.

</Recall>

## Scalars vs vectors — one number vs an ordered list

A **scalar** is a single number: a temperature, a learning rate, a similarity score. Papers write scalars as *italic lowercase*: x, η, λ.

A **vector** is an *ordered* list of numbers, written **bold lowercase**: **x**, **v**. Ordered matters: `(2, 5)` and `(5, 2)` are different vectors — the first slot and the second slot *mean different things* (sci-fi-ness vs romance-ness, or east vs north). The number of entries is the vector's **dimension**, and the set of all n-dimensional real vectors is written ℝⁿ ("R-n"). So "**x** ∈ ℝ³" reads: *x is a vector of 3 real numbers*. ℝ alone (no exponent) is just the real number line — scalars live there.

<RunnableCode title="one vector, three faces" code={threeFacesCode} packages={["numpy"]} />

The pitfall is a word collision: in ML, a "768-**dimensional** embedding" means a vector with 768 *entries* — in NumPy, an array of shape `(768,)` with **one** axis. It does *not* mean a 768-axis array. When an ML person says dimension, they almost always mean *entries in the vector*, not `ndim`.

## Column vectors — the ML default (and NumPy's third option)

By convention, math treats every vector as a **column** unless told otherwise:

$$
\mathbf{x} = \begin{bmatrix} x_1 \\ x_2 \\ \vdots \\ x_n \end{bmatrix} \in \mathbb{R}^n
\qquad\qquad
\mathbf{x}^\top = [\, x_1 \; x_2 \; \cdots \; x_n \,]
$$

The **transpose** symbol ᵀ flips a column into a **row** (and back). Why default to columns? Because matrix-times-vector, the single most common operation in ML, is written `W x` — and the shapes only line up when x stands vertically. Every neural-network layer `y = Wx + b` assumes it.

NumPy quietly adds a third option: the 1-D array of shape `(n,)`, which is *neither* a row nor a column — it has no second axis at all. This is what you'll use 95% of the time, and it's fine: `W @ x` treats it as a column automatically. But the moment 2-D shapes enter, be explicit:

<RunnableCode title="(3,) vs (3,1) vs (1,3) — and the silent (3,3)" code={columnRowCode} packages={["numpy"]} />

The last line is the trap that costs real debugging hours: adding a `(3,)` to a `(3,1)` doesn't error — broadcasting stretches both and hands you a `(3, 3)` **matrix**. If a "vector" in your code suddenly has two axes (a stray `keepdims=True`, a sklearn column), either `ravel()` it back to `(n,)` or keep *everything* 2-D. Never mix.

## Reading the symbols — subscripts, superscripts, indexing

Three rules decode almost every line of vector notation you'll meet:

1. **Subscript = component.** xᵢ (italic — it's a scalar!) is the i-th *entry* of the vector **x**. So for **x** = (7, 2, 5): x₁ = 7, x₃ = 5.
2. **Parenthesized superscript = which sample.** ML papers write x⁽ʲ⁾ for the j-th *data point* in a dataset. So x₂⁽⁷⁾ reads: entry 2 of sample 7. Subscript picks the piece; superscript picks the vector.
3. **Math counts from 1, code counts from 0.** The paper's x₁ is your `x[0]`; xₙ is `x[-1]`. This off-by-one is the single most common bug when translating a formula. Say it out loud when you translate: *math one is code zero*.

And the workhorse symbol Σ (capital sigma) just means "add them up": Σᵢ xᵢ is the sum of all entries — `x.sum()`.

## Arrows vs points — two pictures of the same numbers

Draw **x** = (3, 4) as an **arrow** from the origin when you care about geometric questions — which direction does it point, how long is it, how aligned is it with another vector? Every *operation* you'll learn (adding, scaling, projecting, rotating) is easiest to see on arrows.

Draw it as a **point** — just the arrow's tip — when the vector is a *sample in a dataset*: one movie, one user, one token embedding. A dataset of N samples with d features each is a matrix **X** ∈ ℝᴺˣᵈ, and each **row** is one vector, one point in d-dimensional space. Clusters, outliers, decision boundaries — all of those are point-cloud pictures.

<RunnableCode title="the same 8 vectors, drawn both ways" code={arrowsPointsCode} packages={["numpy", "matplotlib"]} captureFig={true} />

Same eight vectors. The left picture asks "which arrows point the same way?" — that's the *similarity* question the dot-product chapter answers. The right picture asks "which movies cluster together?" — that's the question k-means and every classifier answers. You'll flip between these pictures for the rest of the syllabus; practice flipping now.

<Example title="Decoding a dataset, symbol by symbol">

A paper says: *"Given a dataset X ∈ ℝᴺˣᵈ with N = 3 samples x⁽¹⁾, x⁽²⁾, x⁽³⁾ ∈ ℝ², compute x₂⁽¹⁾ + x₁⁽³⁾."* Let's take

$$
X = \begin{bmatrix} 7 & 2 \\ 4 & 9 \\ 5 & 1 \end{bmatrix}
$$

- **X ∈ ℝᴺˣᵈ = ℝ³ˣ²** — a matrix with 3 rows (samples) and 2 columns (features). In code: `X.shape == (3, 2)`.
- **x⁽¹⁾** is *sample 1* — the first **row**: (7, 2). In code: `X[0]` (math one, code zero).
- **x₂⁽¹⁾** is *entry 2 of sample 1*: the 2 in (7, 2). In code: `X[0, 1]`.
- **x₁⁽³⁾** is *entry 1 of sample 3*: sample 3 is (5, 1), so x₁⁽³⁾ = 5. In code: `X[2, 0]`.
- Answer: x₂⁽¹⁾ + x₁⁽³⁾ = 2 + 5 = **7**.

Every intimidating formula in an ML paper decodes this same way: identify which vector (superscript / row), then which entry (subscript / column), then shift every index down by one for code. Slow the first ten times, automatic forever after.

</Example>

<RealWorldML title="An embedding table — the first thing every LLM does">

Before a transformer can attend to anything, text must become vectors. The mechanism is almost embarrassingly simple: a big matrix **E** with one **row per token** in the vocabulary, and "embedding a token" is *looking up its row*.

<RunnableCode title="tokens → vectors by row lookup" code={embeddingCode} packages={["numpy"]} />

Read the cell:

1. **The table.** `E` has shape `(6, 4)`: 6 tokens in the vocabulary, each owning a 4-dimensional vector. In this chapter's language: each row of E is a vector in ℝ⁴.
2. **The lookup.** `token_to_id` maps a token string to a row number; `E[token_to_id[token]]` fetches that row. That's the entire "embedding layer" of a neural network — indexing.
3. **A sentence becomes a matrix.** Stacking the three token vectors gives shape `(3, 4)` — N = 3 samples in ℝ⁴, exactly the X ∈ ℝᴺˣᵈ shape from this chapter. This matrix is what flows into the transformer.
4. **Scale.** GPT-3's table is 50,257 × 12,288 — about 617 million numbers, learned by gradient descent like every other parameter. Notice "dog" and "cat" were given similar rows here; real training *discovers* that similarity on its own — which is what makes the arrow picture (directions encode meaning) so powerful later.

When Tier 4 builds a transformer, this cell is literally step one of the forward pass.

</RealWorldML>

<Summary>

- Scalar = one number (*italic x*); vector = **ordered** list of numbers (**bold x**). Order carries meaning; the entry count is the dimension; **x** ∈ ℝⁿ says "n real entries."
- Math's default vector is a **column**; xᵀ flips it to a row. NumPy's `(n,)` array is neither — fine on its own, dangerous next to 2-D shapes.
- Subscript = component (xᵢ), parenthesized superscript = sample (x⁽ʲ⁾), and *math one is code zero*: x₁ ↔ `x[0]`.
- Two pictures, same numbers: **arrow** from the origin for geometry and operations; **point** for datasets and clustering.
- A dataset is X ∈ ℝᴺˣᵈ — every row a sample vector. An embedding table is the same thing for tokens, and "embedding" a token is just row indexing.
- Shape traps to say out loud: a "768-dimensional" embedding is shape `(768,)` (one axis); `(n,) + (n, 1)` silently produces `(n, n)`.

</Summary>
