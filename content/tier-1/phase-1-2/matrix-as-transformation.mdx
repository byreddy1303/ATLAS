# A matrix IS its action on the standard basis — columns are Ae1, Ae2, ...

              [0.0,  3.0]])

e1 = np.array([1.0, 0.0])

R1 = lambda t: np.array([[np.cos(t), -np.sin(t)],
                         [np.sin(t),  np.cos(t)]])

A = R1(np.pi / 4)       # rotate 45
B = R1(np.pi / 6)       # rotate 30

# apply B then A to a vector v:
v = np.array([1.0, 0.0])

<Motivation title="Every transformer layer is a matrix acting on vectors">

Open the source of any GPT-style model and count the operations that actually shape the answer: **`x @ W`**, **`Q @ K.T`**, **`attn @ V`**, **`h @ W_out`**. That's it. Layer norm, softmax, and ReLU do the nonlinear seasoning — but the *shape* of an LLM's computation is a chain of matrix multiplications acting on batches of token vectors.

For a GenAI engineer, "matrix" is not "grid of numbers you push through a solver." It's a **function you learn** — a map that pulls an embedding into a query space, or mixes 512-dim residuals into 2048-dim MLP activations, or averages value vectors weighted by attention scores. Every learned parameter tensor in a transformer is a matrix and every matrix is a linear map. This chapter builds the one mental model that turns matrix code into geometry you can picture — then every subsequent chapter (multiplication, eigen, SVD, LoRA) is just refinements of the same picture.

The picture: **a matrix is a verb**. It takes vectors from one space and puts them somewhere. The columns tell you exactly where the standard-basis vectors land, and everything else follows.

</Motivation>

<Intuition title="A matrix is where the basis vectors land">

Picture an m×n matrix A. It's a machine with n input dials (one per column) and m output wires (one per row). Feed it a vector x with n numbers; it hands back a vector Ax with m numbers.

The trick: **you don't need to know what A does to *every* input to know everything about A** — you only need to know what it does to the n standard-basis vectors e₁ = (1, 0, …, 0), e₂ = (0, 1, 0, …), …, eₙ.

Because every x can be written as x = x₁e₁ + x₂e₂ + … + xₙeₙ, and A is *linear* (A(u+v) = Au + Av, A(cv) = cAv), we get

Ax = x₁(Ae₁) + x₂(Ae₂) + … + xₙ(Aeₙ) — **a weighted sum of the images of the basis vectors**.

And the images `Ae₁, Ae₂, …, Aeₙ` are literally the columns of A (proved in the derivation below). So A is fully determined by "where does each standard basis vector go?" — n arrows sketched on paper, and the whole map lives in them.

</Intuition>

<Recall term="Linear map">

A function T : ℝⁿ → ℝᵐ is **linear** if it respects addition and scaling: T(u + v) = T(u) + T(v) and T(cv) = c·T(v) for all vectors u, v and scalars c. Combined, this means T(a·u + b·v) = a·T(u) + b·T(v). Every linear map from ℝⁿ to ℝᵐ can be written as x ↦ Ax for exactly one m×n matrix A — this is the fundamental correspondence the chapter turns on.

</Recall>

## The one identity that unlocks matrices — Ae_j = column j of A

Given a matrix A with columns a₁, a₂, …, aₙ, the product with the j-th standard basis vector e_j is:

$$
A e_j = a_j
$$

That's it. **Applying a matrix to a standard basis vector plucks out the corresponding column** — no calculation required, just look at column j. Every other formula in this chapter falls out of this one.

Consequences:

- The image of the standard basis under A is exactly the set of A's columns: `{Ae₁, Ae₂, …, Aeₙ} = {a₁, a₂, …, aₙ}` — the columns are the *arrows* the map draws in the output space.
- For general x = Σⱼ xⱼ eⱼ, linearity gives Ax = Σⱼ xⱼ (Aeⱼ) = **Σⱼ xⱼ aⱼ** — every matrix-vector product is a **linear combination of the columns of A, using x as coefficients**. This is the "column view" of matrix-vector multiplication; next chapter fleshes out three of these views.

<Derivation title="Why the columns of A ARE Ae_1, …, Ae_n">

We prove the identity Aeⱼ = column j of A directly from the row-column definition of the matrix-vector product. This is the one paragraph in this chapter that's pure symbol-pushing — memorize it once, and every other formula in linear algebra becomes visible.

<Step n={1} math={<span>Setup: A is m×n with entries A<sub>ij</sub>; e<sub>j</sub> is n×1 with a 1 in slot j and zeros elsewhere.</span>}>
    Concretely: e_j = (0, …, 0, 1, 0, …, 0) with the 1 in position j. The i-th entry (e_j)_k is 1 if k = j and 0 otherwise — the <b>Kronecker delta</b> δ<sub>jk</sub>.
</Step>

<Step n={2} math={<span>(Ae<sub>j</sub>)<sub>i</sub> = Σ<sub>k=1..n</sub> A<sub>ik</sub> (e<sub>j</sub>)<sub>k</sub></span>}>
    The definition of the matrix-vector product: the i-th entry of Ax is the dot product of row i of A with x. Here x is e_j, so we sum A<sub>ik</sub> · (e<sub>j</sub>)<sub>k</sub> as k ranges over 1..n.
</Step>

<Step n={3} math={<span>Only the k=j term survives: (Ae<sub>j</sub>)<sub>i</sub> = A<sub>ij</sub></span>}>
    (e<sub>j</sub>)<sub>k</sub> is zero except when k=j, so the entire sum collapses to a single term: A<sub>ij</sub> · 1 = A<sub>ij</sub>. Ae<sub>j</sub>'s i-th entry is A<sub>ij</sub> — <b>exactly</b> the i-th entry of column j of A.
</Step>

<Step n={4} math={<span>Assembling all i together: Ae<sub>j</sub> = (A<sub>1j</sub>, A<sub>2j</sub>, …, A<sub>mj</sub>) = column j of A. ∎</span>}>
    That's the whole proof. Every row-i entry of Ae<sub>j</sub> is A<sub>ij</sub>, so stacking them all up gives column j of A. The consequence — Ax = Σ x<sub>j</sub> a<sub>j</sub> for any x — follows by linearity.
</Step>

</Derivation>

## Building any matrix by specifying where the basis lands

Because A is fully determined by its columns, **you can design any linear map you want by choosing where you want the basis vectors to land**. Pick target arrows a₁, …, aₙ in the output space, put them side by side, and the resulting matrix has exactly the map you asked for. This is how every geometric transformation is constructed.

The five moves every engineer should be able to write by hand:

- **Scale by (s_x, s_y)** — e₁ → (s_x, 0), e₂ → (0, s_y). Matrix: `diag(s_x, s_y)`.
- **Rotate by θ (counter-clockwise)** — e₁ → (cos θ, sin θ), e₂ → (−sin θ, cos θ). Matrix: `[[cos θ, −sin θ], [sin θ, cos θ]]`.
- **Shear (x by k times y)** — e₁ → e₁ (untouched), e₂ → (k, 1). Matrix: `[[1, k], [0, 1]]`.
- **Reflect across the x-axis** — e₁ → e₁, e₂ → (0, −1). Matrix: `[[1, 0], [0, −1]]`.
- **Project onto the x-axis** — e₁ → e₁, e₂ → (0, 0). Matrix: `[[1, 0], [0, 0]]` (a projection is a matrix that squashes one direction to zero — an idempotent that satisfies P² = P).

<RunnableCode title="Five transformations applied to a unit square" code={catalogCode} packages={["numpy"]} />

The corners of the transformed square are exactly the columns of `M @ square` — one column per corner. Rotation preserves shape; scale stretches it; shear tilts it; reflection flips it; projection flattens it onto a line. **The action on the corners is the action on the whole shape** because linearity extends any map on the basis to the entire space.

## Composition = matrix product (order matters)

If A maps ℝⁿ → ℝᵐ and B maps ℝᵖ → ℝⁿ, then "first apply B, then apply A" is the composite map (A ∘ B) : ℝᵖ → ℝᵐ. The matrix that computes this composite map is the **product AB** — an m×p matrix. Applied to a vector:

$$
(A B)\, x = A (B x)
$$

Left-hand side: one matrix multiplication. Right-hand side: two. They agree by design — matrix multiplication is *defined* to be composition of the underlying linear maps. That's why the inner dimensions must match (A's columns = B's rows = n): B outputs vectors in ℝⁿ, and A needs ℝⁿ vectors as input.

**Order is fatal**. AB and BA are usually different matrices — often not even the same shape. Even when both products make sense (both square, same size), they generally give different maps: "rotate then scale" is not "scale then rotate."

<RunnableCode title="Compose transformations by multiplying matrices" code={composeCode} packages={["numpy"]} />

Two rules to burn in:

- **Read chains right-to-left.** `A @ B @ C @ x` means: apply C first (rightmost, closest to x), then B, then A. The rightmost matrix touches the vector; each next one wraps the previous output.
- **Matrix multiplication is associative but non-commutative.** `(AB)C = A(BC)` — you can regroup — but `AB ≠ BA` in general. This is why neural network layer order matters (you can't swap two learned linear layers).

## Non-square matrices — the dimension-changer

Square matrices (n×n) shuffle ℝⁿ into itself — rotate, scale, shear. **Non-square** matrices change the dimension of the space:

- **m×n with m > n** (tall matrix): embeds ℝⁿ into a larger ℝᵐ. Example: an *embedding layer* that takes 1-hot token indices (in a vocab-size dimensional space, but only one active) and produces a d_model-dimensional dense embedding — mathematically that's a `d_model × vocab_size` matrix. In the other direction, `d_model × d_ff` in an MLP up-projection.
- **m×n with m < n** (wide matrix): projects ℝⁿ down to a smaller ℝᵐ. Example: the attention head projections W_Q, W_K, W_V in transformers are `d_model × d_k` matrices with d_k = d_model / n_heads — they take the full residual stream and squeeze it down to per-head query/key/value spaces. LoRA's low-rank updates ΔW = A B are wide × tall — two rectangular matrices whose product is a rank-r approximation of a full square weight update.

<RunnableCode title="Non-square matrices as dimension changers" code={nonSquareCode} packages={["numpy"]} />

The mental model doesn't change with shape: the columns of an m×n matrix are still n vectors in ℝᵐ — n arrows saying "here's where each input-space basis vector lands." What changes is whether that map is invertible: a wide matrix (m < n) *must* have a nullspace (inputs that map to zero) because the output space isn't big enough to hold n independent images. That's a chapter on its own; for now, the intuition is enough.

## The traps that matter

**"Rows or columns?" pick one and stick to it.** Some texts write vectors as row vectors and use xA (right-multiplication); ML libraries almost universally use column vectors and Ax. NumPy is agnostic — both `A @ x` (with x as (n,)) and `x @ A.T` compute the same thing. **In this course, and in almost every ML paper: columns are inputs, matrix is on the left, `Ax` is the standard**. Read every formula through this convention.

**Matrix multiplication is not commutative — but it IS associative.** `A(BC) = (AB)C`, so you can compute either grouping. This associativity is why "fuse two learned matrices into one" is a real optimization: two consecutive linear layers `y = W₂(W₁ x)` = `(W₂ W₁) x` — you can precompute the composed matrix once at inference time. LoRA's ΔW = A B is stored as two thin matrices during training (cheap) and can be materialized as one full matrix at inference time (free).

**Only *linear* maps have matrices.** T(x) = x + 3 (shifting by a constant) is *not* linear — T(0) = 3 ≠ 0. Neural networks use **affine** maps (Wx + b) followed by nonlinearities; the +b piece isn't matrix multiplication, it's a separate translation. Similarly, a nonlinearity like ReLU or softmax is not a matrix. The linear-map/matrix correspondence covers only the linear pieces; the rest of the network is deliberately non-matrix nonlinearity.

**Applying the same matrix to a batch is one matmul.** In code, if `X` is shape `(batch, features)` and `W` is shape `(features, out)`, then `X @ W` applies the linear map to every row in one shot. This is the **fused, vectorized** form the GPU actually executes — never loop over rows. The mental model above (columns as images of basis) still holds row-wise: each row of `X @ W` is a linear combination of the rows of W (because in this convention the *rows of W* are what act on features).

<RealWorldML title="Attention's Q projection is a linear map you can visualize">

Every transformer builds Q, K, V matrices by projecting token embeddings through learned matrices W_Q, W_K, W_V. The Q projection is `Q = tokens @ W_Q` — one matrix, one product, one shape change. Here's the whole operation in eight lines, with the same interpretation you'd give any linear map:

<RunnableCode title="A tiny attention Q-projection" code={attentionProjCode} packages={["numpy"]} />

Read the cell:

1. **Shape story.** `tokens` is `(3 tokens, 6 features)` — three residual-stream vectors in the model's 6-dimensional latent space. `W_Q` is `(6, 2)` — a wide matrix that projects 6 dims down to 2 (a tiny toy head; real models use 64 or 128 per head). The output `Q = tokens @ W_Q` is `(3, 2)` — three query vectors, one per token, living in the 2-dim query space.
2. **What W_Q *is*, geometrically.** Its 2 columns are two vectors in ℝ⁶ — the two query-space "basis directions" expressed in the model's feature vocabulary. Column 0 says "this direction of query space is *this* mixture of the 6 embedding features"; column 1 says the same for the other query dimension. The projection Q[i] just measures how much of each direction is present in token i — literally two dot products (right-hand side of the print).
3. **Learned, not designed.** In a real model, W_Q is initialized randomly and adjusted by gradient descent so that the projections `Q_i` produce *high* dot products with the right K_j — encoding "this token wants to look at that token." No one specifies the columns of W_Q; they emerge. But whatever they end up as, the map they define is the same construction as this chapter's: pick columns, get a linear map.
4. **Why the shape matters.** Small d_k (= 2 here, = 64 in real models) means the head can only track a low-dimensional slice of the residual — perfect for specializing (some heads track syntax, some track long-range coreference, etc.). Multi-head attention runs many such maps in parallel and concatenates — a linear map factory. The chapter you're reading is the entire underlying vocabulary.

Every learned matrix in a transformer is *some* linear map like this one — the columns saying where the basis vectors of the input space are being sent. Understand those columns, and you can start reading model weights the way you read code.

</RealWorldML>

<Summary>

- A **matrix is a linear map**: A ∈ ℝᵐˣⁿ represents a function ℝⁿ → ℝᵐ that respects addition and scaling. There is a one-to-one correspondence between linear maps and matrices (once bases are chosen).
- **Columns of A ARE the images of the standard basis**: `Ae_j = a_j`. Consequence: `Ax = Σ_j x_j a_j` — every matrix-vector product is a linear combination of columns weighted by the input's entries.
- **Design principle**: to build any matrix, pick where you want the basis vectors to land and stack those targets as columns. Rotate/scale/shear/reflect/project — five moves, all built this way.
- **Composition = matrix product**: `(AB)x = A(Bx)`, read right-to-left. Associative but not commutative — the order in a chain is meaningful.
- **Non-square matrices** change dimension: tall matrices embed, wide matrices project. Every attention head's W_Q, W_K, W_V is a d_model × d_k wide-projection matrix; embedding tables and MLP up-projections are tall.
- Every learned parameter tensor in a transformer is a matrix, and every matrix is one of these dimension-changing linear maps — chained together and interleaved with nonlinearities.

</Summary>
