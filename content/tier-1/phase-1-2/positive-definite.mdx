export const symmetricCode = `import numpy as np

# Symmetric matrix: A = A.T
A = np.array([[2.0, 1.0, 0.0],
              [1.0, 3.0, 4.0],
              [0.0, 4.0, 5.0]])

print("A == A.T?", np.allclose(A, A.T))

# Symmetric matrices have REAL eigenvalues and ORTHOGONAL eigenvectors
eigvals, eigvecs = np.linalg.eigh(A)      # eigh is for symmetric/Hermitian
print("eigenvalues (all real):", eigvals)
print("eigenvectors (columns) — orthogonal:")
print(eigvecs)
print("Q.T @ Q =")
print(np.round(eigvecs.T @ eigvecs, 8))   # identity — eigenvectors are orthonormal

# Spectral theorem: A = Q Lambda Q.T
reconstruction = eigvecs @ np.diag(eigvals) @ eigvecs.T
print("A reconstructed from spectral decomposition:", np.allclose(A, reconstruction))
`;

<Motivation title="Positive-definite matrices — the 'well-behaved' matrices ML loves">

Three of the most common matrices in ML share a lucky property. They're all **symmetric** and **positive-definite**. Vocabulary first:

- **Symmetric** = the (i, j) entry equals the (j, i) entry. Or in one symbol: A = Aᵀ (A equals its transpose).
- **Positive-definite (PD)** = the quadratic form xᵀAx is strictly positive for every nonzero x. In plainer language: applying A to any vector never sends it "backwards" past 90°.
- **Positive semi-definite (PSD)** = same but with ≥ instead of > (some directions can flatten to zero).

Where do these show up?

- **Covariance matrices** in Gaussian models. Always symmetric and (typically) PD.
- **Kernel Gram matrices** in kernel methods and Gaussian processes (a Bayesian regression tool).
- **Hessians** (second-derivative matrices) at local minima of well-behaved loss functions.

When a matrix is symmetric-PSD, three amazing things become available:

1. **Real eigenvalues, orthogonal eigenvectors** — a result called the **spectral theorem**. It means the matrix breaks into a rotation, a stretch, and an unrotation.
2. **Cholesky factorization** — the fast, stable factorization for solving A·x = b. Roughly twice as fast as generic methods and never numerically unstable.
3. **The quadratic form xᵀAx ≥ 0 for all x** — this is exactly what makes optimization problems **convex** (bowl-shaped, one clear minimum).

Understanding when a matrix is PSD, why it matters, and how to exploit its structure is a huge chunk of what makes practical ML fast and numerically robust.

</Motivation>

<Intuition title="Symmetric = A equals its own transpose; PD = never rotates any vector past 90°">

A **symmetric** matrix satisfies A = Aᵀ — the (i, j) entry equals the (j, i) entry. Symmetric matrices show up whenever the underlying quantity is naturally *pairwise-defined*: covariance between two features doesn't care about order; distance between two points doesn't care about order; the second derivative of a smooth function with respect to variables x_i, x_j equals the mixed partial in either order (Hessian is always symmetric for C² functions).

A **positive-definite (PD)** symmetric matrix additionally satisfies `xᵀAx &gt; 0` for every nonzero x. Geometrically: for every input vector x, the output A·x lands on the *same side* as x (their dot product is positive) — the transformation never rotates any vector by more than 90°. Its eigenvalues are all strictly positive. Its determinant is positive. It's invertible.

**Positive semi-definite (PSD)** is the same but with ≥ instead of &gt; — allows `xᵀAx = 0` for some nonzero x. Eigenvalues are ≥ 0, potentially some zeros. This is the correct condition for covariance matrices (which can be singular if features are linearly dependent).

The reason all this matters: PD matrices define *convex* quadratic bowls (unique minimum, easy to optimize), while indefinite matrices define saddles (hard to optimize). Whether your loss's Hessian is PD, indefinite, or PSD-singular is the difference between "gradient descent will converge cleanly," "gradient descent might climb a saddle," and "gradient descent has a whole valley of equally-good minima."

</Intuition>

<Recall term="Symmetric matrix">

A square matrix A is symmetric if A = Aᵀ, i.e., A(i, j) = A(j, i) for all i, j. Symmetric matrices always have real eigenvalues and mutually orthogonal eigenvectors, and the spectral theorem gives A = QΛQᵀ with Q orthogonal (columns are the eigenvectors) and Λ diagonal (eigenvalues).

</Recall>

## Symmetric matrices — the spectral theorem

The **spectral theorem** for symmetric matrices is short and load-bearing: *every real symmetric matrix A can be diagonalized as*

$$
A \;=\; Q\, \Lambda\, Q^{\top}
$$

where Q is orthogonal (Qᵀ Q = I, columns are unit-length and mutually perpendicular) and Λ is diagonal (the eigenvalues, all real). Two facts fall out immediately:

- Eigenvalues of a symmetric matrix are always **real** (proof: standard, one page, using the fact that xᵀ A x is real).
- Eigenvectors for distinct eigenvalues are always **orthogonal** (proof: from λ_i xᵀ_i x_j = xᵀ_i A x_j = λ_j xᵀ_i x_j).

Geometrically A is a "rotate to the eigenvector basis, scale each axis by its eigenvalue, rotate back." Every symmetric transformation is just that — no shear, no complex-plane twists, no funny business. The three most useful matrices in ML (covariance, kernel, Hessian) all get this decomposition for free.

<RunnableCode title="Spectral theorem — eigenvalues real, eigenvectors orthogonal" code={symmetricCode} packages={["numpy"]} />

**In code**: for a symmetric matrix always use `np.linalg.eigh` (not `eig`). eigh exploits the symmetry to be faster and — much more importantly — guaranteed to return real eigenvalues in sorted order and truly orthonormal eigenvectors. `eig` on a symmetric matrix can return complex-valued numbers with tiny imaginary parts (rounding artifacts) and slightly non-orthogonal vectors, which will silently break any code that assumes them.

## Positive definite — three equivalent definitions

A symmetric matrix A is **positive definite** if any (and hence all) of:

1. `xᵀ A x &gt; 0` for every nonzero x ∈ ℝⁿ. (The defining condition.)
2. Every eigenvalue λ_i of A is strictly positive.
3. There exists a lower-triangular L with strictly positive diagonal such that A = L·Lᵀ. (Cholesky factorization exists.)

Any one implies the other two — a Rosetta stone with three faces. In practice you use whichever is easiest for the check at hand: for a small numeric matrix, compute eigenvalues; for a symbolic proof, use the quadratic form; to actually solve `Ax = b`, factor L·Lᵀ.

**Positive semi-definite** is the same list with ≥ 0 instead of &gt; 0 and Cholesky-with-permutations. PSD matrices can be singular; PD matrices are strictly invertible.

**Indefinite** matrices have both positive and negative eigenvalues — `xᵀ A x` is positive for some x, negative for others. Their quadratic form is a *saddle* — not a bowl.

## The quadratic form and its level sets

The function `q(x) = xᵀ A x` is called a **quadratic form**. It's the natural generalization of `f(x) = a x²` to n dimensions with an n×n matrix A. Its level sets — the shapes of `q(x) = c` — reveal the character of A:

- **Positive definite** ⇒ level sets are ellipsoids centered at the origin. The eigenvectors are the axes of the ellipsoid; the reciprocals of eigenvalues are the axis half-lengths (for the unit level set). This is the *convex bowl* — optimization loves this shape.
- **Positive semi-definite (singular)** ⇒ level sets are degenerate ellipsoids: infinite cylinders in the direction of the zero-eigenvalue eigenvectors.
- **Indefinite** ⇒ level sets are hyperboloids (saddles). No unique minimizer in ℝⁿ; the origin is a saddle point.

The picture on the left (PD) is the shape gradient descent is trying to descend to the center of. Change of basis to the eigenvector coordinates turns the ellipsoid into perfect axis-aligned axes; that's what Newton's method effectively does (it multiplies gradients by H⁻¹ to precondition the geometry).

## Cholesky — the fast, stable factorization for PD matrices

For a positive-definite A there is a unique **Cholesky factorization** A = L·Lᵀ with L lower-triangular and positive diagonal. Once you have L, everything is easy and fast:

- **Solve A·x = b**: forward-solve L·y = b, then back-solve Lᵀ·x = y. Two O(n²) back-solves — much cheaper than LU's O(n³) each time.
- **Compute det(A)**: det(A) = det(L) · det(Lᵀ) = (product of L's diagonal)². So log-det = 2 · sum of log(L_ii). Numerically stable, no risk of overflow.
- **Sample from a Gaussian N(0, A)**: pick z ~ N(0, I), return L·z. Then E[(Lz)(Lz)ᵀ] = L·I·Lᵀ = A. This is *the* standard way to sample from a correlated multivariate normal.

Cholesky is about **2× faster than LU** for the same-size matrix (it exploits symmetry to store and compute half the entries) and is **always numerically stable** (no pivoting needed — a happy consequence of PD). If a matrix is truly PD, always factor it with Cholesky. If Cholesky *fails* with a "matrix is not positive definite" error, that's actually a useful signal: your matrix has drifted into PSD-singular or indefinite territory (usually due to numerical noise; regularize by adding `ε · I` for tiny ε).

## Where PSD lurks in ML — the three main appearances

**Covariance matrices** — for any dataset X (n samples, d features), the sample covariance `Σ = (1/(n−1)) · (X − mean)ᵀ · (X − mean)` is symmetric PSD by construction (it's a sum of outer products, and each outer product is PSD). Diagonal entries are variances; off-diagonals are covariances. In Gaussian models the covariance defines the shape of the distribution (its level sets are ellipsoids — of course they are, that's the quadratic form of Σ⁻¹).

**Kernel Gram matrices** — for a positive-definite kernel k(x, y) (like Gaussian RBF or dot-product) and data points x_1, …, x_n, the n×n matrix K with K(i, j) = k(x_i, x_j) is symmetric PSD by construction. Kernel methods (SVMs, Gaussian process regression) all reduce to linear algebra with K, and PSD is what makes the underlying optimization convex.

**Hessians at local minima** — the Hessian H(x) is the matrix of second partial derivatives of a scalar function. It's symmetric (Clairaut's theorem — mixed partials commute for C² functions). At a *local minimum* of a smooth function, H is PSD; at a *strict local minimum* (isolated), H is PD. This is why second-order optimization methods (Newton, natural gradient) can precondition gradients using H⁻¹ to speed convergence — but only in regions where H is PD.

<RealWorldML title="Gaussian Process regression — where every step is PD linear algebra">

A Gaussian Process (GP) is a distribution over functions defined by a kernel function k(x, x'). For regression on n training points X = (x_1, …, x_n) with outputs y = (y_1, …, y_n), the posterior mean and variance at a new point x★ are:

`mean(x★) = k★ᵀ · (K + σ² I)⁻¹ · y`
`var(x★)  = k(x★, x★) − k★ᵀ · (K + σ² I)⁻¹ · k★`

where K is the n×n training-kernel matrix (K(i, j) = k(x_i, x_j)), k★ is the n-vector of kernel values between x★ and each training point, and σ² is the observation noise variance.

Both formulas require solving a linear system with `(K + σ² I)` — a symmetric PD matrix (K is PSD by construction, and adding σ² I ensures strict positive-definiteness). In real GP implementations:

1. **Factor once**: `L = cholesky(K + σ² I)`. O(n³) cost, done at training time.
2. **Compute α = solve_triangular(L.T, solve_triangular(L, y))**. Once per training pass; α is the "dual coefficients." O(n²).
3. **At prediction time**: `mean(x★) = k★.T @ α` — one dot product, O(n).
4. **For variance**: `v = solve_triangular(L, k★)`, then `var(x★) = k(x★, x★) − v.T @ v`. One O(n²) triangular solve per prediction.

Every step exploits the PD structure: Cholesky (only exists because K + σ²I is PD), triangular solves (only cheap because we have the Cholesky factor), and log-marginal-likelihood evaluation (uses log-det, computed as 2 · sum(log(diag(L))) — accurate to machine precision).

If you tried the same math with `np.linalg.inv(K + σ² I)` you'd be slower by ~2x, numerically less accurate (accumulated rounding error from forming the inverse), and you'd throw away the beautiful triangular structure of L. This is why every serious GP library (GPy, GPflow, GPyTorch, sklearn's GaussianProcessRegressor) is built around Cholesky, and why "the kernel is symmetric PSD" is a non-negotiable structural requirement.

The same story plays out in Bayesian linear regression (posterior covariance is PSD, factor with Cholesky), kernel ridge regression, Kalman filtering (state covariance is PSD, updated with joseph form for numerical stability), and many other ML methods. Once you see "symmetric PSD" as the license to use Cholesky (and everything cheap that follows from it), you'll recognize the pattern in every algorithm that reasons about correlations or second-order structure.

</RealWorldML>

<Summary>

- **Symmetric matrix**: A = Aᵀ. Always has real eigenvalues and orthogonal eigenvectors. Spectral theorem: A = Q·Λ·Qᵀ with Q orthogonal, Λ diagonal.
- **Positive definite (PD)**: symmetric AND `xᵀAx &gt; 0` for every nonzero x ⇔ every eigenvalue positive ⇔ Cholesky A = L·Lᵀ exists. Three equivalent characterizations, use whichever is easiest.
- **Positive semi-definite (PSD)**: same with ≥ 0. Eigenvalues can be zero (singular case). Covariance matrices are always PSD by construction.
- **Quadratic form xᵀAx**: PD gives an ellipsoidal bowl (convex, unique minimum), indefinite gives a saddle, PSD-singular gives a degenerate cylinder.
- **Cholesky**: A = L·Lᵀ with L lower-triangular and positive diagonal. Only exists for PD matrices. ~2× faster than LU and always numerically stable. Use for solving Ax = b, computing log-det, and sampling from Gaussians.
- In code: `np.linalg.eigh` (not eig) for symmetric matrices; `np.linalg.cholesky` for PD; if Cholesky fails, add `εI` for a tiny ε to regularize.
- ML appearances: **covariance matrices** in Gaussian models, **kernel Gram matrices** in kernel methods and Gaussian processes, **Hessians** at local minima of smooth losses — each gets Cholesky-powered fast paths.

</Summary>
