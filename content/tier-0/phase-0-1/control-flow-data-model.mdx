export const protocolTraceCode = `# Every 'for' loop is really this:
def manual_for(iterable, body):
    it = iter(iterable)          # 1. get an iterator by calling __iter__
    while True:
        try:
            item = next(it)      # 2. pull the next item via __next__
        except StopIteration:    # 3. loop exits cleanly on this signal
            break
        body(item)               # 4. run the body

xs = [10, 20, 30]
manual_for(xs, print)            # exactly the same effect as: for x in xs: print(x)

# Not just lists — strings, dicts, files, generators, sets, ranges all obey the same protocol
manual_for("abc", print)
manual_for({"a": 1, "b": 2}, print)      # iterates keys by default`;

export const customIterCode = `# A custom iterable — you implement one protocol, Python treats it as first-class
class Fibonacci:
    """Yields the first n Fibonacci numbers."""
    def __init__(self, n):
        self.n = n

    def __iter__(self):
        a, b = 0, 1
        for _ in range(self.n):
            yield a          # 'yield' inside __iter__ makes a fresh iterator each call
            a, b = b, a + b

fib = Fibonacci(8)

# Works with every builtin that consumes iterables — no magic, just protocol conformance
print(list(fib))                 # -> [0, 1, 1, 2, 3, 5, 8, 13]
print(sum(fib))                  # -> 33
print(max(Fibonacci(20)))        # -> 4181
print(", ".join(str(x) for x in Fibonacci(6)))

# Even destructuring works
a, b, c, *rest = Fibonacci(6)
print("first three:", a, b, c, "  rest:", rest)`;

export const streamMLCode = `# Streaming an ML dataset — the iteration protocol at work
# In real ML this is exactly how HuggingFace 'datasets' and PyTorch 'IterableDataset' behave
import random

def synthetic_stream(n_samples=1_000_000, seed=0):
    """Pretend infinite dataset — generates (x, y) pairs lazily.
    Notice: no 1M rows in memory. Just a generator with state."""
    rng = random.Random(seed)
    for i in range(n_samples):
        x = rng.gauss(0, 1)
        y = 3 * x + 2 + rng.gauss(0, 0.1)   # linear relationship + noise
        yield x, y

def running_mean(stream, take):
    """Iterate the stream once, compute mean of y in a rolling fashion."""
    s = 0.0
    n = 0
    for i, (x, y) in enumerate(stream):
        if i >= take: break
        s += y
        n += 1
    return s / n

# We can pull 1000 samples out of a "1M-sample" dataset with no upfront cost
print(f"mean of first 1000 y-values : {running_mean(synthetic_stream(), 1000):.3f}")
# Or the first 10,000. Nothing on disk, nothing preallocated in RAM.
print(f"mean of first 10000 y-values: {running_mean(synthetic_stream(), 10000):.3f}")`;

<Motivation title="Why 'for x in xs' is one of the deepest ideas in Python">

Every deep-learning training loop you will ever write is `for batch in loader:`. Every dataset you stream from HuggingFace, every file you read line-by-line, every DataFrame you walk — the same three characters: `for`. In Python, `for x in xs` is not a statement about lists. It's an *ask*: Python asks `xs`, "do you know how to be looped over? If yes, hand me an iterator." Any object that answers — list, dict, file, generator, PyTorch `DataLoader`, your own class — is a first-class citizen of the loop.

That one design choice (behavior via small contracts called **protocols**, not via special-cased types) is the whole "data model" of Python. Learn it once and you can read half the standard library — and most of PyTorch, pandas, and scikit-learn — at a glance.

</Motivation>

<Intuition title="Syntax is a phone call to a dunder method">

**Everything is an object** — ints, strings, functions, classes, modules, even types themselves. All can be assigned, passed, stored, compared. That's why `sorted(xs, key=abs)` works: `abs` is just a value.

And **every piece of syntax delegates to a method**. `len(xs)` calls `xs.__len__()`. `a + b` calls `a.__add__(b)`. `model(x)` calls `model.__call__(x)` — that's literally how PyTorch's `nn.Module` makes `model(batch)` run a forward pass. `for x in xs` calls `iter(xs)` → `xs.__iter__()`, then `next(...)` until `StopIteration`. These double-underscore methods are called **dunders** (`__iter__`, `__eq__`, `__hash__`, ...). Implement the dunder and the syntax works on your class instantly — no inheritance, no registration.

Four sentences to internalize: *everything is an object; behavior is negotiated through dunder protocols; iteration is `__iter__` → `__next__` → `StopIteration`; empty is false, and equality is what `__eq__` says.* Everything below is a consequence.

</Intuition>

## Truthiness — empty is false

<RunnableCode title="what counts as True in Python?" code={truthyCode} />

**Every empty container is falsy. Every non-empty container is truthy.** Falsy values: `None`, `False`, `0`, `0.0`, `""`, `[]`, `()`, `{}`, `set()`. This uniformity hands you one tool that works everywhere:

- `if not results:` works whether `results` is a list, dict, set, or string.
- `while queue:` runs while there's anything in the queue and stops naturally when it drains.

**The pitfall that matters:** when you specifically mean *None*, not *empty*, write `if x is None:`. A function that returns `[]` for "nothing matched" and `None` for "I don't know" carries two different messages — `if not x:` collapses them. This is one of the most common bugs in early ML code: valid empty results silently treated as missing data.

## Iterate the Python way

<RunnableCode title="enumerate + zip" code={forVsIndex} />

`for i in range(len(xs))` followed by `xs[i]` is a C/Java reflex to unlearn. The vocabulary:

- **Just the values:** `for x in xs:`
- **Values with index:** `for i, x in enumerate(xs):`
- **Two lists in lockstep:** `for x, y in zip(xs, ys):` — stops at the shortest input; `zip(..., strict=True)` raises if lengths differ.
- **Sorted / reversed:** `for x in sorted(xs):`, `reversed(xs)`.
- **Dicts:** `for key in d`, `for v in d.values()`, `for k, v in d.items()`. Dict iteration order is **insertion order** — a language guarantee since 3.7, safe to rely on.

Every helper takes *any* iterable — list, generator, DataFrame column, dataset — because they all ride on the same protocol.

## The iteration protocol — what `for` really does

<RunnableCode title="a manually-implemented for loop" code={protocolTraceCode} />

`manual_for` is exactly what Python's `for` compiles to: `iter(xs)` gets an iterator (via `__iter__`), `next(it)` pulls items (via `__next__`), and the `StopIteration` exception — not an error, the official "we're done" signal — ends the loop. Lists, strings, dicts, files, generators: all identical under the hood. When a loop misbehaves, you now know the only two places to look: `__iter__` returned the wrong thing, or `__next__` isn't raising `StopIteration` at the right time.

<Recall term="Iterable vs iterator">

An **iterable** answers `iter()` — list, dict, str, range, or your class with `__iter__`. An **iterator** is what `iter()` returns: it holds a cursor and yields items on each `next()`, raising `StopIteration` when done. **Iterables can usually be looped many times** (each `iter()` makes a fresh iterator). **Iterators are one-shot** — once exhausted, they stay exhausted.

</Recall>

That one-shot behavior is a classic trap: `pairs = zip(xs, ys); list(pairs); list(pairs)` — the second `list` is **empty**, no error raised. `zip`, `map`, `filter`, and generators all return iterators. If you need the result twice, materialize once: `pairs = list(zip(xs, ys))`.

Making your own class iterable takes one method:

<RunnableCode title="a Fibonacci iterable — first-class citizen of the language" code={customIterCode} />

Writing `yield` inside `__iter__` turns it into a generator method — a *fresh* iterator per call, so re-iteration works free, and you never write `__next__` yourself. Now `Fibonacci(8)` works with `list`, `sum`, `max`, `join`, destructuring — every consumer of iterables in the language. That's **duck typing** made concrete: satisfy the contract, get treated as first-class.

## match/case — pattern matching, not switch

<RunnableCode title="destructuring with match/case" code={matchCode} />

Python 3.10's `match/case` is not a switch statement — it's a **destructuring pattern matcher** (the construct that makes Rust and OCaml pleasant). It matches the *shape* of the input and extracts pieces into variables: `case [a, b]:` replaces `isinstance` + length check + unpacking; `case {"status": "ok", "data": [first, *rest]}:` picks apart a JSON response in one line. Guards add constraints (`case x if x > 0:`), `_` matches anything, and first match wins — put specific patterns before general ones.

**The pitfall:** there is no fall-through, and a bare name in a pattern *binds*, never compares. `case DEFAULT:` doesn't check equality against your constant — it matches everything and creates a local variable named `DEFAULT`. Use `case _:` for "anything", or a guard (`case v if v == DEFAULT:`) to compare.

## is vs == — identity vs equality

`a is b` asks "same object in memory?" — a pointer comparison. `a == b` asks "does `a.__eq__(b)` say equal?" — semantic, overridable per class.

<CodeBlock language="python" title="the pit that catches every beginner exactly once" code={`# Numbers with the same value are equal but may not be identical
1 == 1.0        # True (equality: same numeric value)
1 is 1.0        # False (identity: different objects, one int + one float)

# Small ints are cached, so this happens to be True — an implementation detail, don't rely on it
a = 256
b = 256
a is b          # True (CPython caches small ints)

# ... but with a large enough number, the cache stops applying
a = 10_000
b = 10_000
a is b          # False in a fresh interpreter — cached separately per compilation

# None, True, False are singletons — checking identity is the right move
if x is None: ...       # correct
if x == None: ...       # works, but a linter will yell — you meant identity`} />

**The rule:** `is` only for the singletons `None`, `True`, `False`; `==` for everything else. `if x is 3:` can work for years on int-caching luck, then break. One style note: `if x is not None:` is the idiomatic negation — `is not` is a single operator; don't write `if not x is None:`.

## Hashability — the price of being a dict key

Only *hashable* objects can be dict keys or set members — roughly, objects whose value never changes. If a key mutated, its hash would change and the dict would lose it in the wrong bucket. Strings, numbers, tuples-of-hashables, frozensets: hashable. Lists, dicts, sets: **not** — they're mutable.

<CodeBlock language="python" title="hashable = usable as a key" code={`d = {}
d[(1, 2)] = "a tuple key works"            # OK, tuples are hashable
d["hello"] = "strings are hashable"         # OK

d[[1, 2]] = "won't work"                   # TypeError: unhashable type: 'list'
d[{1, 2}] = "also won't work"              # TypeError: unhashable type: 'set'
d[frozenset({1, 2})] = "frozen set is OK"  # frozensets are hashable`} />

**The consistency contract:** if `a == b`, then `hash(a) == hash(b)` must hold. Override `__eq__` on a class and Python auto-sets `__hash__` to `None` — hashing raises `TypeError` until you supply a matching `__hash__` (usually `hash((self.x, self.y))`). That's the language protecting you from sets that silently hold "duplicates". `@dataclass(frozen=True)` derives both correctly for you.

<RealWorldML title="Iteration protocols in a real ML data pipeline">

The single most important use of custom iterables in ML is **streaming datasets** — feeding a model one batch at a time without loading the full dataset into memory. Train on data bigger than RAM, stream from URLs, augment on the fly. PyTorch's `IterableDataset`, TensorFlow's `tf.data`, HuggingFace's streaming mode — all built on this exact protocol.

<RunnableCode title="a streaming dataset — no memory, no preallocation" code={streamMLCode} />

**Line by line:**

1. `synthetic_stream` is a *generator function* — the `yield` inside makes calling it return a generator (an iterator) instead of running the body. Nothing is computed until something pulls.
2. Each `__next__` runs the body to the next `yield`, returns that value, and pauses. State (`i`, `rng`) survives between yields — a coroutine in disguise, and the reason streaming works at all.
3. `running_mean` iterates with a plain `for`. It doesn't know or care that a generator feeds it — it's just some iterable.
4. Because the generator is lazy, "1M samples" never exist simultaneously. One sample is alive at any moment.

Substitute a real file and you have a fully functional PyTorch dataset:

```python
class MyDataset(torch.utils.data.IterableDataset):
    def __iter__(self):
        for line in open("giant_dataset.jsonl"):
            record = json.loads(line)
            yield preprocess(record["input"]), record["label"]
```

Wrap it in a `DataLoader`, write `for batch in loader:`, and the same six-line protocol runs the show. **Every deep-learning training loop is fundamentally a `for` over `iter()` and `next()`. The rest is bookkeeping.**

</RealWorldML>
