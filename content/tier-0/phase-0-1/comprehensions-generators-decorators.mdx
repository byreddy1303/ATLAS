# ---------------------------------------------------------

squares_list = [x*x for x in range(6)]

# 2. generator expression — build values LAZILY, one at a time

export const decoratorTraceCode = `# A decorator is just a function that takes a function and returns another
# ---------------------------------------------------------------

from functools import wraps
import time

def timed(fn):
    """Wrap fn so every call prints its wall-clock time."""
    @wraps(fn)                                   # preserve name, docstring
    def wrapper(*args, **kwargs):
        t0 = time.perf_counter()
        out = fn(*args, **kwargs)
        dt = time.perf_counter() - t0
        print(f"{fn.__name__} took {dt*1e3:.2f}ms")
        return out
    return wrapper

# The @timed line is syntactic sugar for:  slow_sum = timed(slow_sum)
@timed
def slow_sum(n):
    return sum(i*i for i in range(n))

slow_sum(100_000)
slow_sum(500_000)

# Introspection still works because we used @wraps
print("name :", slow_sum.__name__)               # 'slow_sum', not 'wrapper'
print("doc  :", slow_sum.__doc__)                # if slow_sum had a docstring, it survives`;

export const decoratorFactoryCode = `# A decorator WITH ARGUMENTS is a decorator that returns a decorator
# --------------------------------------------------------------

from functools import wraps
import time

def retry(times=3, wait=0.0):
    """Retry a function up to 'times' times if it raises."""
    def decorator(fn):
        @wraps(fn)
        def wrapper(*args, **kwargs):
            last_exc = None
            for attempt in range(times):
                try:
                    return fn(*args, **kwargs)
                except Exception as e:
                    last_exc = e
                    print(f"  attempt {attempt+1} raised {type(e).__name__}, retrying...")
                    time.sleep(wait)
            raise last_exc
        return wrapper
    return decorator

# @retry(times=3) means: decorator = retry(3); flaky = decorator(flaky)
_calls = {"n": 0}

@retry(times=3)
def flaky():
    _calls["n"] += 1
    if _calls["n"] < 3:
        raise ValueError("temporary failure")
    return "success on attempt " + str(_calls["n"])

print(flaky())`;

export const generatorPipelineCode = `# Generators compose into pipelines — memory-cheap, expressive, fast
# --------------------------------------------------------------

import random, itertools
random.seed(0)

def synthetic_events(n):
    """Simulate a stream of user events."""
    for _ in range(n):
        yield {
            "user": random.choice(["a", "b", "c", "d"]),
            "amount": random.expovariate(0.01),      # dollar value
            "kind": random.choice(["click", "buy", "view"]),
        }

def buys_only(events):
    for e in events:
        if e["kind"] == "buy":
            yield e

def with_tax(events, rate=0.09):
    for e in events:
        e["tax"] = e["amount"] * rate
        yield e

# The whole thing is lazy — nothing runs until we consume
pipeline = with_tax(buys_only(synthetic_events(100_000)))

# Consume just the first 5 buys — the other 99,995 events are never generated
for e in itertools.islice(pipeline, 5):
    print(e)`;

export const memoizeCode = `# The most common decorator in Python — cache-by-args
# --------------------------------------------------------------

from functools import wraps

def memoize(fn):
    """Cache fn's return value by its positional args."""
    cache = {}
    @wraps(fn)
    def wrapper(*args):
        if args not in cache:
            cache[args] = fn(*args)
        return cache[args]
    return wrapper

@memoize
def fib(n):
    return n if n < 2 else fib(n-1) + fib(n-2)

# Without memoization, fib(35) is exponential and takes seconds.
# With memoization, each fib(k) is computed once. fib(35) is instant.
print("fib(35) =", fib(35))
print("fib(60) =", fib(60))  # would take forever without cache`;

<Motivation title="Why Python isn't just 'C with indentation'">

Three moves separate scripting Python from **Pythonic** Python: **comprehensions** turn multi-line loops into single expressions, **generators** turn those loops into memory-cheap streams that never materialize their full output, and **decorators** turn common wrappers (timing, caching, retries, registration) into one-line annotations.

These three are the load-bearing idioms of modern ML infrastructure. Every PyTorch `DataLoader` is a generator underneath. Every FastAPI `@app.post("/predict")` route, every `@torch.no_grad()` inference block, every `@dataclass` config, every `@pytest.mark.parametrize` — decorators. Every `[x for x in batch if valid(x)]` in a training script — a comprehension. Skip this chapter and you can't read PyTorch, FastAPI, or Hugging Face fluently; land it and their internals stop looking like ceremony.

</Motivation>

<Intuition title="What each move actually does">

- A **list comprehension** `[x*x for x in xs]` is a `for`-loop that builds a list, written as an expression. Same effect as `out = []; for x in xs: out.append(x*x)` — less noise, and slightly faster (dedicated bytecode).
- A **generator** is that loop but *paused between items* — it hands you one value on demand, keeps its state, and holds nothing else. Perfect for streams, pipelines, infinite sequences, and datasets bigger than RAM.
- A **decorator** `@timed` wraps a function without touching its source: a function that takes a function in, returns a modified function out. Perfect for cross-cutting concerns that would otherwise be sprinkled through every function body.

All three are declarative: you say *what* should happen, Python handles the plumbing. That's what "Pythonic" points at.

</Intuition>

<Recall term="Iterable, iterator, generator">

An **iterable** is anything you can loop over (list, dict, str, file, DataLoader, your custom class). An **iterator** is the stateful cursor produced by `iter(iterable)`; it yields the next item on each `next()` call and raises `StopIteration` when done. A **generator** is a specific *convenient way to write an iterator* — a function with `yield`. Every generator is an iterator, and every iterator is an iterable (iterators return themselves from `__iter__`).

</Recall>

## Comprehensions — the loop as an expression

<RunnableCode title="list / dict / set comprehensions + generator expression" code={compsCode} />

Four sibling shapes — read the bracket as the container being built:

- `[expr for x in xs]` — **list comprehension**, eager list.
- `{expr for x in xs}` — **set comprehension**, uniqueness for free.
- `{k: v for x in xs}` — **dict comprehension**.
- `(expr for x in xs)` — **generator expression**, lazy, one-shot.

Comprehensions build **eagerly**: for 10M rows a list comprehension happily allocates a gigabyte. Reach for the generator expression — same syntax, different brackets — whenever the whole result doesn't need to live at once.

## Filter, map, nest — the full grammar

<RunnableCode title="if clauses, multiple for clauses" code={filterMapCode} />

A comprehension has three parts: the **output expression** on the left, one or more **for clauses**, and optional **if clauses**. Multiple `for`s run as nested loops, left-to-right = outer-to-inner. Two more grammar facts worth knowing: the loop variable does **not** leak into the enclosing scope, and Python 3.8+ allows the walrus `[y for x in xs if (y := expensive(x)) > 0]` to compute a value once and use it in both filter and output.

**The rule of thumb:** if you can't read a comprehension aloud in one breath, write a real `for` loop. Two `for`s and one `if` is the practical limit; beyond that is write-only code.

## Generators — where laziness saves you

<RunnableCode title="lazy memory usage, islice, infinite generator" code={memoryCode} />

The dramatic number: a list of 10 million squares occupies hundreds of megabytes; a generator producing the *same virtual sequence* occupies ~100 bytes regardless of "length". The list *materializes* answers; the generator *promises* them. If the consumer only looks at the first 100 items, the other 9,999,900 were never computed. This is why `open(path)` iterates line-by-line, why HuggingFace `datasets` has `streaming=True`, why PyTorch `IterableDataset` exists.

**The trade — generators are one-shot.** `g = (x*x for x in range(3)); list(g); list(g)` gives `[0, 1, 4]` then `[]`, silently. Need the values twice? Materialize once with `list(g)`, or wrap in a class whose `__iter__` returns a fresh generator per call.

## Generator pipelines — compose without buffers

<RunnableCode title="a lazy pipeline: source → filter → transform → take-5" code={generatorPipelineCode} />

In a list-based world, each stage would build a full intermediate list — 100k events → 20k buys → 20k taxed — before you saw anything. In the generator world, **each stage pulls its input on demand**: `islice` asks `with_tax` for one item, `with_tax` asks `buys_only`, `buys_only` keeps pulling from `synthetic_events` until a `"buy"` appears, then the value flows back down the chain. Total events actually generated: only enough to produce 5 buys — a few dozen. The other ~99,970 never existed.

This is **pull-based dataflow** — the reason Unix pipes work and the shape of every real ML data-loading chain. In Python it's just `yield`.

## Decorators — the wrap and the sugar

<RunnableCode title="@timed — a decorator built from scratch" code={decoratorTraceCode} />

De-sugared, `@timed` above a `def` is exactly `slow_sum = timed(slow_sum)`: Python creates your function, passes it to `timed`, and rebinds the name to whatever `timed` returns. The original function survives only inside `wrapper` — as the captured variable `fn`. That capture is a **closure**: a nested function that remembers variables from its enclosing scope even after that scope has returned. Closures are how decorators carry state (a timer, a cache) without globals. And this all works because functions are first-class objects — passing a function to a function is no different from passing a list.

Two rules that matter:

- **Always use `functools.wraps`.** It copies `__name__`, `__doc__`, and the signature onto the wrapper — without it, tracebacks report `wrapper`, `help()` lies, and framework introspection (FastAPI reads signatures!) breaks.
- **Decorators run at *definition* time**, not call time. The `@decorator` line executes when the `def` is evaluated — so a decorator that registers a route or mutates a global registry does it at import. Design accordingly.

## Decorators with arguments — a factory

<RunnableCode title="parameterized decorators: @retry(times=3)" code={decoratorFactoryCode} />

Read `@retry(times=3)` as **two calls**: first `retry(times=3)` runs and returns a decorator; then that decorator wraps your function. `flaky = retry(3)(flaky)`. This is why configurable decorators have "three levels of `def`" — factory, decorator, wrapper. Once seen, every `@app.get("/path")` and `@pytest.mark.parametrize(...)` reads the same way.

Stacked decorators apply **bottom-up at definition, top-down at call**:

```python
@a
@b
def f(): ...
```

is `f = a(b(f))` — `b` wraps first, `a` wraps the result; a call enters `a`'s wrapper, then `b`'s, then `f`. Order matters when decorators have effects (caching-over-timing ≠ timing-over-caching).

<RealWorldML title="A memoized model loader — the flavor of production ML code">

A scenario in every real deployment: an inference endpoint lazily loads a model the first time it's needed, then reuses it on every subsequent request. That "load once, use many" pattern is what caching decorators were made for.

<RunnableCode title="cache your model load, don't pay it every request" code={memoizeCode} />

**Line by line:**

1. `memoize` creates a dict in its closure, keyed by the function's arguments. First call for a given argument set: run the function, store the result. Every later call: return the stored result immediately.
2. Applied to Fibonacci, it turns exponential recursion into linear time — `fib(35)` no longer recomputes `fib(34)`, `fib(33)`, ... billions of times.
3. Now imagine it on `def load_model(name: str)`. First call: gigabytes off disk into memory. Subsequent calls: microseconds. This is exactly how `transformers.AutoModel.from_pretrained` and every serving framework behaves internally.

In production, don't hand-roll it: **`functools.cache`** (unbounded, 3.9+) and **`functools.lru_cache(maxsize=128)`** (evicts least-recently-used) are the stdlib versions with threading and statistics handled. One caveat both share: arguments become dict keys, so they must be hashable — pass `tuple(...)`/`frozenset(...)`, never a list.

</RealWorldML>

<Summary>

- **Comprehension** = a for-loop as an expression: `[x*x for x in xs if x > 0]`. Brackets pick the container (list/set/dict); parentheses make it a lazy generator. Two `for`s + one `if` is the readability limit.
- **Generator** = a paused loop: `yield` hands out one value on demand, keeps its state, holds nothing else. ~100 bytes whether it "contains" 6 items or 10 million. One-shot — materialize with `list()` if you need two passes.
- **Pipelines**: generator stages compose without buffers; the consumer pulls, and only what's demanded ever gets computed. Every ML data loader is this shape.
- **Decorator** = higher-order wrapper: `@timed` is `f = timed(f)`, applied at *definition* time. The original function lives on inside the wrapper's closure.
- **With arguments** = a factory: `@retry(3)` is `f = retry(3)(f)` — two calls. Stacked decorators apply bottom-up, execute top-down.
- **Always `functools.wraps`** — or tracebacks, `help()`, and framework introspection all lie.
- **Don't hand-roll caches**: `functools.cache` / `lru_cache` are the stdlib memoizers; arguments must be hashable.

</Summary>
