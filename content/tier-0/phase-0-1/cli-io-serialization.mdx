export const parquetCode = `# The size + speed demo you can feel — CSV vs JSON vs Parquet
# uses pandas + pyarrow which Pyodide includes
import pandas as pd, numpy as np, io, time

rng = np.random.default_rng(7)
n = 200_000
df = pd.DataFrame({
    "user_id": rng.integers(0, 100_000, n),
    "ts":      pd.date_range("2026-01-01", periods=n, freq="s"),
    "event":   rng.choice(["click", "view", "buy"], n, p=[0.6, 0.35, 0.05]),
    "value":   rng.normal(10, 2.5, n).round(3),
})

def bytes_of(write):
    buf = io.BytesIO()
    t = time.perf_counter(); write(buf); dt = time.perf_counter() - t
    return len(buf.getvalue()), dt

csv_bytes,  csv_t  = bytes_of(lambda b: df.to_csv(b, index=False))
json_bytes, json_t = bytes_of(lambda b: df.to_json(b, orient="records"))
pq_bytes,   pq_t   = bytes_of(lambda b: df.to_parquet(b, engine="pyarrow", compression="snappy"))

def mb(n): return f"{n/1_048_576:.2f} MB"
print(f"CSV     : {mb(csv_bytes):>10s}  write {csv_t*1000:6.0f} ms")
print(f"JSON    : {mb(json_bytes):>10s}  write {json_t*1000:6.0f} ms")
print(f"Parquet : {mb(pq_bytes):>10s}  write {pq_t*1000:6.0f} ms")
print(f"\\nParquet is {csv_bytes/pq_bytes:.1f}x smaller than CSV,")
print(f"and {json_bytes/pq_bytes:.1f}x smaller than JSON — with dtypes preserved.")`;

export const roundtripCode = `# Round-tripping shows what each format preserves and what it drops
import pandas as pd, numpy as np, io

df = pd.DataFrame({
    "int_col":   [1, 2, 3, 4],
    "float_col": [1.0, np.nan, 3.5, 4.7],
    "str_col":   ["a", "b", None, "d"],
    "date_col":  pd.to_datetime(["2026-01-01", "2026-01-02", "2026-01-03", "2026-01-04"]),
    "cat_col":   pd.Categorical(["x", "y", "x", "y"], categories=["x", "y"]),
})
print("--- original ---")
print(df.dtypes.to_string())

def roundtrip_csv(df):
    buf = io.StringIO()
    df.to_csv(buf, index=False)
    buf.seek(0)
    return pd.read_csv(buf)

def roundtrip_parquet(df):
    buf = io.BytesIO()
    df.to_parquet(buf, engine="pyarrow")
    buf.seek(0)
    return pd.read_parquet(buf)

print("\\n--- after CSV round-trip (types lost) ---")
print(roundtrip_csv(df).dtypes.to_string())

print("\\n--- after Parquet round-trip (types preserved) ---")
print(roundtrip_parquet(df).dtypes.to_string())`;

<Motivation title="Where your program meets the outside world">

Every Python program is a little world with a wall around it. Inside the wall, everything is orderly — objects with types, control flow you designed, invariants you can trust. But nothing useful happens purely inside. Real programs read a CSV, take a `--config` flag, write a checkpoint, ship a log to disk, receive JSON from an API, load a Parquet from S3, dump metrics to a warehouse. Every one of those crossings is a **boundary**, and every boundary is where bugs actually live: the file that isn't there, the argument that's a string when you expected an int, the JSON with a missing field, the 60 GB CSV you tried to load into 16 GB of RAM, the encoding mismatch that turned "café" into "cafÃ©".

If the first six chapters of this tier gave you the *inside* of a Python program, this chapter is about the *wall*. And the wall matters more than most beginners realize. **In every serious ML system, more of the code exists to move data across boundaries than to do the actual training.** Data pipelines convert raw CSVs into typed Parquet. Config files describe experiments. Checkpoints serialize model state. Logs ship metrics to observability platforms. Every one of those is a boundary crossing, and every one of them has a right answer.

The right answers are surprisingly small: **pathlib** for names, **argparse** or **Typer** for arguments, **JSON / TOML / Parquet** for serialization, and **`with` blocks + streaming** for reading things bigger than memory. Master these four moves and the outside world stops being scary. Miss them and you spend your career wrestling `os.path.join`, discovering `pip` installed to the wrong Python, and finding out at 2am that your 500 GB training dataset can't be loaded because you saved it as CSV instead of Parquet.

</Motivation>

<Intuition title="Four boundaries, four right-sized tools">

There are exactly four kinds of moves you make across the wall of a Python program, and each has one right-sized tool:

- **Take arguments in from the command line** → `argparse` (stdlib) for scripts you own, `Typer` for CLIs other people will use.
- **Name files and directories** → `pathlib` — always. Never `os.path.join`, never `"a/" + b`.
- **Serialize objects to a durable format** → `json` for interop with anything else, `toml`/`yaml` for human-authored configs, **Parquet** for large tabular data, safetensors for model weights.
- **Read things bigger than memory** → *stream* with an iterator inside a `with` block. Never `.read()` all at once.

Everything else in this chapter is elaboration on these four reflexes. Get these into your fingers and you can build an entire ML pipeline that reads correctly, runs safely, fails loudly, and reproduces exactly. Miss even one and something will silently corrupt at the boundary.

</Intuition>

<Recall term="Context managers (`with`)">

A `with` block guarantees that the resource opened at the top is **closed on the way out** — even if the code inside raises. It's how you avoid file leaks, half-flushed writes, and dangling database connections. Every `open()` in real code should be inside a `with`. Every custom resource you write (a temp directory, a network session, an S3 client, a database transaction) should support `__enter__`/`__exit__` so callers can do the same. Rule of thumb: *if it acquires something, it needs a `with`.*

</Recall>

<Recall term="Text mode vs binary mode">

`open(path, 'r')` (default) opens in **text mode** — the file's bytes are decoded to Python `str` using an encoding (default is platform-dependent; always pass `encoding="utf-8"` explicitly). `open(path, 'rb')` opens in **binary mode** — reads raw `bytes`. Use text mode for anything textual (JSON, YAML, TOML, CSV, source code, logs). Use binary mode for anything else (Parquet, images, model checkpoints, .pt files, compiled artifacts). Mixing them up produces the classic "utf-8 codec can't decode byte 0x80" error, which is Python telling you a binary file was opened as text.

</Recall>

<Recall term="Encoding">

An **encoding** is how a string of characters becomes a sequence of bytes on disk. UTF-8 is the universal standard in 2026 — every modern system reads and writes it correctly. On older Windows machines, the default encoding was cp1252 (Windows-1252), which produces garbled output for any non-ASCII text. **Always pass `encoding="utf-8"` explicitly** in your `open()` calls, `Path.read_text()`, `json.dump()` (via the file's encoding), and CSV writers. This one habit prevents an entire category of "works on my Mac, garbled on their PC" bugs.

</Recall>

<Angles intro={<>Six lenses on the boundary of your program. Each corresponds to a real habit you'll form: pathlib-first, argparse or Typer, JSON as lingua franca, streaming for scale, format-by-audience, and safe-reading defaults.</>}>

  <Angle type="taste" title="pathlib — paths as objects, not strings">
    <RunnableCode title="paths that behave like paths" code={pathlibCode} />

    `Path` gives you four things that string paths cannot: **operators** (`root / "sub" / "file.json"` is safe on every OS), **methods** (`.exists()`, `.stat()`, `.glob()`, `.read_text()`, `.write_text()`), **anatomy** (`.name`, `.stem`, `.suffix`, `.parent`, `.parts`), and **cross-platform correctness** (Windows vs POSIX separators are handled for you).

    The moment you type `os.path.join` in modern Python, stop. Use `Path(...) / ...`. The only exception is when a third-party API insists on a string — then wrap with `str(p)` at the very last step. Even then, most modern libraries (`pandas`, `scikit-learn`, `torch`, `open()`) accept both `str` and `Path` transparently.

    Common one-liners to memorize:

    - `Path("data").mkdir(parents=True, exist_ok=True)` — create the directory (and its parents), don't error if it already exists.
    - `Path(f).glob("**/*.json")` — recursively find all JSON files under `f`.
    - `Path.home() / ".config" / "my-tool" / "config.toml"` — cross-platform way to name a user-config path.
    - `p.with_suffix(".pt")` — return a new Path with the suffix swapped (`checkpoint.pt` from `checkpoint.json`).

  </Angle>

  <Angle type="workflow" title="argparse — the stdlib CLI you'll actually use">
    <RunnableCode title="argparse basics" code={argparseCode} />

    `argparse` parses a list of strings from `sys.argv` into a typed namespace. Four habits separate a professional CLI from a script that surprises its users:

    1. **Positional arguments are required; `--flags` are optional.** That's the convention across every CLI you'll ever meet (`git commit`, `docker run`, `ffmpeg`). Don't invert it.
    2. **Set `type=int` / `type=float` / `type=Path`** so bad input dies at parse time with a helpful message, not deep inside training with a mysterious `TypeError`.
    3. **Use `choices=[...]`** for enumerable options — the parser validates and the auto-generated `--help` lists the valid values.
    4. **Use `--dry-run` and `--verbose` generously.** They cost nothing to add and save many hours of "did I set it right?" per-year.

    For public tools upgrade to **`Typer`** (type hints become the CLI schema — see next angle) or **`Click`** (more mature ecosystem, more custom widgets). For a private script? argparse is exactly the right amount of ceremony.

  </Angle>

  <Angle type="comparative" title="Typer — the type hints ARE the CLI schema">
    <CodeBlock language="python" title="Typer version of the same CLI" code={typerCode} />

    **Typer** takes a fundamentally different angle: it reads your function's type hints and defaults, and *from those alone* builds the CLI schema, help text, tab completion, and error messages. Your Python function is the CLI; there's no separate "argument parser" to maintain.

    Notice what disappears versus the argparse version:

    - No `add_argument` calls. The parameter list is the argument list.
    - No `type=int` / `type=float`. The hints already declared it.
    - No manual `--help` strings. The docstring and hints generate them.
    - Tab completion is automatic (Typer installs the shell integration for you).

    The trade-off: Typer adds a dependency (`uv add typer`), and it's easier to accidentally break the CLI contract by refactoring the function signature. For long-lived tools you ship to others, this trade is worth it. For quick one-file scripts, argparse is faster to reach for.

    **When to use which:** argparse for private scripts and simple tools; Typer for user-facing CLIs and internal tools with many subcommands. Both are excellent; the choice is stylistic, not correctness-affecting.

  </Angle>

  <Angle type="ecosystem" title="JSON — the lingua franca of every system">
    <RunnableCode title="json.dump / json.load — round-trip and its edges" code={jsonCode} />

    JSON is the format every language speaks. Every language has a native JSON parser; every HTTP API produces or consumes it; every logging platform ingests it. Its type system is minimal by design — **string, number, bool, null, array, object** — which is also its constraint: anything richer needs conversion at the boundary or JSON will raise.

    Three things always to remember:

    1. **`None` becomes `null`**, then comes back as `None`. The round-trip is faithful only for JSON's native vocabulary.
    2. **JSON has no datetime, no Decimal, no set, no bytes.** These must be converted to strings (or arrays) before writing and parsed back on the way in. Common convention: ISO 8601 for datetimes (`"2026-07-12T13:15:00Z"`); base64 strings for bytes.
    3. **JSON is unordered by spec**, but Python's parsers preserve insertion order in practice (dict since 3.7). Never rely on order across languages; if order matters, use an array of pairs.

    Use `indent=2` when a human will read the JSON (configs, small debug outputs); no `indent` when only a machine will (logs, wire formats). Prefer `json.dumps(obj, sort_keys=True)` for anything you want to diff between runs.

  </Angle>

  <Angle type="performance" title="Streaming — reading files bigger than memory">
    <RunnableCode title="iterate a file line by line" code={streamCode} />

    `for line in file:` is not just syntactic sugar. It uses the file's own **iterator protocol** and pulls one line at a time, letting the OS buffer the rest. This is how you scan a 60 GB log on a laptop with 16 GB of RAM. The moment you write `f.read()` or `f.readlines()`, ask yourself: *does this file fit in RAM?* If you're not sure, the answer is **stream**.

    The same idea powers many higher-level tools:

    - **`pd.read_csv(path, chunksize=100_000)`** yields DataFrames of 100k rows at a time.
    - **`csv.reader(open(path))`** yields row tuples.
    - **`ijson`** (third-party) yields JSON values from a huge JSON file without loading it all.
    - **`pyarrow.parquet.ParquetFile(path).iter_batches()`** yields Arrow record batches from a Parquet file.
    - **`torch.utils.data.IterableDataset`** yields training samples from arbitrary sources.

    In every case, the abstraction is the same as the plain file loop: **iterate, don't materialize**. The moment you learn this reflex, files stop having a maximum size.

  </Angle>

  <Angle type="comparative" title="Formats — the trade-off table you'll live by">
    Every format is a bet on **who reads it, how fast, and how big**. This table is worth memorizing:

    | Format  | Human-readable | Schema | Speed  | Size   | Sweet spot |
    | ------- | -------------- | ------ | ------ | ------ | ---------- |
    | JSON    | Yes            | Loose  | OK     | Large  | Web APIs, small configs, logs |
    | YAML    | Yes (comments!)| Loose  | Slow   | Large  | Human configs (Kubernetes, CI, Hydra) |
    | TOML    | Yes            | Strict | Fast   | Small  | `pyproject.toml`, app config |
    | CSV     | Yes            | None   | Slow   | Huge   | Interchange with spreadsheets |
    | Parquet | No             | Strong | Fast   | Tiny   | ML datasets, warehouses |
    | Arrow   | No             | Strong | Fastest| Small  | In-memory or IPC between processes |
    | Pickle  | No             | Python | Fast   | Medium | Model state — *same-Python only* |
    | Safetensors | No         | Strong | Fast   | Small  | Model weights — safe, portable |

    Rules:
    - **Config for humans → TOML or YAML** (comments matter to humans).
    - **API interchange → JSON**.
    - **Tabular ML data → Parquet**.
    - **Model weights → safetensors** (never pickle for weights that will cross machines).

    Never use CSV when you have a choice; it has no types, no null representation, and its byte size is embarrassing. Never use pickle for anything crossing a machine or Python version boundary — it's Python-specific *and* executes arbitrary code on load (a security hole). Use safetensors instead for weights.

  </Angle>

</Angles>

<Convergence>

The rule reads left-to-right: **arguments in → paths → work → serialize out.** `pathlib` names the boundaries. `argparse` (or Typer) crosses them cleanly on the way in. `json`/`toml`/`parquet` cross them on the way out — choosing the right format for the audience. `with` blocks and streaming make sure every crossing closes cleanly even when something fails halfway. Read every ML pipeline this way and you'll find the same skeleton in every framework, from a research paper's repo to a Fortune-500 production stack.

</Convergence>

<Derivation title="What happens when you write `df.to_parquet(path)`">

Parquet feels opaque the first time — bytes go in, bytes come out, everything is faster and smaller. Trace what the format actually does and it stops being magic; it's just a well-designed data layout.

<Step n={1} math={<span>Pandas passes the DataFrame to PyArrow, which infers a schema from the column dtypes.</span>}>
Every Parquet file starts with a schema — a typed description of the columns. `df.dtypes` maps 1:1 to Arrow types: `int64` → INT64, `float64` → DOUBLE, `datetime64[ns]` → TIMESTAMP[NANOSECOND], `object` (strings) → BYTE_ARRAY (UTF8), `category` → DICTIONARY(index_type, value_type). This is why Parquet round-trips types faithfully — the schema is stored in the file.
</Step>

<Step n={2} math={<span>The data is transposed from row-major to column-major layout.</span>}>
CSV and JSON are *row-based*: row 1's values, then row 2's values, then row 3's. Parquet is *column-based*: all of column A, then all of column B, then all of column C. This is the single biggest reason Parquet is fast for analytics — if you only need column A, you skip the bytes for B and C entirely.
</Step>

<Step n={3} math={<span>Each column is split into "row groups" — chunks of typically ~1M rows each.</span>}>
Row groups are the granularity of read/skip. When you filter by `WHERE date > '2026-01-01'`, Parquet's engine reads the min/max metadata for each row group's date column and *skips whole row groups* whose dates are all outside the filter. This "predicate pushdown" is why Parquet can query a 500 GB file on a laptop.
</Step>

<Step n={4} math={<span>Each column of each row group is <em>encoded</em> — dictionary, delta, run-length — before compression.</span>}>
Categorical columns become integer indices into a small dictionary. Sorted timestamps become delta-encoded (store the differences, not the absolute values — much smaller). Long runs of the same value become run-length pairs. These encodings often shrink data 10× before compression even runs.
</Step>

<Step n={5} math={<span>Each encoded column is <em>compressed</em> — Snappy (fast) or Gzip/Zstd (smaller).</span>}>
Snappy is the default because it's fast to decompress; Zstd gives better ratios if size matters more. Either way, the compression is *per-column*, so column-typical patterns (integers close to each other, repeated strings) compress much better than they would in a row-based mix.
</Step>

<Step n={6} math={<span>PyArrow writes the metadata footer — schema, row group offsets, min/max stats.</span>}>
The footer is at the end of the file (an unusual choice, but it lets writers stream data without knowing the total size up front). Readers seek to the footer first, learn the layout, then jump directly to the columns and row groups they need. That's why a 500 GB Parquet file can be opened almost instantly — the reader only touches the footer at first.
</Step>

<Step n={7} math={<span>The whole file is a well-formed Parquet document that any language can read.</span>}>
Parquet is not a Python format — it's an Apache-standard binary format used by Spark, DuckDB, Dask, Rust's polars, PostgreSQL extensions, BigQuery, Redshift, Snowflake, and every other serious analytics tool. Once you write a Parquet, you can query it from any of them. This portability is why ML pipelines standardize on Parquet regardless of what language the pipeline is written in.
</Step>

</Derivation>

<Example title="What does each format actually preserve?">

<RunnableCode title="round-trip a typed DataFrame through CSV and through Parquet" code={roundtripCode} />

Read the output carefully. Before the round-trip, `df` has columns of type `int64`, `float64` (with NaN), string, datetime, and Categorical. After **CSV**, everything is either `int64`, `float64`, or `object` (untyped strings) — the datetime column is a string that pandas has to re-parse, the Categorical is now plain strings, the None became "None". After **Parquet**, every dtype is preserved exactly. `date_col` is still datetime; `cat_col` is still Categorical; nulls are nulls (not "None" strings).

**Why this matters in ML.** If your training data goes through CSV once, you now need to re-detect and re-cast every column type every time you load it. If your data lives in Parquet, `pd.read_parquet(path)` gives you back exactly the DataFrame you saved. Multiply this by 100 pipeline stages and 1000 loads a day, and Parquet-vs-CSV becomes the difference between reproducible ML and constant-firefighting ML.

</Example>

<RealWorldML title="Parquet vs CSV, felt in bytes and milliseconds">

<RunnableCode title="benchmark on 200K rows" code={parquetCode} />

**Line by line, why this is what a real ML pipeline looks like:**

1. We built a small realistic events table — 200,000 rows, four typed columns (int, datetime, categorical string, float). This is the shape of *most* ML input data — user events, sensor readings, transactions, logs.
2. We wrote it three ways in memory using `io.BytesIO` — no disk, so the numbers reflect pure format cost.
3. **CSV** re-encodes every value as text. Every float becomes `"10.428"`, every timestamp becomes a 19-byte ISO string. No types survive the trip — a re-read has to *guess* whether a column is int or string. And CSV has no compression by default.
4. **JSON** is even worse: on top of stringifying, it repeats **every column name in every row** (`{"user_id": 42, ...}` × 200K). That's why it's the fattest of the three.
5. **Parquet** stores each column separately in a compact binary layout (dictionary-encoded categorical, delta-encoded timestamps, snappy-compressed floats). Types are preserved. Result: **often 5–10× smaller and faster to read**, especially when you only need a few columns.

**Why this matters for ML:** almost every training pipeline reads the same dataset thousands of times — once per epoch, once per hyperparameter tune, once per experiment replay. If you save it as CSV, you pay the parse cost *every single time*. Move it to Parquet once and every downstream job — training, EDA, feature engineering, evaluation — reads in a fraction of the time, keeps its dtypes, and uses less disk. **This is the single biggest boring win in data engineering.** Every major ML platform (Weights & Biases, MLflow, Tecton, Databricks) stores data in Parquet or Arrow (Parquet's in-memory cousin) precisely because of this multiplier.

**Bonus for large data:** with a Parquet dataset partitioned by date (`/data/events/date=2026-07-12/*.parquet`), a query "give me events for July" scans only July's partition. On a 500 GB dataset that's minutes, not hours. With CSV, you're scanning the whole thing.

<Nuance>

**Encoding.** Always `open(path, encoding="utf-8")` or `Path.read_text(encoding="utf-8")`. Python's default is platform-dependent (cp1252 on older Windows) and will *silently* mangle non-ASCII text some day when you least expect it. Same for CSV writers — pass `encoding="utf-8"` explicitly.

**Newlines and CSV.** Open CSV files with `newline=""` when using the `csv` module — otherwise Windows double-counts line endings and you get blank rows.

**JSON and NaN.** The JSON spec has no NaN or Infinity. `json.dump` accepts them by default (non-standard) but many parsers will reject them. Pass `allow_nan=False` to catch this at write time, or coerce to `None` first.

**Argument parsing dies loud.** Let argparse die on bad input with a clear message. Don't try/except around `parse_args` — the traceback *is* the UX for a CLI.

**Config libraries.** `pydantic`, `hydra`, and `omegaconf` layer schema + defaults + overrides on top of YAML/JSON. Worth it for large ML projects with many hyperparameters; overkill for a one-file script.

**Big Parquet.** For datasets larger than RAM, use `pyarrow.dataset` or `duckdb` to query them without loading. You can filter by column or row group without ever materializing the full frame. This is how "reading a 500 GB Parquet on a laptop" works in practice.

**Never pickle strangers.** `pickle.load` on untrusted data is equivalent to `eval` — it can execute arbitrary code. Only unpickle files you produced. For model weights specifically, prefer **safetensors** — a Rust-implemented format that stores tensors as pure data with no executable code path.

**Atomic writes.** If you're writing an output file that a downstream consumer might tail, use the "write to temp, then rename" pattern: `path.with_suffix(".tmp").write_text(...); path.with_suffix(".tmp").rename(path)`. A rename is atomic on POSIX — the consumer sees either the old file or the new one, never a half-written mess.

**Compression choices.** Snappy (Parquet's default) is fast to decompress, moderate ratio. Gzip is slower but ~30% smaller. Zstd is the modern default in many pipelines — good ratio and fast. Choose based on read/write frequency: hot data → Snappy; cold archive data → Zstd or Gzip.

**Streaming JSON.** Regular `json.load` reads the whole file. For huge JSONs, use `ijson` (streaming parser) or split your data as JSONL (one JSON object per line) — then plain file iteration works.

**`os.path` still exists** and is fine for legacy code. Every new code path should use `pathlib`. When calling into a library that only accepts strings, wrap: `str(path)`. Most modern libraries accept both.

**TOML in the stdlib.** Python 3.11+ ships `tomllib` for *reading* TOML (`import tomllib; with open(p, 'rb') as f: tomllib.load(f)`). For *writing* TOML you still need `tomli-w` or another package. `pyproject.toml` and app configs are the target audience.

</Nuance>

<LifeLesson title="A program has a wall. Life does too — respect the boundary">

Every clean program in this chapter has one thing in common: it treats its *boundary* with more care than its interior. Inside, code trusts types, invariants, function contracts. At the boundary — where user input, file bytes, network responses, config strings enter — it validates ruthlessly, converts explicitly, fails loudly with helpful messages. That asymmetry — trust-the-inside, verify-at-the-edge — is what makes systems that don't randomly explode at 2am.

The pattern is not unique to code. The strongest personal and organizational systems are structured the same way. **Inside** — your team, your friend group, your home — you can afford deep trust, tight loops, less ceremony. **At the boundary** — where clients, strangers, and unknown situations meet your interior — you have to state expectations clearly, validate inputs (does this new contract make sense? does this person's ask fit our scope?), convert explicit types (a vague "help me with the project" becomes a specific "review the doc by Friday"), and be willing to fail loudly ("sorry, we can't take this on") rather than absorb bad-shape input and hope for the best.

The failure mode in code is: your program silently accepts a malformed JSON, writes garbage to disk, and mysteries appear a week later. The failure mode in life is: you silently accept a scope creep, absorb the extra work, and the resentment or burnout appears a week later. Both are the same bug — treating the boundary as if it were the interior.

**Say `type=Path` at the boundary. Say "utf-8, please" at the boundary. Say `argparse` — parse me, validate me, give me a real object or fail loudly — at the boundary.** Then, inside, trust what's already been converted. Do this consistently and the world stops surprising you at the wall of your program *and* the wall of your life.

</LifeLesson>

<MLConnection>

Serialization and I/O are the invisible spine of every ML system:

- **Datasets.** Raw sensor data → cleaned Parquet is the first move in every clean pipeline. HuggingFace `datasets`, PyTorch DataLoader, Ray Data, TFRecord (Google's equivalent), Arrow — all use column-oriented binary formats for a reason. Parquet is the interchange standard.
- **Configs.** Every training script grows a config file. Small teams start with YAML/TOML. Larger ones use **Hydra** or **OmegaConf** to compose configs from multiple files with CLI overrides. Modern research code often ships with a Hydra config that captures every hyperparameter of the experiment.
- **Checkpoints.** Model weights are serialized. Classic pipelines use `torch.save` (pickle-based); modern pipelines are migrating to **safetensors** — Rust-implemented, no arbitrary-code execution risk, faster to load. Optimizer state, scheduler state, RNG state get saved alongside so training can resume.
- **Logs.** Every experiment writes JSONL logs — one metric event per line — because JSONL streams beautifully and any tool can read it. Weights & Biases, MLflow, TensorBoard, Comet all ingest this shape.
- **APIs.** Every model inference server has a JSON request/response schema. FastAPI + Pydantic is the modern default: your Pydantic model is the request type, the API doc, and the runtime validator, all in one.
- **Big data.** Column-oriented formats (Parquet, Arrow) exist because ML often reads a few columns from a huge table. Column layouts turn a full-table scan into a targeted read. Combined with **partitioning** (`/data/events/date=2026-07-12/`), this is how you query a 500 GB dataset in seconds.
- **Reproducibility.** Alongside `uv.lock` (environment reproducibility), Parquet + Hydra configs give you *dataset reproducibility* and *experiment reproducibility*. Together they let a paper's code + data + config reproduce results down to the decimal.
- **Streaming datasets.** `datasets.load_dataset("...", streaming=True)` yields records without downloading the full dataset. Under the hood: a generator that pulls Parquet row groups on demand, exactly the streaming pattern in this chapter.

Learn the boundary tools once; they show up in every ML system you'll ever touch.
