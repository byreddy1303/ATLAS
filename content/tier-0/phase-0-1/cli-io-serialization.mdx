export const parquetCode = `# The size + speed demo you can feel — CSV vs JSON vs Parquet
# uses pandas + pyarrow which Pyodide includes
import pandas as pd, numpy as np, io, time

rng = np.random.default_rng(7)
n = 200_000
df = pd.DataFrame({
    "user_id": rng.integers(0, 100_000, n),
    "ts":      pd.date_range("2026-01-01", periods=n, freq="s"),
    "event":   rng.choice(["click", "view", "buy"], n, p=[0.6, 0.35, 0.05]),
    "value":   rng.normal(10, 2.5, n).round(3),
})

def bytes_of(write):
    buf = io.BytesIO()
    t = time.perf_counter(); write(buf); dt = time.perf_counter() - t
    return len(buf.getvalue()), dt

csv_bytes,  csv_t  = bytes_of(lambda b: df.to_csv(b, index=False))
json_bytes, json_t = bytes_of(lambda b: df.to_json(b, orient="records"))
pq_bytes,   pq_t   = bytes_of(lambda b: df.to_parquet(b, engine="pyarrow", compression="snappy"))

def mb(n): return f"{n/1_048_576:.2f} MB"
print(f"CSV     : {mb(csv_bytes):>10s}  write {csv_t*1000:6.0f} ms")
print(f"JSON    : {mb(json_bytes):>10s}  write {json_t*1000:6.0f} ms")
print(f"Parquet : {mb(pq_bytes):>10s}  write {pq_t*1000:6.0f} ms")
print(f"\\nParquet is {csv_bytes/pq_bytes:.1f}x smaller than CSV,")
print(f"and {json_bytes/pq_bytes:.1f}x smaller than JSON — with dtypes preserved.")`;

export const roundtripCode = `# Round-tripping shows what each format preserves and what it drops
import pandas as pd, numpy as np, io

df = pd.DataFrame({
    "int_col":   [1, 2, 3, 4],
    "float_col": [1.0, np.nan, 3.5, 4.7],
    "str_col":   ["a", "b", None, "d"],
    "date_col":  pd.to_datetime(["2026-01-01", "2026-01-02", "2026-01-03", "2026-01-04"]),
    "cat_col":   pd.Categorical(["x", "y", "x", "y"], categories=["x", "y"]),
})
print("--- original ---")
print(df.dtypes.to_string())

def roundtrip_csv(df):
    buf = io.StringIO()
    df.to_csv(buf, index=False)
    buf.seek(0)
    return pd.read_csv(buf)

def roundtrip_parquet(df):
    buf = io.BytesIO()
    df.to_parquet(buf, engine="pyarrow")
    buf.seek(0)
    return pd.read_parquet(buf)

print("\\n--- after CSV round-trip (types lost) ---")
print(roundtrip_csv(df).dtypes.to_string())

print("\\n--- after Parquet round-trip (types preserved) ---")
print(roundtrip_parquet(df).dtypes.to_string())`;

<Motivation title="Where your program meets the outside world">

Inside your program everything is orderly — objects with types, invariants you can trust. But nothing useful happens purely inside. Real programs read CSVs, take `--config` flags, write checkpoints, receive JSON from APIs. Every crossing is a **boundary**, and boundaries are where bugs actually live: the file that isn't there, the argument that's a string when you expected an int, the 60 GB CSV you tried to load into 16 GB of RAM, the encoding mismatch that turned "café" into "cafÃ©".

In every serious ML system, more code exists to move data across boundaries than to do the actual training — pipelines convert raw CSVs to typed Parquet, configs describe experiments, checkpoints serialize model state, logs ship metrics. The right answers are surprisingly small: **pathlib** for names, **argparse/Typer** for arguments, **JSON/TOML/Parquet** for serialization, **`with` blocks + streaming** for anything bigger than memory.

</Motivation>

<Intuition title="Four boundaries, four right-sized tools">

There are exactly four kinds of moves you make across the wall of a Python program, and each has one right-sized tool:

- **Take arguments in from the command line** → `argparse` (stdlib) for scripts you own, `Typer` for CLIs other people will use.
- **Name files and directories** → `pathlib` — always. Never `os.path.join`, never `"a/" + b`.
- **Serialize objects to a durable format** → `json` for interop with anything else, `toml`/`yaml` for human-authored configs, **Parquet** for large tabular data, safetensors for model weights.
- **Read things bigger than memory** → *stream* with an iterator inside a `with` block. Never `.read()` all at once.

Everything else in this chapter is elaboration on these four reflexes. Miss even one and something will silently corrupt at the boundary.

</Intuition>

<Recall term="Context managers (`with`)">

A `with` block guarantees that the resource opened at the top is **closed on the way out** — even if the code inside raises. It also ensures written data is flushed before the block exits. Every `open()` in real code lives inside a `with`; every custom resource (temp directory, network session, DB transaction) should support `__enter__`/`__exit__` so callers can do the same. Rule of thumb: *if it acquires something, it needs a `with`.*

</Recall>

## pathlib — paths as objects, not strings

<RunnableCode title="paths that behave like paths" code={pathlibCode} />

`Path` gives you four things string paths cannot: **operators** (`root / "sub" / "file.json"`, safe on every OS), **methods** (`.exists()`, `.stat()`, `.glob()`, `.read_text()`, `.write_text()`), **anatomy** (`.name`, `.stem`, `.suffix`, `.parent`, `.parts`), and **cross-platform correctness**.

The moment you type `os.path.join` in modern Python, stop and use `Path(...) / ...`. The only exception: a third-party API that insists on strings — wrap with `str(p)` at the last step (most modern libraries accept both).

One-liners to memorize:

- `Path("data").mkdir(parents=True, exist_ok=True)` — create the directory and parents; no error if it exists.
- `Path(f).glob("**/*.json")` — recursively find all JSON files under `f`.
- `Path.home() / ".config" / "my-tool" / "config.toml"` — cross-platform user-config path.
- `p.with_suffix(".pt")` — same name, different extension.

## argparse — the stdlib CLI

<RunnableCode title="argparse basics" code={argparseCode} />

`argparse` parses `sys.argv` into a typed namespace. Four habits separate a professional CLI from a script that surprises its users:

1. **Positional arguments are required; `--flags` are optional.** That's the convention across every CLI you'll meet (`git commit`, `docker run`). Don't invert it.
2. **Set `type=int` / `type=float` / `type=Path`** so bad input dies at parse time with a helpful message, not deep inside training with a mysterious `TypeError`. And let it die — don't try/except around `parse_args`; for a CLI, the parse error *is* the UX.
3. **Use `choices=[...]`** for enumerable options — the parser validates and `--help` lists the valid values.
4. **Add `--dry-run` and `--verbose` generously.** They cost nothing and save hours of "did I set it right?"

## Typer — the type hints ARE the schema

<CodeBlock language="python" title="Typer version of the same CLI" code={typerCode} />

**Typer** reads your function's type hints and defaults, and *from those alone* builds the CLI schema, help text, error messages, and tab completion. Compare with the argparse version: no `add_argument` calls (the parameter list is the argument list), no `type=int` (the hints declared it), no manual help strings (docstring + hints generate them).

The trade-off: a dependency (`uv add typer`), and it's easier to accidentally break the CLI contract by refactoring the signature. **When to use which:** argparse for private scripts and simple tools; Typer (or Click) for user-facing CLIs and tools with many subcommands. The choice is stylistic, not correctness-affecting.

## JSON — the lingua franca

<RunnableCode title="json.dump / json.load — round-trip and its edges" code={jsonCode} />

JSON is the format every language speaks — every HTTP API, every logging platform. Its type vocabulary is minimal by design — **string, number, bool, null, array, object** — which is also its constraint. Three things to always remember:

1. **`None` becomes `null`** and comes back as `None`. The round-trip is faithful only for JSON's native vocabulary.
2. **No datetime, Decimal, set, or bytes.** Convert at the boundary: ISO 8601 strings for datetimes, base64 for bytes. The cell shows a datetime going out as a string and *staying* a string on the way back.
3. **No NaN or Infinity in the spec.** `json.dump` writes them anyway (non-standard) and other parsers reject them — pass `allow_nan=False` to catch it at write time.

Style: `indent=2` when a human will read it; no indent for machine-only output; `sort_keys=True` for anything you want to diff between runs. For huge JSON data, prefer **JSONL** (one object per line) — then plain file iteration streams it.

## Streaming — files bigger than memory

<RunnableCode title="iterate a file line by line" code={streamCode} />

`for line in file:` uses the file's own iterator protocol and pulls one line at a time — this is how you scan a 60 GB log on a 16 GB laptop. The moment you write `f.read()` or `f.readlines()`, ask: *does this fit in RAM?* If unsure, stream. The same **iterate, don't materialize** idea powers `pd.read_csv(path, chunksize=100_000)`, `csv.reader`, `ijson` for huge JSON, `pyarrow`'s `iter_batches()`, and PyTorch's `IterableDataset`.

Two file-opening rules that prevent whole bug categories:

- **Text vs binary mode.** `open(path)` decodes bytes to `str` (text mode) — right for JSON, CSV, TOML, logs. `open(path, 'rb')` reads raw `bytes` — right for Parquet, images, checkpoints. The classic "utf-8 codec can't decode byte 0x80" error is Python telling you a binary file was opened as text.
- **Always pass `encoding="utf-8"` explicitly** in `open()`, `Path.read_text()`, and CSV writers. The platform default (cp1252 on older Windows) silently mangles non-ASCII text. And open CSVs with `newline=""` when using the `csv` module, or Windows double-counts line endings.

## Formats — the trade-off table you'll live by

Every format is a bet on **who reads it, how fast, and how big**:

| Format  | Human-readable | Schema | Speed  | Size   | Sweet spot |
| ------- | -------------- | ------ | ------ | ------ | ---------- |
| JSON    | Yes            | Loose  | OK     | Large  | Web APIs, small configs, logs |
| YAML    | Yes (comments!)| Loose  | Slow   | Large  | Human configs (Kubernetes, CI, Hydra) |
| TOML    | Yes            | Strict | Fast   | Small  | `pyproject.toml`, app config |
| CSV     | Yes            | None   | Slow   | Huge   | Interchange with spreadsheets |
| Parquet | No             | Strong | Fast   | Tiny   | ML datasets, warehouses |
| Arrow   | No             | Strong | Fastest| Small  | In-memory or IPC between processes |
| Pickle  | No             | Python | Fast   | Medium | Model state — *same-Python only* |
| Safetensors | No         | Strong | Fast   | Small  | Model weights — safe, portable |

Rules: **config for humans → TOML or YAML** (comments matter; Python 3.11+ reads TOML via stdlib `tomllib`); **API interchange → JSON**; **tabular ML data → Parquet**; **model weights → safetensors**. Never use CSV when you have a choice — no types, no null representation, embarrassing byte size. Never unpickle data you didn't produce: `pickle.load` is equivalent to `eval` and executes arbitrary code on load. Large ML projects layer `pydantic`/`hydra` on top of these formats for schema + overrides.

## What each format preserves

<RunnableCode title="round-trip a typed DataFrame through CSV and through Parquet" code={roundtripCode} />

Read the output carefully. Before the trip, `df` has `int64`, `float64` (with NaN), string, datetime, and Categorical columns. After **CSV**, everything is `int64`, `float64`, or untyped `object` — the datetime is a string pandas must re-parse, the Categorical is plain strings. After **Parquet**, every dtype is preserved exactly, nulls included.

If training data goes through CSV once, you re-detect and re-cast every column type on every load. In Parquet, `pd.read_parquet(path)` returns exactly the DataFrame you saved. Multiply by 100 pipeline stages and 1000 loads a day, and Parquet-vs-CSV is the difference between reproducible ML and constant firefighting.

<RealWorldML title="Parquet vs CSV, felt in bytes and milliseconds">

<RunnableCode title="benchmark on 200K rows" code={parquetCode} />

Line by line:

1. The table is 200,000 rows of typed events (int, datetime, categorical string, float) — the shape of most ML input data.
2. All three writes go to `io.BytesIO`, so the numbers reflect pure format cost, no disk.
3. **CSV** re-encodes every value as text — every float becomes `"10.428"`, every timestamp a 19-byte string. No types survive; a re-read has to guess.
4. **JSON** is fatter still: it repeats every column name in every row, 200,000 times.
5. **Parquet** comes out 5–10× smaller with types preserved.

What Parquet actually does under the hood (why it wins):

- The **schema** is stored in the file — `df.dtypes` map 1:1 to Arrow types, which is why round-trips are faithful.
- Data is laid out **column-major**: all of column A, then all of B. Reading only the columns you need skips the rest entirely.
- Columns are split into **row groups** (~1M rows) with min/max stats — a filter like `date > '2026-01-01'` skips whole row groups without reading them ("predicate pushdown").
- Each column is **encoded** (dictionary for categoricals, delta for sorted timestamps, run-length for repeats) — often 10× smaller before compression even runs — then **compressed** per column (Snappy = fast default; Zstd = smaller, for cold data).
- A metadata **footer** lets readers learn the whole layout instantly, then jump straight to what they need. And it's an Apache standard — Spark, DuckDB, polars, BigQuery, Snowflake all read it.

Why this matters: a training pipeline reads the same dataset thousands of times — every epoch, every tune, every replay. CSV pays the parse cost every single time; Parquet pays once at conversion. With date-partitioned layout (`/data/events/date=2026-07-12/*.parquet`), a "give me July" query scans only July's partition — and `pyarrow.dataset` or `duckdb` can query datasets far larger than RAM this way. Every major ML platform (W&B, MLflow, Databricks) stores data in Parquet or Arrow because of this multiplier.

</RealWorldML>

<Summary>

- pathlib for filesystem work — `Path` objects and the `/` operator, never `os.path` or string concat in new code.
- argparse for scripts you own; Typer/Click for tools you ship. Positional args required, flags optional, types enforced at parse time so bad input dies early and loud.
- JSON for interop, TOML/YAML for human configs, Parquet for tabular data, safetensors for model weights. Pickle only for same-Python state you produced yourself.
- Every `open()` lives inside a `with` block; every possibly-large file is streamed line-by-line, never `.read()` whole.
- Parquet is columnar, typed, encoded, and compressed — 5–10× smaller than CSV, reads by column, skips row groups by min/max stats.
- Reflexes: `encoding="utf-8"` always explicit, `newline=""` for CSV, `allow_nan=False` for strict JSON, binary mode `'rb'` for non-text files.

</Summary>

        with tmp.open("w", encoding="utf-8") as f:
            json.dump(obj, f)
            f.flush()                              # ensure userspace buffer -> OS
            import os
            os.fsync(f.fileno())                   # ensure OS buffer -> disk

        tmp.unlink(missing_ok=True)
