export const closureCode = `# A closure captures its enclosing variables — and keeps them alive after outer returns
def make_counter(start=0):
    count = start                       # captured variable
    def tick():
        nonlocal count                  # without this, tick would rebind a local shadow
        count += 1
        return count
    return tick                         # the returned function 'closes over' count

a = make_counter()
b = make_counter(100)

print(a(), a(), a())     # 1 2 3   — a has its own count cell
print(b(), b())          # 101 102 — b has a SEPARATE count cell
print(a())               # 4       — a's state is untouched by b

# Under the hood: 'a' carries a reference to a cell object holding count.
# You can inspect the closure cells directly:
print("cells on a:", a.__closure__)
print("count in a:", a.__closure__[0].cell_contents)`;

export const closurePitfallCode = `# The classic Python trap: late binding in a loop
# ----------------------------------------------
handlers = []
for i in range(3):
    handlers.append(lambda: i)         # every lambda closes over the SAME i

# By the time we call any of them, the loop is done and i == 2
print("late-binding (buggy):", [h() for h in handlers])       # [2, 2, 2]

# Fix 1: default-argument capture (bind at definition time)
handlers2 = []
for i in range(3):
    handlers2.append(lambda i=i: i)    # i=i evaluates i NOW, stores it as a default
print("default-arg fix:    ", [h() for h in handlers2])       # [0, 1, 2]

# Fix 2: functools.partial
from functools import partial
handlers3 = [partial(lambda i: i, i) for i in range(3)]
print("partial fix:        ", [h() for h in handlers3])       # [0, 1, 2]

# Fix 3: put the loop body inside a function so each iteration has its own scope
def bind(i):
    return lambda: i
handlers4 = [bind(i) for i in range(3)]
print("inner-def fix:      ", [h() for h in handlers4])       # [0, 1, 2]`;

export const trainingLoopCode = `# A training loop with all four concepts in one small program:
# 1) functions as arguments (loss_fn, optim_step, on_epoch_end callback)
# 2) closures (the trainer holds captured state across epochs)
# 3) default + keyword args (readable configuration)
# 4) type hints (self-documenting APIs)

from typing import Callable

def make_trainer(loss_fn: Callable[[float, float], float],
                 on_epoch_end: Callable[[int, float], None] | None = None):
    """Return a train(epochs, lr) function that carries its own history."""
    history: list[float] = []          # captured by closure — trainer's memory

    def train(epochs: int = 5, lr: float = 0.1) -> list[float]:
        w = 0.0
        for epoch in range(epochs):
            # pretend one 'batch' of loss: comparing w to a target of 3
            loss = loss_fn(w, 3.0)
            grad = 2 * (w - 3.0)       # d/dw of (w-3)^2 = 2(w-3)
            w -= lr * grad             # one gradient step
            history.append(loss)
            if on_epoch_end is not None:
                on_epoch_end(epoch, loss)
        return history

    return train

# Use different loss functions & callbacks without changing the trainer

absolute_error = lambda pred, target: abs(pred - target)

def print_progress(epoch, loss):
    print(f"  epoch {epoch}: loss={loss:.4f}")

train_sq = make_trainer(squared_error, on_epoch_end=print_progress)
print("training with squared error:")
train_sq(epochs=5, lr=0.3)

train_ab = make_trainer(absolute_error)
print("\\ntraining with absolute error (silent):")
history = train_ab(epochs=4, lr=0.5)
print("final loss history:", [round(h, 3) for h in history])`;

<Motivation title="The one chapter that unlocks decorators, callbacks, and training loops">

If Python has a *master idea*, it is this: **functions are first-class values**. Assign them to variables, put them in dicts, pass them to other functions, return them from other functions. Decorators are functions that take a function and return a modified one. Callbacks — the `on_epoch_end` you register with a trainer, the `key=len` you hand to `sorted`, the loss function you plug into a training loop — are functions passed around like data. Closures are how a returned function keeps private state, which is what makes `functools.lru_cache` and half the standard library work.

Once functions-as-values feels obvious, 80% of what looks "advanced" in Python stops being advanced — and `torch.optim.Adam`, `sklearn.make_pipeline`, `datasets.map`, and `transformers.Trainer` become readable: every one is functions calling functions holding functions.

</Motivation>

<Intuition title="Three ideas that make Python callable magic">

1. **Functions are objects.** `def foo(): ...` is essentially `foo = <function object>`. You can `bar = foo`, store functions in containers, pass them anywhere. That's *first-class functions*.

2. **Scope resolution follows LEGB.** Reading a name searches **L**ocal → **E**nclosing → **G**lobal → **B**uilt-in, first hit wins. Assignment is different: it writes to the current *local* scope unless you say `nonlocal` or `global`.

3. **A closure is a function that remembers.** When an inner function references a variable from its enclosing scope, Python keeps the *cell* containing that variable alive as long as the inner function lives. Each `make_counter()` call gets a fresh cell; two counters never share state.

Every "advanced" Python trick you'll meet is one, two, or all three of these in a costume.

</Intuition>

## Functions as values — dispatch, callback, factory

<RunnableCode title="functions as data — the strategy pattern in one line" code={higherOrderCode} />

Read the cell three ways. **Storage:** functions live in a dict keyed by symbol — behavior picked at runtime, the "strategy pattern" in one line. **Arguments:** `sorted(words, key=len)` passes the builtin `len` as a value; `sorted` calls it per element to decide order. **Return values:** `multiplier(2)` returns a fresh function that has captured `factor=2` — a factory. Dispatch, callback, factory: every serious codebase is built from these three moves.

One note on `lambda`: it's just a short function literal — `lambda x: x*2` is `def <anon>(x): return x*2`. Use it for one-liners passed as arguments; anything longer deserves a named `def`, which is equally fast and vastly more debuggable.

## The five kinds of function arguments

<RunnableCode title="positional / default / *args / keyword-only / **kwargs" code={argsCode} />

Read a signature as a menu of parameter positions:

- **Positional** (`name`): required, in order.
- **Default** (`greeting="Hello"`): optional; passable by position or name. Beware **mutable defaults** — `out=[]` is evaluated once at definition time and shared across calls; use `out=None` and assign inside.
- **`*extras`**: captures extra positional arguments into a tuple.
- **Keyword-only** (`formal=False` *after* `*` or `*args`): must be passed by name — the modern way to keep call sites readable (`greet("Sarath", formal=True)`, never `greet("Sarath", True)`).
- **`**kwargs`**: captures leftover keyword arguments as a dict.

The markers `/` and `*` split the menu: `def f(a, b, /, c, *, d):` — `a, b` positional-only, `c` either, `d` keyword-only. Real APIs use this deliberately: PyTorch's `Adam(params, lr=1e-3, ...)` makes hyperparameters keyword-only so users can't silently pass them in the wrong order, and FastAPI reads your signature to parse the URL. The signature is a *contract* that tools read.

## LEGB — how Python finds a name

<RunnableCode title="Local / Enclosing / Global / Built-in — step by step" code={legbCode} />

LEGB is a lookup algorithm: to *read* a name, Python walks outward — current function, enclosing functions, module globals, built-ins — and stops at the first hit. To *write* a name, the rule flips: assignment creates or updates a **local**, unless declared `nonlocal` (write to enclosing) or `global` (write to module).

**The asymmetry is the classic gotcha.** `x = x + 1` inside a function makes `x` local *for the whole function body* — so the read on the right-hand side hits an unassigned local and raises `UnboundLocalError`, even though a global `x` exists. If you truly want to rebind an outer name, say so with `nonlocal`/`global` — but reach for them rarely. Returning a value and re-assigning outside is cleaner, and mutable containers dodge the need entirely: with `counts = {"n": 0}` outside, `counts["n"] += 1` inside mutates *contents* without rebinding the name — no declaration required.

## Closures — functions with private state

<RunnableCode title="counter closure — separate state per instance" code={closureCode} />

Each `make_counter` call creates a fresh `count` and a fresh `tick`. Because `tick` references `count`, Python allocates a **cell** for it and stores a reference in `tick.__closure__`. When `make_counter` returns, its frame is gone — but the cell survives, because `tick` still points at it. `a` and `b` were built by separate calls, so they own separate cells: incrementing one never touches the other. This is the functional way to do what OOP does with instance variables, and it's exactly what powers `functools.lru_cache` and every counter/timer/rate-limiter decorator.

<Recall term="Cell">

A **cell** is Python's implementation of a captured variable. When a nested function references an enclosing name, both functions share one cell object holding the value; the cell outlives the outer function's frame. Inspect via `fn.__closure__` (a tuple of cells) and `.cell_contents`. This is why "closures capture *variables*, not values" — they capture a shared cell whose contents can change.

</Recall>

Why is `nonlocal count` required? Without it, `count += 1` inside `tick` would follow the write-locally rule — Python would treat `count` as a new local, and the read before assignment would raise `UnboundLocalError`. `nonlocal` redirects the write into the captured cell. (Read-only captures need no declaration.)

## The late-binding trap

<RunnableCode title="the [2, 2, 2] surprise and three fixes" code={closurePitfallCode} />

Closures capture cells, not values — so every `lambda: i` created in the loop closes over *the same `i` cell*. By the time any lambda runs, the loop has finished and `i == 2`: hence `[2, 2, 2]`. Not a bug; a direct consequence of how cells work. Three fixes, all giving each lambda its own binding:

- `lambda i=i: i` — the default `i=i` evaluates *now*, at definition time.
- `functools.partial(fn, i)` — freezes the current value into the callable.
- A helper `def bind(i): return lambda: i` — each call creates a fresh scope with its own cell.

This trap lives in every "why do all my buttons do the same thing" question about GUI/web callback registration. Once you feel *why*, you'll spot it in code review at a glance.

## Type hints — signatures tools can read

<RunnableCode title="Callable, generics, PEP 695 syntax" code={typedCode} />

Type hints are optional annotations with four consumers:

- **Editors** — autocomplete, refactoring, go-to-definition.
- **Static checkers** (`mypy`, `pyright`) — catch a huge class of bugs before the code runs.
- **Frameworks at runtime** — Pydantic validates from them, FastAPI parses requests from them (via `inspect.signature`, which turns any signature into inspectable data).
- **Not the interpreter.** `def add(x: int, y: int) -> int:` happily concatenates strings — hints are documentation, not enforcement. Need runtime enforcement? Pydantic.

Modern spellings are compact and import-free: `list[T]`, `dict[K, V]`, `X | None`, and PEP 695 generics `def first[T](xs: list[T]) -> T:` (3.12+). Adopt hints from day one — a hundred-file project without them is nearly impossible to refactor safely.

<RealWorldML title="A training loop with all four concepts stacked">

<RunnableCode title="a mini trainer with closures, callbacks, and type hints" code={trainingLoopCode} />

**Line by line, how this shows up in production ML code:**

1. `make_trainer(loss_fn, on_epoch_end=...)` is a *factory* returning a customized `train` — the exact shape of `pytorch_lightning.Trainer(callbacks=[...])` and `transformers.Trainer(compute_metrics=...)`: strategies in, configured procedure out.
2. `loss_fn: Callable[[float, float], float]` says "any function taking two floats, returning a float". The trainer doesn't care if it's squared error or cross-entropy — which is why swapping `nn.MSELoss()` for `nn.CrossEntropyLoss()` is a one-line change.
3. `history: list[float] = []` lives in the enclosing scope and is captured by closure — every call to `train` appends to the same list. The trainer "remembers" across epochs with no class and no global. (JAX/Flax code leans on exactly this style.)
4. `on_epoch_end` is a **callback** — a function passed in and invoked at a fixed point. `keras.callbacks.EarlyStopping` and `transformers.TrainerCallback` are the same idea: the framework owns the loop, you supply the "what to do at each checkpoint".
5. `train_sq` and `train_ab` come from two separate `make_trainer` calls, so their closures hold different cells — two independent trainers, zero shared state.

Every ML framework reduces to this shape: **first-class functions + captured configuration + call at the right moment.**

<Summary>

- **Functions are objects** — assign, store, pass, return them. Dispatch tables, callbacks, and factories are all this one fact.
- **LEGB:** reads walk Local → Enclosing → Global → Built-in; writes stay local unless `nonlocal`/`global`. `x = x + 1` with no local `x` raises `UnboundLocalError` — the asymmetry in action.

- **Arguments:** positional, default (never a mutable one), `*args`, keyword-only after `*`, `**kwargs`; `/` and `*` mark position-only/keyword-only zones.
- **Type hints** are contracts for editors, checkers, and frameworks — not enforced by the interpreter.
- **Every trainer/optimizer/callback API** is first-class functions + captured configuration + call at the right moment.

</Summary>
