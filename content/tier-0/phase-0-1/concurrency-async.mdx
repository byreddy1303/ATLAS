export const threadsCode = `# threads for I/O — waiting on network in parallel
import time, threading, urllib.request

urls = ["https://httpbin.org/delay/1"] * 5

def fetch(url):
    urllib.request.urlopen(url).read()

# sequential
t = time.perf_counter()
for u in urls: fetch(u)
print(f"sequential: {time.perf_counter() - t:.2f}s")

# threaded
t = time.perf_counter()
ts = [threading.Thread(target=fetch, args=(u,)) for u in urls]
for th in ts: th.start()
for th in ts: th.join()
print(f"threaded  : {time.perf_counter() - t:.2f}s")`;

export const asyncCode = `import asyncio, time

async def slow_task(name, secs):
    print(f"  {name} start")
    await asyncio.sleep(secs)    # yields control while it 'sleeps'
    print(f"  {name} done")
    return f"{name}={secs}"

async def main():
    t = time.perf_counter()
    results = await asyncio.gather(
        slow_task("A", 1),
        slow_task("B", 2),
        slow_task("C", 1),
    )
    print("results:", results)
    print(f"total: {time.perf_counter() - t:.2f}s (would be 4 sequentially)")

asyncio.run(main())`;

export const poolCode = `# processes for CPU — no GIL, real parallelism
from concurrent.futures import ProcessPoolExecutor
import time, math

def is_prime(n):
    if n < 2: return False
    for i in range(2, int(math.isqrt(n)) + 1):
        if n % i == 0: return False
    return True

nums = list(range(10_000_000, 10_000_050))

t = time.perf_counter()
seq = [is_prime(n) for n in nums]
print(f"sequential: {time.perf_counter() - t:.2f}s")

t = time.perf_counter()
with ProcessPoolExecutor(max_workers=4) as pool:
    par = list(pool.map(is_prime, nums))
print(f"pool(4)   : {time.perf_counter() - t:.2f}s")

assert seq == par`;

<Motivation title="Why one loop is sometimes not enough">

You want your Python program to do more than one thing at a time. Maybe you're fetching 100 URLs — waiting on each individually is unbearable. Maybe you're crunching numbers on a 16-core CPU — using one core is wasteful. Maybe you're building a web server — each request should not block others. These are three completely different situations, and Python gives you three completely different tools: `asyncio`, `threading`, and `multiprocessing`. Confusing them is the classic Python performance mistake. Understanding which is which turns "I/O should be fast" and "why isn't NumPy faster?" from mysteries into decisions.

</Motivation>

<Intuition title="The one decision chart you actually need">

- Is your workload **waiting on I/O** (network, disk, subprocess)? → `asyncio` if the library supports it, else `threading`.
- Is your workload **CPU-bound in pure Python**? → `multiprocessing`.
- Is your workload **CPU-bound in NumPy/PyTorch**? → *Don't* reach for concurrency in Python — the underlying C/BLAS libraries already use all your cores. Adding threads on top usually makes things slower.

Everything else in this chapter is why.

</Intuition>

<Recall term="The Global Interpreter Lock (GIL)">

CPython (the reference Python implementation) has a **single lock** — the GIL — that only lets one thread execute Python bytecode at a time. That means `threading` in CPython does NOT give you true parallelism for CPU-bound Python code. It *does* give you parallelism for I/O, because a thread waiting on a socket releases the GIL. This one fact explains 90% of "why is my threaded Python not faster?" questions. (Python 3.13 introduced an experimental *no-GIL* build. If you're running that, revisit these rules; otherwise the classic behavior applies.)

</Recall>

<Angles>
  <Angle type="computational" title="threading — parallelism for waiting">
    <RunnableCode title="threads on I/O" code={threadsCode} />

    Five HTTP calls take ~5 seconds sequentially and ~1 second threaded. The GIL doesn't hurt here because `urllib.request.urlopen` blocks on the socket — Python releases the GIL while waiting, other threads run. Threads are perfect for embarrassingly-parallel I/O: crawling, hitting many APIs, reading many files.
  </Angle>

  <Angle type="computational" title="asyncio — cooperative single-thread I/O">
    <RunnableCode title="asyncio.gather" code={asyncCode} />

    `asyncio` gives you the same speedup as threads for I/O, without the overhead of OS threads. The trick: `await` is a *cooperative yield point*. When one coroutine hits `await`, the event loop runs another. This scales to **10,000+ concurrent operations** in a single thread — a place threads can't reach because each OS thread costs a few MB of stack.

    Rule: everything in the chain must be `async`. Mixing sync I/O into an asyncio program is the #1 performance foot-gun.
  </Angle>

  <Angle type="computational" title="multiprocessing — real parallelism for real CPU work">
    <RunnableCode title="ProcessPoolExecutor for primes" code={poolCode} />

    Each process has its own Python interpreter and its own GIL. So 4 processes = 4 cores. This is how you actually get parallel *pure-Python* CPU work. Downsides: each process has memory overhead (100+ MB for a NumPy import), and passing data between processes requires serialization (pickling). If your task is small, the overhead can dwarf the gain.

    Sweet spot: CPU-heavy tasks that operate on chunks of independent data and produce small results. Bad fit: shared large arrays (use `multiprocessing.shared_memory` for that).
  </Angle>

  <Angle type="code" title="concurrent.futures — the friendly interface">
    Both threads and processes have a unified high-level API in `concurrent.futures`:

    <CodeBlock language="python" title="ThreadPoolExecutor / ProcessPoolExecutor" code={`from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

# I/O bound: threads
with ThreadPoolExecutor(max_workers=8) as pool:
    for result in pool.map(fetch, urls):
        handle(result)

# CPU bound: processes
with ProcessPoolExecutor(max_workers=4) as pool:
    for result in pool.map(cpu_heavy, chunks):
        handle(result)`} />

    This is often the *right* place to start — it hides Thread/Process construction, handles exceptions cleanly, and lets you swap between them by changing one class name.
  </Angle>
</Angles>

<Convergence>

Three tools, one decision. I/O → asyncio (best when library supports it) or threading (when it doesn't). CPU → multiprocessing. NumPy-heavy → let NumPy handle it. The rest of your career you'll answer *"is this I/O-bound or CPU-bound?"* first and pick the tool second.

</Convergence>

<Nuance>

**Race conditions.** When threads share mutable state — a counter, a dict, a list — they can interleave badly and produce wrong answers. Use `threading.Lock`, `queue.Queue`, or immutable data. asyncio avoids this because there's only ever one coroutine running at a time (until it hits an `await`).

**Async everywhere.** Once you go async, every I/O call in the same task must be async, or you'll block the whole event loop. Use `httpx` (async), not `requests`. Use `aiofiles`, not builtin `open()` for large files. Use async database drivers, not sync ones.

**Structured concurrency.** Modern Python (3.11+) has `asyncio.TaskGroup` and third-party `anyio` for *structured* concurrency — child tasks that always finish before the parent, with grouped exception handling. Reach for these instead of raw `asyncio.gather` when a task might raise and you need clean shutdown.

**Multiprocessing on macOS.** Since 3.8, the default start method is `spawn`, which is slow to boot. If you're launching lots of short pool tasks, batch them into a smaller number of longer ones.

**Don't nest event loops.** Calling `asyncio.run()` from inside an already-running loop errors. In a notebook or FastAPI handler, use `await` directly.

</Nuance>

import httpx

async def download_all(urls, concurrency=10):
    sem = asyncio.Semaphore(concurrency)
    async with httpx.AsyncClient() as client:
        async def fetch(u):
            async with sem:
                r = await client.get(u)
                return r.text
        return await asyncio.gather(*(fetch(u) for u in urls))

# usage: asyncio.run(download_all(urls, concurrency=20))`} />}
