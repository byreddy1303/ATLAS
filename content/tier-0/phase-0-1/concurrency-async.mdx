export const threadsCode = `# threading for I/O — parallelism while waiting on the network
import time, threading, urllib.request

urls = ["https://httpbin.org/delay/1"] * 5

def fetch(url):
    urllib.request.urlopen(url).read()

# Sequential — each request waits for the previous to finish
t = time.perf_counter()
for u in urls: fetch(u)
print(f"sequential: {time.perf_counter() - t:.2f}s   (about 5s)")

# Threaded — all five requests wait in parallel
t = time.perf_counter()
ts = [threading.Thread(target=fetch, args=(u,)) for u in urls]
for th in ts: th.start()
for th in ts: th.join()
print(f"threaded  : {time.perf_counter() - t:.2f}s   (about 1s)")

# Same speedup with a friendlier pool interface
from concurrent.futures import ThreadPoolExecutor
t = time.perf_counter()
with ThreadPoolExecutor(max_workers=5) as pool:
    list(pool.map(fetch, urls))
print(f"pool      : {time.perf_counter() - t:.2f}s")`;

import asyncio, time

async def slow_task(name, secs):
    print(f"  {name} start")
    await asyncio.sleep(secs)         # yields control while it 'sleeps'
    print(f"  {name} done")
    return f"{name}={secs}"

async def main():
    t = time.perf_counter()
    # gather runs all three concurrently and returns the list of results
    results = await asyncio.gather(
        slow_task("A", 1),
        slow_task("B", 2),
        slow_task("C", 1),
    )
    print("results:", results)
    print(f"total: {time.perf_counter() - t:.2f}s (would be 4s sequentially)")

export const asyncPatternCode = `# Two async patterns that show up in every LLM pipeline:
# 1) run many concurrent tasks with a concurrency LIMIT (Semaphore)
# 2) collect results as they finish (as_completed) so slow ones don't block fast ones
import asyncio, random, time

async def flaky_embed(text: str) -> int:
    """Pretend to call an embedding API. Latency varies."""
    delay = random.uniform(0.05, 0.4)
    await asyncio.sleep(delay)
    return len(text)                     # 'embedding' is just its length

async def embed_all(texts, concurrency=5):
    """At most 'concurrency' embed calls in flight at any moment."""
    sem = asyncio.Semaphore(concurrency)

    async def worker(text):
        async with sem:                  # only enters if the sem has a slot
            return await flaky_embed(text)

    return await asyncio.gather(*(worker(t) for t in texts))

async def main():
    texts = [f"doc-{i}" for i in range(20)]
    t = time.perf_counter()
    results = await embed_all(texts, concurrency=5)
    print(f"embedded {len(results)} texts in {time.perf_counter()-t:.2f}s")
    print("first five results:", results[:5])

asyncio.run(main())`;

export const poolCode = `# multiprocessing — real parallelism for real CPU work
from concurrent.futures import ProcessPoolExecutor
import time, math

def is_prime(n: int) -> bool:
    if n < 2: return False
    for i in range(2, int(math.isqrt(n)) + 1):
        if n % i == 0:
            return False
    return True

nums = list(range(10_000_000, 10_000_050))

# Sequential — one core, all Python bytecode, GIL locked to one thread
t = time.perf_counter()
seq = [is_prime(n) for n in nums]
print(f"sequential: {time.perf_counter() - t:.2f}s")

# Multiprocessing — one Python interpreter per worker, one GIL each,
# so 4 cores can actually run 4 CPU-bound Python jobs in parallel.
t = time.perf_counter()
with ProcessPoolExecutor(max_workers=4) as pool:
    par = list(pool.map(is_prime, nums))
print(f"pool(4)   : {time.perf_counter() - t:.2f}s")

assert seq == par
print("results agree; primes found:", sum(par))`;

<Motivation title="Why one loop is sometimes not enough">

Three flavors of "do many things at once" show up over and over: **fetch 100 URLs from a slow API** (almost all the time is spent waiting, so waiting in parallel is nearly free), **crunch numbers on a 16-core CPU** (real computation — you need real parallelism or 15 cores sit idle), and **serve many concurrent clients** (each request should progress while others wait).

These look similar but are radically different underneath, and Python has a different tool for each: `asyncio`, `threading`, `multiprocessing`. Confusing them is *the* classic Python performance mistake — threading a CPU-bound loop gives zero speedup; multiprocessing an I/O job pays huge startup cost for nothing. Every LLM pipeline, RAG system, and serving stack you'll build sits on the right choice here.

</Motivation>

<Intuition title="One decision chart that answers everything">

Memorize this decision tree. Three questions, in order.

1. **Is your workload *waiting* on I/O — network, disk, a subprocess, a database?**
   - Yes, and the library has an async version: → **`asyncio`**. Thousands of concurrent operations in one thread. The modern default for LLM APIs, RAG pipelines, and web servers.
   - Yes, but the library is sync-only (`requests`, sync drivers, legacy code): → **`threading`** via `ThreadPoolExecutor`. Drop-in parallelism for existing sync code.

2. **Is your workload *CPU-bound in pure Python*** — arithmetic, string parsing, tree traversal, all in Python bytecode?
   - Yes: → **`multiprocessing`** via `ProcessPoolExecutor`. Each process has its own interpreter and its own GIL, so N processes really use N cores.

3. **Is your workload CPU-bound but the actual work happens in NumPy, PyTorch, or another C-level library?**
   - Yes: → **do nothing.** Those libraries release the GIL internally and already use all your cores. Wrapping them in threads or processes usually *slows things down*.

That is the entire chapter. Everything below is *why* these answers are correct.

</Intuition>

<Recall term="The Global Interpreter Lock (GIL)">

CPython has a **single lock** — the GIL — that permits only one thread to execute Python bytecode at a time. So `threading` gives **no** true parallelism for CPU-bound pure-Python code: 16 threads on 16 cores still run one at a time, just interleaved.

The reason threading still matters: a thread waiting on a socket, disk, or subprocess **releases the GIL** while it waits. While thread A blocks on `urlopen(url_1)`, thread B grabs the GIL and issues `urlopen(url_2)` — ten threads can all be in flight on the network simultaneously. Threads: great for I/O, useless for CPU. (Python 3.13's experimental free-threaded build removes the GIL; until that's the default, the classic rules apply.)

</Recall>

<Recall term="Coroutine">

A **coroutine** is a function that can *pause itself mid-execution and be resumed later*. `async def` defines one; `await` inside it pauses the coroutine and hands control to the event loop, which runs other coroutines until the awaited thing is ready. Nothing runs in parallel — there's one thread — but everything is *concurrent* because the loop juggles them. Threads top out around a thousand (each OS thread carries a 1–8 MB stack); coroutines scale to hundreds of thousands because they're just Python objects.

</Recall>

## threading — parallelism for *waiting*

<RunnableCode title="threads on I/O + a ThreadPoolExecutor variant" code={threadsCode} />

Five HTTP calls take ~5s sequentially and ~1s threaded. The GIL doesn't hurt because `urlopen` blocks on the socket — Python releases the GIL while waiting, other threads pick it up, and all five requests fly in parallel from the network's point of view. Threads are perfect for embarrassingly-parallel I/O: crawling, hitting many APIs, reading many files.

Prefer **`ThreadPoolExecutor`** over manual `threading.Thread` — it manages the pool, propagates worker exceptions cleanly, and gives you `.map(...)`. One caution that comes free with threads: if they share mutable state (a counter, a dict), interleaving can corrupt it — use `threading.Lock`, `queue.Queue`, or keep workers independent.

Threading breaks down past a few hundred concurrent operations — thread creation and OS scheduling become the bottleneck. At that scale, switch to asyncio: same wall-clock behavior for I/O, radically lower overhead.

## asyncio — I/O concurrency in one thread

<RunnableCode title="asyncio.gather — three tasks concurrent in one thread" code={asyncCode} />

Same wall-clock speedup as threads for I/O work, **without any threads.** What actually happens at each `await`:

1. The coroutine *suspends* — its locals, current line, everything preserved — and control returns to the event loop.
2. The loop picks another ready coroutine and runs it until *that* one awaits or finishes.
3. When the awaited thing is ready — a timer fires, the OS reports the socket has data — the loop moves the original coroutine back to "ready".
4. It resumes right after the `await`, with the awaited value as the expression's result. From the coroutine's view, `await` was a slow function call; from the loop's view, it juggled many things in that time.

Three practical rules separate asyncio wins from asyncio pain:

1. **Everything in the chain must be async.** One synchronous blocking call — `time.sleep(5)`, `requests.get(url)` — freezes the *entire* loop, because there's only one thread and it's stuck inside the call. This is the #1 async foot-gun. Use `httpx` not `requests`, `aiofiles` not `open`, `asyncpg` not sync drivers; if you *must* call sync code, escape with `await asyncio.to_thread(sync_fn, arg)`.
2. **`asyncio.gather(*coros)` is your for loop.** It launches all passed coroutines concurrently and returns the list of results. An exception in any coroutine cancels the rest (unless `return_exceptions=True`).
3. **`asyncio.Semaphore(n)` limits concurrency** — without it, gathering 10,000 coroutines fires all 10,000 at once and gets you rate-limited.

Two operational notes: never call `asyncio.run()` inside an already-running loop (in Jupyter or a FastAPI handler, just `await` directly — `asyncio.run` is only for the top of a program), and set `PYTHONASYNCIODEBUG=1` to get slow-callback warnings that catch accidental blocking calls.

## The pattern you'll write forever — Semaphore + gather

<RunnableCode title="rate-limited concurrent API calls" code={asyncPatternCode} />

This is the pattern behind every "embed 10,000 documents" script, every benchmark runner, every RAG bulk-ingest. `Semaphore(5)` says "at most 5 calls in flight." Each worker does `async with sem:` — acquire a slot before the call, release on exit. `gather` launches all 20 workers, but only 5 run the API call at any moment; the rest wait on the semaphore.

Why it's load-bearing: hitting a rate-limited API without a semaphore fails immediately (`429 Too Many Requests`). Semaphore + gather makes you throughput-optimal; add retry-with-backoff from the errors chapter and you're resilient too. Every serious LLM SDK ships this combination.

Two neighbors worth knowing: **`asyncio.as_completed(coros)`** yields results as they finish (fast ones first) instead of in submission order, and **`asyncio.TaskGroup`** (3.11+) is the modern structured-concurrency replacement for raw `gather` — child tasks complete or fail together, with grouped exceptions and no orphan tasks. Reach for `TaskGroup` in new code.

## multiprocessing — real parallelism for real CPU work

<RunnableCode title="ProcessPoolExecutor for prime checks" code={poolCode} />

Each worker is a **separate Python process** — separate interpreter, separate memory, separate GIL — so 4 processes on 4 cores really run 4 CPU-bound Python jobs in parallel. That's the only way to get true speedup from pure-Python CPU work. The costs you must budget for:

- **Startup.** Each worker boots a fresh interpreter (~50–150 ms; more on macOS, where `spawn` is the default start method). Use processes for work measured in seconds, not milliseconds.
- **Memory.** Each worker re-imports your libraries — NumPy/pandas can be 100+ MB per worker at startup.
- **Serialization.** Data crosses process boundaries via `pickle`. Shipping a 1 GB DataFrame means 1 GB of pickling, IPC, and unpickling. For huge data, `multiprocessing.shared_memory` or memory-mapped files skip the copy.
- **No shared state.** A dict mutated in one worker is invisible to others. Design work to be embarrassingly parallel: independent input in, small result out.

Sweet spot: CPU-heavy Python on chunks of independent data — feature engineering loops, custom scoring, hashing, simulations. Bad fit: anything whose real work is in NumPy/PyTorch/BLAS — those already parallelize internally, and processes just add coordination overhead.

## concurrent.futures — one interface for both

<CodeBlock language="python" title="ThreadPoolExecutor / ProcessPoolExecutor — same shape" code={`from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

# I/O-bound → threads
with ThreadPoolExecutor(max_workers=16) as pool:
    for result in pool.map(fetch_url, urls):
        handle(result)

# CPU-bound → processes
with ProcessPoolExecutor(max_workers=4) as pool:
    for result in pool.map(is_prime, big_numbers):
        handle(result)

# Non-blocking submit: get a Future you can query
with ThreadPoolExecutor() as pool:
    futures = [pool.submit(fetch_url, u) for u in urls]
    for f in futures:
        if f.done():
            print(f.result())`} />

This is usually the right place to start. It hides raw thread/process construction, propagates worker exceptions on `.result()`, and lets you swap thread↔process by changing one class name. `.map` for embarrassingly-parallel work; `.submit` when you want individual futures. Rule of thumb: sync-library I/O at modest concurrency → `ThreadPoolExecutor`; CPU-bound pure Python → `ProcessPoolExecutor`; reach for asyncio when concurrency is high or the codebase is already async.

<RealWorldML title="An async RAG pipeline — every load-bearing pattern in one script">

Here's what a production RAG pipeline looks like — the pattern behind every "chat with your docs" application. It combines every asyncio idiom worth knowing: gather for parallel I/O, a semaphore for rate-limiting, timeouts, TaskGroup for structured concurrency, and a typed exception.

<RunnableCode title="an async RAG pipeline sketch" code={`import asyncio, random, time

class ServiceError(RuntimeError):
    """An external service call failed after retries."""

# --- fake service clients (in production these hit real APIs)
async def embed(text: str) -> list[float]:
    await asyncio.sleep(random.uniform(0.02, 0.15))
    return [len(text)]

async def vector_search(query_vec: list[float], k: int = 3) -> list[str]:
    await asyncio.sleep(random.uniform(0.03, 0.1))
    return [f"doc-{i}" for i in range(k)]

async def fetch_doc(doc_id: str) -> str:
    await asyncio.sleep(random.uniform(0.05, 0.25))
    return f"body of {doc_id}"

async def call_llm(prompt: str) -> str:
    await asyncio.sleep(random.uniform(0.5, 1.0))
    return f"answer to: {prompt[:60]}..."

# --- the pipeline: bounded concurrency + timeouts + structured concurrency
async def answer_one(query: str, sem: asyncio.Semaphore) -> str:
    """Answer a single query. Uses the semaphore to cap concurrent LLM calls."""
    try:
        q_vec = await asyncio.wait_for(embed(query), timeout=2.0)
        doc_ids = await asyncio.wait_for(vector_search(q_vec, k=3), timeout=2.0)
        # fetch the k doc bodies in parallel
        bodies = await asyncio.gather(*(fetch_doc(d) for d in doc_ids))
        prompt = f"Q: {query}\\nContext:\\n" + "\\n".join(bodies)
        async with sem:                          # rate-limit LLM concurrency
            return await asyncio.wait_for(call_llm(prompt), timeout=10.0)
    except asyncio.TimeoutError as e:
        raise ServiceError(f"timeout on query: {query!r}") from e

async def answer_many(queries: list[str], llm_concurrency=3):
    """Answer many queries, but never more than llm_concurrency LLM calls in flight."""
    sem = asyncio.Semaphore(llm_concurrency)
    async with asyncio.TaskGroup() as tg:
        tasks = [tg.create_task(answer_one(q, sem)) for q in queries]
    return [t.result() for t in tasks]

async def main():
    queries = [f"what is topic {i}?" for i in range(8)]
    t = time.perf_counter()
    answers = await answer_many(queries)
    print(f"answered {len(answers)} queries in {time.perf_counter()-t:.2f}s")
    for a in answers[:3]:
        print("  ->", a)

asyncio.run(main())`} />

Line by line, why this is a genuine production shape:

1. **`embed`, `vector_search`, `fetch_doc`, `call_llm`** are the four canonical RAG steps. In production they hit external services (embedding API, Pinecone/pgvector, S3, LLM API) — all I/O-bound, all `async def`.
2. **`answer_one`** is one query's whole journey. `asyncio.wait_for(...)` puts a hard timeout on every external call so a hung request can't freeze the pipeline; `gather` fetches all retrieved documents in parallel, saving hundreds of milliseconds over a sequential loop.
3. **`async with sem:` around the LLM call.** The LLM API is expensive and rate-limited; the semaphore caps in-flight calls regardless of how many queries are being answered.
4. **`asyncio.TaskGroup`** gives structured concurrency: if any query fails, siblings are cancelled and the exceptions are grouped at the `async with` exit — no orphan tasks, no cleanup bugs.
5. **`ServiceError`** is domain-typed, so a caller can catch it specifically (fall back to a cached answer) without hiding unrelated bugs.

You'll see this exact shape in LangChain's async pipelines, LlamaIndex's async retrieval, and every commercial LLM app's inference backend. The pattern is stable because the problem is stable — I/O-bound calls to expensive rate-limited services under latency budgets.

<Summary>

- The GIL allows one thread of Python bytecode at a time — but waiting on I/O releases it, so threads work for I/O and do nothing for CPU-bound Python.
- I/O-bound → asyncio (async libraries, scales to 100k+ concurrent ops) or `ThreadPoolExecutor` (sync libraries, up to ~1k).
- CPU-bound pure Python → `ProcessPoolExecutor`: each process has its own interpreter and GIL. Budget for startup, memory, and pickling costs — batch small tasks.
- NumPy/PyTorch/BLAS already use all your cores at the C level — don't wrap them in Python-level concurrency.
- asyncio's foot-gun: one sync blocking call freezes the whole loop. Async libraries everywhere, or `await asyncio.to_thread(...)` to escape.
- The universal LLM-app shape: `gather` + `Semaphore(n)` + `wait_for` timeouts + retry — with `TaskGroup` for structured concurrency in new code.

</Summary>

                try:
