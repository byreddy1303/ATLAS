<Motivation title="The boring hour that saves the next six months">

The default outcome of "just start coding" Python is a machine with four Python installs, a `site-packages` museum of conflicting numpy versions, and repos that take forty minutes of dependency Tetris before anything runs. It compounds brutally in ML: PyTorch pinned to a CUDA version, numpy pinned because 2.0 broke things, scikit-learn pinned to whatever pickled your model six months ago. One careless `pip install` bricks a working project; papers you try to reproduce won't even start.

In 2024 Astral shipped **uv** — one Rust binary that replaces pip, virtualenv, pyenv, pip-tools, and poetry, 10–100× faster than any of them — and **Ruff**, one binary replacing black, isort, flake8, pyupgrade, and most of pylint. With **pytest** and **git**, that's the whole modern workstation: five tools, one config file, no ceremony. Every serious ML team has adopted this stack or is migrating to it. Set it up once today; never have another "why does this only work on Sarah's machine" week.

</Motivation>

<Intuition title="A venv is just a folder, and uv is the thing that builds and runs it">

Strip the mystique: a **virtual environment** is a folder containing a Python interpreter (usually a symlink to a real one) and a `site-packages/` directory holding *this project's* libraries. Nothing else — no daemon, no registry. "Activating" one is a `PATH` trick: `PATH` is the ordered list of folders your shell searches for executables, and activation just prepends the venv's `bin/` so `python` resolves to the venv's Python. `uv run` skips the trick entirely and invokes the venv's Python directly — same effect, zero ceremony, works from any shell.

Every project gets its own venv, so `numpy 1.26` for one course and `numpy 2.1` for another coexist without ever meeting. **uv** builds the sandbox, fills it, and runs commands inside it; `pyproject.toml` declares what you want, `uv.lock` records exactly what you got. That one sentence — *uv reads `pyproject.toml`, writes `uv.lock`, populates `.venv/`, and `uv run` executes inside it* — is the entire mental model.

</Intuition>

## The workstation in seven commands

<CodeBlock language="bash" title="from empty terminal to running ML project" code={setupCode} />

Notice what's *missing*: no `activate`, no `pip install --upgrade pip`, no `python -m venv`, no `deactivate`. `uv add` writes the dependency into `pyproject.toml` **and** installs it in one step — so the config file and reality can't drift apart. `uv run` finds the venv adjacent to `pyproject.toml`, executes inside it, and exits cleanly. And `uv python install 3.12` gives uv its own private interpreter downloads — your system Python (which the OS itself uses) is never touched.

When old tutorials tell you to `pip install`, `python -m venv`, `pyenv install`, or `poetry add` — mentally translate: it's all `uv` now. Same for `black`/`isort`/`flake8` → `ruff`.

## What's on disk — the anatomy

<CodeBlock language="text" title="the project's file tree" code={treeExample} />

`uv init` did nothing magical — it created the folder, wrote a stub `pyproject.toml`, picked (or downloaded) an interpreter matching `.python-version`, created `.venv/` with a symlinked Python and empty `site-packages/`, and scaffolded `src/my_atlas_lab/` with an `__init__.py`. All filesystem operations, under a second.

The things worth naming: **`.venv/` is disposable** — delete it any time; `uv sync` rebuilds it from the two committed files. Never commit it (gigabytes, machine-specific binaries). **The `src/` layout** keeps your package (a folder of modules — single `.py` files — marked importable by `__init__.py`) *inside* `src/` rather than at repo root, which prevents "works because the current directory happens to be importable" bugs that only surface in CI. And after `uv sync`, your own package is installed in **editable mode**: edits to `src/.../math.py` take effect immediately, no reinstall — which is why `uv run pytest` just works after every edit.

## pyproject.toml — the contract

<CodeBlock language="toml" title="the complete contract, one file" code={pyprojectExample} />

Everything about the project lives in **one file** — not scattered across `requirements.txt`, `setup.py`, `.flake8`, and a Makefile like the old days. **`[project]`** is identity: name, version, minimum Python, and dependencies at their *loosest acceptable* bounds (`numpy>=1.26` — the lockfile handles exactness). **`[project.optional-dependencies]`** holds what users don't need but developers do (pytest, ruff) — added via `uv add --dev`. **`[tool.ruff]`** — every modern tool reads its config from a `[tool.<name>]` table here, so formatter, linter, and eventually pytest/mypy settings all live in this single file. Every modern Python repo you will ever clone has this file at its root; being able to read one top-to-bottom *is* the skill.

## The two-file reproducibility guarantee

<CodeBlock language="bash" title="the entire onboarding ritual, from scratch" code={reproExample} />

The pair does different jobs: `pyproject.toml` states your *intent* (`numpy>=1.26`); **`uv.lock`** records the *exact resolved outcome* — every package, every transitive dependency, precise versions and hashes. Without the lockfile committed, two teammates running `uv sync` on Tuesday and Friday can get different versions (a patch released Wednesday), and "works on my machine" is back. With it, `uv sync` reproduces the same bytes on any machine — upgrades become explicit events (`uv lock --upgrade numpy`, commit the diff), never accidents.

This is not team hygiene alone — it's how ML survives time. A pickled model saved eight months ago can fail to unpickle under a newer scikit-learn; with `uv.lock` committed alongside the checkpoint, `uv sync` returns you to *exactly* the environment that produced it. Same story in CI: every serious ML pipeline starts with `uv sync` (often `--frozen`, which fails loudly if lock and config disagree) before running tests or training.

## Ruff and pytest — the quality loop

<CodeBlock language="python" title="tests/test_math.py — tests as executable documentation" code={pytestExample} />

**Ruff** is two reflexes: `uv run ruff format .` (canonical layout — replaces black + isort) and `uv run ruff check .` (catches bugs-in-waiting: unused imports, undefined names, outdated idioms — replaces flake8 + pyupgrade + most of pylint). Run both before every commit until it's muscle memory.

**pytest** discovers any `tests/test_*.py` file and runs every `test_*` function; a bare `assert` that's False is a failure, with pytest printing the offending values. `@pytest.mark.parametrize` turns one test body into a table of cases — three inputs, three independent tests, one function. `uv run pytest -q` is the whole invocation. Tests earn their keep the first time you "improve" something and a red test catches what you broke.

## The rules that keep it clean

- **Never `pip install` into system Python.** The OS uses it; breaking it breaks tools you didn't know existed. Every install goes through `uv add` inside a project.
- **Notebooks are exploration, not production.** They hide execution order (cell 3 depending on a deleted cell 7), diff terribly, and can't be tested. Prototype in a notebook, then move working code into `src/<package>/*.py` and import it back — the notebook stays a thin driver over tested modules. Jupyter itself is a dev dep: `uv add --dev jupyter`, `uv run jupyter lab` (the kernel picks up the project venv automatically).
- **Conda's one remaining niche** is research clusters needing non-Python native builds you can't get as wheels. Know it exists; don't mix it with uv in one project.

The payoff cell: what a working "first model" looks like once the workstation is right — the exact shape of every classical-ML pipeline: **load → split → fit → predict → evaluate**.

<RunnableCode title="an iris classifier — the 'hello world' of ML" code={irisCode} packages={["scikit-learn"]} />

Line by line, connected back to the setup:

1. `load_iris(return_X_y=True)` — the classic 150-flower dataset (4 measurements, 3 species). The import works only because `uv add scikit-learn` put it in *this project's* venv.
2. `train_test_split(..., random_state=42)` — deterministic 75/25 partition. The fixed seed means the same split every run: reproducibility lives inside training scripts, not just around them.
3. `LogisticRegression(...).fit(Xtr, ytr)` — the learning step, used as a black box today; Tier 1 opens it (it's dot products and gradient steps all the way down).
4. `clf.predict(Xte)` — a species guess for each of the 38 held-out flowers the model never saw.
5. `accuracy_score` + `confusion_matrix` — the two most-run diagnostics in ML: the top-line "how often right," and the per-class breakdown showing *which* species get confused.

Why this chapter had to come first: if any dependency were mispinned — sklearn changed an API, numpy broke compatibility — this cell would error or, worse, silently produce different numbers. That's the ML reproducibility crisis in miniature, and `pyproject.toml` + `uv.lock` are the two-file vaccine.

<Summary>

- **One project = one folder**: `pyproject.toml` (intent) + `uv.lock` (exact truth) + `src/` layout + disposable `.venv/`.
- **The daily loop**: `uv init` scaffolds; `uv add` records *and* installs; `uv run` executes inside the venv with no activation; `uv sync` rebuilds everything from the two files.
- **A venv is just a folder + a PATH trick** — per-project `site-packages` so versions never collide; uv keeps private interpreters so system Python stays untouched.
- **Ruff** = format (`black`+`isort`) + check (`flake8`+`pyupgrade`+most `pylint`), one binary, config in `[tool.ruff]`. **pytest** = `test_*` functions with bare asserts; `parametrize` scales one test into many.
- **Commit the recipe, not the sandwich**: `pyproject.toml` + `uv.lock` + `.python-version` + `src/` + `tests/`; never `.venv/`. Lockfile committed = teammates, CI, and future-you get identical bytes — checkpoints stay unpicklable-proof, papers stay reproducible.
- **Hygiene**: never `pip install` into system Python; notebooks are thin drivers over tested `src/` modules; onboarding forever = `git clone && uv sync && uv run pytest`.

</Summary>
