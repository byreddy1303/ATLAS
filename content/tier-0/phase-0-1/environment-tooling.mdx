<Motivation title="The boring hour that saves the next six months">

Every guide to Python starts the same way. *"Install Python 3, open a REPL, print hello world."* Fifteen minutes in you're writing loops, ignoring the environment. Six months later you have four Python installs on your Mac — the system Python (never touch it), the one Homebrew put there, the one that came with Anaconda that a course made you install, and the one you can't remember installing but suspect was pyenv. Your `site-packages` is a museum: `numpy 1.19` for one course, `numpy 1.26` because a tutorial said so, `numpy 2.1` for the newest project. Nothing works. Every fresh clone of a repo takes forty minutes of dependency Tetris before anything runs.

This is not a rare tragedy; it's the default outcome of the "just start coding" philosophy. And it *compounds* in ML. ML projects have big, heavy dependencies with strict version ties: PyTorch pinned to CUDA 12.4, NumPy pinned to 1.26 (2.0 broke things), scikit-learn pinned to whatever the pickle format was when you saved that model six months ago. One bad `pip install` and a working project is bricked. Papers you're trying to reproduce won't even *start*. This is what people mean by "the Python packaging crisis" — it's real, it costs weeks per year, and until 2024 there was no clean fix.

Then in 2024, Astral shipped **`uv`** — a Rust-native, ten-to-a-hundred-times-faster replacement for pip, virtualenv, pipenv, poetry, and pyenv, all in one binary. Combined with **Ruff** (their all-in-one formatter/linter, also Rust), **pytest** (the testing standard), and **git** (version control since forever), you now have the modern Python workstation: five tools, one config file, no ceremony. Every serious ML team in 2026 either has adopted this stack or is migrating to it. Set it up correctly *once* today and you will never have another "why does this only work on Sarah's machine" week. That is the trade this chapter offers you.

</Motivation>

<Intuition title="What a Python 'environment' actually is">

Strip away the mystique: a **virtual environment** (venv) is just a folder. It contains a Python interpreter (usually a symlink into a real one), a `site-packages/` directory holding the libraries this specific project uses, and a tiny config file. When your shell "uses" that venv, it means the very first `python` in your `PATH` points at that folder's Python. That is the *entire* mechanism. There is no daemon, no server, no registry — just a folder and a `PATH` trick.

Every project you build gets its own venv. That way `numpy 1.26` for the class you're taking and `numpy 2.1` for the tutorial you're following can coexist without ever seeing each other. The venv is the sandbox. **`uv`** is the tool that builds, populates, and runs commands inside that sandbox for you, without you having to remember activation dances.

</Intuition>

<Recall term="The PATH environment variable">

`PATH` is an ordered list of folders your shell searches to find executables. When you type `python`, the shell looks in each folder of `PATH`, top to bottom, and runs the first `python` it finds. That's how "activating" a venv works: it prepends the venv's `bin/` folder to `PATH`, so `python` now resolves to *the venv's* Python instead of the system one. `uv run` skips this dance entirely — it invokes the venv's Python directly without touching your shell's `PATH`. Same effect, zero ceremony.

</Recall>

<Recall term="Package vs module">

A **module** is a single `.py` file. A **package** is a folder containing modules, marked either by an `__init__.py` file (the classic marker) or by simply being importable (implicit namespace packages, added in Python 3.3). When you `import numpy` you're importing the package; `import numpy.linalg` reaches into a sub-package inside it. Your own project should be structured the same way — one top-level package folder inside `src/`, with sub-modules for logically-separate pieces. That's what `uv init` scaffolds for you.

</Recall>

<Recall term="Lockfile">

A lockfile is a machine-generated file that records the *exact resolved version* of every package your project uses, including every transitive dependency (the packages your packages depend on). It answers the question *"if I install this on another machine tomorrow, exactly which bytes of code will end up on disk?"* `uv.lock` is uv's lockfile. Commit it. Without it, "works on my machine" is guaranteed.

</Recall>

  <Angle type="historical" title="A short history — why so many tools existed">

  For fifteen years, Python packaging looked like this: **pip** installs packages, **virtualenv** creates sandboxes, **pyenv** manages Python versions, **pip-tools** locks dependencies, **poetry** (or **pipenv**) tries to unify the previous four, and **conda** does all of the above *but only for its own separate universe of packages*. Each tool solved a real problem in its era. Each shipped with its own configuration format. Learning Python meant learning six tools' quirks, plus the failure modes of moving between them.

  **`uv`** (Astral, 2024) is the consolidation. One Rust binary does the job of pip + virtualenv + pyenv + pip-tools + poetry, and does it faster than any of them individually. It reads `pyproject.toml` — the modern official standard — as its single source of truth, and produces `uv.lock` as its lockfile. There is nothing to configure separately. If you started Python in 2020, this feels like the future arriving early; if you're starting today, this is simply *how it works* and none of the earlier tools need to enter your daily life.

  The other consolidation is **Ruff** (also Astral, also Rust). It replaces `black` (formatting), `isort` (import sorting), `flake8` (linting), `pyupgrade` (Python-version modernization suggestions), and most of `pylint`. One binary, one config section in `pyproject.toml`, one hundredfold speedup over the tools it replaces. In modern Python, "run the formatter" and "run the linter" are one command each: `ruff format` and `ruff check`.

  </Angle>

<Derivation title="What actually happens when you type `uv init my-atlas-lab`">

The setup command feels like magic, but each step is a small, understandable filesystem operation. Trace them once and the whole tool stops being mysterious.

<Step n={1} math={<span>uv checks that the folder <code>my-atlas-lab/</code> does not already exist, then creates it.</span>}>
Fail-early check: uv refuses to overwrite an existing folder. This prevents the "oops, I just erased my previous project" class of bug. If the folder is there, uv exits with a clear error and does nothing. Good tools have short, loud failure modes.
</Step>

<Step n={2} math={<span>Writes a fresh <code>pyproject.toml</code> with default <code>[project]</code> and <code>[build-system]</code> tables.</span>}>
The default file is intentionally sparse — name, version, requires-python, an empty dependencies list. This is your project's constitution as a stub; you will edit it (or let `uv add` edit it) as the project grows. Everything downstream — sync, run, build — reads from this file.
</Step>

<Step n={3} math={<span>Reads <code>.python-version</code> (creating one at <code>3.12</code> if absent) and resolves an interpreter that satisfies it.</span>}>
If uv has already installed Python 3.12 (via `uv python install 3.12`), it uses that. If not, it downloads a self-contained Python build from the Python Build Standalone project and stashes it in `~/.local/share/uv/`. Notice: your system Python is not touched. Your Homebrew Python is not touched. uv has its own private Pythons — one less thing that can silently drift.
</Step>

<Step n={4} math={<span>Creates <code>.venv/</code> — a folder with a Python symlink and an empty <code>site-packages/</code>.</span>}>
Concretely: `.venv/bin/python` becomes a symlink to the interpreter chosen in step 3, and `.venv/lib/python3.12/site-packages/` is created empty. From now on any package installed into this project lands *here*, not in your system Python. Delete `.venv/` and step 4 reruns cleanly; the sandbox is disposable by design.
</Step>

<Step n={5} math={<span>Scaffolds a <code>src/my_atlas_lab/</code> package and an <code>__init__.py</code>.</span>}>
The `src/` layout is a convention: your code lives *inside* `src/`, not at the repo root. This is what makes `import my_atlas_lab` work only after installation — surfacing packaging bugs that would otherwise hide until you deploy. `__init__.py` marks the folder as an importable package.
</Step>

<Step n={6} math={<span>Writes a placeholder <code>uv.lock</code> and initializes a fresh <code>git init</code> (unless <code>--no-vcs</code>).</span>}>
The lockfile is empty at scaffold time because you have no dependencies yet. The moment you `uv add numpy`, uv resolves numpy + all its transitive dependencies against your Python version, records the exact versions and hashes in `uv.lock`, and writes a `[dependencies]` entry to `pyproject.toml`. Two files, always in sync, always describing the truth.
</Step>

<Step n={7} math={<span>Prints a friendly banner telling you what happened.</span>}>
That is it. The whole ceremony took under a second because uv is native code, no Python startup, no per-package HTTP round-trip. What used to be a five-command ritual (`mkdir`, `cd`, `python -m venv .venv`, `source .venv/bin/activate`, `pip install --upgrade pip`) is now one command, and the resulting state is *strictly better* — reproducible, testable, version-pinned.
</Step>

</Derivation>

<Nuance>

**Never `pip install` into your system Python.** On macOS the system Python is used by parts of the OS; on Linux the same. Random `pip install` into it can break tools you didn't know existed. If you've already done this once or twice, it's fine — but from today forward, every install goes through `uv add` inside a project.

**Notebooks (`.ipynb`) are exploration, not production.** They hide execution order (cell 3 might depend on cell 7 that you already deleted), make git diffs unreadable, and can't be tested. Rule of thumb: prototype in a notebook, then move the working code into `src/<your-package>/*.py` modules and import back into a small driver notebook. This is how professional ML teams keep their notebooks tidy — the notebook is a thin driver, the logic lives in tested modules.

**Ruff replaces many tools at once.** You may see old tutorials reference `black`, `isort`, `flake8`, `pyupgrade`, `pylint`. In 2026 you install exactly one of them: `ruff`. Its `format` subcommand replaces `black` + `isort`; its `check` subcommand replaces `flake8` + `pyupgrade` + most of `pylint`. Configuration lives in `[tool.ruff]` and `[tool.ruff.lint]` inside `pyproject.toml`.

**Lockfiles matter more than you think.** Without `uv.lock`, `uv sync` picks whatever version of each dependency satisfies your loose bounds *at the moment of running*. Two teammates who run `uv sync` on Tuesday and Friday can end up with different versions if a package released a new one on Wednesday. Commit the lockfile.

**Editable installs.** After `uv sync`, your own package is installed in *editable mode* into the venv — meaning changes to `src/my_atlas_lab/math.py` are picked up immediately without reinstall. This is the reason `uv run pytest` works right after you edit a file. If someone tells you "you have to reinstall your package after edits", they're using a stack from ten years ago.

**Notebook kernels.** If you use Jupyter, install it as a dev dep (`uv add --dev jupyter`) and run with `uv run jupyter lab`. The kernel it launches automatically picks up the project venv. Never `pip install jupyter` globally — same reason as never installing into system Python.

**Conda's remaining niche.** Conda still owns one legitimate corner: research environments where you need non-Python native libraries (CUDA-specific PyTorch builds sometimes, specialized numerical stacks) *and* you can't get them via wheels. If you're just doing normal ML work in 2026, uv covers 99% of cases. If you're doing GPU-heavy research on institutional clusters, you'll sometimes still meet Conda; know it exists, don't mix stacks in one project.

</Nuance>

<LifeLesson title="Setup discipline: a small tax that compounds into freedom">

There is a temptation — every time — to skip the boring hour. *"I'll just get something running first, I'll structure it later."* The trouble is that "later" is a promise made by a version of you with more time, and that person doesn't exist. Every messy start becomes a debt whose interest is paid in one-off bugs, weird import errors, and lost work.

The uv/Ruff/pytest/git ritual is a small **investment** that pays back every time you touch the project: another dependency added is one command; another test is one function; another machine set up is one `uv sync`. Compare this to the alternative — hand-editing `requirements.txt`, guessing why `pip install` fails on Sarah's laptop, discovering that your notebook only works if cells are run in a particular non-obvious order — and the trade is not close.

The pattern generalizes far outside coding. **A well-set-up workstation, a well-set-up morning routine, a well-set-up shared calendar** — these are all small taxes now that convert big future costs into no cost. The people who look like they get twice as much done in the same day usually just paid the setup tax where you didn't. Pay it here. Then forget you paid it, because you'll never think about your Python environment again.

</LifeLesson>

<MLConnection>

Environment discipline is where research reproducibility comes from — and reproducibility is the difference between science and folklore.

- **Every ML paper you read from 2024 forward** likely ships a `pyproject.toml` (or a `pyproject.toml` + a Conda `environment.yml`) *and a `uv.lock`*. That is why you can now sometimes clone a paper's repo and reproduce its results on your laptop in an afternoon; five years ago this was unheard of.
- **Model checkpoints have a shelf life** proportional to how tightly you've pinned the code that saved them. Save a scikit-learn model as pickle without a lockfile, come back in a year, and unpickling can fail because internal object layouts changed between versions. With `uv.lock` committed, `uv sync` returns you to *exactly* the environment that produced the checkpoint. This is career-saving.
- **CI/CD** in every serious ML shop starts with `uv sync` (or `docker build` which internally runs `uv sync`). Then the CI runs tests and, often, the training run itself. If it doesn't reproduce, the CI job fails loudly, and someone finds out on a Monday morning instead of at a Friday-afternoon demo.
- **Dockerized deployments** copy your `pyproject.toml` + `uv.lock`, run `uv sync`, and ship. Ruff runs in CI as the linter step. Pytest runs in CI as the correctness step. This is not a per-team invention; it is the industry template.
- **Research groups** with tight versioning practices publish their pinned envs alongside the paper (`uv.lock` in the paper's GitHub) so referees can rerun. Groups without them are getting harder to publish in top venues, because reviewers now genuinely test "does this compile?" before accepting.

Learn the workstation once; every ML project you touch — school, research, industry — will use variations of this same stack.
