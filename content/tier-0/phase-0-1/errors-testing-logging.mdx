export const exceptionsCode = `# Custom exceptions — cheap to write, huge for clarity of intent
class ConfigError(Exception):
    """Raised when configuration is missing or invalid."""

class MissingKeyError(ConfigError):
    """The specific config key was not found."""

class InvalidRangeError(ConfigError):
    """Value is present but outside the acceptable range."""

def get_temperature(cfg: dict) -> float:
    if "temperature" not in cfg:
        raise MissingKeyError("temperature not in config")
    t = cfg["temperature"]
    if not (0 < t <= 2):
        raise InvalidRangeError(f"temperature {t} out of range (0, 2]")
    return t

for cfg in [{"temperature": 0.7}, {"temperature": 3.0}, {}]:
    try:
        print("ok:", get_temperature(cfg))
    except MissingKeyError as e:
        print("MISSING:", e)
    except InvalidRangeError as e:
        print("OUT OF RANGE:", e)
    except ConfigError as e:                # base class — catches anything above
        print("BAD CONFIG:", e)`;

export const trainingRobustCode = `# What a robust ML training loop looks like: try/except + logging + tests
import logging, random
from dataclasses import dataclass

logging.basicConfig(level=logging.INFO, format="%(levelname)s | %(message)s")
log = logging.getLogger("train")

class TrainingDivergedError(RuntimeError):
    """Loss went to NaN/inf — abort and let the caller decide what to do."""

@dataclass
class Batch:
    x: float
    y: float

def load_batch() -> Batch:
    x = random.gauss(0, 1)
    return Batch(x=x, y=3 * x + 2 + random.gauss(0, 0.1))

def train_step(w: float, batch: Batch, lr: float) -> tuple[float, float]:
    pred = w * batch.x
    loss = (pred - batch.y) ** 2
    grad = 2 * (pred - batch.y) * batch.x
    if loss != loss:                                     # NaN check (NaN != NaN)
        raise TrainingDivergedError(f"loss is NaN at w={w}, lr={lr}")
    return w - lr * grad, loss

def train(epochs: int = 5, lr: float = 0.05):
    random.seed(0)
    w = 0.0
    for epoch in range(epochs):
        try:
            for step in range(20):
                w, loss = train_step(w, load_batch(), lr)
            log.info(f"epoch {epoch}: w={w:.3f} loss={loss:.4f}")
        except TrainingDivergedError as e:
            log.error("training diverged, aborting", exc_info=e)
            raise                                        # let caller decide

train()

# a lr that's TOO high — diverges
try:
    train(epochs=3, lr=1e6)
except TrainingDivergedError:
    log.warning("we caught the divergence at the outer level; would trigger auto-lr-halving")`;

<Motivation title="Debugging is what you'll actually do most days">

Writing new code is fun. Writing *correct* code that stays correct across a hundred future edits — and diagnosing it when it explodes at 2am on a Tuesday — is the job. The industry secret is that senior engineers don't write code faster than juniors; they spend less time on the wrong things, because they've built (or joined) systems where **errors surface cleanly, tests document behavior, and logs tell the story of what happened.**

Three tools carry that entire load. **Exceptions** are the language's way of saying "something went wrong here, and here's what and why." Handled well, they turn a stack trace from a punishment into a diagnostic. **Tests** are the code that proves your code. A well-named test says "here is what this function is *supposed* to do"; if it passes, that statement is still true, forever. **Logs** are the runtime's narrative — the story your future self reads to reconstruct what happened when the code met the real world. Every senior engineer's first move on an unfamiliar codebase is the same trio of questions: are there tests, are there logs, how do errors surface? If any answer is *no*, that's the first thing they fix.

In ML work these three tools become especially load-bearing. **Training runs are long and unattended** — a bug that surfaces two hours in must produce a diagnosable message, or you'll waste the whole run. **Data pipelines fail on edge cases** — missing columns, corrupt bytes, unexpected schemas — and the fix has to be findable, not "restart and hope." **Model behavior is stochastic** — logs are how you tell "diverged because lr too high" from "diverged because bad seed" from "diverged because someone shipped a bug last week." Every serious ML platform (Weights & Biases, MLflow, HuggingFace Trainer) is built on this exact stack: typed exceptions, tested code, structured logs. Get comfortable with it now and you save yourself years of wandering.

</Motivation>

<Intuition title="Three mental models that make debugging tractable">

- **Exceptions are values that carry a message and a stack trace.** They aren't just "error signals" — they're *messages* between one part of your code and another. Well-designed exceptions communicate not only *that* something went wrong but *what* went wrong, *where*, and *why* — structured enough that upstream code can decide (retry? fallback? abort?) based on the type.

- **Tests are executable documentation.** A test named `test_mean_of_empty_list_raises_ValueError` is a sentence in English. Any developer can read it and know what the function is supposed to do in that case. And unlike English documentation, it's checked continuously — the moment behavior drifts from the description, CI fails.

- **Logs are the model's story.** Every meaningful state transition should leave a trace: what config was used, what data was loaded, when a checkpoint was written, why a step was skipped. When something misbehaves, logs are your only view into the past — the debugger isn't attached, the training already ran. Structured logs (key-value fields) beat free-form strings because they can be filtered, aggregated, and searched in production observability tools.

Together these three form a **feedback loop**: exceptions surface bugs, tests prove fixes, logs identify what broke and where. Every serious codebase has all three. If any is missing, the missing one is where the next mystery bug will hide.

</Intuition>

<Recall term="Traceback">

When an exception isn't caught, Python prints a **traceback** — the chain of function calls that led to the failure, most-recent-call *last*. The bottom line of the traceback is the actual exception (`ValueError: ...`); the lines above it show the call stack. Read tracebacks bottom-up: start at the exception, walk *up* to see how you got there. The frame *just above* the exception is usually where the fix goes; frames further up are usually just relaying blame. This one habit — start at the bottom of the traceback — will save you thousands of hours over a career.

</Recall>

<Recall term="Exception hierarchy">

Every exception in Python inherits from `BaseException`, but almost every exception you'll actually raise or catch inherits from **`Exception`** — a subclass of `BaseException` that specifically *excludes* `KeyboardInterrupt` (Ctrl-C) and `SystemExit` (`sys.exit()`). Rule of thumb: **raise from and catch `Exception` and its subclasses; leave `BaseException` alone** so that Ctrl-C still works. Well-known standard exceptions form their own mini-hierarchy: `ValueError`, `TypeError`, `KeyError`, `IndexError`, `AttributeError`, `RuntimeError`, `IOError` (now aliased to `OSError`), and so on. Learn to reach for the right one instead of always raising `Exception`.

</Recall>

<Recall term="Log level">

The `logging` module has five standard **levels**, in increasing severity: `DEBUG` (details only useful when investigating), `INFO` (normal operation events), `WARNING` (something odd but recoverable), `ERROR` (something failed but the program continues), `CRITICAL` (something failed and the program may not continue). A logger's level acts as a **filter** — messages below the current level are dropped without formatting cost. In production, INFO is usually the default; you turn on DEBUG when investigating.

</Recall>

<Angles intro={<>Five lenses on error handling and observability. Each corresponds to a real habit you'll form: raising precise exceptions, chaining causes, writing minimal expressive tests, choosing EAFP over LBYL, and emitting structured logs your future self can search.</>}>

  <Angle type="code" title="Custom exceptions communicate intent">
    <RunnableCode title="a small exception hierarchy" code={exceptionsCode} />

    Two rules make custom exceptions worth their weight:

    - **Inherit from `Exception`, never `BaseException`.** `BaseException` is the root and includes `KeyboardInterrupt` and `SystemExit`. Catching or raising at that level breaks Ctrl-C and clean shutdown.
    - **Name the class after the *concept*, not the mechanism.** `MissingKeyError` says what happened. `Exception("bad config")` says nothing structured. When a caller wants to react to *specifically* missing keys, they can `except MissingKeyError` — but if you'd raised a bare `Exception`, they'd have to string-match.

    Notice the shape in the example: a base `ConfigError`, with `MissingKeyError` and `InvalidRangeError` as more specific subclasses. Now the caller can catch at whatever granularity they want:

    - `except MissingKeyError:` — only handle missing keys.
    - `except InvalidRangeError:` — only handle out-of-range.
    - `except ConfigError:` — handle any config problem generically (the base class catches all subclasses).

    This is called an **exception hierarchy**, and every good library (requests, sqlalchemy, pytorch) has one. It's the difference between *forcing* a caller to catch-and-string-match versus *offering* a menu of precision levels they can choose from.

  </Angle>

  <Angle type="physical" title="`raise X from Y` — chain the cause into the traceback">
    <RunnableCode title="the wrapping pattern for domain-specific errors" code={raiseFromCode} />

    A common design need: your code calls a low-level API (a filesystem, a network library, a database driver), and that API raises a low-level exception (`PermissionError`, `ConnectionError`, `IntegrityError`). You don't want to leak those *domain-level* to your callers — they should see something like `ModelLoadError`, not `PermissionError`. But you also don't want to *hide* the original cause: if the loader failed because of permissions, the debugger needs to see that.

    `raise ModelLoadError(...) from e` is the pattern. It sets `ModelLoadError.__cause__ = e`, and Python's traceback formatter prints both: "The above exception was the direct cause of the following exception:". Your caller sees `ModelLoadError`; when they print the full traceback, they see `PermissionError` too. **Best of both worlds — abstract at the interface, transparent for debugging.**

    Two variants worth knowing: `raise X from None` explicitly suppresses the chain (say, if the original exception was expected and you want a clean re-raise). `raise X` inside an `except` block *implicitly* chains too (`__context__` instead of `__cause__`) — Python does this for you when you re-raise inside a handler.

  </Angle>

  <Angle type="code" title="pytest — the assertion is the test">
    <RunnableCode title="tests, parametrize, fixtures" code={pytestCode} />

    pytest's genius is that it treats the language's own `assert` as the assertion primitive. You don't call `assertEqual(a, b)`; you write `assert a == b`, and if it fails, pytest *rewrites* the bytecode of the test to introspect both sides and print a helpful diff. This is why pytest tests read like ordinary Python — because they are.

    Three constructs cover 95% of what you'll write:

    - **`test_*` functions.** Any function in a `test_*.py` (or `*_test.py`) file whose name starts with `test_` is a test. That's it. No decorators, no base classes.
    - **`pytest.raises(ExceptionType, match=...)`.** Assert that a block *does* raise the expected exception, optionally with a message that matches a regex. This is how you write "does this function fail on bad input" tests.
    - **`@pytest.mark.parametrize`** turns one test function into many by looping through argument tuples. Instead of copy-pasting the same test structure with different data, you list the cases in a table. This is the single biggest readability win when testing pure functions with many edge cases.

    Beyond the basics, **fixtures** (`@pytest.fixture`) are pytest's way of injecting shared setup into tests. Any test whose parameter list contains a name matching a fixture receives the fixture's return value. This lets you build DB connections once per session, temp directories once per test, mocks once per class — no need for `setUp`/`tearDown` boilerplate. Fixtures also compose: a fixture can request other fixtures by name, and pytest resolves the dependency graph automatically.

  </Angle>

  <Angle type="algebraic" title="EAFP vs LBYL — the Python happy-path preference">

    Two styles of handling potentially-absent things:

    - **LBYL** (Look Before You Leap): `if 'key' in d: value = d['key']`.
    - **EAFP** (Easier to Ask Forgiveness than Permission): `try: value = d['key'] except KeyError: ...`.

    Python's community strongly prefers EAFP. Two reasons:

    1. **Atomicity in the face of concurrency.** With LBYL, between `if 'key' in d` and `value = d['key']`, another thread could delete the key. The check-then-use races. `try: d['key'] except KeyError:` is atomic in the single-thread sense — you either get the value or catch the exception; nothing in between.

    2. **The happy path is faster.** `try/except` is very cheap when *no* exception fires; the JIT knows this. `if 'key' in d` does a hash lookup, then `d['key']` does another hash lookup — twice the work in the happy case. In a tight loop over millions of items where the key is usually present, EAFP is measurably faster.

    LBYL wins in one specific case: when the **check is cheap** and the **action is very expensive or has side effects**. E.g., `if user.has_permission("write"): expensive_write(...)` — you want to fail before doing the expensive action, not partway through with a rollback. Otherwise, reach for EAFP.

  </Angle>

  <Angle type="computational" title="Structured logging beats `print`">
    <RunnableCode title="JSON logs with `extra={...}`" code={loggingCode} />

    `print` is the developer's default and it's fine for scratch code. In anything that will run more than once, use `logging`. Three wins:

    - **Levels.** `log.debug(...)` for details, `log.info(...)` for milestones, `log.warning(...)` for oddities, `log.error(...)` for failures, `log.exception(...)` for errors *with* traceback. One line at the top (`log.setLevel(...)`) filters everything else.
    - **Logger names.** `logging.getLogger("atlas.train")` gives you a named logger you can filter on. In production this is how you turn on DEBUG for one subsystem without drowning in everything else's DEBUG.
    - **Structured fields.** Pass `extra={"step": 42, "loss": 0.812}` and, with a JSON formatter, those fields become searchable columns in your log storage. In Datadog, Grafana Loki, CloudWatch, or Splunk, you can `filter by loss > 1.0 in last 6h across all training runs` — the exact query you can't do with a freeform `print(f"step 42, loss 0.812")`.

    The JSON formatter in the cell is ~20 lines and enough for real work. Production ML platforms use libraries like `structlog` or `loguru` that handle this and more (context binding, per-request context, redaction of secrets). But even the stdlib `logging` is a huge upgrade over `print` the moment you go past prototype.

  </Angle>

</Angles>

<Derivation title="What happens when `raise` is executed inside a `try` block">

Exceptions aren't magic. They're a controlled flow-of-control mechanism. Trace what actually happens step by step and the machinery stops being mysterious.

<Step n={1} math={<span>The <code>raise ExceptionType(msg)</code> statement creates an exception instance.</span>}>
`raise MissingKeyError("temperature not in config")` first constructs `MissingKeyError("temperature not in config")` — that's a normal function-call-like operation producing an object. This object has attributes like `args`, `__traceback__` (populated shortly), and `__cause__` (if `from Y` was used).
</Step>

<Step n={2} math={<span>Python captures the current stack frame's line info into the exception's <code>__traceback__</code>.</span>}>
The traceback is a linked list of frame objects — one per active call, from innermost to outermost. Each frame records the file, line number, and function name where the raise happened.
</Step>

<Step n={3} math={<span>Python unwinds the stack, searching for a matching <code>except</code> clause.</span>}>
Starting from the current frame, Python walks *outward*. At each frame, it checks for an active `try` block whose `except` clauses could catch this exception. A clause matches if the exception is an instance of the listed type (checked via `isinstance`, so subclasses match parent-class handlers too).
</Step>

<Step n={4} math={<span>The first matching <code>except</code> clause receives the exception.</span>}>
Python jumps into the matched `except` block, binding the exception to the `as e:` name (if any). All frames it unwound through are gone — their local variables freed, their execution abandoned. The exception is now "handled" (though the handler can still re-raise).
</Step>

<Step n={5} math={<span>If no handler matches by the time the stack is empty, the interpreter itself catches it.</span>}>
Python's top-level exception hook prints the traceback and the exception message, then exits with a non-zero code. In an interactive REPL, it prints the traceback and returns you to the prompt. In a Jupyter cell, it prints the traceback in the cell output and the notebook keeps running. Same mechanism, different environment.
</Step>

<Step n={6} math={<span>Handler options: swallow, re-raise, or wrap.</span>}>
Inside the `except` block, three moves are canonical. **Swallow** — do nothing more; the exception is gone. **Re-raise** — `raise` alone (no arguments) preserves the original with its chain intact. **Wrap** — `raise MyError(...) from e` chains the original as `__cause__` and lets a caller see both.
</Step>

<Step n={7} math={<span>The <code>finally</code> clause always runs, even on <code>raise</code>.</span>}>
If the `try` had a `finally` block, it runs regardless of whether the try succeeded, raised and was caught, raised and *wasn't* caught, or returned. This is why `finally` is used for cleanup that must happen no matter what — closing files, releasing locks, restoring state. The modern replacement, however, is a `with` block whose context manager handles cleanup for you.
</Step>

</Derivation>

**Where this shape shows up in the real world:**

- **PyTorch Lightning's `Trainer`** wraps every training step in a try/except that logs and can auto-checkpoint before propagating.
- **HuggingFace's `Trainer.train`** logs metrics every N steps via a structured logger, catches OOM errors and can auto-reduce batch size.
- **Ray Tune** catches trial exceptions and reports them to a central scheduler, which can restart failed trials.

<Nuance>

**Don't catch bare `except:`.** It catches `KeyboardInterrupt`, `SystemExit`, and every subclass of `BaseException`. Suddenly Ctrl-C is caught and swallowed, and your users can't stop the program. Catch specific types, or `except Exception:` (which excludes those two). If you must catch broadly (e.g., a top-level supervisor), catch `Exception` explicitly.

**Assertions are not error handling.** `assert x > 0` is stripped when Python runs with `-O` (optimization flag). Use `assert` only for **invariants during development** — things you're saying "this should never be false; if it is, my logic is broken." For real user-facing error paths (bad input, missing config), raise a proper exception.

**Logging is expensive if you format eagerly.** `log.debug(f"tensor: {big_tensor}")` interpolates the tensor into a string *every time the line executes*, even if DEBUG is disabled. Use `log.debug("tensor: %s", big_tensor)` (older style) or wrap the log call in a level check. `logging`'s lazy `%s` formatting only pays cost when the message will actually be emitted.

**pytest fixtures are scoped.** Default scope is `function` (fresh per test); `class`, `module`, `session` scopes reuse across bigger units. Use `session` for expensive setup like DB connections that shouldn't be rebuilt per test. Use `function` (the default) when a test could taint the fixture state for the next one.

**`pytest.raises(match=...)`** takes a regex, not a literal string. Escape metacharacters if you're matching a specific value.

**Re-raise inside an `except` with a bare `raise`.** `raise` alone (no argument) re-raises the *current* exception, preserving its traceback. `raise e` (with the caught exception) works but is subtly different — it can double-up the chain. Prefer the bare `raise`.

**Chained exceptions read bottom-up.** `raise X from Y` prints Y first, then "The above exception was the direct cause of the following exception:", then X. Read the *bottom* exception to find the domain-level error your code raised; read *up* to find the root cause.

**Log rotation.** Production log files must rotate. `logging.handlers.RotatingFileHandler` (size-based) and `TimedRotatingFileHandler` (time-based) are the stdlib options. In containerized environments, log to stdout and let the platform (Kubernetes, Docker) rotate for you.

**Sensitive data in logs.** Never log passwords, API keys, or personally identifiable information. Structured loggers make this easier because you can whitelist which `extra` fields are safe to log. Consider automated redaction (`structlog`, `loguru`) if the codebase has many hands.

**pytest markers.** `@pytest.mark.slow`, `@pytest.mark.gpu` let you tag tests and run subsets: `uv run pytest -m "not slow"`. Great for keeping the fast test suite fast.

</Nuance>

<RealWorldML title="Retry-with-backoff for a flaky embedding API">

Almost every production ML system depends on an external service — an embedding API, a model gateway, a vector database. These services rate-limit, occasionally return 5xx errors, and occasionally just time out. Naïve code fails every N-hundredth call, silently drops that request, and produces mysterious downstream bugs. The industry-standard fix is **retry with exponential backoff, structured logging of each attempt, and a typed exception for permanent failures.**

<RunnableCode title="A production-shape retry decorator" code={`import logging, random, time
from functools import wraps

logging.basicConfig(level=logging.INFO, format="%(levelname)s | %(message)s")
log = logging.getLogger("embed")

class APIError(Exception):
    """Permanent error from the embedding API."""

class TemporaryAPIError(APIError):
    """Transient error — safe to retry."""

def with_backoff(*, tries=4, base=0.1, cap=2.0, on=TemporaryAPIError):
    """Retry a function with exponential backoff and jitter."""
    def decorator(fn):
        @wraps(fn)
        def wrapper(*args, **kwargs):
            last = None
            for attempt in range(1, tries + 1):
                try:
                    return fn(*args, **kwargs)
                except on as e:
                    last = e
                    sleep_for = min(cap, base * 2 ** (attempt - 1))
                    sleep_for += random.random() * base   # jitter
                    log.warning(
                        f"attempt {attempt}/{tries} failed: {e} — sleeping {sleep_for:.2f}s"
                    )
                    time.sleep(sleep_for)
            log.error(f"gave up after {tries} attempts, re-raising last error")
            raise last
        return wrapper
    return decorator

# --- fake embedding API that fails the first 2 times, then succeeds
_state = {"n": 0}

@with_backoff(tries=4, base=0.05, cap=0.5)
def embed(text: str) -> list[float]:
    _state["n"] += 1
    if _state["n"] < 3:
        raise TemporaryAPIError(f"503 service unavailable (call {_state['n']})")
    log.info(f"embed('{text}') succeeded on call {_state['n']}")
    return [0.1, 0.2, 0.3]

vec = embed("hello world")
print("got vector:", vec)`} />

**Line by line, why this is the actual pattern shipped in production:**

1. **A two-level exception hierarchy** — `APIError` (permanent) and `TemporaryAPIError` (transient). The retry decorator only catches the temporary kind; permanent errors bubble up immediately (no point retrying a `401 Unauthorized`).

2. **Exponential backoff with a cap.** Wait 0.1s, then 0.2s, then 0.4s, ... capped at `cap`. This is the standard TCP-inspired backoff that gives the downstream service time to recover without hammering it.

3. **Jitter.** Adding a small random component prevents "thundering herds" — if 1000 clients all fail at t=0 and all wait exactly 100ms, they all retry at t=100ms simultaneously and knock the service over again. Jitter spreads the retries out.

4. **Structured logging per attempt.** Every retry emits a `WARNING` with the attempt number and sleep time. In production, aggregating these tells you which endpoints are flakiest, which retry counts are exhausted, and whether your SLO budget is being burned.

5. **Give up cleanly.** After `tries` attempts, re-raise the last exception. The caller now knows this operation is genuinely broken (not just slow) and can decide policy — fall back to a different model, queue for later, alert on-call.

**Where you'll see this shape in production ML code:** OpenAI/Anthropic API clients wrap every call in retry-with-backoff. HuggingFace Inference API SDK does it. Every LangChain LLM adapter does it. Every embedding-server SDK does it. **The pattern is universal because the failure mode is universal.** Ship this decorator once and you protect every downstream service call in your system.

</RealWorldML>

import logging

def retry(n=3, on=Exception, log=None):
    log = log or logging.getLogger("retry")

    def decorator(fn):
        @wraps(fn)
        def wrapper(*args, **kwargs):
            last = None
            for attempt in range(1, n + 1):
                wrapper.stats["attempts"] += 1
                try:
                    return fn(*args, **kwargs)
                except on as e:
                    last = e
                    log.warning(

                        extra={"attempt": attempt, "of": n, "err": str(e)},
                    )
            raise last
        wrapper.stats = {"attempts": 0}
        return wrapper
    return decorator

    state["calls"] += 1
    if state["calls"] < 3:
        raise RuntimeError("network blip")

<LifeLesson title="Handle failures loudly, silently — never in the middle">

Software error handling teaches a design principle whose reach is much wider than code: **failures should either be handled loudly at the boundary (where you have context to decide policy) or handled silently at the source (where the failure is recoverable and unremarkable) — but never in the middle, where you have neither context nor authority.**

Watch the pattern in a well-designed system. Deep in a function that does one small thing, a low-level failure (network glitch, disk hiccup, integer overflow) is either quietly recovered from *right there* (a small retry, a clamp to zero, a skip) or bubbled up as an exception with a clear name. The middle layers don't *guess* what to do — they don't wrap error-handling logic around every call, they don't decide whether to retry or crash. They just forward. And at the top — at the "trainer decides what to do about divergence" layer, or the "app decides what to show the user" layer, or the "operator decides whether to page on-call" layer — the decision is made *once*, with full context.

The same shape applies to teams and to personal life. Small setbacks either get *quietly absorbed* by the person best positioned (skip a set at the gym; that's a source-level fix) or *bubble up loudly* to whoever can decide policy (a project is off-track; the lead calls a scope reset). Middle-layer heroics — pretending everything is fine, patching over broken decisions, silently absorbing costs that belong elsewhere — is where teams and relationships accumulate hidden debt that later explodes. When you find yourself wrapping a mysterious failure with `try/except: pass`, or covering for a broken process at work, ask: *is this the right layer to decide what to do?* If not, forward the signal upward.

The strongest people, systems, and teams are not the ones that never fail. They're the ones where failures are **legible and load-balanced** — small failures die quietly at their source, and big ones surface loudly to the person who can act on them. The middle stays clean. That's what "error handling" really means, and it's why this chapter deserves as much care as the ones on training loops or transformers.

</LifeLesson>

<MLConnection>

Every mature ML system leans hard on the three tools in this chapter:

- **Training loops** wrap steps in typed exception handling (`OutOfMemoryError`, `NaNLossError`, `DivergenceError`) — the framework catches specific types and can react (halve batch size on OOM, checkpoint and abort on NaN, log and continue on divergent step).
- **Data pipelines** catch and log per-record parse failures with `extra={"row": row_num, "err": str(e)}` — you get a searchable log of "which rows failed" without stopping the pipeline for one bad record.
- **Model registries** raise `ModelNotFoundError` and `ModelIncompatibleError` — typed exceptions that let downstream orchestration retry, fall back, or alert.
- **Pytest is the standard test runner across every ML shop.** scikit-learn, PyTorch, transformers all publish their test suites publicly, all use pytest, all use fixtures and parametrize. Reading their tests is often the fastest way to learn a library.
- **`pytest.raises`** is how ML libraries test that `fit(X, y)` with mismatched shapes raises the right exception — "you can't train a 100-sample X against a 200-sample y."
- **Structured logs** are what Weights & Biases, MLflow, TensorBoard, and Comet ingest. Every metric event is a JSON record with a step, a metric name, and a value. This chapter's `extra={...}` pattern is literally the shape that flows into those systems.
- **Retry decorators with backoff** wrap every external-service call in production LLM pipelines — OpenAI, Anthropic, HuggingFace Inference, embedding APIs, vector DBs. All of them rate-limit; all of them occasionally 5xx. The pattern is universal.
- **Log-based observability** in ML platforms (Datadog for latency, Loki for logs, Sentry for exceptions) all consume structured logs and exception chains. The structured practices in this chapter are what those tools need to work well.
- **CI/CD gates on tests.** Every push that fails the `pytest` step blocks merge. Every ML repo of any size has hundreds of tests that assert dataset shapes, loss values on tiny examples, deterministic behavior with seeded RNG.

Errors, tests, and logs aren't the exciting part of ML. They're the part that makes the exciting parts possible.
