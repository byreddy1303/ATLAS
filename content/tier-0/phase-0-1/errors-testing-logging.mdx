export const exceptionsCode = `# Custom exceptions — cheap to write, huge for clarity of intent
class ConfigError(Exception):
    """Raised when configuration is missing or invalid."""

class MissingKeyError(ConfigError):
    """The specific config key was not found."""

class InvalidRangeError(ConfigError):
    """Value is present but outside the acceptable range."""

def get_temperature(cfg: dict) -> float:
    if "temperature" not in cfg:
        raise MissingKeyError("temperature not in config")
    t = cfg["temperature"]
    if not (0 < t <= 2):
        raise InvalidRangeError(f"temperature {t} out of range (0, 2]")
    return t

for cfg in [{"temperature": 0.7}, {"temperature": 3.0}, {}]:
    try:
        print("ok:", get_temperature(cfg))
    except MissingKeyError as e:
        print("MISSING:", e)
    except InvalidRangeError as e:
        print("OUT OF RANGE:", e)
    except ConfigError as e:                # base class — catches anything above
        print("BAD CONFIG:", e)`;

class JsonFormatter(logging.Formatter):
    """Emit each record as one line of JSON with any extra={} fields."""
    RESERVED = {"name","msg","args","levelname","levelno","pathname","filename",
                "module","exc_info","exc_text","stack_info","lineno","funcName",
                "created","msecs","relativeCreated","thread","threadName",

import logging, random
from dataclasses import dataclass

logging.basicConfig(level=logging.INFO, format="%(levelname)s | %(message)s")
log = logging.getLogger("train")

class TrainingDivergedError(RuntimeError):
    """Loss went to NaN/inf — abort and let the caller decide what to do."""

@dataclass
class Batch:
    x: float
    y: float

def load_batch() -> Batch:
    x = random.gauss(0, 1)
    return Batch(x=x, y=3 * x + 2 + random.gauss(0, 0.1))

def train_step(w: float, batch: Batch, lr: float) -> tuple[float, float]:
    pred = w * batch.x
    loss = (pred - batch.y) ** 2
    grad = 2 * (pred - batch.y) * batch.x
    if loss != loss:                                     # NaN check (NaN != NaN)
        raise TrainingDivergedError(f"loss is NaN at w={w}, lr={lr}")
    return w - lr * grad, loss

def train(epochs: int = 5, lr: float = 0.05):
    random.seed(0)
    w = 0.0
    for epoch in range(epochs):
        try:
            for step in range(20):
                w, loss = train_step(w, load_batch(), lr)
            log.info(f"epoch {epoch}: w={w:.3f} loss={loss:.4f}")
        except TrainingDivergedError as e:
            log.error("training diverged, aborting", exc_info=e)
            raise                                        # let caller decide

train()

# a lr that's TOO high — diverges
try:
    train(epochs=3, lr=1e6)
except TrainingDivergedError:
    log.warning("we caught the divergence at the outer level; would trigger auto-lr-halving")`;

<Motivation title="Debugging is what you'll actually do most days">

Writing new code is fun. Diagnosing it when it explodes at 2am is the job — and three tools carry that entire load. **Exceptions** turn "something went wrong" into a typed, catchable message. **Tests** are the code that proves your code, re-checked on every commit. **Logs** are the story your future self reads to reconstruct what happened when no debugger was attached.

In ML these three are load-bearing, not optional. Training runs are long and unattended — a bug two hours in must produce a diagnosable message or the whole run is wasted. Data pipelines fail on edge cases — the fix has to be findable, not "restart and hope." And every LLM app wraps its API calls in retry logic built from exactly this chapter's parts. W&B, MLflow, and HuggingFace Trainer are all built on typed exceptions, tested code, and structured logs.

</Motivation>

<Intuition title="Everything in this chapter is a message">

One mental model covers all three tools: **each one is a message from your code to a different audience.**

- An **exception** is a message to the *caller*: what went wrong, where, and why — typed, so the caller's code can decide (retry? fallback? abort?) by catching the type instead of string-matching.
- A **test** is a message to the *future editor*: "here is what this function is supposed to do." Unlike a comment, it's checked continuously — the moment behavior drifts, CI fails.
- A **log line** is a message to the *operator*: the past-tense narrative of a run. Structured key-value fields beat freeform strings because they can be filtered and aggregated later.

The three form a feedback loop — exceptions surface bugs, tests prove fixes, logs tell you where to look. If a codebase is missing one, that's where the next mystery bug hides.

</Intuition>

<Recall term="Traceback">

When an exception isn't caught, Python prints a **traceback** — the chain of calls that led to the failure, most-recent-call *last*. Read it bottom-up: the bottom line is the actual exception, and the frame just above it is usually where the fix goes. This one habit — start at the bottom — will save you thousands of hours.

</Recall>

## Custom exceptions — name the failure

<RunnableCode title="a small exception hierarchy" code={exceptionsCode} />

Two rules make custom exceptions worth their weight:

- **Inherit from `Exception`, never `BaseException`.** `BaseException` also covers `KeyboardInterrupt` (Ctrl-C) and `SystemExit`; raising or catching at that level breaks clean shutdown. Same reason you never write a bare `except:` — it swallows Ctrl-C too. Broad supervisors catch `except Exception:` explicitly.
- **Name the class after the concept, not the mechanism.** `MissingKeyError` says what happened; `Exception("bad config")` forces callers to string-match.

The base-plus-subclasses shape is an **exception hierarchy**, and every good library (requests, sqlalchemy, pytorch) has one. It offers callers a menu of precision: `except MissingKeyError:` for one case, `except ConfigError:` for any config problem — the base class catches all subclasses because matching uses `isinstance`.

What actually happens when `raise` runs:

1. The exception instance is constructed and the current call stack is captured into its `__traceback__`.
2. Python unwinds the stack outward, frame by frame, checking each active `try` for an `except` clause whose type matches (`isinstance`, so subclasses match parent handlers).
3. The first match wins; every frame unwound along the way is gone — locals freed, execution abandoned.
4. If nothing matches, the interpreter prints the traceback and exits non-zero.
5. Any `finally` block runs no matter which of those paths was taken — that's why it's used for must-happen cleanup (though a `with` block is the modern form).

For everyday raising, the built-ins `ValueError`, `TypeError`, `KeyError`, `IndexError`, `AttributeError`, and `RuntimeError` cover 90% of cases — reach for the right one instead of a bare `Exception`.

## `raise X from Y` — wrap without hiding

<RunnableCode title="the wrapping pattern for domain-specific errors" code={raiseFromCode} />

The design need: your code calls a low-level API and it raises a low-level error (`PermissionError`, `ConnectionError`). You don't want to leak that to your callers — they should see `ModelLoadError` — but you also can't *hide* the original cause, or debugging becomes archaeology.

`raise ModelLoadError(...) from e` is the pattern. It sets `__cause__ = e`, and Python's traceback formatter prints both, joined by "The above exception was the direct cause of the following exception:". Abstract at the interface, transparent for debugging. Chained tracebacks read bottom-up too: the bottom exception is your domain-level error; walk up to find the root cause.

Two variants worth knowing:

- **Bare `raise`** (no argument) inside an `except` block re-raises the current exception with its traceback intact. Prefer it over `raise e`, which can double up the chain. (Re-raising inside a handler chains implicitly via `__context__` even without `from`.)
- **`raise X from None`** explicitly suppresses the chain when the original error was expected and would only add noise.

## pytest — the assertion is the test

<RunnableCode title="tests, parametrize, fixtures" code={pytestCode} />

pytest's genius is that plain `assert` *is* the assertion primitive: it rewrites the test's bytecode so a failing `assert a == b` prints both sides and a diff. Tests read like ordinary Python because they are.

Three constructs cover 95% of what you'll write:

- **`test_*` functions** in a `test_*.py` file. That's the whole discovery rule — no decorators, no base classes.
- **`pytest.raises(ExceptionType, match=...)`** asserts a block *does* raise — the "does this fail correctly on bad input" test. `match` takes a **regex**, not a literal string; escape metacharacters.
- **`@pytest.mark.parametrize`** turns one test into many by looping through a table of argument tuples — the biggest readability win when testing pure functions with many edge cases.

**Fixtures** (`@pytest.fixture`) inject shared setup: any test whose parameter list names a fixture receives its return value, and fixtures can request other fixtures — pytest resolves the graph. No `setUp`/`tearDown` boilerplate.

One trap: `assert` is for development invariants only. Python's `-O` flag strips every assert, so `assert user.is_admin` becomes a no-op in production. User-facing checks must raise real exceptions.

## EAFP vs LBYL

Two styles for potentially-absent things: **LBYL** (Look Before You Leap — `if 'key' in d: v = d['key']`) and **EAFP** (Easier to Ask Forgiveness than Permission — `try: v = d['key']` / `except KeyError:`). Python strongly prefers EAFP, for two reasons:

1. **Atomicity.** With LBYL, the state can change between the check and the use (another thread deletes the key). The try/except either gets the value or catches — nothing in between.
2. **The happy path is faster.** `try` costs almost nothing when no exception fires; LBYL does the lookup twice.

LBYL wins in one case: the check is cheap and the action is expensive or has side effects (`if user.has_permission("write"): expensive_write(...)`). Otherwise, reach for EAFP.

## Logging — levels, names, structure

<RunnableCode title="JSON logs with `extra={...}`" code={loggingCode} />

`print` is fine for scratch code. Anything that runs more than once uses `logging`, which gives you three things `print` can't:

- **Levels.** Five, in rising severity: `DEBUG` (investigation detail), `INFO` (normal milestones), `WARNING` (odd but recoverable), `ERROR` (failed but continuing), `CRITICAL` (may not continue). A logger's level is a filter — messages below it are dropped before any formatting cost. `log.exception(...)` is `ERROR` plus the current traceback, for use inside `except` blocks.
- **Names.** `logging.getLogger("atlas.train")` gives a named logger in a dot-hierarchy — in production this is how you turn on DEBUG for one subsystem without drowning in everything else's noise.
- **Structured fields.** Pass `extra={"step": 42, "loss": 0.812}` and a JSON formatter turns those into searchable columns. In Datadog, Loki, or CloudWatch you can then query "loss &gt; 1.0 across all runs, last 6h" — impossible with a freeform `print`.

Two habits to build now: format lazily — `log.debug("tensor: %s", big_tensor)` only pays the interpolation cost if DEBUG is enabled, while an f-string pays it every time — and never log secrets (API keys, tokens, PII); structured `extra` fields make whitelisting what's safe much easier. The ~20-line JSON formatter in the cell is enough for real work; production codebases graduate to `structlog` or `loguru` for context binding and redaction.

## Putting it together — a robust training loop

<RunnableCode title="try/except + logging + custom exception, in a training loop" code={trainingRobustCode} />

Read it as a design showcase. `TrainingDivergedError` is domain-specific, so a caller can catch it without swallowing unrelated bugs. The NaN check exploits `NaN != NaN` — the only Python value not equal to itself. `log.info` at epoch boundaries writes a searchable narrative; `log.error(..., exc_info=e)` embeds the traceback for post-mortems. And the outer call catches divergence and *decides policy* — here it re-raises so a supervisor could halve the learning rate and retry.

That layering — **low-level code raises typed exceptions, a middle layer logs and forwards, an outer layer applies policy** — is the whole discipline. PyTorch Lightning wraps every step this way; HuggingFace `Trainer` catches OOM and auto-reduces batch size; Ray Tune reports trial exceptions to a scheduler that restarts them. Same shape every time.

<RealWorldML title="Retry-with-backoff for a flaky embedding API">

Almost every production ML system depends on an external service — an embedding API, a model gateway, a vector database. These services rate-limit, occasionally 5xx, occasionally time out. The industry-standard fix is **retry with exponential backoff, structured logging of each attempt, and a typed exception for permanent failures.**

<RunnableCode title="A production-shape retry decorator" code={`import logging, random, time
from functools import wraps

logging.basicConfig(level=logging.INFO, format="%(levelname)s | %(message)s")
log = logging.getLogger("embed")

class APIError(Exception):
    """Permanent error from the embedding API."""

class TemporaryAPIError(APIError):
    """Transient error — safe to retry."""

def with_backoff(*, tries=4, base=0.1, cap=2.0, on=TemporaryAPIError):
    """Retry a function with exponential backoff and jitter."""
    def decorator(fn):
        @wraps(fn)
        def wrapper(*args, **kwargs):
            last = None
            for attempt in range(1, tries + 1):
                try:
                    return fn(*args, **kwargs)
                except on as e:
                    last = e
                    sleep_for = min(cap, base * 2 ** (attempt - 1))
                    sleep_for += random.random() * base   # jitter
                    log.warning(
                        f"attempt {attempt}/{tries} failed: {e} — sleeping {sleep_for:.2f}s"
                    )
                    time.sleep(sleep_for)
            log.error(f"gave up after {tries} attempts, re-raising last error")
            raise last
        return wrapper
    return decorator

# --- fake embedding API that fails the first 2 times, then succeeds
_state = {"n": 0}

@with_backoff(tries=4, base=0.05, cap=0.5)
def embed(text: str) -> list[float]:
    _state["n"] += 1
    if _state["n"] < 3:
        raise TemporaryAPIError(f"503 service unavailable (call {_state['n']})")
    log.info(f"embed('{text}') succeeded on call {_state['n']}")
    return [0.1, 0.2, 0.3]

vec = embed("hello world")
print("got vector:", vec)`} />

Line by line, why this is the pattern shipped in production:

1. **A two-level exception hierarchy** — `APIError` (permanent) vs `TemporaryAPIError` (transient). The decorator only catches the temporary kind; permanent errors bubble up immediately (no point retrying a `401 Unauthorized`).
2. **Exponential backoff with a cap.** Wait 0.1s, then 0.2s, 0.4s, ... capped at `cap` — gives the downstream service time to recover without hammering it.
3. **Jitter.** A small random component prevents thundering herds: if 1000 clients fail at t=0 and all wait exactly 100ms, they all retry simultaneously and knock the service over again.
4. **Structured logging per attempt.** Every retry emits a `WARNING` with attempt number and sleep time — aggregated, these tell you which endpoints are flakiest and whether retry budgets are being exhausted.
5. **Give up cleanly.** After `tries` attempts, re-raise the last exception so the caller knows the operation is genuinely broken and can decide policy — fall back, queue for later, alert on-call.

OpenAI and Anthropic API clients wrap every call in this. So does the HuggingFace Inference SDK, and every LangChain LLM adapter. The pattern is universal because the failure mode is universal.

</RealWorldML>
