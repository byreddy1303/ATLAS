export const coreCode = `import pandas as pd
import numpy as np
from io import StringIO

RAW = """order_id,region,amount,status
1,North,120.0,complete
2,south ,80.5,complete
3,North,-999,complete
4,East,200.0,refunded
5,SOUTH,95.0,complete
5,SOUTH,95.0,complete
6,,60.0,complete
7,East,310.0,complete
"""

def clean(df: pd.DataFrame) -> pd.DataFrame:
    df = df.drop_duplicates()
    df["region"] = df["region"].str.strip().str.lower()
    df["amount"] = df["amount"].mask(df["amount"] < 0)   # sentinel sweep
    return df

def summarize(df: pd.DataFrame, by: str, top: int) -> pd.DataFrame:
    return (df.groupby(by, dropna=False)                 # NaN keys stay visible
              .agg(orders=("order_id", "size"), revenue=("amount", "sum"))
              .sort_values("revenue", ascending=False)
              .head(top))

df = clean(pd.read_csv(StringIO(RAW)))
print(summarize(df, "region", 5))
# every EDA-chapter move is here: dedup, normalize, sentinel->NaN, dropna=False.
# And none of it is a "script" - each piece is a callable, testable function.`;

<Motivation title="Skills become an artifact — and the artifact's shape is the shape of every ML script you'll ever run">

Everything so far ran in a notebook-like cell and vanished when you left. This chapter turns Volume 0 into a **tool**: `insights sales.csv --group-by region --top 5 --save-chart` — a command that ingests a messy CSV, cleans it with logged decisions, prints a report, writes a chart, and tells the shell whether it succeeded. Something you can install, run on a schedule, and put in front of another human.

The reason this matters for your actual destination: **every serious ML program is this exact shape.** `train.py --lr 3e-4 --epochs 10`, an eval harness with `--model` and `--dataset` flags, a data-prep job on a cron schedule — argparse front door, functions doing the work, logging narrating, exit code reporting back. Learn the anatomy on 60 lines of pandas and you've learned the anatomy of a training script before ever writing one.

It's also the difference between "I know pandas" and *proof*: a pip-installable CLI with tests is the kind of artifact you can show anyone.

</Motivation>

<Intuition title="Thin shell, thick core — a 10-line script wrapped around testable functions">

The architecture of every good command-line tool is the same sandwich. **The shell layer is thin**: argparse translates the words a human typed into typed Python values, and `main()` returns an integer the shell understands (0 = success, anything else = failure). **The core is thick and pure**: `clean(df)`, `summarize(df, by, top)`, `save_chart(report, path)` — ordinary functions that take data and return data, never touching `sys.argv`, never calling `input()`, never knowing a terminal exists.

Why the separation is the whole game: the core can be **tested** (call `clean` on a 2-row fixture — no subprocess needed), **reused** (import `summarize` from a notebook tomorrow), and **rewired** (the same core could back a web API next month). Meanwhile logging gives the tool a narrator — `print` is for the user's *report*, logging is for the *operator's* diagnosis — and the exit code makes it a good citizen of pipelines, cron, and CI. One sentence to build by: **the script is 10 lines; everything interesting is a function.**

</Intuition>

## argparse — the front door

<RunnableCode title="declare the interface; parsing, types, defaults and --help come free" code={argparseCode} />

You *declare* the interface and argparse builds the parser: a **positional** argument (`csv_path` — required, no dashes), **options** with defaults (`--group-by region`), **typed options** (`type=int` means `"3"` arrives as the integer 3 — leave it off and every argument is a string, and `head("3")` breaks), and **flags** (`action="store_true"`: present → True, absent → False). Note the rename: `--group-by` becomes `args.group_by`, dashes turning to underscores because Python identifiers can't contain dashes.

The line that makes tools testable: `parse_args(argv)` accepts a **list**. The shell normally supplies `sys.argv[1:]`, but you can hand it `["sales.csv", "--top", "3"]` from a test — which is exactly how the whole tool gets exercised without ever opening a terminal. And `--help` (plus argument validation with readable errors) is generated from your declarations: documentation that cannot drift out of sync with behavior. (Typer and click are popular third-party wrappers over this same idea with decorators; argparse is stdlib, universal, and the one you'll meet in every ML repo — learn it first.)

## The core — functions, not a script

<RunnableCode title="clean() and summarize(): the EDA chapter as two named contracts" code={coreCode} />

Resist the beginner default of one long top-to-bottom script. The core is **two functions with names that state contracts**: `clean` dedups, normalizes the category, and sweeps sentinels to NaN (all three moves from the EDA chapter — including `dropna=False` in the groupby so unlabeled rows stay *visible* in the report instead of silently vanishing). `summarize` is the groupby-chapter's named-aggregation pattern with a sort and a head.

Neither function knows about argparse, files, or terminals — `clean` takes a DataFrame and returns a DataFrame, full stop. That's what makes the next two sections possible: a function with no hidden inputs can be logged around, tested in isolation, and reused anywhere. The pitfall being dodged: a 200-line script where parsing, cleaning, and printing interleave is untestable except by running the whole thing and eyeballing the output — which is to say, untestable.

## Logging — the operator's channel

<RunnableCode title="the tool narrates its own decisions, at adjustable volume" code={logCode} />

A tool that silently drops duplicates and masks sentinels is a tool nobody can trust or debug. Logging makes every cleaning decision **narrated**: how many duplicates dropped, how many sentinels masked, how many rows had no region. Run it in six months, and the log tells you what happened to *this month's* file without touching the code.

The discipline has three parts. **Channel separation**: `print` writes the user's actual product (the report); `logging` writes the operator's diagnosis. Keep them distinct and the report stays clean enough to pipe into another program. **Levels as a volume knob**: `INFO` for normal narration, `WARNING` for "handled, but you should know" (sentinels found), `ERROR` for failures — and the threshold is set once at startup (`--verbose` → `DEBUG`), never by editing code. **Lazy formatting**: `log.info("dropped %d", n)` — logging fills in the `%d` only if the message will actually be emitted, so silenced levels cost nothing (and it's the style every production codebase enforces).

## Tests — the contract, enforced

<RunnableCode title="four fixtures, four behaviors, four asserts" code={testCode} />

Each test is the same three beats: **a tiny inline fixture** (2–3 rows, built in the test — small enough to verify by eye), **one call**, **one assertion about the contract**. Test *behaviors*, not implementation: "duplicates are dropped", "region variants collapse to one", "sentinels become NaN, *not zero*" (a zero would silently poison every sum downstream — the assertion encodes the reasoning), and the sleeper: **idempotence** — cleaning already-clean data changes nothing, which is what makes re-running a pipeline safe.

`pd.testing.assert_frame_equal` is pandas' purpose-built comparison (dtype-aware, NaN-aware — `NaN == NaN` is False, so `==` alone can't verify frames). In a real repo these functions live in `test_insights.py` and `pytest` discovers and runs anything named `test_*` — the loop at the bottom is just this page's stand-in for that. The payoff compounds: next month, when you "improve" `clean`, four green tests are the difference between confidence and hope.

<RealWorldML title="The assembled tool — argparse to exit code, end to end">

<RunnableCode title="one file: parser, core, chart, main() — then two shell runs" code={realWorldCode} captureFig />

This is the whole tool in one file, then two simulated shell invocations. The assembly, piece by piece:

- **`build_parser()`** — the interface as a function, so tests can build a fresh parser too.
- **`clean` / `summarize`** — the tested core, unchanged from their sections. Notice they still know nothing about the CLI.
- **`save_chart(report, path)`** — the matplotlib-chapter rules in miniature: makes its *own* Figure (never `plt.plot` inside a function), horizontal bars inverted so the biggest sits on top, `savefig` before anything else — **the file is the artifact**; a chart nobody can re-open never happened.
- **`main(argv=None) -> int`** — the thin shell. Parse; configure logging *once* at the volume the user asked for; do the work by calling functions; return an integer. The `argv=None` parameter is the testability trick from the argparse section: production passes nothing (argparse falls back to `sys.argv`), tests pass a list.
- **The `try/except FileNotFoundError` → `return 2`** — failure handling for a *tool*: catch the specific expected failure, log an `ERROR` a human can act on, and return non-zero. No traceback vomit, no silent `print` — the shell, cron, and CI all read that number (`echo $?`), and a pipeline like `insights sales.csv && deploy` only proceeds on 0.
- **The two runs at the bottom** — the happy path (report + chart + `exit 0`) and the failure path (clean error + `exit 2`). A tool is only done when its *failure* behavior is designed.

The missing last mile — making `insights` a real command — is packaging, and it's ~10 lines:

```toml
# pyproject.toml, next to insights.py
[project]
name = "insights"
version = "0.1.0"
dependencies = ["pandas", "matplotlib"]

[project.scripts]
insights = "insights:main"
```

`pip install -e .` reads this, installs the dependencies, and generates an `insights` command on your PATH that calls `sys.exit(main())` — which is why `main` returning an int was the right shape all along. The `-e` ("editable") flag means edits to `insights.py` take effect without reinstalling. That's the entire distance from script to installed tool.

</RealWorldML>
