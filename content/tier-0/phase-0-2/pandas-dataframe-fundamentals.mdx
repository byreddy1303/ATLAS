export const alignCode = `import pandas as pd

# THE signature pandas behavior: operations align on labels, not positions.
q1 = pd.Series({"apples": 10, "bananas": 4, "cherries": 7})
q2 = pd.Series({"bananas": 5, "apples": 12, "dates": 3})   # different order, different keys!

total = q1 + q2
print(total)
# apples   22.0   <- 10+12, matched BY LABEL even though positions differ
# bananas   9.0   <- 4+5
# cherries  NaN   <- only in q1 -> missing partner -> NaN
# dates     NaN   <- only in q2

# NumPy would have added position 0 to position 0 — silently wrong.
# pandas refuses to lie: unmatched labels become NaN, and you decide.
print()
print(q1.add(q2, fill_value=0))   # treat missing as 0 instead
#
# Alignment also happens between a DataFrame and a Series, and
# between two DataFrames — on both rows AND columns.`;

export const locIlocCode = `import pandas as pd

df = pd.DataFrame(
    {"temp": [31.5, 33.2, 30.1, 29.8, 35.0],
     "humid": [45, 38, 52, 71, 30]},
    index=["mon", "tue", "wed", "thu", "fri"],
)
print(df)
print()

# .loc = LABEL world.  .iloc = POSITION world.  Never mix them up.
print("loc['wed','temp']  :", df.loc["wed", "temp"])     # by name
print("iloc[2, 0]         :", df.iloc[2, 0])             # by position — same cell
print()

# slicing — the trap: label slices are INCLUSIVE on both ends
print(df.loc["mon":"wed"])       # mon, tue, AND wed  (3 rows)
print()
print(df.iloc[0:2])              # positions 0,1 only (Python half-open, 2 rows)
print()

# boolean masks — the workhorse of all data filtering
hot = df["temp"] > 31
print(hot)                       # a bool Series aligned to df's index
print()
print(df.loc[hot])                          # rows where mask is True
print()
print(df.loc[df["temp"] > 31, "humid"])     # filter rows AND pick a column, one loc
print()

# combining conditions: & | ~ with parentheses (not 'and'/'or'!)
print(df.loc[(df["temp"] > 30) & (df["humid"] < 50)])`;

export const pitfallCode = `import pandas as pd

df = pd.DataFrame({
    "city":  ["Austin", "Dallas", "Austin", "Houston"],
    "temp":  [31.5, 33.2, 30.1, 29.8],
})

# THE classic bug: chained indexing.  Two operations back-to-back:
#   step 1: df[df.city == "Austin"]   -> may return a COPY
#   step 2: ...["temp"] = 0           -> writes into that copy, which is thrown away
try:
    df[df["city"] == "Austin"]["temp"] = 0     # looks fine. does nothing (or warns).
except Exception as e:
    print("raised:", type(e).__name__)

print(df)          # temp is UNCHANGED for Austin rows on modern pandas
print()

# The fix: ONE indexer, row selector and column selector together.
df.loc[df["city"] == "Austin", "temp"] = 0
print(df)          # now it actually changed
print()

# Same rule for creating columns from a filtered frame:
sub = df[df["temp"] > 0]           # a filtered sub-frame
sub = sub.copy()                   # make ownership EXPLICIT
sub["flag"] = "warm"               # now safe — you own sub
print(sub)`;

<Motivation title="Every dataset you feed a model passes through a DataFrame first">

Real data is tables: user ids, dates, plan names, spend, churn flags — five columns, five *different* types. An ndarray physically can't hold that without degrading everything to `object` dtype, killing NumPy's speed. Worse, ndarrays have no row identity: delete twelve corrupt rows and every parallel array you kept is silently misaligned by twelve positions. Pandas adds exactly what tables need: **one dtype per column**, **labels that travel with the data**, and **table verbs** (filter, group, join) that replace pages of index bookkeeping.

For a GenAI engineer this is unavoidable, not optional: data preparation is 45–80% of applied ML work. Fine-tuning datasets are prompt/completion tables cleaned in pandas before becoming JSONL; eval results (model × prompt × score) land in DataFrames; RAG document metadata is tabular. Every scikit-learn tutorial starts from a clean `X` matrix — this chapter is the tool that *makes* it. And pandas is a layer *on top* of NumPy, not a replacement: each column literally contains an ndarray, so vectorization, ufuncs, and no-loops all still apply.

</Motivation>

<Intuition title="A Series is an ndarray with name tags; a DataFrame is a team of them">

Two objects. That's the whole library.

A **Series** is a 1-D ndarray plus an **index** — one label pinned to each value, like name tags on people standing in a line. Positions can change (sort, filter) but tags stay pinned. `s["alice"]` asks by *tag*; `s.iloc[3]` asks by *position in line* — two different questions.

A **DataFrame** is a dict of Series that all **share the same row index** — several columns pinned to one tag list, each column keeping its own dtype. The index buys you three things: **identity** (row 103 is customer 103 through every filter and sort), **alignment** (operations match elements by label, not position — mismatches surface as NaN instead of silently pairing strangers), and **fast lookup** (a hash table underneath, dict-speed).

One structural fact to internalize: pandas is **column-oriented**. Each column is a contiguous ndarray; rows are stitched together on demand. Column math is a single C sweep; row-by-row iteration (`df.iterrows()`) is famously slow. Think in columns, exactly like the no-loops reflex from NumPy.

</Intuition>

## Series, DataFrame, and the index

<RunnableCode title="Series and DataFrame — the two objects, opened up" code={introCode} />

Read the printed Series: left column is tags, right column is values, traveling together. The dict-of-lists you pass to `pd.DataFrame({...})` *is* the mental model — keys become column names, lists become columns, and pandas staples a `RangeIndex` (0,1,2,3) down the side because you didn't provide tags. Note `df.dtypes`: `city` is `object`, `temp` is `float64`, `humid` is `int64` — three dtypes in one table, each column internally as fast as any ndarray. `.to_numpy()` is the exit hatch back to raw NumPy.

The pitfall: `df["col"]` (single brackets) returns a 1-D **Series**; `df[["col"]]` (a list of one column) returns a 1-column **DataFrame**. scikit-learn cares: `X` should be 2-D (`df[["a", "b"]]`), `y` should be 1-D (`df["target"]`).

## Alignment — pandas matches by label, not position

<RunnableCode title="alignment — the index at work" code={alignCode} />

The single most surprising behavior for NumPy-trained eyes: `q1 + q2` matches by *tag*. The fruits are listed in different orders — pandas doesn't care; `apples` finds `apples`. Labels without a partner get `NaN` — pandas telling you *"no match found, you decide"* (`q1.add(q2, fill_value=0)` to treat missing as 0). The failure mode this prevents is NumPy's: position 0 added to position 0, producing a confident wrong number. Loud missingness beats quiet wrongness.

The pitfall hiding in the output: NaN is a *float*, so aligning integer Series produces float64 results. This float-promotion theme returns in the dtype section.

## loc, iloc, and boolean masks

<RunnableCode title="loc vs iloc vs boolean masks — the complete vocabulary" code={locIlocCode} />

Two accessors, one sentence each: **`df.loc[rows, cols]` is the label world** ("the temp of wednesday"); **`df.iloc[rows, cols]` is the position world** (pure integers, exactly like NumPy). Both take the same four selectors in each slot: single item, list, slice, or boolean mask. `df["temp"] > 31` builds a bool Series in one vectorized pass; feed it to `.loc` and add a column selector in the same bracket to filter *and* project in one line. Bare `df[...]` is shorthand for exactly two things — a column by name, or rows by mask; anything fancier gets `.loc`/`.iloc`.

Two pitfalls to burn in now:

- **`.loc` slices are inclusive on both ends** — `df.loc["mon":"wed"]` is 3 rows — while `.iloc` keeps Python's half-open rule (`df.iloc[0:2]` is 2). Reason: in label-world you usually can't name "the label after wed", so exclusive endpoints would be unusable.
- **Combine conditions with `&` `|` `~` and parentheses around each comparison.** Python's `and`/`or` try to collapse a whole Series to one truth value and raise "truth value of a Series is ambiguous" — and `&` binds *tighter* than `>`, so dropping the parentheses gives a different, wronger error.

## One indexer per write — the chained-indexing bug

<RunnableCode title="the write that vanishes — and the one-indexer fix" code={pitfallCode} />

<Recall term="View vs copy">

From the broadcasting chapter: a **view** shares the original's memory; a **copy** is independent. Pandas historically couldn't tell you which a filtered selection was — the root of this bug class. Modern pandas (Copy-on-Write) makes every selection behave as a copy.

</Recall>

The most infamous bug in pandas history looks completely innocent, so here it is desugared one interpreter step at a time:

<Derivation title="Why `df[mask]['temp'] = 0` silently does nothing">

<Step n={1} math={<code>df[df["city"] == "Austin"]["temp"] = 0</code>}>
  The line you wrote. It reads like one sentence — "set temp to 0 where city is Austin" — but Python sees two bracket operations and desugars them mechanically, left to right.
</Step>

<Step n={2} math={<code>tmp = df.__getitem__(df["city"] == "Austin")</code>}>
  First brackets: a <em>read</em>. It builds a <strong>new DataFrame</strong> of the Austin rows — under Copy-on-Write, an independent copy. Crucially, this intermediate has <em>no variable name</em>; nothing holds onto it.
</Step>

<Step n={3} math={<code>tmp.__setitem__("temp", 0)</code>}>
  Second brackets, with <code>= 0</code> attached: a <em>write</em> — but aimed at <code>tmp</code>, not <code>df</code>. The Austin rows <em>inside tmp</em> dutifully get temp = 0. <code>df</code> hasn't been touched.
</Step>

<Step n={4} math={<span>{"refcount(tmp) → 0 — the write is garbage-collected"}</span>}>
  The statement ends. <code>tmp</code> was never assigned to a name, so Python frees it — along with your freshly-written zeros. Net effect on <code>df</code>: <strong>none</strong>. Modern pandas detects the write into an unnamed intermediate and raises a <code>ChainedAssignmentError</code> warning; old pandas could only emit the vague <code>SettingWithCopyWarning</code>.
</Step>

<Step n={5} math={<code>df.loc[df["city"] == "Austin", "temp"] = 0</code>}>
  The fix: <strong>one</strong> <code>__setitem__</code> call, on an accessor bound to <code>df</code> itself, receiving rows <em>and</em> column <em>and</em> value together. No intermediate exists, so there is nothing wrong to write into.
</Step>

</Derivation>

The rule: *reads may chain freely* (`df[df.a > 0]["b"].mean()` is safe); *a write must be a single indexer on the frame you mean to change*. Second habit from the same family: when you filter a frame and plan to keep working on it, make ownership explicit — `sub = df[mask].copy()`. Under CoW the copy is lazy (no memory duplicated until one side mutates), so it costs nothing and documents intent.

## dtype discipline — category, datetime64, nullable Int64

<RunnableCode title="the 10× memory cut hiding in one astype call" code={dtypesCode} />

`read_csv` guesses dtypes conservatively, and the guesses cost real memory and speed. Three upgrades cover 90% of what matters:

- **`category`** — for low-cardinality strings. `object` stores a full Python string object per row; `category` stores each *unique* value once plus a tiny integer code per row. Run the cell: ~10× memory drop, and comparisons/groupbys get faster too. Rule: unique values ≪ rows → categorize; near-unique free text → don't.
- **`datetime64`** — for anything time-shaped. String dates can only compare alphabetically; parsed dates unlock the `.dt` accessor (`.dt.month`, `.dt.days` on differences) and all date arithmetic — the raw material of features like "days since signup".
- **Nullable `Int64`** (capital I) — for integer columns with holes. The gotcha chain: NaN is a float → an int column that gains one missing value silently becomes float64 → user IDs print as `101.0` and merges get flaky. `Int64` keeps missingness in a separate mask, so integers stay integers (`<NA>` for holes).

Two habits: treat any unexpected `object` column in `df.info(memory_usage="deep")` as a bug to investigate — it means "arbitrary Python objects at Python speed". And skip `inplace=True` everywhere: it rarely saves memory, breaks chaining, and is headed for deprecation — write `df = df.drop(columns=["junk"])`.

## read_csv is an argument list, not a default

<RunnableCode title="one messy CSV, tamed at the boundary" code={csvCode} />

Real files are hostile: junk comment lines, missingness spelled four ways, dates as strings, trailing spaces, `yes`/`no` booleans. The professional move is to *push the mess to the boundary* — declare what you know at read time:

- **`na_values=[...]`** — what "missing" looks like *in this file*; otherwise `"N/A"` is a literal string and the column silently turns `object`.
- **`parse_dates=[...]`** — the column is born datetime64; every later step can trust it.
- **`dtype={...}`** — pin what you know (`zipcode` must be a *string* — leading zeros die as int).
- **`comment="#"`, `usecols=[...]`** — skip junk lines; load only the columns you need (big files shrink at the source).

After the load: `.str.strip().str.lower()` for string chaos, `.map({"yes": True, "no": False})` for pseudo-booleans. Then — every time — the **after-load checklist**: `df.shape` (expected row count?), `df.isna().sum()` (where are the holes?), `df.duplicated(key).sum()` (is the key actually unique?). Three lines that catch the bug class which otherwise surfaces as a mysteriously wrong model metric two weeks later.

## The first-look ritual

<RunnableCode title="shape, head, info, describe — run on every dataset, forever" code={firstLookCode} />

Four read-only commands before any cleaning, plotting, or modeling: **`df.shape`** (scale check — did the load get the rows you expected?), **`df.head(3)`** (eyeball actual values — secret strings and junk columns reveal themselves in three rows), **`df.info()`** (dtype-and-missingness x-ray — `950 non-null` instantly says 50 missing), **`df.describe()`** (distribution sniff test — a `min` of −1 in an age column announces itself here, before it poisons a model).

<RealWorldML title="From messy CSV to model-ready X, y — the pipeline every ML project starts with">

<RunnableCode title="churn dataset → features → matrices for scikit-learn" code={realWorldCode} />

Every supervised-learning project has the same first mile: a raw table of business facts in, two numeric arrays out — `X` (features) and `y` (target). Models don't eat plans and dates; they eat floats. Line by line:

- `pd.read_csv(io.StringIO(raw), parse_dates=["signup_date"])` — the boundary. `io.StringIO` makes the inline string behave like a file; `parse_dates` makes date arithmetic possible two lines later.
- `df["churned"] = (df["churned"] == "yes").astype("int64")` — target encoding. The comparison is a vectorized ufunc producing a boolean Series; `.astype("int64")` gives the 1/0 integers scikit-learn expects. No loop.
- `df["tenure_days"] = (snapshot - df["signup_date"]).dt.days` — **feature engineering**: datetime minus timestamp gives a timedelta column; `.dt.days` extracts integers. "How long has this user been around" is classic churn signal.
- `df["tickets_per_month"] = ...` — a *rate* feature. Raw ticket counts mislead (old users accumulate more just by existing); dividing by tenure-in-months (30.44 = average month length) makes users comparable. Column ÷ column is elementwise and index-aligned.
- `pd.get_dummies(df, columns=["plan"], dtype="int64")` — **one-hot encoding**. A model can't multiply the string `"pro"` by a weight, so `plan` explodes into `plan_free`/`plan_pro`/`plan_team` columns of 0s and 1s. You'll meet the industrial version (`OneHotEncoder`) in the scikit-learn chapter.
- `X = df[feature_cols].to_numpy(dtype="float64")` — the handoff. Select *only* numeric feature columns, drop the labels, hand scikit-learn the bare `(10, 6)` ndarray it wants. `y = df["churned"].to_numpy()` is the 1-D target, same move.

The division of labor to remember: **pandas owns everything with labels, types, and mess; NumPy owns the final matrices; the model never sees the chaos.** The index kept every row aligned through five transformations, then retired at `.to_numpy()`.

<Summary>

- **Series = ndarray + index** (values with name tags). **DataFrame = dict of Series sharing one row index** — one dtype *per column*, which is exactly what NumPy couldn't give you.
- **The index is identity.** Labels survive filtering and sorting; operations align by label and surface mismatches as NaN — loud missingness over quiet wrongness.
- **Selection grammar:** `df.loc[rows, cols]` = label world (slices *inclusive*); `df.iloc[rows, cols]` = position world (half-open); boolean masks with `&` `|` `~` + parentheses.
- **Writes take ONE indexer:** `df.loc[mask, "col"] = value`. Chained `df[mask]["col"] = value` writes into a doomed temporary. Own your sub-frames with `.copy()`.
- **dtype discipline:** `category` for repeated strings (~10× memory), `datetime64` + `.dt` for time, nullable `Int64` for integers with holes; treat `object` columns as bugs to investigate.
- **read_csv is an argument list** (`na_values=`, `parse_dates=`, `dtype=`, `usecols=`), followed by the after-load checklist (`shape`, `isna().sum()`, `duplicated().sum()`) and the first-look ritual (`shape` → `head` → `info` → `describe`).
- **The ML handoff:** pandas owns labels, types, and mess → `df[cols].to_numpy()` → NumPy matrices → model.

</Summary>
