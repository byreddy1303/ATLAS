<Motivation title="Statistics summarize; plots reveal">

You can now compute almost anything about a table — but **nobody, including you, can read the raw output.** Human working memory holds about four numbers; your datasets hold millions. In 1973 Francis Anscombe built four small datasets with *identical* means, variances, correlations, and regression lines — plotted, they are obviously four different worlds (a clean trend, a curve, an outlier dragging a line, a vertical stack). Every statistic you'll learn calls them the same; one scatter plot each exposes the truth in half a second. You will meet real Anscombes constantly: the "flat" metric that's two populations moving oppositely, the "strong correlation" that's one outlier.

For an ML engineer, one chart family runs the show: **training curves** — train and validation loss per epoch. That picture is how you diagnose overfitting (the gap opens), underfitting (both stuck high), a broken learning rate (loss explodes). TensorBoard and Weights & Biases are, at heart, machines for drawing it; you'll build it from raw arrays in this chapter. And matplotlib specifically because it's the substrate: pandas' `.plot()`, seaborn, scikit-learn's displays — all matplotlib underneath, all returning `Axes` objects you finish with the API you learn here.

</Motivation>

<Intuition title="A Figure is a sheet of paper; an Axes is one chart drawn on it">

The **Figure** is the sheet — the canvas, the file you'll save, the window that pops up. It has a size in inches (`figsize`) and it *contains* things. The **Axes** is one complete chart on that paper: the x/y axes, the plotted lines and bars, title, labels, legend, grid, and the four border lines (**spines**). Everything you draw is an **artist** (a Line2D, a Rectangle, a Text) living on an Axes. The whole containment hierarchy: **Figure → Axes → artists.**

The first line of essentially every real matplotlib script on Earth hands you both objects:

```python
fig, ax = plt.subplots()
```

Division of labor: talk to `ax` for anything *inside* the chart (`ax.plot`, `ax.set_title`, `ax.legend`); talk to `fig` for anything about the *sheet* (`fig.savefig`, `fig.tight_layout`). Want six charts? `fig, axs = plt.subplots(2, 3)` — one sheet, six Axes, drawn on explicitly. Hold the picture and the API stops being 3,000 functions: *get the objects, then set properties on them.*

</Intuition>

## The pyplot trap — never `plt.plot` inside a function

<RunnableCode title="two chart functions, one current axes — watch them collide" code={stateCode} captureFig />

Every tutorial starts `plt.plot([1, 2, 3])` — no Figure, no Axes — and it works because `pyplot` is a **state machine**: `plt.plot` is essentially `plt.gca().plot(...)`, where `gca()` ("get current axes") consults a hidden module-level registry and **auto-creates** a figure and axes if none exist. Every `plt.title`, `plt.legend`, `plt.xlabel` walks the same chain — "…on whatever is current."

The cell stages the accident: `plot_revenue()` manufactures the current axes; `plot_signups()` finds it already exists and draws *onto the same one* — two unrelated series mashed together, the second title overwriting the first. Neither function is wrong locally; the shared hidden global did it (same disease as pandas' chained-assignment temporary: invisible shared state plus innocent calls). In the wild the bug wears worse costumes: a chart function called in a loop stacks twenty series onto one axes; notebook cells re-run out of order swap titles between charts.

The fix is a one-sentence policy the whole ecosystem converged on: **functions never touch `plt.*` for drawing — they accept an `ax` parameter and draw on it.** The *caller* owns layout (`fig, (a1, a2) = plt.subplots(1, 2); plot_revenue(a1); plot_signups(a2)`). Reserve `plt` for exactly three jobs: `plt.subplots()` to create, `plt.show()` to display, `plt.close()` to dispose.

## The five-line skeleton

<RunnableCode title="the ritual, on a loss curve" code={skeletonCode} captureFig />

Every finished chart for the next several years is this skeleton with different data:

1. **Create** — `fig, ax = plt.subplots(figsize=(w, h))`. Size in inches, picked deliberately: figsize controls how big fonts look relative to data.
2. **Draw** — `ax.plot(x, y, ...)` with style at the call: `color`, `lw`, `marker`, and crucially `label=`. Different question, different verb (`bar`, `hist`, `scatter`) — same skeleton.
3. **Label** — `ax.set_xlabel` / `set_ylabel` with units, plus a title. Non-negotiable: an unlabeled chart is readable only by present-you, who expires in about a week.
4. **Legend** — `ax.legend()` whenever two or more series share an Axes.
5. **Annotate** — `ax.annotate(text, xy=point, xytext=..., arrowprops=...)` plants a labeled arrow on *the one point that matters*. A chart states facts; the annotation states the **finding**.

Then the exit: `fig.tight_layout()` (fixes clipped labels — just always call it), and `fig.savefig("name.png", dpi=200, bbox_inches="tight")` or `plt.show()`. The `dress(ax)` helper is worth stealing as a pattern: styling you type three times becomes a function taking `ax` — the ax-passing discipline now working *for* you.

## Choose the chart from the question

<RunnableCode title="four question types → four charts, as small multiples" code={selectCode} captureFig />

The matplotlib gallery has hundreds of chart types; working data science runs on four. Ask what *question* the chart answers:

- **"How are values distributed?"** → **histogram**. One numeric column's shape: where values pile up, the tail, whether there are two humps (two populations hiding in one column). `bins=` matters: too few hides structure, too many is noise — try 20–50 and look.
- **"How do categories compare?"** → **bar**. A few named things, one number each. Bars encode *length* — which is why they must start at zero (next section).
- **"Are two quantities related?"** → **scatter**. One dot per row; correlation, clusters, outliers all visible instantly. Thousands of points: drop `alpha` to ~0.3 and shrink `s` so density reads as darkness.
- **"How does it change over time/order?"** → **line**. Lines imply *continuity between points* — exactly the claim of a trend, and exactly why a line through unordered categories is wrong.

The cell also smuggles in **small multiples** — `plt.subplots(2, 2)` returns an *array* of Axes, one honest little chart per question. Four modest panels beat one heroic chart-of-everything. Note `axs` is a 2-D **ndarray of Axes** (`axs[0, 0]` — your NumPy indexing works on charts now); `axs.flat` is the loop idiom, and beware that `subplots(1, 2)` returns a 1-D array while `subplots(1, 1)` returns a bare Axes.

## Honest scales — the axis is part of the claim

<RunnableCode title="one dataset, two stories: the truncated-axis lie, side by side with the truth" code={honestCode} captureFig />

Same two numbers — 96.1 and 97.3 — told two ways. Left: y starts at 95.8 and "ours" towers at ~4× the baseline's height — a chart that screams breakthrough. Right: y starts at zero and you see the truth, a real but modest 1.2-point gain. The axis limits are part of the claim, and the left panel's claim is false; the crime is one line: `set_ylim`.

The rule, precisely: **bar charts must start at zero** — bars encode value as *length*, and truncation clips every bar by the same amount, distorting the ratios bars exist to show. **Line charts may zoom** — lines encode value as *position*, and zooming into ±0.5° of signal is often the only way to see it; just label the axis so the zoom is visible. Three more cheap honesty habits: **compared panels share scales** (`sharey=True` — auto-scaling is per-Axes and silently lies across small multiples); **annotate the takeaway with units** (the `+1.2 pts` does the reader's arithmetic); **color is meaning, so spend it** — one accent for the thing you're arguing about, gray for context, and never meaning in color alone (~1 in 12 men can't separate red from green).

## The ecosystem speaks Axes

<RunnableCode title="pandas draws the data, matplotlib finishes the chart" code={ecosystemCode} captureFig />

The payoff of learning the object model: **every Python plotting tool draws on matplotlib Axes.** pandas' `.plot()` accessor reads the DataFrame's *index* as x, plots each *column* as a series, uses column names as legend labels — and draws onto the Axes you pass via `ax=`. Then you finish the chart with raw matplotlib on the same `ax`: titles, honest limits, annotations. ## Housekeeping — the four bugs everyone hits

- **Blank saved files:** `plt.show()` can flush/tear down the figure when its window closes, so a later `savefig` writes an empty canvas. **Save first, then show** — treat `show()` as the last line a script runs.
- **Figure leaks:** every `plt.subplots()` registers a live figure in pyplot's global registry, so Python can't garbage-collect it. A chart-per-epoch loop accumulates figures until the OOM killer arrives (matplotlib warns at 20). Pattern: create → `savefig` → **`plt.close(fig)`**, every iteration — or create once before the loop and `ax.clear()` per iteration.
- **Clipped labels:** rotated ticks or a sliced-off ylabel means layout ran with defaults — `fig.tight_layout()` fixes 95% of cases; just always call it.
- **NaN breaks the line — usefully.** `ax.plot` skips NaN points, leaving a visible gap instead of bridging missing data. A sensor that died for two days should show a hole, not a confident straight line — so keep timelines complete and set missing readings to NaN rather than deleting rows (deleted rows make `plot` connect across the gap, fabricating data). Interpolate explicitly, styled dashed, if you must bridge.

Two smaller ones: `figsize` is inches, `dpi` is dots per inch, but *fonts* are points — if text looks wrong at final size, fix figsize, not font sizes. And `ax.set_yscale("log")` is right for decade-spanning quantities (loss curves, scaling laws — a straight line on log axes means exponential/power-law change) but errors on zeros and negatives.

<Summary>

- **Two objects:** Figure = the sheet (size, saving); Axes = one chart on it (data, labels, legend, spines). Everything drawn is an artist owned by an Axes. `fig, ax = plt.subplots()` is line one, forever.
- **Never `plt.plot` inside a function.** pyplot routes calls through hidden "current figure/axes" globals (`plt.plot` ≈ `plt.gca().plot`, auto-creating on empty); functions take an explicit `ax`. Reserve `plt` for `subplots`, `show`, `close`.
- **The five-line skeleton:** create → draw (with `label=`) → label axes (with units) → legend → **annotate the finding**. Then `tight_layout` and `savefig(dpi=200, bbox_inches="tight")` *before* any `show`.
- **Chart = question:** distribution → hist · comparison → bar · relationship → scatter · trend → line. Small multiples (`plt.subplots(r, c)`) beat one chart-of-everything; `axs` is an ndarray of Axes.
- **Honest axes:** bars start at zero (length encoding); lines may zoom (position encoding) but label it; compared panels share scales (`sharex`/`sharey`); color is meaning — one accent, gray context, never meaning by color alone.
- **Housekeeping:** figures leak — `plt.close(fig)` in loops; NaN breaks lines (honestly); log scales for decade-spanning data; `figsize` controls font-to-data proportion, `dpi` controls sharpness.
- **The ecosystem speaks Axes:** `df.plot(ax=ax)`, seaborn, scikit-learn displays all draw on your canvas — specialist library for the first 80%, this chapter's API for the finishing 20%.
- **The chart that runs ML:** train/val loss per epoch — gap opening = overfitting, `argmin` of val loss = early stopping. You built it from raw arrays; TensorBoard just automates it.

</Summary>
