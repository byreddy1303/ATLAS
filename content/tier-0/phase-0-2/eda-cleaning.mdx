export const missingCode = `import pandas as pd
import numpy as np

df = pd.DataFrame({
    "age":    [23.0, 41, 25, 55, np.nan, np.nan, 27, 46],
    "plan":   ["free", "pro", "free", "pro", "team", "free", "free", "pro"],
    "spend":  [0.0, 60.0, np.nan, 99.0, 12.5, 8.0, np.nan, 75.0],
    "income": [np.nan, 82_000, np.nan, 120_000, np.nan, 55_000, np.nan, 98_000],
})

# 1 - how much is missing, per column?
print(df.isna().mean().round(2))       # fraction missing, not just count
print()

# 2 - is missingness RELATED to other columns? (the MAR question, in code)
print(df.groupby(df["income"].isna())["age"].mean().round(1))
# users who skipped the income field are ~25; those who filled it are ~47.
# Missing NOT at random -> dropping those rows would quietly age the dataset.
print()

# 3 - impute with the median, and RECORD that you did (indicator column)
df["spend_missing"] = df["spend"].isna().astype(int)
df["spend"] = df["spend"].fillna(df["spend"].median())

# 4 - group-aware imputation: fill age with the median FOR THAT PLAN...
df["age"] = df["age"].fillna(df.groupby("plan")["age"].transform("median"))
# ...with a global fallback, because a whole group can be empty (team is)
df["age"] = df["age"].fillna(df["age"].median())

export const outlierCode = `import pandas as pd
import numpy as np

rng = np.random.default_rng(0)
spend = pd.Series(rng.lognormal(mean=3.0, sigma=0.5, size=200).round(2))
spend.iloc[7]  = 4200.0    # one whale: decimal slip? genuinely big account?
spend.iloc[42] = -50.0     # impossible: negative spend

# Rule 0 - DOMAIN rules first. Impossible values are ERRORS, not outliers.
print("impossible (spend < 0):", (spend < 0).sum())
spend = spend.mask(spend < 0)          # unknowable -> honest NaN

# IQR fences - built from quartiles, which extremes can barely move
q1, q3 = spend.quantile([0.25, 0.75])
lo, hi = q1 - 1.5 * (q3 - q1), q3 + 1.5 * (q3 - q1)
flagged = spend[(spend < lo) | (spend > hi)]
print(f"IQR fences [{lo:.1f}, {hi:.1f}] flag {len(flagged)} points for review")
print()

# Why mean +/- 3*std betrays you: the whale corrupts the fence meant to catch it
z_hi_with = spend.mean() + 3 * spend.std()
no_whale = spend.drop(spend.index[7])
z_hi_without = no_whale.mean() + 3 * no_whale.std()
q1n, q3n = no_whale.quantile([0.25, 0.75])
print(f"z fence, whale in the stats: {z_hi_with:7.1f}")
print(f"z fence, whale excluded:     {z_hi_without:7.1f}   <- moved ~15x")
print(f"IQR fence, with vs without:  {hi:7.1f} vs {q3n + 1.5*(q3n-q1n):.1f}")
print()

# Tame, don't delete: clip caps the leverage; log1p tames the whole tail
clipped = spend.clip(upper=hi)
print("skew raw / clipped / log1p:",
      spend.skew().round(1), "/", clipped.skew().round(1), "/",
      np.log1p(clipped).skew().round(1))`;

<Motivation title="The model learns whatever the data says — including the lies">

A model is a mirror of its training table. Feed it ages of −999 and it learns that some customers are a thousand years young; leave user 4 duplicated and that user votes twice in every statistic; let `"Pro"`, `"pro "` and `"PRO"` stay three separate plans and the model splits one signal across three columns. **No error will be raised at any point.** The model trains, the metrics print, the numbers are just quietly wrong — and you ship a system that confidently encodes the dirt.

This is why practitioners repeat that cleaning is "60% of the job": not because it's hard, but because it's *always there* and skipping it poisons everything downstream. It matters double in GenAI work — fine-tuning sets, eval benchmarks, and RAG corpora are all tables, and duplicated rows there cause memorization, inflated eval scores (test questions leaked into training), and retrieval that returns the same document three times.

The skills in this chapter are the difference between "I ran the model" and "I can defend every number that went in."

</Motivation>

<Intuition title="Every column is a witness — interrogate it, then write the verdict down as code">

Treat a fresh dataset as testimony from a witness who might be lying. You ask each column three questions: **Is anything absent?** (missing values — including the ones disguised as −999 or `"unknown"`). **Is everything possible?** (a 250-year-old customer isn't an outlier, it's an error; a $4,200 order might be your best customer). **Is everything consistent?** (duplicated rows, `"Pro"` vs `"pro "`, dates stored as strings).

The second half of the mental model is just as important: **every fix is a decision, recorded as code.** You never edit the CSV, never fix a value by hand, never let a default silently decide for you. The cleaning script *is* the audit trail — six months later, `df["age"].mask(df["age"] < 0)` still tells everyone exactly what you did and why. Interrogate, decide, write it down.

</Intuition>

## The audit — interrogate before you touch

<Recall term="The first-look ritual">
From the DataFrame chapter: on any new table, run `shape` → `dtypes` → `head()` → `isna().sum()` → `describe()` before doing anything else. This chapter turns that ritual into a full interrogation — because the ritual's official answers routinely understate the damage.
</Recall>

<RunnableCode title="the official answers vs the truth — one messy table, fully audited" code={auditCode} />

The official report says: one missing value. The interrogation finds: two `-999` ages (a *sentinel* — a made-up code meaning "didn't answer"), one impossible age of 250, one exact duplicate row, one `"not_available"` hiding in the date column, and a plan column where the same three plans masquerade as five. `isna()` can only see genuine `NaN` — **sentinels wear valid dtypes as camouflage**, which is why `describe()` (impossible min/max, absurd mean) and `sorted(unique())` on every suspicious column are part of the audit. The `describe()` output is the smoking gun: a mean age of −197 means every downstream statistic is already poisoned.

Two closing moves: `pd.to_datetime(..., errors="coerce")` converts unparseable date-strings to `NaT` — junk becomes *honestly missing* instead of secretly corrupt. And check duplicates twice: `df.duplicated()` for exact repeated rows, `df["user_id"].duplicated()` for repeated keys — different diseases, as you'll see below.

## Missingness — three reasons data is absent, three different responses

<RunnableCode title="measure it, test whether it's random, impute with a paper trail" code={missingCode} />

Statisticians sort missingness into three patterns, and the pattern decides what you're allowed to do:

- **MCAR — missing completely at random.** A sensor occasionally drops a reading; a form sometimes fails to save. The gap is unrelated to anything. Dropping those rows loses data but introduces *no bias* — the survivors still look like the population.
- **MAR — missing at random, given other columns.** The gap is predictable from data you *have*: in the demo, income is missing almost exclusively for young users (skippers average 25, fillers 47). Dropping those rows would silently age your dataset by two decades. Instead, impute *using* the related columns — that's the group-median move.
- **MNAR — missing not at random.** The gap depends on the missing value *itself*: high earners decline to state income; struggling students skip the survey. No imputation can recover what's systematically hidden — the honest move is an **indicator column** (`spend_missing`), because here the *fact of absence is itself signal* a model can use.

You can't formally prove which pattern you have, but the groupby-on-`isna()` probe in the demo catches the common MAR cases in one line. The workflow that survives all three: impute with the **median** (robust to the outliers you haven't cleaned yet), add the indicator column *before* filling, use group medians when a related category exists — with a global fallback, because a whole group can be empty and `transform` will hand you NaN right back.

Two hard rules. **Never impute the target** — a row with a missing label is not training data; drop it. And imputation statistics are *learned from data*, so in real pipelines they must be learned from the training set only — the next chapter's Pipeline machinery exists largely to enforce that.

## Outliers — domain rules first, robust fences second

<RunnableCode title="the whale, the impossible value, and the fence that moves" code={outlierCode} />

Order of operations: **domain rules → robust detection → a decision that isn't deletion.**

*Domain rules first.* Negative spend, age 250, a timestamp from 1970 — these aren't outliers, they're **errors**, and no statistics are needed to convict them. Encode the rule (`spend.mask(spend < 0)`), route the finding back to whoever owns the data, and move on.

*Then robust fences.* The classic `mean ± 3·std` rule has a fatal flaw the demo makes visible: **the outlier inflates the very mean and std used to judge it** — with the whale included the fence sits near 900, without it near 60. The whale moves its own goalposts. **IQR fences** — `Q1 − 1.5·IQR` to `Q3 + 1.5·IQR`, the same rule behind box-plot whiskers — are built from quartiles, which barely move when one value goes berserk: the demo's IQR fence shifts by pennies. That robustness is the entire argument. (And before any formula: `spend.hist()` — a histogram shows you the tail in one glance.)

*Then decide — and deletion is the last resort.* A fence **flags for review**; it doesn't convict. The demo flags several perfectly legitimate heavy spenders alongside the whale, and in fraud detection *the outliers are the entire point* — delete them and you've deleted the signal. The gentler tools: **clip/winsorize** (`spend.clip(upper=hi)`) keeps the row but caps its leverage, and **log1p** tames a whole right-skewed tail at once (watch skew fall from double digits to near zero) — often the right "fix" for money-like columns is a transform, not a removal.

## Duplicates and category chaos

<RunnableCode title="dedup, normalize, then count — in that order" code={textCode} />

**Duplicates come in two species.** *Exact duplicate rows* are almost always mechanical (a double export, a retried upload) — `drop_duplicates()` is safe and should happen **before any statistic is computed**, or the duplicated row votes twice in every mean, median, and imputation value. *Duplicate keys with disagreeing data* (order 2 appearing twice with different amounts) are a real inconsistency — investigate which record is true; a blind drop just picks a winner at random.

**Categories lie through formatting.** `"Pro"`, `"pro "`, `"PRO"` are one plan wearing three costumes, and every tool downstream — `value_counts`, groupby, one-hot encoding — will treat them as three. The two-step cure: **normalize** (`.str.strip().str.lower()` collapses six "plans" to three), then **consolidate known variants with an explicit dict** (`.replace({"huston": "houston"})`). The dict looks manual because it *should* be — it's auditable, greppable, and reviewable in a way no fuzzy-matching magic ever is. `value_counts()` before and after is the proof the cleanup worked; run it on every categorical column you intend to use.

## Encoding categories as numbers

<RunnableCode title="one-hot, ordinal, and the fake-arithmetic trap" code={encodeCode} />

Models eat numbers, so every category must become numeric — and the encoding you pick is a claim about the world:

- **One-hot** (`pd.get_dummies`) for **nominal** categories — no order exists among `free/pro/team`, and one column per value means no fake arithmetic can be done. For linear models, pass `drop_first=True` (the dropped column is implied by the others; keeping all three creates redundancy that destabilizes coefficients).
- **Ordinal mapping** (`.map({"S": 0, "M": 1, ...})`) only when a **true order exists** — sizes, ratings, education levels — and *you* write the order down explicitly. The distances are still a modeling assumption (is XL−L the same gap as M−S?), but at least the ranking is real.
- **The trap:** `pd.factorize` or any "just assign each city a number" scheme on nominal data. `austin=0, dallas=1, houston=2` quietly claims Austin is less than Dallas and that their average is meaningful — a linear model will happily learn from that nonsense geometry. Arbitrary codes are only acceptable for things that treat them as labels, never as quantities.
- **High cardinality** (thousands of cities): one-hot explodes into thousands of columns. The simple starter is **frequency encoding** — replace each value with its count. The powerful upgrade, *target encoding* (replace each city with its average churn rate), is also the easiest way to leak your labels into your features — park it until the cross-validation chapter shows you how to do it safely.

Bigger picture: `get_dummies` is fine for learning and one-off analyses, but it has a production flaw — if next month's data is missing a plan (or has a new one), you get a *different set of columns* and the model breaks. The next chapter's `ColumnTransformer` fixes exactly that.

<RunnableCode title="audit → dedup → sentinel sweep → impute with indicators → encode" code={realWorldCode} />

<Summary>

- **The audit before the touch:** `shape`, `dtypes`, `isna().sum()`, `duplicated().sum()`, `describe()`, `sorted(unique())`. Official missingness understates the truth — **sentinels** (−999, `"unknown"`, dates-as-strings) hide inside valid dtypes; `describe()`'s impossible min/max/mean is the smoking gun.
- **Order of operations:** dedup → sentinel sweep (`mask`, `to_datetime(errors="coerce")`) → normalize text → impute → encode. Every step *before* imputation exists so the imputation statistics aren't computed from garbage.
- **Missingness has three species:** MCAR (drop is unbiased), MAR (predictable from other columns — impute with group medians via `transform`, plus a global fallback), MNAR (the gap is the signal — keep an **indicator column**). Never impute the target.
- **Outliers:** domain rules convict errors instantly; **IQR fences** flag the rest (robust — unlike mean±3σ, where the outlier moves its own goalposts). Fences flag for *review*: clip to cap leverage or `log1p` the tail before you ever delete — in fraud, outliers *are* the signal.
- **Duplicates:** exact rows → drop before any statistic; duplicate keys that disagree → investigate, don't coin-flip.
- **Categories:** normalize (`.str.strip().str.lower()`), consolidate with explicit dicts, verify with `value_counts()`. Encode: one-hot for nominal (`drop_first=True` for linear models), explicit maps for true orders, frequency encoding for high cardinality — never arbitrary integer codes.
- **Cleaning is code, not edits:** every fix is a recorded, greppable, re-runnable decision. That's what makes the resulting model defensible.

</Summary>
