export const pipeCode = `import pandas as pd
import numpy as np
from sklearn.pipeline import make_pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer
from sklearn.linear_model import LogisticRegression
from sklearn.model_selection import train_test_split

df = pd.DataFrame({
    "age":   [23, 41, 25, 55, 34, 38, 27, 46, 31, 52],
    "spend": [0.0, 60.0, np.nan, 99.0, 12.5, 8.0, np.nan, 75.0, 20.0, 88.0],
    "plan":  ["free", "pro", "free", "pro", "team", "free", "free", "pro", "team", "pro"],
    "churned": [1, 0, 1, 0, 0, 1, 1, 0, 0, 0],
})
X, y = df.drop(columns="churned"), df["churned"]

# declare WHAT happens to WHICH columns - preprocessing as configuration
prep = ColumnTransformer([
    ("num", make_pipeline(SimpleImputer(strategy="median"), StandardScaler()),
     ["age", "spend"]),
    ("cat", OneHotEncoder(handle_unknown="ignore"), ["plan"]),
])
pipe = make_pipeline(prep, LogisticRegression())
print("steps:", list(pipe.named_steps))

X_tr, X_te, y_tr, y_te = train_test_split(X, y, test_size=0.3,
                                          random_state=0, stratify=y)
pipe.fit(X_tr, y_tr)   # ONE fit: imputer, scaler, encoder, model - train-only, in order
print("test accuracy:", pipe.score(X_te, y_te).round(2))
print()

# the payoff: a RAW customer goes straight in - NaN spend, never-seen plan and all
newbie = pd.DataFrame({"age": [29], "spend": [np.nan], "plan": ["enterprise"]})
print("churn probability:", pipe.predict_proba(newbie)[0, 1].round(2))`;

export const realWorldCode = `import pandas as pd
import numpy as np
import joblib
from sklearn.pipeline import make_pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer
from sklearn.linear_model import LogisticRegression
from sklearn.model_selection import cross_val_score

# a churn table like last chapter's, synthesized at 60 users
rng = np.random.default_rng(7)
n = 60
plans = rng.choice(["free", "pro", "team"], size=n, p=[0.5, 0.3, 0.2])
age = rng.integers(19, 65, size=n).astype(float)
spend = np.where(plans == "free", 0.0, rng.lognormal(3, 0.6, n)).round(2)
spend[rng.random(n) < 0.1] = np.nan            # some users hide their spend
tickets = rng.poisson(2, n)
churn_logit = 1.2 * (plans == "free") + 0.35 * tickets - 0.02 * age - 0.6
churned = (rng.random(n) < 1 / (1 + np.exp(-churn_logit))).astype(int)

df = pd.DataFrame({"age": age, "spend": spend, "tickets": tickets,
                   "plan": plans, "churned": churned})
X, y = df.drop(columns="churned"), df["churned"]
print("data:", X.shape, "| churn rate:", y.mean().round(2))

# every cleaning decision from last chapter, frozen into one object
prep = ColumnTransformer([
    ("num", make_pipeline(SimpleImputer(strategy="median"), StandardScaler()),
     ["age", "spend", "tickets"]),
    ("cat", OneHotEncoder(handle_unknown="ignore"), ["plan"]),
])
pipe = make_pipeline(prep, LogisticRegression(max_iter=1000))

# evaluate the WHOLE pipeline: preprocessing is re-fit inside every fold
scores = cross_val_score(pipe, X, y, cv=5, scoring="accuracy")
print(f"5-fold accuracy: {scores.mean():.2f} +/- {scores.std():.2f}")

# ship it: fit on all data, save, reload, score a raw customer
pipe.fit(X, y)
joblib.dump(pipe, "churn_model.joblib")
loaded = joblib.load("churn_model.joblib")

new_customer = pd.DataFrame(
    {"age": [24.0], "spend": [np.nan], "tickets": [6], "plan": ["free"]})
p = loaded.predict_proba(new_customer)[0, 1]
print(f"new raw customer -> churn probability {p:.0%}")`;

<Motivation title="One API to rule a decade of ML — learn the grammar once, speak every library">

You don't yet know how any model works inside — that's Tier 2's job. You don't need to. scikit-learn's genius was separating the *grammar* of machine learning from the *math* of it: every model, from a one-line linear regression to a gradient-boosted forest, is driven through the same three methods — `fit`, `predict`, `transform`. Learn the grammar now, on models treated as black boxes, and every algorithm you meet later arrives with its interface already familiar.

The grammar escaped the library. XGBoost, LightGBM, and countless industrial systems clone the sklearn API; its metrics (`accuracy`, `roc_auc`, F1) are the vocabulary of every LLM eval harness; and the professional reflex of *baseline first* — before fine-tuning anything expensive, fit a logistic regression to learn whether the problem is even hard — runs entirely on what's in this chapter.

Just as important: sklearn is where the discipline lives. Train/test splits, the fit-on-train-only rule, cross-validation, pipelines that make leakage structurally impossible — these habits separate numbers you can defend from numbers that merely look good. This is the chapter where you train, evaluate, save, and reload your first real model.

</Motivation>

<Intuition title="Every estimator is an appliance with the same three buttons — and fit is the only one that learns">

Picture every sklearn object as an appliance with three buttons. **`fit(X, y)` — learn.** The appliance studies the data and stores everything it learned in attributes ending with an underscore (`coef_`, `mean_`, `categories_`); the trailing `_` literally means *"computed during fit."* **`predict(X)` — answer.** Replay what was learned on new rows; nothing is learned here. **`transform(X)` — reshape.** Same replay, but the output is modified data instead of answers (scalers, encoders, imputers). One workflow drives everything: *instantiate → fit → use*.

Two consequences carry the whole chapter. First, **the golden rule**: since `fit` is where learning happens, it is only ever allowed to see the *training* data — calling it on test data lets the future contaminate the model's view of the past. Second, **a Pipeline is appliances bolted into an assembly line** that share one set of buttons: call `fit` on the pipeline and each stage fits then passes its output down the line; call `predict` and raw data flows through every learned transformation into the model. The line is one object — one thing to cross-validate, one thing to save, one thing that cannot leak.

</Intuition>

## The contract — three buttons and an underscore

<Recall term="The X, y handoff">
The NumPy and pandas chapters both ended at the same doorstep: features as a float matrix `X` with one row per sample and one column per feature, labels as a length-matching `y`. That handoff shape is exactly what every sklearn estimator demands.
</Recall>

<RunnableCode title="instantiate, fit, inspect the learned state, predict" code={contractCode} />

The data is exactly the line `score = 43 + 9·hours`, and after `fit` the model's learned state says so: `coef_ = [9.]`, `intercept_ = 43.` — recovered from data, not told. The three-step rhythm never changes: **instantiate** (hyperparameter choices, no data), **fit** (all learning, stored in `_`-suffixed attributes), **use** (`predict` on rows the model never saw). Accessing `coef_` *before* fit raises `NotFittedError` — the underscore attributes don't exist until learning happens.

The pitfall that greets every beginner within the first hour: **X must be 2-D even with one feature.** `df["age"]` is a 1-D Series and crashes with *"Expected 2D array"*; `df[["age"]]` (list inside the brackets — last chapter's selection rule) keeps it a one-column DataFrame and works. Fixes when you're in NumPy-land: `x.reshape(-1, 1)`.

## Train/test split — the exam must be unseen

<RunnableCode title="a memorizer aces the homework and flunks the exam" code={splitCode} />

A model's score on data it trained on is a student grading their own homework — the demo makes it vivid with `KNeighborsClassifier(n_neighbors=1)`, which literally memorizes the training set (every training point's nearest neighbor is itself): **1.0 on training data, every time, for any dataset**, telling you nothing. The score on the held-out 25% is the only number that estimates performance on data the model will actually face. That gap — perfect homework, imperfect exam — is your first sighting of **overfitting**, and the entire reason evaluation discipline exists.

Three arguments to never omit: **`random_state=42`** makes the shuffle reproducible (any integer works; consistency is the point — rerunning with a different split and calling the improvement "progress" is self-deception). **`stratify=y`** keeps class proportions equal in both halves — with imbalanced classes, an unlucky split can leave the test set with almost no positives. And the split happens **before anything else touches the data** — before scaling, before imputation statistics, before feature selection. The test set is the future; the future hasn't happened yet.

`.score(X, y)` is the estimator's built-in default metric — accuracy for classifiers, R² for regressors. Handy, but it answers only one question; `scoring=` below opens the rest.

## Transformers and the golden rule — fit on train only

<RunnableCode title="the scaler learns on train, then judges test by train's numbers" code={scalerCode} />

Transformers are the appliances whose output is *data*: `StandardScaler` learns each column's mean and std during `fit` (see `mean_`, `scale_`) and `transform` re-expresses every value as "how many stds from the mean." `fit_transform(X_tr)` is the two-step shortcut — **legal only on training data**.

The test set gets `transform` *only* — judged by the **train** set's numbers. That's not pedantry; it's the whole point, and the demo shows why: the test row with `10.0` lands at roughly +6 stds — correctly flagged as extreme relative to what the model trained on. Call `fit_transform` on the test set instead and the scaler re-centers around the *test* mean, the outlier gets laundered back toward normal, and information about the test distribution has silently flowed into your features. That's **data leakage** — the future whispering to the past — and it inflates every score it touches while destroying the model in production, where there is no batch of future data to re-fit on.

The same rule binds every transformer: the imputer's median, the encoder's category list, the scaler's mean — all learned from train, all *replayed* on test. Keeping that straight by hand across five transformers is exactly the bookkeeping humans botch. Hence:

## Pipeline + ColumnTransformer — preprocessing as one object

<RunnableCode title="impute + scale + encode + model, fused into one appliance" code={pipeCode} />

`make_pipeline(prep, LogisticRegression())` bolts the stages into one estimator with the standard three buttons. **One `fit` call** runs the assembly line left to right on training data only: imputer learns medians → scaler learns means/stds → encoder learns categories → model learns weights. **One `predict` call** replays every learned transformation on raw input. The golden rule stops being a discipline you maintain and becomes a property of the object — there is *no way* to accidentally fit preprocessing on test data, because you never touch the pieces individually.

`ColumnTransformer` handles the reality that columns need different treatment: numeric columns route through impute-then-scale, categorical through one-hot — declared as configuration, by column name. Two arguments do heavy lifting:

- **`handle_unknown="ignore"`** — last chapter flagged `get_dummies`' production flaw: new month, new plan value, different columns, broken model. `OneHotEncoder` fixes it: categories are *learned at fit* and frozen; a never-seen `"enterprise"` at predict time encodes as all-zeros instead of crashing. Watch the final lines: a raw customer with a NaN spend **and** an unknown plan flows straight through to a probability.
- **`SimpleImputer(strategy="median")`** — the median from the *training* rows, stored, replayed forever. The eda-chapter's paper-trail imputation, now leak-proof by construction.

(The accuracy printed on 3 test rows is statistical noise — with data this small only the *shape* of the code is the lesson. Fixing the noise is next.)

## Cross-validation — never trust one split

<RunnableCode title="three seeds, three different 'truths' — then five folds settle it" code={cvCode} />

The first loop is quietly devastating: the *same model* on the *same data* scores noticeably differently depending on which rows the seed happened to deal into the test set. Any single split is one draw from a lottery — report it alone and you're reporting luck.

**Cross-validation** deals the lottery fairly: split the data into 5 folds, train on 4, test on the 5th, rotate until every fold has taken one turn as the exam, and report **all five scores**. `cross_val_score(model, X, y, cv=5)` does the whole dance. The deliverable is `mean ± std` — the std is honesty about how much the number wobbles (a model scoring 0.78 ± 0.02 is credibly better than 0.74 ± 0.02; at ± 0.15 the comparison is noise). This is the format every serious practitioner expects, and single-number claims without it are a smell.

`scoring=` swaps the question: `"accuracy"`, `"roc_auc"`, `"f1"`, `"neg_mean_squared_error"` — same models, same folds, different metric. Accuracy in particular lies under class imbalance (a 95%-negative dataset gives "predict nothing ever happens" 95% accuracy), which is when `roc_auc` or recall earn their keep — Tier 2 dissects the metric zoo.

The clause that ties the chapter together: **cross-validate the pipeline, not the bare model.** `cross_val_score(pipe, X, y)` re-fits the *entire assembly line* inside every fold, so each fold's imputer and scaler learn only from that fold's training portion. Scale first and cross-validate after, and every fold's test data has already leaked into the preprocessing — the subtle version of the crime from the scaler section, committed at scale by people who know the golden rule and forgot the pipeline.

<Summary>

- **The contract:** instantiate (config, no data) → `fit(X, y)` (all learning; results stored in `_`-suffixed attributes) → `predict`/`transform` (pure replay). One grammar for every model in the library — and most beyond it.
- **Shapes:** `X` is 2-D `(n_samples, n_features)` — always, even for one feature (`df[["col"]]`, `reshape(-1, 1)`); `y` is 1-D and row-aligned.
- **Split first, judge on unseen:** training scores are homework grades (k=1 KNN scores 1.0 forever); `train_test_split(..., stratify=y, random_state=...)` before any statistic is computed. The test set is the future.
- **The golden rule:** transformers `fit` on train only; test gets `transform` with train's learned numbers. Violations are *leakage* — scores inflate, production dies.
- **Pipeline = the rule made structural:** `make_pipeline` fuses impute → scale → encode → model into one estimator; `ColumnTransformer` routes columns to treatments; `handle_unknown="ignore"` survives categories the future invents.
- **Cross-validation:** one split is a lottery ticket; `cross_val_score(pipe, X, y, cv=5)` rotates the exam and reports **mean ± std** — and it must wrap the *pipeline* so preprocessing re-fits inside each fold.
- **Persistence:** `joblib.dump/load` ships the whole assembly line — learned statistics, categories, weights — as one file (pinned to its sklearn version). Raw row in, prediction out.

</Summary>
