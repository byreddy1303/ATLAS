export const closedFormCode = `# Linear regression via the normal equations: fit y = X @ w + noise on synthetic data.
# Closed-form: w_hat = (XᵀX)⁻¹ X^T y  (assuming X^T X is invertible).
import numpy as np
import matplotlib.pyplot as plt

np.random.seed(0)
N, D = 100, 3

# Synthetic ground truth: 3 features + a bias, y = X @ w_true + noise
X_raw = np.random.randn(N, D)
X     = np.hstack([X_raw, np.ones((N, 1))])          # append bias column
w_true = np.array([2.0, -3.0, 0.5, 1.0])             # last is bias
y = X @ w_true + 0.5 * np.random.randn(N)            # sigma = 0.5

# The normal equations, spelled out
XtX = X.T @ X
Xty = X.T @ y
w_hat = np.linalg.solve(XtX, Xty)                    # solve is more stable than @ inv

print(f"true    w = {w_true}")
print(f"fitted  w = {w_hat.round(3)}")
print()

# Sanity check against numpy's canonical solver
w_lstsq, *_ = np.linalg.lstsq(X, y, rcond=None)
print(f"lstsq   w = {w_lstsq.round(3)}   (should match fitted w)")

# Residual sum of squares and R^2
y_hat = X @ w_hat
rss   = ((y - y_hat) ** 2).sum()
tss   = ((y - y.mean()) ** 2).sum()
r2    = 1 - rss / tss
print(f"\\nRSS = {rss:.3f}   TSS = {tss:.3f}   R^2 = {r2:.4f}")

# Plot the fit versus target (2D projection: first feature vs y)
fig, ax = plt.subplots(figsize=(7, 4))
order = np.argsort(X[:, 0])
ax.plot(X[order, 0], y[order], "o", color="royalblue", alpha=0.6, label="data")
ax.plot(X[order, 0], y_hat[order], "-", color="crimson", lw=2, label="fitted line (marginal)")
ax.set_xlabel("feature 0"); ax.set_ylabel("y"); ax.grid(alpha=0.3); ax.legend()
ax.set_title("Linear regression: first-feature projection of the fit")
plt.tight_layout()`;

export const gdVsClosedFormCode = `# Gradient descent finds the SAME solution as the normal equations,
# but iteratively — the two agree in the limit.
import numpy as np, matplotlib.pyplot as plt

np.random.seed(0)
N, D = 300, 4
X = np.hstack([np.random.randn(N, D), np.ones((N, 1))])
w_true = np.array([1.5, -2.0, 0.3, 0.8, 1.0])
y = X @ w_true + 0.4 * np.random.randn(N)

def loss(w): return 0.5 * np.mean((X @ w - y) ** 2)
def grad(w): return X.T @ (X @ w - y) / N

# Closed form
w_star = np.linalg.solve(X.T @ X, X.T @ y)
print(f"Closed-form w = {w_star.round(3)}")
print(f"Closed-form loss = {loss(w_star):.6f}")

# Gradient descent from zero init
w = np.zeros(X.shape[1])
lr = 0.03
losses = [loss(w)]
for step in range(1, 5001):
    w = w - lr * grad(w)
    if step % 500 == 0:
        print(f"step {step:>4}: loss = {loss(w):.6f}  |w - w*| = {np.linalg.norm(w - w_star):.6f}")
    losses.append(loss(w))

print(f"\\nFinal GD  w = {w.round(3)}")
print(f"Distance to closed-form: {np.linalg.norm(w - w_star):.6f}")

plt.figure(figsize=(7, 3.6))
plt.semilogy(losses, color="crimson", lw=2)
plt.axhline(loss(w_star), linestyle="--", color="teal", label=f"closed-form loss ({loss(w_star):.4f})")
plt.xlabel("GD step"); plt.ylabel("loss (log)"); plt.title("GD converges to the closed-form loss")
plt.legend(); plt.grid(alpha=0.3); plt.tight_layout()`;

<Motivation title="Linear regression — the one model worth deriving all the way down">

**Linear regression** is the simplest possible **supervised model** — meaning a model that learns from examples where you already know the right answer. You show it a bunch of `(input, output)` pairs and it learns to guess the output from a new input.

The guess is a one-liner:

$$
\hat{y} = w \cdot x + b
$$

That's it. Given features `x` (a list of numbers describing one example), the prediction `ŷ` (read "y-hat") is a **dot product** with a weight vector `w`, plus a bias number `b`. If any of this sounds fuzzy, quick refresher:

- **Features** = the numbers describing one example. For predicting house price: square footage, bedroom count, neighborhood score.
- **Target** = what you're predicting. Here, price.
- **Dot product** = multiply matching entries and add them all up. `w · x = w₁x₁ + w₂x₂ + …`.
- **Bias** = a constant offset. Without it, every prediction would have to pass through zero.

One dot product per prediction. That's genuinely all it is. And yet: **every big idea in modern ML has a "linear regression version"** — regularization, gradient descent, maximum likelihood, feature engineering, cross-validation, bias-variance decomposition, interpretability, uncertainty. Learning them once here — where the math is clean and you can solve everything by hand — is the fastest way to understand their harder cousins later.

For a GenAI engineer, linear regression shows up in four surprisingly practical places:

1. **Debugging a big model.** When your fancy model is misbehaving, one of the fastest sanity checks is: "does plain linear regression on the same features give sensible answers?" If yes → the signal is there and the big model is buggy. If no → the features themselves are the problem.
2. **Linear probing.** LLM and vision-model research measures how good a pretrained model's **embeddings** (its internal number-summaries of each input) are by freezing the model and fitting a linear regression on top. If the linear model can predict the target well from those embeddings, the pretrained model already "gets it" — no fine-tuning needed.
3. **The last layer of almost every neural net.** Almost every deep model ends with a linear layer: `W · h + b`, where `h` is the last hidden vector. That final layer *is* linear regression (or its classification cousin), trained jointly with the rest.
4. **The baseline you have to beat.** Before deploying anything fancy, compare it to linear regression. It's free, fast, and often shockingly close on tabular data. A paper claiming a 5% win over a linear baseline is trivial; a 50% win is the real deal.

Here's what we'll do in this chapter:

- The **model** and its **squared-error loss** — what "best fit" actually means, and why we square the errors.
- The **closed-form solution** — derived by hand. Yes, this model has an exact formula for the optimal weights.
- The **gradient-descent path** to the same answer, and when it's worth using instead of the formula.
- The **four classical assumptions** and how their violations show up as tell-tale shapes in the residual plots.
- **Interpreting the coefficients** — the real reason linear regression is still the backbone of statistics.

</Motivation>

<Intuition title="Draw the line that misses the points the least — 'least' measured by squared distance">

Picture a scatterplot of `(x, y)` points on a page. Your job: draw a straight line through the cloud that misses the points as little as possible. That line's slope and intercept are your `w` and `b`.

But what counts as "misses the least"? We need a number to compare candidate lines. Here's the recipe:

1. For each point, the line predicts a `ŷᵢ`. The **residual** — how much we missed — is `ŷᵢ − yᵢ`.
2. **Square it.** Squaring does two useful things: it makes negatives positive (missing high or missing low both count as "missing"), and it *punishes big misses way more than small ones*. A miss of 4 counts as 16; a miss of 1 counts as 1.
3. **Average across all points.** The final number is called **MSE** (mean squared error):

$$
L(w, b) = \frac{1}{N} \sum_{i=1}^N (\hat{y}_i - y_i)^2 = \frac{1}{N} \sum_i (w \cdot x_i + b - y_i)^2
$$

The line that makes MSE smallest is the answer. Under mild conditions there's exactly one — no ties — and (this is the magical part) there's an **exact formula** for it. No iterative training, no learning rate, no waiting. Just linear algebra.

The same picture works in higher dimensions. If each `x` has 5 features, you're fitting a flat 5-dimensional surface (a **hyperplane** — the flat generalization of a line and a plane) that best passes through your 6-dimensional cloud of points. Same rule: minimize squared error, get a unique answer.

</Intuition>

<Recall term="Mean squared error (MSE)">

Given predictions `ŷ_i` and true values `y_i` for i = 1..N, the **mean squared error** is `(1/N) · Σ (ŷ_i − y_i)²`. It's the loss function used by linear regression and the "standard" loss for continuous-valued targets. Two properties: (1) it's differentiable everywhere (unlike absolute error), so gradient methods work; (2) it corresponds to maximum likelihood under Gaussian noise — a fact the assumptions section formalizes.

</Recall>

<Recall term="Dot product">

For two vectors `a` and `b` of the same length, the dot product is `a · b = Σ_i a_i b_i`. It's the single operation behind linear regression's prediction (`ŷ = w · x`), attention (`softmax(Q·Kᵀ/√d)·V` uses dot products), classification (each output logit is a dot product), and every "score" in ML. When you optimize a linear regression, you're optimizing a set of dot-product-produced predictions.

</Recall>

## The model, spelled out carefully

Let's write everything down once so nothing surprises us later.

**One example at a time.** If a single example has `D` features (imagine `D = 3` — square footage, bedrooms, neighborhood score), the prediction is:

$$
\hat{y} = w_1 x_1 + w_2 x_2 + \dots + w_D x_D + b = w \cdot x + b
$$

Every feature gets its own weight; the weights say how much that feature pulls the prediction up or down.

**A whole batch at once.** For `N` examples at once, stack the feature vectors as rows of a matrix `X` (shape `N × D`) and stack the true targets in a vector `y` (length `N`):

$$
\hat{y} = X w + b \mathbf{1}
$$

The `𝟏` is just "a vector of all 1s" — the way we add the same bias to every prediction.

**The bias trick — a tidy shortcut.** Carrying `b` around separately is annoying. So we pull a small trick: tack an extra column of 1s onto `X`, and treat `b` as one more entry of `w`. Now the bias magically disappears from every formula because it's just weight number `D+1` multiplied by the constant 1:

```python
X_aug = np.hstack([X, np.ones((N, 1))])
```

From here on, when we write `X`, assume the bias column is already there and `w` includes the bias slot.

**The loss function, made pretty.** We square the errors and average — that's MSE. Convention: throw a `1/2` in front so the derivative later comes out clean:

$$
L(w) = \frac{1}{2N} \|Xw - y\|^2 = \frac{1}{2N} \sum_{i=1}^N (\hat{y}_i - y_i)^2
$$

The `1/2` doesn't change the optimum — you can scale a loss by any positive number and the same `w` still minimizes it.

**Why square the error instead of just taking the absolute value?** Fair question. Three reasons:

1. **Smooth everywhere.** Absolute value `|ŷ − y|` has a sharp kink at zero, where the derivative jumps. Squared error curves smoothly through zero, which means gradient methods work without hitting weird cliffs.
2. **You can solve it with a formula.** MSE leads to a system of linear equations — solvable in one shot with matrix algebra. Absolute error leads to linear programming, which is much fiddlier.
3. **It matches a real story about noise.** If you assume the true relationship is `y = w · x + ε` where `ε` (the noise) is Gaussian ("bell-curve") noise, then finding the `w` that minimizes MSE is *literally the same thing* as **maximum likelihood** — the most-probable answer under that noise model. Different noise assumptions give different losses. If you thought the noise was Laplace-shaped (heavier tails) you'd use absolute error instead — that's called **median regression**.

## The normal equations — the exact formula

Here's where we cash in on all the setup. To find the best `w`, take calculus's most basic move: **compute the gradient of the loss and set it to zero.** Any minimum of a smooth function has to sit at a zero-gradient point.

The gradient works out to:

$$
\nabla_w L(w) = \frac{1}{N} X^\top (X w - y) = 0
$$

Rearrange:

$$
X^\top X w = X^\top y \quad \Longrightarrow \quad w^* = (X^\top X)^{-1} X^\top y
$$

That formula is called the **normal equations**. Two things to keep in mind:

- **The formula only works cleanly if `Xᵀ X` is invertible.** That in turn requires `rank(X) = D` — i.e., no feature is a perfect linear combination of the others. If two features are strongly correlated (a situation called **multicollinearity** — "multiple features carrying the same information"), `Xᵀ X` is nearly singular and the answer becomes unstable. More on that in the pitfalls section.
- **Never actually compute the inverse.** Even though the formula has `(Xᵀ X)⁻¹` in it, computing that inverse explicitly is numerically wasteful and unstable. In code, use `np.linalg.solve(X.T @ X, X.T @ y)` (a specialized solver that skips inverting anything) or, even better, `np.linalg.lstsq(X, y)` (which uses **QR** or **SVD** — matrix factorizations that are even more robust). Sklearn uses `lstsq` under the hood.

**How expensive is this?** Computing `Xᵀ X` costs roughly `N · D²` operations; solving the `D × D` system costs another `D³`. For small `D` (say fewer than 10,000 features), that's *instant* on a laptop. For `D` in the millions (rare, but happens in some genomics or huge-vocab problems), it becomes prohibitive — that's when gradient descent wins.

**One last picture — a satisfying geometric one.** The normal equations say `Xᵀ (Xw* − y) = 0`. Read that literally: the **residual vector** `Xw* − y` (the leftover after your best fit) is **perpendicular to every column of `X`**. Geometrically, `Xw*` is the *orthogonal projection* of `y` onto the space of vectors reachable by combining X's columns. That's the deep reason people call linear regression **least-squares projection**: you're literally projecting `y` onto the "space of predictable things."

## Gradient descent lands at the same answer

Sometimes the closed-form formula isn't practical — maybe you have so many features that `Xᵀ X` won't fit in memory, or new data keeps arriving in a stream. In that case, use **gradient descent**: take a bunch of small steps, each downhill on the loss surface.

The update rule is exactly what you'd expect from the gradient we already computed:

$$
w \leftarrow w - \eta \cdot \nabla L(w) = w - \eta \cdot \frac{1}{N} X^\top (X w - y)
$$

Here `η` (Greek "eta") is the **learning rate** — how big a step to take. Small = safe but slow; big = fast but risks overshooting.

Here's the lovely part: the loss surface for linear regression is a big bowl (**convex** — meaning any straight line between two points on the surface stays above the surface). No local minima to get trapped in. Gradient descent from *any* starting point winds up at the same `w*` the formula would give you. No initialization worries, no random restarts.

**Which method should you actually use?** Quick decision tree:

- **Few features (< 10k), data fits in RAM, no streaming:** the closed-form formula. Fastest, exact, no hyperparameters to tune.
- **Lots of features or streaming data:** gradient descent (or **SGD** = stochastic gradient descent — GD on small random batches). Same answer, more flexible.
- **Need regularization?** Both work. The formula version for **ridge regression** (linear regression with an L2 penalty — coming next chapter) is just `w* = (XᵀX + λI)⁻¹ Xᵀy`. Adding one term to the matrix, still one equation.

## What linear regression secretly assumes — and how you spot violations

Linear regression comes with fine print. Four **assumptions** — things about your data that need to be true for the fit to be trustworthy. They're not always obvious, and violations show up as visible patterns in the **residuals** (the leftover errors after your best fit).

Here's the fine print, translated:

1. **Linearity.** The real relationship between `x` and `y` actually *is* a straight line (or hyperplane) — not a curve, not a wiggle. If reality is quadratic and you fit a line, you're bringing a ruler to a curve.
2. **Independent errors.** One example's noise doesn't tell you anything about another example's noise. Violated by things like time series (today's stock error is correlated with yesterday's) or measurements from the same person (repeated readings share person-level quirks).
3. **Homoscedasticity** — long word, simple idea: "same-sized noise everywhere." The size of the wobble in `y` shouldn't depend on `x`. Violated when, say, high-income spending is much more variable than low-income spending. The Greek roots literally mean "same-scatter."
4. **Normally-distributed errors.** The noise follows a bell curve. This one you only need if you want **confidence intervals** (how sure am I about this coefficient?) or **p-values** — not for the point estimate itself.

Small side note: for just *finding* the best `w` (the point estimate), only assumptions 1 and 2 matter. There's a famous result called the **Gauss-Markov theorem** that says: under just linearity + independent errors + finite noise variance, the least-squares answer is the best possible answer among a broad class of methods. That class-winning property is called **BLUE** — **Best Linear Unbiased Estimator**. (Unbiased = "on average, gives back the true answer.") For confidence intervals though, you need the normality piece too.

**How do you check if the assumptions hold?** Look at the residuals. Plot them against `x` (or against the predictions `ŷ`) and see if you can spot a pattern. If residuals scatter uniformly around zero → you're fine. If they show shape → the assumption is broken.

**Common violations and what to do:**

- **Residuals form a U (or upside-down U)** → the real relationship is *curved*, not straight. Fix: add polynomial features like `x²`, `x · y`, or switch to a nonlinear model (tree, neural net, etc.).
- **Residuals fan outward** (small near the middle, big at the extremes) → *heteroscedasticity* — non-constant noise size. Fix: **weighted least squares** (give bigger-variance points less say), or transform the target (a common trick: fit to `log(y)` instead of `y`, if `y` is always positive).
- **Residuals correlated over time** (typical in time-series data) → use time-series models like **ARIMA** (autoregressive integrated moving average) or add lagged features (`x` at time `t-1`, `t-2`, …).
- **Non-normal residuals but you still need CIs** → use the **bootstrap**: resample your data many times, refit each time, and use the spread of resulting `w` estimates as your confidence intervals.

## Reading the coefficients — the real reason statisticians love linear regression

Once you've fit `ŷ = w · x + b`, each weight `wᵢ` has a *specific, clean* meaning:

$$
w_i = \text{how much } y \text{ changes when you bump } x_i \text{ up by one unit, holding every other feature fixed}
$$

That "holding every other feature fixed" is called **ceteris paribus** — a Latin phrase from economics meaning "all else equal." It's why linear regression is still a workhorse of statistics and social science: **no other ML model gives you such a clean per-feature story.** A random forest or neural net can be more accurate, but ask "how much does one more bedroom cost?" and you get a shrug. Ask a linear regression and you get: "\$12,400, holding square footage and neighborhood fixed."

Three practical warnings:

- **Standardize before comparing coefficients across features.** If `x₁` is in dollars and `x₂` is in years, their raw coefficients are on wildly different scales — you can't directly say "years matters more." Standardizing (subtract the mean, divide by the standard deviation, per feature) puts everything on the same footing: each coefficient now says "change in `y` per one-standard-deviation change in this feature."
- **Watch out for multicollinearity.** When two features carry the same information (say, one column is "years of experience" and another is "months of experience"), individual coefficients become unstable — one can be +2,000 and the other −1,990, canceling out. The predictions are fine, but the per-feature story is nonsense. A diagnostic called **VIF** (variance inflation factor) flags this: any feature with VIF above ~10 is at risk.
- **Correlation is not causation.** The coefficient tells you the *association* between `xᵢ` and `y` in your data given the model. Whether changing `xᵢ` would *cause* `y` to change is a separate question no regression can settle by itself. A carefully-designed regression (with proper controls) can be *evidence* for a causal claim, but only alongside domain knowledge and study design.

<RealWorldML title="Linear probing — how researchers check if an LLM 'gets it'">

A modern LLM or vision model turns each input into an **embedding** — a dense vector of numbers (usually 512, 768, or several thousand entries) that summarizes what the input is "about." How do you tell whether these embeddings actually capture some property you care about — say, sentiment, or topic, or safety-relevance?

The standard trick is called **linear probing**:

1. Freeze the pretrained model (don't touch its weights).
2. Feed it a labeled dataset and collect the embeddings.
3. Fit a *linear regression* (or logistic regression for yes/no labels) on top of those embeddings, predicting your target.
4. Check how well the linear model does.

If the linear model does well → the pretrained embeddings already encode the target. The big model "figured out" this property just from pretraining, without ever being told to. If the linear model does badly → either the target is too subtle for a straight line to pick out, or the embeddings aren't capturing it.

<RunnableCode title="Linear probing on 64-dim frozen embeddings" code={linearProbingCode} packages={["numpy"]} />

**Why researchers reach for linear probing all the time:**

1. **Cheap.** Fitting a linear model on 768-dim embeddings takes seconds. Fine-tuning a whole 7B-parameter LLM takes hours and a lot of GPU money.
2. **Interpretable.** The probe's weights tell you *which* embedding dimensions carry the signal. On a big labeled task, a high probe score is strong evidence that the pretrained model has genuinely captured the concept.
3. **Standard.** Papers routinely report linear-probe results as a benchmark — you'll see terms like "**ImageNet linear evaluation**" (image models) or "**STS**" (semantic-similarity tasks for language models). Beat a linear probe by a small margin with fine-tuning and you've barely added anything; beat it by a lot and there's real task-specific value.

**One clean connection to sit with:** the *last layer* of every neural classifier is itself a linear layer sitting on top of the previous layer's embeddings. So "linear-probe accuracy" and "final-layer accuracy" are the same thing when the earlier layers are frozen. **Fine-tuning is just "linear probing, but you also let the encoder learn."**

</RealWorldML>

<CodeItYourself title="Three drills — normal equations by hand, GD variants, linear probing">

1. **Fit by hand.** For the tiny dataset `X = [[1, 1], [1, 2], [1, 3]]` (bias column + one feature) and `y = [1.5, 2.5, 3.4]`, compute w* = (XᵀX)⁻¹ X^T y by hand. Verify with `np.linalg.solve`.

2. **GD sensitivity to LR.** For the synthetic problem above, sweep learning rates lr = 0.001, 0.01, 0.1, 1.0 and watch what happens. At lr = 1.0 the largest eigenvalue of the Hessian exceeds 1/lr and GD diverges; at lr = 0.001 it converges but slowly. Find the sweet spot.

3. **Linear probe on toy embeddings.** Generate synthetic "embeddings" `E = np.random.randn(500, 128)` and a target `y = E[:, :10].sum(1) + noise` — only the first 10 dimensions carry signal. Fit a linear probe on the first 250 examples, evaluate on the rest. R² should be high because the linear model can rediscover the "signal is in first 10 dims" structure automatically.

<Summary>

- **Model:** ŷ = w · x + b (with `x, w ∈ R^D`, `b ∈ R`). Batch: ŷ = Xw using bias trick.
- **Loss:** MSE L(w) = (1/2N) · ‖Xw - y‖². Squared for smoothness, closed-form solvability, and MLE justification under Gaussian noise.
- **Normal equations:** w* = (XᵀX)⁻¹ X^T y. Requires rank(X) = D. Compute via np.linalg.solve or np.linalg.lstsq — never inv().
- **Gradient descent** gets the same w* (loss is convex). Better when D exceeds ~10⁴ or data is streaming.
- **R² = 1 - RSS/TSS** measures fraction of variance explained. Standardize features before comparing coefficients.
- **Four classical assumptions:** linearity, independent errors, homoscedasticity, normality. Coefficient point estimates only need the first two.
- **Residual plots diagnose violations:** U-shape = nonlinearity, fan-out = heteroscedasticity.
- **Linear probing** is linear regression on top of a pretrained encoder's frozen embeddings — the standard evaluation for representation quality.

</Summary>
