export const confusionCode = `# Confusion matrix + all the derived metrics from one source of truth.
import numpy as np

# Toy dataset: 100 samples, 90 negatives + 10 positives.
# Model predictions -- both classes correctly detected most of the time, some errors.
y_true = np.array([1]*10 + [0]*90)
y_pred = np.array([1]*7 + [0]*3 + [1]*4 + [0]*86)   # 7 TP, 3 FN, 4 FP, 86 TN

def confusion(y, y_hat):
    "Return TP, FP, TN, FN as a 2x2 matrix."
    return np.array([
        [((y_hat == 1) & (y == 1)).sum(),  ((y_hat == 1) & (y == 0)).sum()],  # TP, FP
        [((y_hat == 0) & (y == 1)).sum(),  ((y_hat == 0) & (y == 0)).sum()],  # FN, TN
    ])

C = confusion(y_true, y_pred)
TP, FP = C[0]
FN, TN = C[1]
print(f"Confusion matrix (rows: predicted, cols: actual):")
print(f"                  actual+  actual-")
print(f"  predicted+:  TP={TP:>3}   FP={FP:>3}")
print(f"  predicted-:  FN={FN:>3}   TN={TN:>3}")
print()

# All the derived metrics
accuracy    = (TP + TN) / (TP + FP + FN + TN)
precision   = TP / (TP + FP) if (TP + FP) else 0
recall      = TP / (TP + FN) if (TP + FN) else 0
specificity = TN / (TN + FP) if (TN + FP) else 0
f1          = 2 * precision * recall / (precision + recall) if (precision + recall) else 0

print(f"accuracy    = {accuracy:.3f}   ({TP + TN}/{TP+FP+FN+TN} correct)")
print(f"precision   = {precision:.3f}   (of what I called +, {int(precision*100)}% really were)")
print(f"recall      = {recall:.3f}   (of the real +, I caught {int(recall*100)}%)")
print(f"specificity = {specificity:.3f}   (of the real -, I correctly said - to {int(specificity*100)}%)")
print(f"F1          = {f1:.3f}   (harmonic mean of precision & recall)")
print()

# Naive baseline: always predict negative -> 90% accuracy!
print(f"Baseline 'always predict negative': accuracy = 0.90, recall = 0.00, F1 = 0.00.")
print(f"Accuracy alone hides the fact that a 'do nothing' model beats a real model.")`;

from sklearn.datasets import make_classification

ax.set_ylabel("actual fraction positive")
ax.set_title("Reliability diagram — perfect calibration lies on the diagonal")
ax.legend(); ax.grid(alpha=0.3)
plt.tight_layout()

# Expected Calibration Error (a summary of miscalibration)
def ece(y, p, n_bins=10):
    bins = np.linspace(0, 1, n_bins + 1)
    err = 0.0
    for lo, hi in zip(bins[:-1], bins[1:]):
        mask = (p >= lo) & (p < hi)
        if mask.sum() > 0:
            gap = abs(p[mask].mean() - y[mask].mean())
            err += mask.mean() * gap

<Motivation title="Accuracy lies — pick the right metric or ship the wrong system">

Say you build a classifier for a rare disease. Base rate: 1% of people have it. You train a "model" that always predicts "no disease." Its accuracy: **99%**. That's not a great model. That's a useless one — it never catches a single sick person. But if all you look at is accuracy, you'd think you shipped a winner.

Same story with a fraud classifier at 0.1% base rate — a trivial "always negative" baseline scores 99.9%. Same story with an LLM guessing "A" on multiple-choice — 25% right just by luck.

**Accuracy hides too much.** Real ML systems have three things that break the "accuracy is enough" story:

- **Costs aren't symmetric.** False positives cost one thing (customer service call), false negatives cost another (undetected cancer). They're almost never equal.
- **Classes aren't balanced.** Someone always outnumbers the others, often by a lot.
- **Predictions are actually probabilities, not hard labels.** A model that says "60% positive" is meaningfully different from one that says "99% positive" — and neither one collapses cleanly into "yes" or "no."

For a GenAI engineer, choosing the right metric literally determines what your system will optimize for. Some concrete cases:

1. **Safety classifiers** (hate speech, harmful requests). You want extremely high **recall** (catch essentially all bad stuff), even if it means some false alarms. Better to review a benign request than to let a bad one through.
2. **Spam filters / RLHF reward models.** You want balance: high **precision** (don't quarantine real emails) *and* reasonable recall. `F1` or `F0.5` is the headline metric.
3. **Retrieval / RAG.** You care about *ordering*, not classification. Metrics like **Recall@K**, **MRR** (mean reciprocal rank), and **NDCG** (normalized discounted cumulative gain) matter — accuracy doesn't apply.
4. **RLHF preference modeling.** Accuracy on preference pairs is the standard headline — but **calibration** (do the predicted preference probabilities match reality?) matters even more, because the reward's magnitude directly drives policy updates.
5. **Anomaly detection.** Very rare positives. Use **PR-AUC** over ROC-AUC.

This chapter builds every classification metric from a single object — the **confusion matrix** — then layers on ROC/PR curves, calibration, and cost-sensitive threshold selection.

</Motivation>

<Intuition title="One little 2×2 table — the confusion matrix — spawns every metric you need">

Take any binary classifier's predictions and drop them into a 2×2 grid based on what the model said vs. what was true:

|                   | actual **positive** | actual **negative** |
|-------------------|-------------------|-------------------|
| predicted **positive** | **True Positive (TP)** | **False Positive (FP)** |
| predicted **negative** | **False Negative (FN)** | **True Negative (TN)** |

Four numbers. That grid is called the **confusion matrix** (it doesn't confuse *you* — it shows where the model is confusing classes with each other). From those four numbers you can compute every classification metric anyone has ever invented. The important ones:

- **Accuracy** = `(TP + TN) / total` — "what fraction did I get right?"
- **Precision** = `TP / (TP + FP)` — "when I said positive, how often was I right?"
- **Recall** (also called **sensitivity** or **true positive rate, TPR**) = `TP / (TP + FN)` — "of the actual positives, how many did I catch?"
- **Specificity** (or **TNR**) = `TN / (TN + FP)` — "of the actual negatives, how many did I correctly identify?"
- **False positive rate (FPR)** = `FP / (FP + TN)` — "of the actual negatives, how many did I *wrongly* flag?" This equals `1 − specificity`.
- **F1 score** = `2 · P · R / (P + R)` — the harmonic mean of precision and recall (a "balanced" summary).

**Which one do you care about?** Depends on the *cost*:

- **Precision matters** when **false positives are expensive**. Examples: arresting an innocent person, quarantining a legitimate email, telling a healthy person they have cancer.
- **Recall matters** when **false negatives are expensive**. Examples: missing a real cancer diagnosis, missing an actual security attack, letting spam or fraud through.
- **F1 matters** when you want a single "how good, overall" number that punishes low performance on either axis.

**Now the twist — everything above assumes a specific threshold.** A classifier usually outputs a *probability*, not a hard label. The default "predict positive if p ≥ 0.5" is *a choice*, not a mathematical necessity. Move the threshold up to 0.8 and you get *higher precision* but *lower recall*. Move it down to 0.2 and vice versa. Every point along that trade-off gives you a different confusion matrix. **ROC and PR curves** are exactly the tools for visualizing that full trade-off across every possible threshold.

</Intuition>

<Recall term="Sigmoid probability">

Logistic regression (and every neural-net classifier's last layer) outputs a probability `p̂ = σ(w·x + b) ∈ (0, 1)`. To convert to a hard label, apply a threshold — usually 0.5, but that's a modeling choice, not a mathematical necessity. All classification metrics implicitly assume you've chosen a threshold; ROC and PR curves let you visualize the full trade-off across thresholds.

</Recall>

## The confusion matrix and its 4 core metrics

**Accuracy** measures "overall correctness":

$$
\text{Accuracy} = \frac{TP + TN}{TP + FP + FN + TN}
$$

**Precision** measures "when I predict positive, how often am I right?":

$$
\text{Precision} = \frac{TP}{TP + FP}
$$

**Recall** (aka sensitivity) measures "of the actual positives, how many did I find?":

$$
\text{Recall} = \frac{TP}{TP + FN}
$$

**F1** is the harmonic mean of precision and recall (harsher than arithmetic mean — punishes if either is low):

$$
F_1 = \frac{2 \cdot P \cdot R}{P + R} = \frac{2 TP}{2 TP + FP + FN}
$$

**Beware accuracy on imbalanced classes.** For a rare-disease classifier with 1% positive base rate, "always predict negative" achieves 99% accuracy. Use precision/recall/F1/PR-AUC/MCC in imbalanced settings — accuracy is misleading.

## ROC and PR curves — visualizing the full threshold trade-off

Instead of committing to one threshold, sweep every possible threshold from 0 to 1 and see how precision, recall, TPR, and FPR change. That gives you a whole *curve* of possible operating points — and lets you pick the one that fits your problem.

**The ROC curve.** For every possible threshold `t`, compute `(FPR, TPR)` — the false positive rate and true positive rate at that threshold. Plot TPR against FPR. A **perfect classifier** hugs the top-left corner (high recall, no false alarms). A **random classifier** is the diagonal `y = x`.

**ROC-AUC** — the *area under the ROC curve*. Beautiful interpretation: given a random positive example and a random negative example, ROC-AUC is the probability the classifier scores the positive higher than the negative. Ranges from 0.5 (random) to 1 (perfect ranking).

**The PR curve.** For every threshold `t`, compute `(recall, precision)`. Plot precision vs. recall. Different shape than ROC, useful in different situations.

**PR-AUC** (also called **Average Precision** or **AP**) — the area under the PR curve. Ranges from the *base rate* (random guess) to 1 (perfect).

<RunnableCode title="ROC and PR curves for the same classifier" code={rocPrCode} packages={["numpy", "sklearn", "matplotlib"]} captureFig={true} />

**When to prefer which:**

- **Balanced data** (roughly equal positives and negatives). ROC-AUC and PR-AUC give similar signals. ROC-AUC is the standard by convention.
- **Imbalanced data** (rare positives — fraud, disease, anomaly detection). **Use PR-AUC.** ROC-AUC can look great even for a not-so-great classifier when negatives dominate — the huge negative pool makes FPR stay tiny with modest effort, inflating AUC. PR-AUC directly measures precision as recall varies, which is what you care about when positives are scarce.

## Calibration — do the probabilities actually mean what they say?

Here's a question people rarely ask: when your model says "90% probability," is it actually right 90% of the time? A model is **well-calibrated** if the answer is "yes" — its predicted probabilities match empirical frequencies. Modern deep neural nets are famously *not* well-calibrated. They're typically **overconfident** — they'll report 95% confidence for examples that are actually right only 70% of the time.

**How to check — the reliability diagram.** Bin the predicted probabilities (say into 10 equal-width bins). For each bin, plot the bin midpoint on the x-axis and the *actual* fraction of positives in that bin on the y-axis. A perfectly calibrated model's points lie on the `y = x` diagonal. A miscalibrated model's points wander off.

**How to summarize with one number — Expected Calibration Error (ECE).** For each bin, compute `|predicted probability − actual frequency|`, then average those gaps weighted by how many points are in each bin. `ECE = 0` = perfect calibration. Bigger `ECE` = worse.

**Common fixes for miscalibrated models:**

- **Platt scaling** — fit a small logistic function `p_cal = σ(a · p_raw + b)` on a held-out set. Two parameters. Popular for SVMs and boosted trees.
- **Isotonic regression** — fit a non-parametric monotonic function from raw scores to calibrated probabilities. More flexible than Platt (can bend more), but needs more data.
- **Temperature scaling** — divide the raw logits by a single scalar `T` (fit on a held-out set): `p_cal = softmax(logits / T)`. `T > 1` spreads probabilities out (less confident), `T < 1` sharpens them. Single parameter, works remarkably well for neural nets, doesn't change the argmax (so accuracy stays the same, only confidences change). *This is the standard fix for overconfident deep models.*

**Why calibration is worth caring about:**

- **Cost-sensitive decisions.** If you're picking a threshold based on FP vs FN cost, the optimal threshold assumes calibrated probabilities. Uncalibrated probs → wrong threshold → suboptimal decisions.
- **Ensembling.** Averaging probabilities across models only makes sense if each model's probs are calibrated. Otherwise you're averaging apples and oranges.
- **Downstream systems.** Anything that uses "confidence" from a classifier (routing, escalation, prioritization) will misbehave when the confidences are wrong.

## Picking the threshold based on cost — not defaulting to 0.5

The default 0.5 threshold implicitly assumes false positives and false negatives cost the same amount. That's almost never true in practice.

- **Fraud detection.** Missing a fraud (FN) → $10,000 loss. Wrongly flagging a legitimate transaction (FP) → $50 customer service call. Threshold should be *much lower* than 0.5 — catch even weak signals.
- **Medical screening (rare disease).** Missed diagnosis (FN) is catastrophic. Unnecessary follow-up test (FP) is inconvenient. Low threshold.
- **Content moderation.** Very high volume. Some FPs are OK if they get human review. Threshold depends on the specific harm.

**The formula for the cost-optimal threshold** (assuming a calibrated classifier):

$$
t^* = \frac{\text{cost}_{FP}}{\text{cost}_{FP} + \text{cost}_{FN}}
$$

Fraud example: `t* = 50 / (50 + 10000) ≈ 0.005`. Predict fraud any time the calibrated probability exceeds 0.5%. That sounds aggressive, but the math backs it up — the cost of missing fraud so vastly outweighs the cost of a false alarm that you want to be very sensitive.

**In practice**, pick the threshold by minimizing empirical cost on a validation set. The ROC or PR curve gives you the trade-off; you pick the operating point that matches your business costs.

<RealWorldML title="Classification metrics for LLM systems — safety, RLHF, judges, retrieval">

**LLM safety classifiers** (hate speech, harmful requests). Very high **recall** is the target — you want to catch essentially all bad stuff, even if it means some benign requests get flagged for review. The threshold is set aggressively (predict "unsafe" at low probabilities). Standard reports: *"recall at 95% recall"*, precision at that operating point, and PR-AUC as the summary.

**RLHF preference modeling.** The reward model learns from human labels like "response A is preferred over response B." Headline metric: accuracy on held-out preference pairs. But **calibration matters even more** — the RLHF loss depends on the reward's *absolute value*, not just the sign. A miscalibrated reward creates a phenomenon called **reward hacking**, where the policy learns to exploit regions of the reward function that no longer match actual human preference. Careful calibration of the reward model is one of the guards against this.

**LLM-as-judge.** GPT-4 (or Claude) grades outputs of GPT-3.5 (or another model). Standard practice: report *agreement rate with human raters* and per-category accuracy. Watch out — the judge inherits the biases of the models it's grading (they were trained on overlapping data), so this is not a fully independent evaluation.

**Retrieval metrics for RAG.**

- **Recall@K** — fraction of relevant documents in the top `K` retrieved. Standard RAG headline: "we retrieve the relevant document in the top 5 90% of the time."
- **MRR** (mean reciprocal rank) — `1 / (rank of the first relevant document)`, averaged over queries. Rewards putting relevant docs near the top.
- **NDCG** (normalized discounted cumulative gain) — a more sophisticated ordering metric that weights higher-ranked hits more heavily and handles graded (not just binary) relevance.

**LLM evaluation on multiple-choice benchmarks** (MMLU, HellaSwag, etc.). Reported metric is accuracy. But look at the confusion matrix — is the model systematically wrong on certain subject areas? Is the "accuracy" driven by lucky letter-guessing (e.g., always answering A) rather than actual reasoning?

**Confidence-based abstention.** Modern eval setups increasingly let LLMs say "I don't know" instead of guessing. Metrics: *accuracy conditional on the model choosing to answer* + *abstention rate*. A well-calibrated model would abstain exactly when its confidence is low. This is where ECE and reliability diagrams matter directly for LLMs — an overconfident LLM abstains too rarely and gets penalized for confident wrong answers.

</RealWorldML>

<CodeItYourself title="Three drills — confusion matrix by hand, threshold sweep, reliability diagram">

1. **Confusion matrix from raw predictions.** Given `y_true = [1, 0, 1, 1, 0, 0, 1]` and `y_pred = [1, 0, 0, 1, 1, 0, 1]`, compute TP, FP, TN, FN by hand. Verify precision = 3/4 = 0.75, recall = 3/4 = 0.75, F1 = 0.75.

2. **Threshold sweep and ROC.** Given a probability model and a set of predictions, sweep the threshold from 0 to 1 in steps of 0.01, compute (FPR, TPR) at each. Plot the ROC curve, compute AUC by trapezoidal integration.

3. **Reliability diagram.** For a trained model, bin its predicted probabilities into 10 bins. For each bin, compute the actual fraction of positives. Plot on a diagonal. If your model is overconfident, points fall BELOW the diagonal in the middle bins.
