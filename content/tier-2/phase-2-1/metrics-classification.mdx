export const confusionCode = `# Confusion matrix + all the derived metrics from one source of truth.
import numpy as np

# Toy dataset: 100 samples, 90 negatives + 10 positives.
# Model predictions -- both classes correctly detected most of the time, some errors.
y_true = np.array([1]*10 + [0]*90)
y_pred = np.array([1]*7 + [0]*3 + [1]*4 + [0]*86)   # 7 TP, 3 FN, 4 FP, 86 TN

def confusion(y, y_hat):
    "Return TP, FP, TN, FN as a 2x2 matrix."
    return np.array([
        [((y_hat == 1) & (y == 1)).sum(),  ((y_hat == 1) & (y == 0)).sum()],  # TP, FP
        [((y_hat == 0) & (y == 1)).sum(),  ((y_hat == 0) & (y == 0)).sum()],  # FN, TN
    ])

C = confusion(y_true, y_pred)
TP, FP = C[0]
FN, TN = C[1]
print(f"Confusion matrix (rows: predicted, cols: actual):")
print(f"                  actual+  actual-")
print(f"  predicted+:  TP={TP:>3}   FP={FP:>3}")
print(f"  predicted-:  FN={FN:>3}   TN={TN:>3}")
print()

# All the derived metrics
accuracy    = (TP + TN) / (TP + FP + FN + TN)
precision   = TP / (TP + FP) if (TP + FP) else 0
recall      = TP / (TP + FN) if (TP + FN) else 0
specificity = TN / (TN + FP) if (TN + FP) else 0
f1          = 2 * precision * recall / (precision + recall) if (precision + recall) else 0

print(f"accuracy    = {accuracy:.3f}   ({TP + TN}/{TP+FP+FN+TN} correct)")
print(f"precision   = {precision:.3f}   (of what I called +, {int(precision*100)}% really were)")
print(f"recall      = {recall:.3f}   (of the real +, I caught {int(recall*100)}%)")
print(f"specificity = {specificity:.3f}   (of the real -, I correctly said - to {int(specificity*100)}%)")
print(f"F1          = {f1:.3f}   (harmonic mean of precision & recall)")
print()

# Naive baseline: always predict negative -> 90% accuracy!
print(f"Baseline 'always predict negative': accuracy = 0.90, recall = 0.00, F1 = 0.00.")
print(f"Accuracy alone hides the fact that a 'do nothing' model beats a real model.")`;

from sklearn.metrics import roc_auc_score, average_precision_score

np.random.seed(0)

def evaluate_at_imbalance(pos_frac, N=1000):
    n_pos = int(N * pos_frac)
    n_neg = N - n_pos
    y = np.concatenate([np.ones(n_pos), np.zeros(n_neg)])

        np.random.beta(3, 2, size=n_pos),

ax.legend(); ax.grid(alpha=0.3)
plt.tight_layout()

# Expected Calibration Error (a summary of miscalibration)
def ece(y, p, n_bins=10):
    bins = np.linspace(0, 1, n_bins + 1)
    err = 0.0
    for lo, hi in zip(bins[:-1], bins[1:]):
        mask = (p >= lo) & (p < hi)
        if mask.sum() > 0:
            gap = abs(p[mask].mean() - y[mask].mean())
            err += mask.mean() * gap

<Motivation title="Accuracy is almost never enough — every metric hides a story you'd rather see">

Take a classifier for a rare disease with 1% base rate. A "model" that always predicts "no disease" achieves **99% accuracy** — while being completely useless. Take a fraud classifier with 0.1% base rate. Same story, 99.9% accuracy from a trivial baseline. Take LLM eval on a benchmark: a model that always answers "A" might score 25% on multiple-choice.

**Accuracy hides too much.** Every ML system has cost structure: false positives cost X, false negatives cost Y, and X != Y. Every ML system has class imbalance in some direction. Every ML system provides a probability, not a hard label — and how well-calibrated that probability is matters more than which side of 0.5 it falls on.

For a GenAI engineer, choosing the right metric determines what your system optimizes for:

1. **Safety classifiers** (hate speech, harmful requests). Very high recall (catch all bad requests) at the cost of some precision (some false alarms) — you'd rather review a legitimate request than let a bad one through.
2. **Spam filters** (or RLHF reward models). Balance: high precision (don't quarantine real mail) with reasonable recall (still catch obvious spam). F0.5 or F1 as headline.
3. **Retrieval / RAG** (search-relevance ranking). Precision-at-K, MRR, NDCG — none of them use "accuracy" directly. What matters is ordering.
4. **Preference / reward modeling for RLHF.** Accuracy on preference pairs is standard — but calibration matters even more, because the reward signal drives policy updates.
5. **Anomaly detection** (drift, unusual usage). Very rare positives; PR-AUC over ROC-AUC.

This chapter builds every classification metric from a single object — the confusion matrix — then adds ROC/PR curves, calibration, and cost-sensitive thresholding.

</Motivation>

<Intuition title="Everything comes from the confusion matrix — 4 numbers, 20 metrics">

A binary classifier's predictions can be tabulated in a 2×2 grid:

|                   | actual **positive** | actual **negative** |
|-------------------|-------------------|-------------------|
| predicted **positive** | **True Positive (TP)** | **False Positive (FP)** |
| predicted **negative** | **False Negative (FN)** | **True Negative (TN)** |

From these 4 numbers, every classification metric follows:

- **Accuracy** = (TP + TN) / total. What fraction did we get right?
- **Precision** = TP / (TP + FP). Of my POSITIVE predictions, how many are actually positive?
- **Recall (sensitivity, TPR)** = TP / (TP + FN). Of the actual positives, how many did I find?
- **Specificity (TNR)** = TN / (TN + FP). Of the actual negatives, how many did I correctly identify?
- **False Positive Rate (FPR)** = FP / (FP + TN) = 1 − specificity.
- **F1 score** = 2 · P · R / (P + R). Harmonic mean of precision and recall.

**Which one do you care about?** It depends on the cost:

- **Precision matters** when FALSE POSITIVES are costly (arresting the wrong person, quarantining legitimate emails, false rare-disease diagnosis).
- **Recall matters** when FALSE NEGATIVES are costly (missing a real disease case, missing an actual attack, letting spam through).
- **F1 matters** as a balanced summary.

**But there's a hidden variable: the DECISION THRESHOLD.** A classifier usually outputs a probability p̂; the standard "predict positive if p̂ ≥ 0.5" threshold produces one confusion matrix. Move the threshold up (predict positive only if p̂ ≥ 0.8) and you get higher precision but lower recall. Move it down (p̂ ≥ 0.2) and vice versa. ROC and PR curves visualize the entire threshold sweep.

</Intuition>

<Recall term="Sigmoid probability">

Logistic regression (and every neural-net classifier's last layer) outputs a probability `p̂ = σ(w·x + b) ∈ (0, 1)`. To convert to a hard label, apply a threshold — usually 0.5, but that's a modeling choice, not a mathematical necessity. All classification metrics implicitly assume you've chosen a threshold; ROC and PR curves let you visualize the full trade-off across thresholds.

</Recall>

## The confusion matrix and its 4 core metrics

**Accuracy** measures "overall correctness":

$$
\text{Accuracy} = \frac{TP + TN}{TP + FP + FN + TN}
$$

**Precision** measures "when I predict positive, how often am I right?":

$$
\text{Precision} = \frac{TP}{TP + FP}
$$

**Recall** (aka sensitivity) measures "of the actual positives, how many did I find?":

$$
\text{Recall} = \frac{TP}{TP + FN}
$$

**F1** is the harmonic mean of precision and recall (harsher than arithmetic mean — punishes if either is low):

$$
F_1 = \frac{2 \cdot P \cdot R}{P + R} = \frac{2 TP}{2 TP + FP + FN}
$$

**Beware accuracy on imbalanced classes.** For a rare-disease classifier with 1% positive base rate, "always predict negative" achieves 99% accuracy. Use precision/recall/F1/PR-AUC/MCC in imbalanced settings — accuracy is misleading.

## ROC and PR curves — the threshold sweep

**ROC curve:** for each possible decision threshold t ∈ [0, 1], compute (FPR, TPR). Plot TPR vs FPR. Perfect classifier goes through (0, 1); random classifier is the diagonal y = x.

**ROC-AUC:** area under the ROC curve. Interpretable as: given a random positive and a random negative, what's the probability that the classifier scores the positive higher? Value in [0.5, 1] (0.5 = random, 1 = perfect).

**PR curve:** for each threshold t, compute (recall, precision). Plot precision vs recall.

**PR-AUC (Average Precision, AP):** area under the PR curve. Similar interpretation but ranges from base-rate (random) to 1 (perfect).

<RunnableCode title="ROC and PR curves for the same classifier" code={rocPrCode} packages={["numpy", "sklearn", "matplotlib"]} captureFig={true} />

**When to use each:**

- **Balanced data (roughly equal positive/negative):** ROC-AUC and PR-AUC give similar signals. Use ROC by convention.
- **Imbalanced data (rare positives):** PR-AUC is more informative. ROC-AUC can be misleadingly high because most predictions are correctly negative.

## Calibration — do the probabilities mean what they say?

A well-calibrated classifier says "70% probability" for examples that are actually positive 70% of the time. Modern deep neural nets are famously **overconfident** — they might report 95% confidence for examples that are correct only 70% of the time.

**Diagnostic: reliability diagram.** Bin the predicted probabilities. For each bin, plot (bin midpoint, actual fraction of positives in that bin). A perfectly calibrated model lies on the diagonal.

**Summary metric: Expected Calibration Error (ECE).** Weighted average of |predicted − actual| across bins, weighted by bin size. ECE = 0 means perfect calibration.

<RunnableCode title="Reliability diagram — visualize calibration" code={calibrationCode} packages={["numpy", "sklearn", "matplotlib"]} captureFig={true} />

**Fixes for miscalibration:**

- **Platt scaling:** fit a logistic function `p_cal = σ(a · p_raw + b)` to a held-out set. Popular for SVMs and boosted trees.
- **Isotonic regression:** fit a non-parametric monotonic function from raw scores to calibrated probabilities. More flexible than Platt.
- **Temperature scaling (deep learning):** divide logits by a scalar T fit on a held-out set: `p_cal = softmax(logits / T)`. Simple, single-parameter, works well for neural nets.

**Why calibration matters:**

- **Cost-sensitive decisions.** If you decide based on "cost of FP vs cost of FN," the optimal threshold depends on the calibrated probability, not the raw score.
- **Ensembling.** Combining probabilities across models only makes sense if each is calibrated.
- **Downstream systems.** Any system that uses "confidence" from a classifier (routing, escalation, prioritization) will misbehave if confidences are wrong.

## Cost-sensitive thresholding

The default 0.5 threshold assumes equal cost of FP and FN. In reality:

- **Fraud detection:** FN (missing fraud) costs $10,000 in losses; FP (blocking a legitimate transaction) costs $50 in customer service. Threshold should be MUCH LOWER than 0.5 (catch even weak signals).
- **Medical screening (rare disease):** FN (missed diagnosis) is catastrophic; FP (unnecessary follow-up test) is inconvenient. Low threshold.
- **Content moderation:** high volume, some FPs OK. Balance depends on the harm.

**Cost-optimal threshold:** minimize expected cost `C(t) = cost_FN · P(FN | t) + cost_FP · P(FP | t)`. If the classifier is calibrated, this gives:

$$
t^* = \frac{\text{cost}_{FP}}{\text{cost}_{FP} + \text{cost}_{FN}}
$$

In the fraud example: t* = 50 / (50 + 10000) ≈ 0.005. Predict positive whenever calibrated probability of fraud exceeds 0.5% — very sensitive.

**In practice**, pick threshold by minimizing empirical cost on a validation set. The ROC or PR curve gives you the trade-off; you pick the operating point.

<RealWorldML title="Classification metrics for LLM systems — safety, RLHF, LLM-as-judge">

**LLM safety classifiers.** For content-moderation classifiers (harmful/unsafe requests), the target is very high **recall** (catch essentially all harmful requests) with acceptable precision. The threshold is set aggressively (predict "unsafe" at low probabilities). Standard reports: recall @ 95%, precision @ that recall, PR-AUC.

**RLHF preference modeling.** The reward model learns from labels of the form "A is preferred over B." Metric: accuracy on held-out preference pairs. But **calibration matters more** — the RLHF loss depends on the reward's absolute value, not just the sign. A well-calibrated reward avoids "reward hacking" where the policy exploits regions of the reward function that don't match human preference.

**LLM-as-judge.** GPT-4 grades outputs of GPT-3.5 (or similar). Standard practice: report agreement rate with human raters, along with per-category accuracy. Beware: the judge inherits the biases of the models it's evaluating (both were trained on similar data), so this is not a fully-independent evaluation.

**Retrieval metrics for RAG.**
- **Recall@K:** fraction of relevant documents in top-K retrieved. Standard RAG headline.
- **MRR (Mean Reciprocal Rank):** 1 / (rank of first relevant document), averaged over queries.
- **NDCG (Normalized Discounted Cumulative Gain):** more sophisticated, weights higher-ranked hits more.

**LLM evaluation on multiple-choice benchmarks (MMLU, HellaSwag).** Reported metric is accuracy. But look at the confusion matrix: is the model systematically wrong on certain subject areas? Is the accuracy driven by lucky guessing vs. genuine reasoning?

**Confidence-based abstention.** Modern eval setups let LLMs say "I don't know." Metric: accuracy conditional on the model choosing to answer, plus abstention rate. A well-calibrated model would abstain when its confidence is low. This is where ECE and reliability diagrams matter for LLMs directly.

</RealWorldML>

<CodeItYourself title="Three drills — confusion matrix by hand, threshold sweep, reliability diagram">

1. **Confusion matrix from raw predictions.** Given `y_true = [1, 0, 1, 1, 0, 0, 1]` and `y_pred = [1, 0, 0, 1, 1, 0, 1]`, compute TP, FP, TN, FN by hand. Verify precision = 3/4 = 0.75, recall = 3/4 = 0.75, F1 = 0.75.

2. **Threshold sweep and ROC.** Given a probability model and a set of predictions, sweep the threshold from 0 to 1 in steps of 0.01, compute (FPR, TPR) at each. Plot the ROC curve, compute AUC by trapezoidal integration.

3. **Reliability diagram.** For a trained model, bin its predicted probabilities into 10 bins. For each bin, compute the actual fraction of positives. Plot on a diagonal. If your model is overconfident, points fall BELOW the diagonal in the middle bins.

Big neural nets often exploit the second option: they push logits to large values, making σ(z) essentially 0 or 1. Optimal T for a big classification model is often ~1.5-2.5. Overconfident models have T &gt; 1 that better matches the empirical frequency distribution. This doesn't change the argmax (so classification accuracy is preserved), only the PROBABILITIES. It's the standard calibration fix for neural nets — a single scalar makes reliability diagrams look much better.</>
